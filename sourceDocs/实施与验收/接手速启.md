## 3. 开工顺序
### 3.1 先确认 4 个固定边界
1. `Loom`
   - 唯一治理真相源
2. `Harness`
   - 核心治理运行时
3. `loom-openclaw`
   - 薄 adapter
4. `runtime/loom/`
   - 唯一 authoritative truth root

### 3.2 再确认 4 个核心对象
1. `interactionLane`
   - 当前输入进 chat 还是 managed lane
2. `managedTaskRef`
   - 单个受管任务 owner
3. `managedTaskClass`
   - `COMPLEX / HUGE / MAX`
4. `decision_token`
   - 当前决策窗口绑定令牌

再补一条输入分层硬约束：
1. `CurrentTurnEnvelope`
   - 宿主原始入站事实
2. `HostSemanticBundle`
   - 宿主语义层综合判断
3. `HostCapabilitySnapshot`
   - 宿主现实能力快照
4. `ControlActionIngress`
   - 宿主回传控制动作的 transport carrier
5. `SemanticDecisionEnvelope`
   - Loom 真正消费的 bounded judgment
6. `ControlAction`
   - Loom 真正消费的结构化控制动作

再补一条输出分层硬约束：
1. `PendingDecisionWindow / SpecBundle / IsolatedTaskRun / ReviewResult / ProofOfWorkBundle / ResultContract`
   - 这些是 Loom domain product
2. `TaskScopeSnapshot / RiskAssessment / ExecutionAuthorization`
   - 这些是 execute 前后会进入治理主链的范围、风险与授权对象
3. `StartCardPayload / ToolDecisionPayload / ResultSummaryPayload`
   - 这些是宿主用户可见 payload
4. `OutboundDelivery`
   - 这是 durable outbox 的正式投递单元

## 4. 当前推荐的第一条实现线
第一条真实代码 spike 固定为：
1. `coding_pack`
2. `COMPLEX`
3. `LocalHttpBridge`
4. `HostSemanticBundle -> SemanticDecisionEnvelope -> PendingDecisionWindow(StartCandidate) -> StartCard -> /loom approve -> approve_start -> TaskScopeSnapshot(scope_version=1) -> RiskAssessment(TaskBaseline) -> IsolatedTaskRun + ExecutionAuthorization -> execute -> review -> result`

这条线能同时验证：
1. host semantic ingress
2. `managedTaskRef` 创建
3. `TaskScopeSnapshot` 冻结第一版工程上下文
4. `RiskAssessment(TaskBaseline)` 在首份授权前生成
5. `SpecBundle / IsolatedTaskRun / ExecutionAuthorization` 创建
6. start card 决策消费
7. review/result 主链
8. `ProofOfWorkBundle` 编译
9. durable outbox 到宿主文本输出
10. 第一条 spike 不引入独立 `AcceptanceLedger / ResponsibilityGraph / AttentionPolicy`
11. `LocalHttpBridge` 由 Loom runtime 持有，plugin 只负责 peer 接入

### 4.1 当前三层推进状态
当前实现节奏固定为：
1. 第一层风险治理
   - 正式推进
   - 直接进入当前实现与验收主线
2. 第二层责任协作
   - 只做最小冻结
   - 先停在 `AgentBinding + HandoffContract`
3. 第三层注意力治理
   - 只做文档冻结
   - 当前不进入代码实现主线

固定取舍：
1. 第一层当前必须打透 `TaskScopeSnapshot -> RiskAssessment -> ExecutionAuthorization`
2. 第二层当前只锁死边界，避免后续长成 `ResponsibilityGraph`
3. 第三层当前继续由 `watchdogNoticeState + ...escalation_policy` 承接，不引入独立 `AttentionPolicy`

### 4.2 官方验收入口
正式产品验收固定只认：
1. [OpenClaw WebUI执行测试方案.md](OpenClaw WebUI执行测试方案.md)
2. `OpenClaw WebUI`
3. Playwright 驱动的真实聊天交互
4. 执行前先走标准接入恢复流程：官方 dashboard URL + 设备配对

不被承认为产品验收的方式：
1. 只跑 `vitest`
2. 只跑内部脚本
3. 直接调用 runtime 或 gateway 内部接口

补充说明：
1. `LocalHttpBridge` 的 runtime owner 仍然是 Loom
2. 但截至 `2026-03-11`，当前仓库尚未把 bridge 自动拉起接回默认启动链路
3. 所以本地验收时，bridge 目前仍需要显式启动；这代表当前实现缺口，不代表 owner 已回退给 plugin

## 5. 不要再做的事
1. 不要绕开 Loom 主链，回头在实现里临时发明另一套分类链。
2. 不要再让 adapter 猜自由文本语义。
3. 不要再让宿主兼容投影充当产品真相源。
4. 不要在代码里临时发明 pack 默认值和结果文本结构。
