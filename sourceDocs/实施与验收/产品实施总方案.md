# 产品实施总方案

更新时间：2026-03-11

---

## 1. 文档定位
这份文档是 Loom 的正式实施主文档。

它回答：
1. 开发顺序怎么排
2. 哪些文档必须先冻结
3. 第一条实现线怎么切
4. 哪些默认行为不允许在代码里自由发明

---

## 2. 实施原则
1. 文档先行
   - 先有 target/spec/contract，再有实现
2. pack 与 result 先行
   - 默认产品形状不能靠代码里“临时猜”
3. adapter 薄层
   - 只做 ingress/egress/映射，不做治理判断
4. authoritative truth 单写
   - 所有治理状态只写 `runtime/loom/`
5. review 先行
   - 第一条 spike 也必须带 review

   - 指 Loom runtime 持有的本地 transport boundary
2. `loom-openclaw`
   - 只负责 bridge peer 的 bootstrap、health check、capability sync 与 reconnect
3. 自动拉起 bridge 可以由 Loom launcher / supervisor / dev tooling 承接
   - 但这不改变 bridge runtime owner 仍然属于 Loom

阶段接缝固定说明：
1. `workflowStage`
   - 仍只使用 `candidate / execute / review / result / closed`
2. `clarify / execute / review / deliver`
   - 指的是 `coding_pack` preset 生成的 `PhasePlan` 默认阶段骨架
3. 实现时不得把 `deliver` 或 `clarify` 抬成新的顶层 `workflowStage`

### 3.4.1 当前三层推进状态
当前推进状态固定为：
1. 第一层风险治理
   - 正式推进
   - 直接进入当前实现与验收主线
2. 第二层责任协作
   - 最小冻结
   - 当前只冻结 `AgentBinding + HandoffContract` 边界
3. 第三层注意力治理
   - deferred
   - 这轮只写进文档，不进入当前代码主线

固定取舍：
1. 第一层当前必须做实 `TaskScopeSnapshot / RiskAssessment / ExecutionAuthorization`
2. 第二层当前不引入 `ResponsibilityGraph`、quorum、review 独立性系统
3. 第三层当前不引入独立 `AttentionPolicy`，继续并入 `watchdogNoticeState + ...escalation_policy`

固定说明：
1. 这里的“分层治理装配”
   - 指的是 Loom 内核后续引入的可装配治理模块
2. 它不是首条 spike 的默认负担
3. 第一条 spike 仍然只要求打透：
   - `coding_pack / COMPLEX`
   - start -> execute -> review -> result
4. 六个治理深化方向的讨论入口见：
   - [分层治理装配模型.md](../研究与讨论/分层治理装配模型.md)

### 4.1 `coding_pack / COMPLEX` 的范围、风险与授权基线
这条基线回答三件事：
1. `TaskScopeSnapshot`
   - 进入 execute 前，任务边界先冻结成哪一版
2. `RiskAssessment`
   - 第一份风险结论什么时候生成、什么时候 supersede
3. `ExecutionAuthorization`
   - 第一张 run lease 怎样被能力、范围与风险共同收紧

固定规则：
1. `approve_start` 被正式消费后，必须先冻结第一版 `TaskScopeSnapshot(scope_version=1)`。
2. 第一版 `TaskScopeSnapshot` 至少包含：
   - `workspace_ref`
   - `repo_ref`
   - `allowed_roots`
   - `secret_classes`
3. 当前有效 `HostCapabilitySnapshot` 与 `TaskScopeSnapshot(scope_version=1)` 就绪后，必须先生成首份 `RiskAssessment(subject_kind=TaskBaseline)`，再发首份 `ExecutionAuthorization`。
4. 首份 `ExecutionAuthorization` 的生成输入至少包含：
   - `HostCapabilitySnapshot`
   - `TaskScopeSnapshot`
   - 当前有效 `RiskAssessment.derived_consequences`
5. 当前 run 的实际可执行范围，必须收敛成：
   `HostCapabilitySnapshot ∩ TaskScopeSnapshot ∩ ExecutionAuthorization`
6. 首份 `ExecutionAuthorization.granted_areas` 第一版至少要写清：
   - `allowed_tools`
   - `readable_roots`
   - `writable_roots`
   - `allowed_secret_classes`
   - `budget_band`
7. 只要正式接受了让 `scope_version` 递增的 task change，就必须刷新 `TaskBaseline`，并让新 assessment 的 `supersedes` 指向旧 baseline。
8. 范围变化若进一步影响工程上下文、secret class、路径边界或执行面，还必须同步收紧或重发 `ExecutionAuthorization`。
9. 高风险动作、review 升级或显式审批需求出现时，可以生成 `RiskAssessment(subject_kind=ActionOverride)`。
10. `ActionOverride` 不替代 `TaskBaseline` 作为任务基线；它只对当前动作或当前审批区间生效，并可以要求：
   - `NarrowExecutionAuthorization`
   - `RequireApprovalWindow`
   - `StopSilentExecution`

### 4.1.1 `LocalHttpBridge` 的当前运行责任
第一条 spike 当前固定责任拆分为：
1. Loom runtime
   - 负责启动 / 停止 `LocalHttpBridge`
   - 负责生成 `bridge_instance_id`
   - 负责持有 bootstrap / session credential authoritative state
2. `loom-openclaw`
   - 负责读取 `bridge.baseUrl`
   - 负责 bootstrap
   - 负责 capability sync
   - 负责 peer health check 与 reconnect
3. Loom launcher / supervisor / dev tooling
   - 负责自动拉起 bridge 的工程体验

固定说明：
1. plugin 不持有 bridge 进程生命周期
2. bridge 不可达时，plugin 必须显式报前置失败并 fail-closed
3. 相关 owner 细节见 [桥接运行责任方案.md](../核心设计/宿主接入/桥接运行责任方案.md)

### 4.2 第一版明确不进入最小 spike 的重对象
第一条 `coding_pack / COMPLEX` spike 固定不引入：
1. 独立 `AcceptanceLedger`
2. 独立 `ResponsibilityGraph`
3. 独立 `AttentionPolicy`

固定取舍：
1. 整体结果接受快照直接进入 `ProofOfWorkBundle`
2. 正式责任交接只在 `HUGE / MAX` 需要时引入最小 `HandoffContract`
3. 注意力治理第一版继续并入 `watchdogNoticeState + ...escalation_policy`

### 4.3 第一层和第二层的当前执行边界
第一层当前必须落实：
1. `TaskBaseline` 在首份授权前生成
2. `ActionOverride` 只绑定当前高风险动作或审批区间
3. `critical` 风险不得继续 silent execution
4. 结构化日志至少包含：
   - `risk_assessment.created`
   - `risk_assessment.superseded`
   - `execution_authorization.issued`
   - `execution_authorization.narrowed`
   - `execution_authorization.reissued`

第二层当前只冻结：
1. `coding_pack / COMPLEX` 默认路径不得创建 `HandoffContract`
2. `HUGE / MAX` 或显式责任转移时，才允许创建最小 `HandoffContract`
3. 结构化日志至少包含：
   - `agent_binding.issued`
   - `agent_binding.superseded`
   - `handoff.proposed`
   - `handoff.accepted`
   - `handoff.cancelled`

## 5. 代码里不允许自由发明的默认值
### 5.1 `Harness`
不允许在代码里自己硬写：
1. `coding_pack` 默认阶段序列
2. `coding_pack` 默认角色骨架
3. `review` 是否存在

这些必须来自 preset。

### 5.2 `loom-openclaw`
不允许在代码里自己硬写：
1. start card 文本的核心结构
2. result summary 的核心字段
3. review summary 如何被拼接

这些必须来自 contract + example。

### 5.3 `approval-gate`
不允许自己定义：
1. 任务是否启动
2. managed class
3. review 是否通过

它只管执行授权和高风险动作边界。
