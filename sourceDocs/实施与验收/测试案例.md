# 测试案例

更新时间：2026-03-11

---

## 1. 文档定位
这份文档给出 Loom 的正式测试案例。

固定原则：
1. 这里只写可追溯到 `sourceDocs/` 的正式约束。
2. 自本轮起，进入当前正式执行 gate 的 WebUI 案例都必须写清楚：
   - 测试目的
   - 前置条件
   - WebUI 输入脚本
   - 期间行为
   - 预期用户可见结果
   - 预期运行时状态
   - 预期日志
   - 证据采集
   - 失败判定
   - 当前是否可执行
3. 官方产品验收只认 [OpenClaw WebUI执行测试方案.md](OpenClaw WebUI执行测试方案.md)。
4. 自动化脚本只能辅助回归，不能替代这些案例。
5. 旧的规格型案例保留语义约束，后续逐步补齐到同一执行模板。

## 3. Lane 场景
本组对齐 [验收规范.md](验收规范.md) 的 `Suite A: Lane`，并补充 [目标规格.md](../产品规范/目标规格.md) 与 [端到端示例.md](../产品规范/端到端示例.md) 的行为链。

### `L-01 chat-default`
约束来源：
1. [目标规格.md](../产品规范/目标规格.md)
2. [验收规范.md](验收规范.md)
3. [端到端示例.md](../产品规范/端到端示例.md)

测试目的：
1. 验证普通解释型输入默认留在 `chat lane`，不会误创建 managed task。

前置条件：
1. 当前没有 active `managedTaskRef`。

用户输入：
```text
解释一下 WorkHorizon。
```

期间行为：
1. 宿主先给出 `HostSemanticBundle`。
2. 其中 `interactionLane` 应保持为 `chat`。
3. adapter 只做归一化，不补造 managed judgment。
4. Loom 不创建 candidate。

预期状态：
1. `interactionLane=chat`
2. 无 `managedTaskRef`
3. 无 `pendingUserDecision`

预期用户可见结果：
1. 正常聊天回复
2. 无 start card

证据点：
1. `runtime/loom/` 中没有新建任务聚合
2. 输出不是 `managed task start card`

### `L-02 explicit-managed-candidate`
约束来源：
1. [目标规格.md](../产品规范/目标规格.md)
2. [验收规范.md](验收规范.md)
3. [端到端示例.md](../产品规范/端到端示例.md)
4. [OpenClaw适配验证方案.md](../核心设计/宿主接入/OpenClaw适配验证方案.md)
5. [决策窗口合同.md](../核心设计/治理策略/决策窗口合同.md)

测试目的：
1. 验证明确定义的任务输入进入 `managed_task_candidate`，并且先停在 start card，而不是直接执行。

前置条件：
1. 当前没有 active `managedTaskRef`。

用户输入：
```text
开始一个任务：整理文档导航，交给子 agent 持续推进。
```

期间行为：
1. 宿主先输出 `HostSemanticBundle`。
2. bundle 至少给出：
   - `interactionLane=managed_task_candidate`
   - `managedTaskClass`
   - `WorkHorizon`
   - `taskActivationReason`
3. adapter 把 bundle 归一化成 `SemanticDecisionEnvelope`。
4. Loom 创建 candidate。
5. Loom 打开 `PendingDecisionWindow(kind=StartCandidate)`。
6. Loom 写入 `pendingUserDecision` projection，并发出 start card。

预期状态：
1. `interactionLane=managed_task_candidate`
2. 创建 `managedTaskRef`
3. `workflowStage=candidate`
4. 存在 `PendingDecisionWindow(kind=StartCandidate)`
5. `pendingUserDecision` 指向该 open window
6. 生成 `decision_token`

预期用户可见结果：
1. 第一条用户可见 managed 消息是 start card
2. start card 至少能验证：
   - `managed_task_ref`
   - `decision_token`
   - `managed_task_class`
   - `work_horizon`

证据点：
1. `runtime/loom/` 中可追到 candidate 任务与 `PendingDecisionWindow(kind=StartCandidate)`
2. start card 携带的 `decision_token` 对应当前 open window
3. start card 字段满足 [验收规范.md](验收规范.md) 的必测字段

### `L-03 active-task-chat-open`
约束来源：
1. [目标规格.md](../产品规范/目标规格.md)
2. [验收规范.md](验收规范.md)

测试目的：
1. 验证 active task 存在时，普通聊天仍保持开放，不被错误吸进任务治理主线。

前置：
1. 已有 active `COMPLEX` task

用户输入：
```text
顺便解释一下 术语表 里的 decision_token。
```

期间行为：
1. 宿主对这条输入给出 `interactionLane=chat`。
2. adapter 不把普通提问映射成 `request_task_change` 或新 candidate。
3. Loom 保持当前 active task 不变。

预期状态：
1. 继续 `chat lane`
2. active task 不被打断
3. 不创建新的 `PendingDecisionWindow(kind=BoundaryConfirmation)` 或其扩展子实体 `pendingBoundaryConfirmation`

预期用户可见结果：
1. 用户收到术语解释
2. 当前 active task 没有被静默取消、替换或 reopen

证据点：
1. `activeManagedTaskRef` 保持不变
2. 没有新的 candidate 或 boundary card

### `L-04 second-managed-task-boundary`
约束来源：
1. [目标规格.md](../产品规范/目标规格.md)
2. [验收规范.md](验收规范.md)
3. [端到端示例.md](../产品规范/端到端示例.md)
4. [控制动作合同.md](../核心设计/治理策略/控制动作合同.md)
5. [决策窗口合同.md](../核心设计/治理策略/决策窗口合同.md)

测试目的：
1. 验证 active task 存在时，第二项重任务不会静默覆盖当前任务，而是先进入边界确认。

前置：
1. 已有 active `COMPLEX` task

用户输入：
```text
再开始一个任务：重写 adapter bridge。
```

期间行为：
1. 宿主把这条输入理解成第二项 managed task 候选。
2. Loom 打开 `PendingDecisionWindow(kind=BoundaryConfirmation)`。
3. Loom 创建 `pendingBoundaryConfirmation` 扩展子实体。
4. Loom 输出边界确认卡，而不是直接替换当前任务。

预期状态：
1. 不静默替换当前任务
2. 先存在 `PendingDecisionWindow(kind=BoundaryConfirmation)`
3. 再创建 `pendingBoundaryConfirmation` 扩展子实体
4. 当前 active task 仍保持 active

预期用户可见结果：
1. 发边界确认卡
2. 后续只允许结构化动作：
   - `keep_current_task`
   - `replace_active`

证据点：
1. 能验证边界窗口的 `decision_token`
2. 原 `activeManagedTaskRef` 在用户明确决策前保持不变
3. `PendingDecisionWindow(kind=BoundaryConfirmation)` 仍处于 open
4. `pendingBoundaryConfirmation` 能验证同时绑定：
   - 当前 active task owner
   - 当前 boundary candidate owner
5. 边界确认卡可验证 `candidate_managed_task_ref`

## 4. Start Card 场景
本组对齐 [验收规范.md](验收规范.md) 的 `Suite B: Candidate / Start Card`，并受 [控制动作合同.md](../核心设计/治理策略/控制动作合同.md) 与 [决策窗口合同.md](../核心设计/治理策略/决策窗口合同.md) 的窗口/token 约束直接约束。

### `S-01 approve-start`
约束来源：
1. [目标规格.md](../产品规范/目标规格.md)
2. [验收规范.md](验收规范.md)
3. [控制动作合同.md](../核心设计/治理策略/控制动作合同.md)
4. [OpenClaw适配验证方案.md](../核心设计/宿主接入/OpenClaw适配验证方案.md)

测试目的：
1. 验证 `approve_start` 只能消费当前决策窗口，并把 candidate 推进到 execute。

前置：
1. candidate 已创建
2. 当前 start card 对应的 `decision_token` 仍有效
3. 当前 `HostCapabilitySnapshot` 已同步

用户动作：
1. `approve_start`

期间行为：
1. 宿主给出结构化 `control_action=approve_start`。
2. adapter 把它映射成正式 `ControlAction`。
3. 该动作必须带有效 `decision_token`。
4. Loom 消费后关闭当前 pending decision window。

预期状态：
1. `activeManagedTaskRef = managedTaskRef`
2. `workflowStage = execute`
3. 当前 `pendingUserDecision` 被消费
4. 创建 `TaskScopeSnapshot(scope_version=1)`
5. 创建 `RiskAssessment(subject_kind=TaskBaseline)`
6. 创建 `SpecBundle`
7. 创建 `IsolatedTaskRun`
8. 创建首份 `ExecutionAuthorization`

预期用户可见结果：
1. 不再重复发新的 start card
2. 进入执行中反馈

证据点：
1. `TaskScopeSnapshot` 至少包含：
   - `workspace_ref`
   - `repo_ref`
   - `allowed_roots`
   - `secret_classes`
2. `RiskAssessment.subject_kind=TaskBaseline`
3. 首份 `ExecutionAuthorization` 引用当前 `HostCapabilitySnapshot` 与当前 `TaskScopeSnapshot`
4. 首份 `ExecutionAuthorization` 在当前 `TaskBaseline` 生成后发放
5. 首份 `ExecutionAuthorization.granted_areas` 至少包含：
   - `readable_roots`
   - `writable_roots`
   - `allowed_secret_classes`
   - `budget_band`
6. `SpecBundle` 至少包含 `scope_doc / plan_doc / verification_doc`
7. `IsolatedTaskRun` 可追溯到当前 `managedTaskRef`

### `S-02 modify-candidate`
约束来源：
1. [目标规格.md](../产品规范/目标规格.md)
2. [验收规范.md](验收规范.md)
3. [控制动作合同.md](../核心设计/治理策略/控制动作合同.md)

测试目的：
1. 验证用户修改 candidate 时，系统会重开一个新的决策窗口，而不是复用旧 token。

用户动作：
1. `modify_candidate`

期间行为：
1. 宿主给出结构化 `control_action=modify_candidate`。
2. 该动作必须带当前窗口的 `decision_token`。
3. Loom 更新 candidate 并重新发卡。

预期状态：
1. candidate 被更新
2. 重新发 start card
3. 新卡带新 `decision_token`
4. 旧 `decision_token` 不再可消费新窗口

预期用户可见结果：
1. 用户看到新的 candidate 摘要
2. 用户不会被旧窗口迟到回复串线

证据点：
1. 新旧 `decision_token` 不同
2. 旧窗口被关闭，新窗口被创建

预期状态：

### `C-01 completed`
测试目的：
1. 验证第一条真实代码 spike 的最小闭环能从 execute 走到 result，并且 review 与结果证据都正式落地。

前置条件：
1. `S-01 approve-start` 已通过。
2. `pack_ref = coding_pack`
3. `managedTaskClass = COMPLEX`

期间行为：
1. `Harness` 依据 `coding_pack` preset 生成 `PhasePlan`。
2. 初始阶段骨架来自 preset，而不是实现层自由硬编码。
3. Loom 在 execute 期创建 `IsolatedTaskRun`。
4. review 阶段至少落一个正式 `ReviewResult`。
5. result 阶段生成 `ProofOfWorkBundle` 与 `ResultSummaryPayload`。

预期状态：
1. 默认阶段：
   - `clarify -> execute -> review -> deliver`
2. 默认角色：
   - `net + recorder + 1 worker`
3. `approve_start` 后生成 `SpecBundle`
4. execute 期创建 `IsolatedTaskRun`
5. 产出 `ReviewResult`
6. 产出 `ProofOfWorkBundle`
7. 产出 `ResultSummaryPayload`

预期用户可见结果：
1. 用户先见到 start card，再进入执行，再收到结果摘要
2. 结果摘要必须能追溯到正式 `ProofOfWorkBundle`，不能只是一段自由文本

证据点：
1. `ReviewResult` 至少可验证 `review_verdict` 与 `ReviewSummary`
   - 这里的 `ReviewSummary` 只作为 `ReviewResult` 的内嵌值对象存在
2. `ProofOfWorkBundle` 至少包含：
   - `run_summary`
   - `evidence_refs`
   - `review_summary`
   - `artifact_manifest`
   - `acceptance_basis`
3. 第一版整体结果接受快照直接体现在 `ProofOfWorkBundle`：
   - `accepted_by_ref`
   - `accepted_at`
   - `accepted_scope_version`
4. `ResultSummaryPayload` 只是摘录，不替代完整证据包

### `C-02 rework-required`
测试目的：
1. 验证 review 结论为 `ReworkRequired` 时，任务回到执行链，而不是伪装成完成态。

前置条件：
1. 任务已进入 review。

期间行为：
1. review 产生正式 `ReviewResult`。
2. 其 `review_verdict` 为 `ReworkRequired`。
3. Loom 按返工规则回退到 execute。

预期状态：
1. review 返回 `rework_required`
2. 任务 reopen execute
3. 不直接进入 result

预期用户可见结果：
1. 用户看到返工结论
2. 不出现伪装成成功完成的结果摘要

证据点：
1. 事件链可追到 `review_completed` 与返工动作
2. `ProofOfWorkBundle.review_summary` 不得伪装成 approved

### `C-03 blocked`
测试目的：
1. 验证 blocked 不是“没有输出就结束”，而是会形成正式阻塞结论。

前置条件：
1. 当前任务在 execute 或 review 期间遇到无法继续的外部阻塞。

期间行为：
1. 系统产出 `ReviewResult` 或等价阻塞结论。
2. 若当前切片已接入最小 `StatusNotice`，则可向宿主发升级 notice。
3. 结果链仍然要形成正式阻塞态摘要。

预期状态：
1. review 或 execute 进入 `blocked`
2. 若已接入最小 `StatusNotice`，则可发升级 notice
3. 最终结果包可以产出阻塞态摘要

预期用户可见结果：
1. 用户知道任务被什么条件阻塞
2. 用户知道当前没有形成可交付完成态

证据点：
1. 阻塞态结果与 [结果样例.md](../核心设计/工作模式与结果/结果样例.md) 的 blocked 示例一致
2. 若无 notice，也必须有正式阻塞态结果包

### `C-04 scope-changed-then-completed`
测试目的：
1. 验证运行中范围变更会留下正式影响映射，不会静默改写历史。

前置条件：
1. 当前已有 active task 正在执行。

期间行为：
1. 宿主把这条输入结构化成 `request_task_change`。
2. adapter 映射成正式 `ControlAction`。
3. Loom 生成 `ChangeImpactAssessment`。
4. 只对未来未执行的小修订允许自动吸收。
5. 如变更被正式接受且提升 `scope_version`，Loom 生成新的 `TaskScopeSnapshot`。
6. 只要变更被正式接受且提升 `scope_version`，Loom 就生成新的 `RiskAssessment(subject_kind=TaskBaseline)`，并让它 `supersedes` 旧 baseline。
7. 如当前授权边界已不再满足新范围或新风险后果，Loom 收紧或重发 `ExecutionAuthorization`。
8. 如变更还影响工程上下文、路径边界、secret class 或执行面，新的 `ExecutionAuthorization` 必须同步反映这些边界。
9. 如变更影响当前或已完成工作，则按 contract 触发确认或返工映射。

预期状态：
1. 中途接受 `request_task_change`
2. 更新 `TaskScopeSnapshot`
3. 旧 `TaskBaseline` 被新 baseline supersede
4. 当前 `ExecutionAuthorization` 按新范围和新风险后果被 keep / narrow / reissue
5. 结果包正确填写 `final_scope_snapshot` 和 `scope_revision_summary`

预期用户可见结果：
1. 用户能看到范围已被修订或需确认
2. 最终结果明确说明按哪版范围交付

证据点：
1. 能看到 `ChangeImpactAssessment`
2. 能看到新的 `TaskScopeSnapshot.scope_version`
3. 能看到新的 `RiskAssessment.supersedes`
4. 最终结果中的 `scope_revision_summary` 与变更历史一致

测试目的：

   - `approve_request`
   - `reject_request`

2. 不直接进入最终 `result`

1. 投递失败进入 `retry_scheduled`
2. backoff 后重新尝试
3. 达到上限后进入 `terminal_failed` 或 `expired`

预期状态：
1. delivery 失败可重试
2. bounded retry 后进入 `terminal_failed` 或 `expired`

预期用户可见结果：
1. 可以没有立即的用户成功消息
2. 但系统必须保留正式失败状态

证据点：
1. `attempt_count`、`max_attempts`、`next_attempt_at`、`last_error` 被记录
2. 失败终态不是只有日志，没有 authoritative status

### `D-03 capability-drift`
测试目的：
1. 验证宿主能力变化后，Loom 会做正式 drift assessment，而不是继续拿旧授权静默运行。

前置条件：
1. 当前 task 存在 active `ExecutionAuthorization`。
2. 宿主同步了新的 `HostCapabilitySnapshot`。

2. 若能力缩减碰到当前授权边界，则至少进入：
   - `Narrow`
   - `RevokeAndReissue`
   - `PauseTask`
   - `RequireUserDecision`
3. 若 drift 结论要求暂停或取消，则进一步进入 worker interruption path。

预期状态：
1. `HostCapabilitySnapshot` 变化后触发 `CapabilityDriftAssessment`

预期用户可见结果：
1. 不允许 silent execution
2. 若需要用户重新决策，应有明确升级或提示

证据点：
1. `capability_snapshot_synced` 与 `capability_drift_assessed` 可追溯
2. 不会继续使用已经失效的旧 lease
