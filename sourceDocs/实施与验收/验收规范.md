# 验收规范

更新时间：2026-03-11

---

## 1. 文档定位
这份文档定义 Loom 的正式验收目标。

固定规则：
1. 官方产品验收入口只认 [OpenClaw WebUI执行测试方案.md](OpenClaw WebUI执行测试方案.md)。
2. `OpenClaw WebUI + Playwright` 是唯一正式执行路径。
3. `vitest` 和脚本桩只能作为工程回归，不替代产品验收。

---

## 2. 验收主轴
### 2.0 官方执行方式
必须成立：
1. 所有正式验收都从 `OpenClaw WebUI` 发起。
2. 用例执行者必须通过 Playwright 驱动真实聊天交互。
3. 每条验收用例都要同时给出：
   - 用户可见结果
   - 运行时状态
   - 结构化日志
   - 证据采集点
4. 执行前必须先完成 [OpenClaw WebUI执行测试方案.md](OpenClaw WebUI执行测试方案.md) 定义的标准接入恢复流程。
5. 如果完成恢复流程后 WebUI 仍显示 `Disconnected from gateway.`、health 不是 `OK`、输入框 disabled，或日志仍出现 `token_missing / pairing required`，本次验收直接判为前置条件失败，不得继续执行。

### 2.1 Chat 默认态
必须成立：
1. 普通聊天和轻量工作默认留在 `chat lane`
2. 不创建 `managedTaskRef`
3. 不出现 start card

### 2.2 Managed task 启动
必须成立：
1. managed lane 的第一条用户可见消息是 `managed task start card`
2. start card 绑定 `decision_token`
3. `approve_start / modify_candidate / cancel_candidate` 必须结构化消费
4. start card 前必须先存在 authoritative `PendingDecisionWindow(kind=StartCandidate)`
5. `pendingUserDecision` 只能作为该窗口的 task projection pointer

### 2.3 `COMPLEX` 最小闭环
第一条真实代码 spike 必须成立：
1. `coding_pack / COMPLEX`
2. `clarify -> execute -> review -> deliver`
3. `net + recorder + 1 worker`
4. `approve_start` 后先冻结 `TaskScopeSnapshot(scope_version=1)`
5. 首份 `RiskAssessment(subject_kind=TaskBaseline)` 在首份授权前生成
6. `approve_start` 后会创建 `SpecBundle`
7. execute 期会创建 `IsolatedTaskRun`
8. execute 期会创建首份 `ExecutionAuthorization`
9. 最终有 `ReviewResult`
10. 最终有 `ProofOfWorkBundle`
11. 最终有 `ResultSummaryPayload`

阶段层级说明：
1. 这里的 `clarify / execute / review / deliver`
   - 指 `PhasePlan` 的默认阶段骨架
2. 顶层 `workflowStage`
   - 仍只使用 `candidate / execute / review / result / closed`

### 2.3.1 第一条 spike 的范围、风险与授权基线
必须成立：
1. 第一版 `TaskScopeSnapshot` 至少持有：
   - `workspace_ref`
   - `repo_ref`
   - `allowed_roots`
   - `secret_classes`
2. 首份 `RiskAssessment` 必须是 `TaskBaseline`
3. 首份 `ExecutionAuthorization` 只能在当前有效：
   - `HostCapabilitySnapshot`
   - `TaskScopeSnapshot`
   - `RiskAssessment`
   都存在后发放
4. 当前 run 的实际可执行范围必须收敛成：
   `HostCapabilitySnapshot ∩ TaskScopeSnapshot ∩ ExecutionAuthorization`
5. 首份 `ExecutionAuthorization.granted_areas` 第一版至少写清：
   - `allowed_tools`
   - `readable_roots`
   - `writable_roots`
   - `allowed_secret_classes`
   - `budget_band`
6. 只要正式接受的 task change 让 `scope_version` 递增，就必须刷新 `TaskBaseline` 并 supersede 旧 baseline
7. 高风险动作或 review 升级时，可以生成 `ActionOverride`
8. `ActionOverride` 生效期间，不得继续 silent execution

### 2.3.2 第一层风险治理的验收要求
必须成立：
1. `TaskBaseline` 在首份 `ExecutionAuthorization` 之前生成
2. 只要正式接受的 task change 让 `scope_version` 递增，就必须生成 supersede 旧 baseline 的新 `TaskBaseline`
3. 高风险动作、secret 使用、外部副作用、不可逆动作、预算超阈值动作，会生成 `ActionOverride`
4. `critical` 风险不会静默继续执行，而是显式打开审批窗口或停跑
5. 结构化日志中可追到：
   - `risk_assessment.created`
   - `risk_assessment.superseded`
   - `execution_authorization.issued`
   - `execution_authorization.narrowed`
   - `execution_authorization.reissued`

### 2.3.3 第二层和第三层的当前阶段状态
必须成立：
1. 第二层当前只冻结到 `AgentBinding + HandoffContract`
2. `coding_pack / COMPLEX` 默认路径不得创建 `HandoffContract`
3. `HUGE / MAX` 或显式责任转移时，才允许进入最小 handoff 路径
4. 第三层当前仍由 `watchdogNoticeState + ...escalation_policy` 承接
5. 第一版不引入独立 `AttentionPolicy`

### 2.4 active task 并行聊天
必须成立：
1. active task 执行期间，主聊天保持开放
2. 普通聊天不被错误吸进任务治理主线

### 2.5 第二项 managed task 边界
必须成立：
1. active task 存在时，第二项重任务先进入边界确认
2. 不允许静默覆盖当前任务
3. 必须先存在 `PendingDecisionWindow(kind=BoundaryConfirmation)`
4. `BoundaryCardPayload` 只能投影当前 `pendingBoundaryConfirmation`

### 2.6 结果与 review
必须成立：
1. review 阶段产出 `ReviewResult`
2. `ReviewSummary` 只作为 `ReviewResult` 的内嵌值对象被消费
3. 最终结果包引用 `ProofOfWorkBundle`
4. `ProofOfWorkBundle` 必须包含 `review_summary`
5. `acceptance_verdict` 与 review/verification 对齐
6. `escalate_to_user` 必须先打开正式审批窗口，再输出 `ApprovalRequestPayload`
7. `approve_request / reject_request` 之外不得消费该审批窗口
8. 第一版整体结果接受快照必须来自 `ProofOfWorkBundle`，不得另起独立 `AcceptanceLedger`

### 2.7 authoritative truth
必须成立：
1. `runtime/loom/` 是唯一 authoritative truth
2. compatibility projection 不是真相源

### 2.8 host semantic first
必须成立：
1. 宿主先产出 `HostSemanticBundle`
2. adapter 归一化成 `SemanticDecisionEnvelope`
3. 缺少 `interaction_lane` 时，默认 `fail-open to chat`
4. managed candidate 缺 `managed_task_class / work_horizon` 时，不允许静默进入 execute
5. low-confidence 或 major schema mismatch 时，必须保守退化
6. adapter 不得从 free-text 猜 control action

## 4. 必测字段
### 4.1 start card
必须能看到或验证：
1. `managed_task_ref`
2. `decision_token`
3. `managed_task_class`
4. `work_horizon`
5. 用户可见摘要

### 4.2 result
必须能看到或验证：
1. `final_scope_snapshot`
2. `scope_revision_summary`
3. `proof_of_work.run_summary`
4. `proof_of_work.evidence_refs`
5. `proof_of_work.review_summary`
6. `proof_of_work.artifact_manifest`
7. `proof_of_work.acceptance_basis`
8. `proof_of_work.accepted_by_ref`
9. `proof_of_work.accepted_at`
10. `proof_of_work.accepted_scope_version`
11. `acceptance_verdict`

### 4.3 outbound
必须能看到或验证：
1. `delivery_id`
2. `delivery_status`
3. retry / ack / terminal_failed / expired 语义

### 4.4 approval request
必须能看到或验证：
1. `decision_token`
2. `approval_scope`
3. `allowed_actions`
4. `why_now`
5. `risk_summary`
