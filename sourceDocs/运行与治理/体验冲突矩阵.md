# 体验冲突矩阵

更新时间：2026-03-10

---

## 1. 文档定位
这份文档只描述 Loom 未来实现必须守住的 UX 质量面。

文件名保留为 `运行与治理/体验冲突矩阵.md` 只是为了链接稳定。
它当前实际承接的是 Loom 的 UX 质量矩阵，不再是旧体系的现象回放清单。

---

## 2. 质量面
| 编号 | 质量面 | 必须成立 |
| --- | --- | --- |
| `UX-01` | 默认静默 | 普通聊天和轻量工作默认不出现 Loom 任务卡 |
| `UX-02` | 明确启动 | 进入 managed lane 的第一条用户可见消息必须是 start card |
| `UX-03` | 边界清晰 | 第二项重任务先出边界确认，不静默覆盖 active task |
| `UX-04` | 主聊天开放 | active task 执行期间，主聊天仍继续可用 |
| `UX-05` | review 可感知 | 结果摘要必须体现 review 与 acceptance，不只是一句“完成了” |
| `UX-06` | 结果可交代 | 结果摘要必须能交代 scope 变化和 artifact 清单 |
| `UX-07` | 内部治理不可见 | 宿主聊天区不直接暴露内部结构化 payload 和治理旁白 |

---

## 3. 用法
1. 当新实现让用户重新看到 Loom 内部治理噪音时，先回到本矩阵判断破坏了哪条 UX。
2. 当评估 start/review/result 渲染时，先检查是否满足 `UX-02 / UX-05 / UX-06`。
