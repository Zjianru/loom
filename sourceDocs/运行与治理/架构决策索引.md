# 架构决策索引

更新时间：2026-03-11

---

## 1. 文档定位
这份文档只记录 Loom 主线已经固定下来的架构决策。

---

## 2. 已冻结决策
### ADR-001 默认聊天优先
1. 默认先走 `chat lane`
2. Loom 默认静默

### ADR-002 宿主负责语义理解
1. kernel 不做自然语言语义推断
2. 宿主产出 `HostSemanticBundle`
3. adapter 归一化为 `SemanticDecisionEnvelope`

### ADR-003 受管任务正式档位
1. `managedTaskClass = COMPLEX / HUGE / MAX`

### ADR-004 单活跃任务，主聊天并行
1. `activeManagedTaskRef` 只保留单活跃任务约束
2. `taskConversationPolicy=chat_open`

### ADR-005 Loom 真相源
1. `runtime/loom/` 是唯一 authoritative truth
2. 宿主兼容投影只做 compatibility projection

### ADR-006 Harness 负责治理主链
1. `Harness` 负责 start/execute/review/result/boundary 主链
2. `approval-gate` 只负责高风险动作边界
3. `watchdog` 只负责 notice

### ADR-007 Pack 驱动默认产品形状
1. pack preset 决定默认阶段骨架和角色骨架
2. 代码实现不得临时发明 pack 默认值

### ADR-008 review 是正式阶段
1. 第一条真实 spike 也必须带 review
2. `ReviewResult` 是正式 review 产物
3. `ReviewSummary` 作为 `ReviewResult` 内嵌值对象被结果链消费

### ADR-009 结果必须结构化后再渲染
1. 先有 `ResultContract`
2. 再有 `ResultSummaryPayload`
3. adapter 只做文本渲染

### ADR-010 bridge-first
1. 逻辑上 daemon-ready
2. 工程上 bridge-first
3. v0 先用 `LocalHttpBridge`

### ADR-011 Spec 是 Loom 通用能力
1. `SpecBundle` 是 Loom 核心对象
2. 它不只是 `coding_pack` 技巧
3. owner 固定为 `managedTaskRef`

### ADR-012 Proof-of-Work 是结果链核心对象
1. `ProofOfWorkBundle` 是正式证据包
2. `ResultContract` 必须显式消费它
3. `ResultSummaryPayload` 只摘录，不得替代它

### ADR-013 run 与 task/session 分层
1. `IsolatedTaskRun` 是 Loom 运行时边界对象
2. task 不等于 session
3. run 不等于 authorization

### ADR-014 入站对象分层
1. 宿主 transport 先形成 `HostInboundTurn`
2. adapter 再归一化成 `CurrentTurnEnvelope`
3. kernel 不直接把宿主 transport 对象当正式领域对象消费

### ADR-015 `AgentBinding` 是正式运行时对象
1. `Pack / AgentPool` 只给骨架和候选画像
2. `Harness` 必须收敛出当前实际生效的 `AgentBinding`
3. capability drift 和重授权可以 supersede 当前 binding

### ADR-016 决策窗口必须 authoritative
1. `pendingUserDecision / pendingBoundaryConfirmation` 必须回到统一决策窗口 contract
2. `StartCardPayload / BoundaryCardPayload / ApprovalRequestPayload` 只做 projection
3. `decision_token` 的生命周期由窗口对象持有

### ADR-017 宿主能力快照必须宿主无关
1. `HostCapabilitySnapshot` 是 Loom 正式输入合同
2. 它不能只散落在 OpenClaw 接口稿中
3. `HostWorkerControlCapabilities` 由该合同持有 owner

### ADR-018 治理深化按模块装配，不全局全开
1. 更深治理能力按模块挂载，不作为所有 managed task 的默认负担
2. 挂载优先级固定为：`HostCapabilitySnapshot` 约束 -> 内核强制安全规则 -> `PackContract` 默认 -> `managedTaskClass` 叠加 -> 风险升级 -> 用户显式覆盖
3. 用户只能在 `override_policy` 允许范围内做显式覆盖
4. 被能力约束挡住的层不能被强行启用，被安全规则强制拉起的层不能被用户关闭

### ADR-019 任务作用域与运行授权分层
1. 第一版由 `TaskScopeSnapshot` 统一持有任务级范围与工程上下文边界
2. `ExecutionAuthorization` 定义当前 run 的执行租约
3. 当前 run 的授权只能是 `TaskScopeSnapshot` 的子集，不能反向扩大 scope

### ADR-020 独立 noun 必须有独立生命周期与真相源
1. 只有同时具备独立生命周期、独立真相源、独立状态转移或独立消费关系的概念，才升级成正式对象
2. 其余概念优先降为值对象、子实体、嵌入字段族或枚举
3. 先复用既有对象族，再考虑新增 noun

### ADR-021 Loom 固定采用五个 bounded contexts
1. 宿主输入 context：`CurrentTurnEnvelope / HostSemanticBundle / SemanticDecisionEnvelope / HostCapabilitySnapshot`
2. 任务治理 context：`managedTaskRef / TaskScopeSnapshot / PhasePlan / PendingDecisionWindow / ChangeImpactAssessment`
3. 执行与授权 context：`AgentBinding / IsolatedTaskRun / ExecutionAuthorization / CapabilityDriftAssessment / RiskAssessment`
4. review / result context：`ReviewResult / ProofOfWorkBundle / ResultContract`
5. notice / outbox context：`watchdogNoticeState / KernelOutboundPayload / OutboundDelivery`

### ADR-022 风险以 `RiskAssessment` 持有，不另起平行 aggregate
1. `risk_profile` 只作为 `RiskAssessment` 的结果视角存在，不升格为独立 aggregate
2. 风险固定分成两层：`TaskBaseline` 与 `ActionOverride`
3. 总体风险档位第一版固定为 `Low / Medium / High / Critical`
4. `critical` 风险不得继续 silent execution，用户只能上调风险，不能压到系统底线以下

### ADR-023 会改变治理消费结果的深化层必须 durable
1. 只影响解释和推荐的内容，可以停留在 policy 或摘要层
2. 只要会影响 `ControlAction` 是否有效、`review/result` 是否成立、或 run 是否允许继续执行，就必须进入 durable state
3. 不允许同一治理判断同时存在持久 owner 和平行的临时内存 owner

### ADR-024 责任协作层第一版先复用 `AgentBinding`
1. 第一版不引入 `ResponsibilityGraph`
2. 当前“谁实际上场”继续由 `AgentBinding` 持有
3. 如果确有 handoff 需求，只补最小 `HandoffContract`
4. 若未来确需更重责任对象，owner 必须放在任务级，阶段只允许持有 overlay，不得形成平行真相源

### ADR-025 证明链第一版不独立 ledger 化
1. 第一版继续以 `AcceptancePolicy + ReviewResult + ProofOfWorkBundle + verification_doc` 形成闭环
2. 第一版不引入独立 `AcceptanceLedger`
3. 第一版整体结果接受快照直接收进 `ProofOfWorkBundle`
4. 若后续确需正式接受记录，只能作为 `ProofOfWorkBundle` 的引用对象或快照来源，不得替代结果主链 owner

### ADR-026 注意力治理第一版并入 notice 链
1. 第一版不引入独立 `AttentionPolicy`
2. 注意力治理先扩 `watchdogNoticeState` 与现有 `...escalation_policy`
3. 若未来引入用户级注意力对象，优先级固定为：安全强打断 > 用户硬偏好 > 任务级收紧 > pack 默认

### ADR-027 第一版最小形状先冻结在现有对象族里
1. `RiskAssessment` 第一版最小字段固定为：`overall_risk_band / risk_dimensions / trigger_reason / evidence_refs / derived_consequences / supersedes`
2. `TaskScopeSnapshot` 第一版工程上下文字段固定为：`workspace_ref / repo_ref / allowed_roots / secret_classes`
3. `ExecutionAuthorization.granted_areas` 第一版继续承接：工具范围、读路径、写入范围、secret class、时间范围、预算带
4. 第一版不新建 `PolicyEvaluationRecord / DecisionBasis / AuthorizationLeaseScope / AcceptanceLedger / AttentionPolicy / WorkspaceBinding / RepoBinding / ResourceScope / SecretScope`

### ADR-028 `LocalHttpBridge` 运行责任归 Loom
1. `LocalHttpBridge` 是 Loom runtime 的本地 transport boundary
2. bridge 进程生命周期、listener、`bridge_instance_id` 与 bridge credential authoritative state 由 Loom 持有
3. `loom-openclaw` 只负责 bridge peer 的 bootstrap、health check、capability sync 与 reconnect
4. Loom launcher / supervisor / dev tooling 可以自动拉起 bridge，但这属于 Loom 的部署责任，不改变 owner 归属
5. 第一版不让 plugin 持有 bridge 进程生命周期，也不把纯外部预启动当成默认产品责任

## 3. 当前结论
未来若要修改 Loom 主线边界，必须先修改这份 ADR 索引，再修改实现或更细的 contract。
