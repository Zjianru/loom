## 已关闭问题回顾

### P0-01 顶层对象表仍把决策窗口家族写成错误层级（已关闭）
问题：
1. 顶层对象表仍把 `pendingUserDecision` 写成“当前待消费的结构化决策窗口”
2. `概念模型` 仍把它写成“当前仍待消费的 start card 或其它决策窗口”
3. `pendingBoundaryConfirmation` 也仍被顶层写成“边界窗口”，而不是 `BoundaryConfirmation` 的边界扩展子实体

根因：
1. 决策窗口主链已经在 `运行时语义 / 验收规范 / 测试案例 / 决策窗口合同` 中修正
2. 但顶层对象表、概念词条和状态机总表没有一起回填

影响范围：
1. 实现者会把 `pendingUserDecision` 当成 authoritative state，而不是 task projection pointer
2. `replace_active` 可能被实现成直接消费 `BoundaryCardPayload` 或 `pendingBoundaryConfirmation` 的模糊窗口
3. 会破坏 `PendingDecisionWindow` 作为唯一窗口 owner 的约束

证据：
1. `产品规范/治理架构.md:66-67`
2. `产品规范/状态机附录.md:51-52`
3. `产品规范/概念模型.md:159-165`

建议修复：
1. 顶层统一改成：
   - `pendingUserDecision` 是 task projection pointer
   - `pendingBoundaryConfirmation` 是 `BoundaryConfirmation` 的边界扩展子实体
   - `decision_token` 生命周期由 `PendingDecisionWindow` 持有
2. 修复范围必须包含：
   - `治理架构.md`
   - `状态机附录.md`
   - `概念模型.md`
   - `运行时基线.md`
   - `术语表.md`

### P1-02 `watchdog` 相关对象没有唯一 owner（已关闭）
问题：
1. 顶层规范写明 `watchdog` 消费：
   - `TaskEvent`
   - `WorkItemState`
   - `watchdogNoticeState`
   - `EscalationPolicy`
2. 但当前合同层只有：
   - `WorkItemState` 作为 `WipPolicy` 里的枚举
   - `WipEscalationPolicy / ReworkEscalationPolicy / AcceptanceEscalationPolicy`
3. `watchdogNoticeState` 在主线里被引用，但没有 formal contract owner
4. 泛化的 `EscalationPolicy` 也没有 formal contract owner

根因：
1. 研究材料里的方法论概念被抬进了顶层规范
2. 但对应的 contract 化工作没有补齐

影响范围：
1. `watchdog` 的 notice 去重状态会被实现层自由发明
2. 升级条件可能会被错误实现成单一泛化 `EscalationPolicy`
3. 后续 `watchdog`、WIP、返工、验收之间的关系会重新长出双真相源

证据：
1. `产品规范/治理架构.md:125-129`
2. `核心设计/任务与协作/阶段包合同.md:60-64`
3. `核心设计/治理策略/在制约束合同.md:72-80`
4. `核心设计/治理策略/返工策略合同.md:68-75`
5. `核心设计/治理策略/验收策略合同.md:102-109`

建议修复：
1. 不要继续保留泛化 `EscalationPolicy` 这个顶层名词
2. 顶层改成显式来源：
   - `WipPolicy.escalation_policy`
   - `ReworkPolicy.escalation_policy`
   - `AcceptancePolicy.escalation_policy`
3. 为 `watchdogNoticeState` 补 dedicated contract
4. 在顶层文档与术语表中明确它是：
   - `watchdog` 去重与已通知进度的正式状态对象

### P1-03 `ResultSummaryPayload` 的顶层字段形状漂移（已关闭）
问题：
1. 顶层链路图把 `ResultSummaryPayload` 写成至少包含：
   - `managed_task_ref`
   - `acceptance_verdict`
   - `summary`
   - `proof_of_work_excerpt`
   - `review_summary_excerpt`
2. 但正式合同里并没有 `review_summary_excerpt` 这个字段
3. 正式合同要求的是：
   - `proof_of_work_excerpt.review_summary`

根因：
1. 顶层链路图在结果链重构后没有同步回到正式 payload 形状

影响范围：
1. adapter 或测试可能照着链路图实现错误字段名
2. 会把顶层图示变成第二套 payload 真相源

证据：
1. `产品规范/治理链路图.md:196-204`
2. `核心设计/宿主接入/内核出站载荷合同.md:255-270`
3. `核心设计/宿主接入/内核出站载荷合同.md:407-417`

建议修复：
1. 顶层链路图不得再发明字段别名
2. 统一改成正式字段路径：
   - `proof_of_work_excerpt.review_summary`
3. 同时补一句：
   - `ResultSummaryPayload` 只是 `ProofOfWorkBundle` 的用户可见摘录

### P2-04 实施主文档把 transport、domain、payload、outbox 混成同一层（已关闭）
问题：
1. “第一条 spike 的固定输入输出”里，输入侧列了：
   - `CurrentTurnEnvelope`
   - `HostSemanticBundle`
   - `ControlAction`
   - `HostCapabilitySnapshot`
2. 同一节却漏掉了核心正式输入：
   - `SemanticDecisionEnvelope`
3. 输出侧又把：
   - `StartCardPayload`
   - `ToolDecisionPayload`
   - `ReviewSummary`
   - `ResultSummaryPayload`
   - `OutboundDelivery`
   混放在一起

根因：
1. 实施主文档在接口层级收紧后，没有把切片清单一起按层分开

影响范围：
1. TDD 切片可能直接从 `HostSemanticBundle` 写进 kernel
2. 也可能把 `ReviewSummary` 误当成宿主正式出站接口
3. 会削弱 adapter / kernel / payload / delivery 的层级边界

证据：
1. `实施与验收/产品实施总方案.md:99-113`
2. `实施与验收/接手速启.md:83-88`

建议修复：
1. 重写成四层清单：
   - host transport input
   - kernel formal input
   - domain product
   - outbound payload / delivery
2. 显式补上 `SemanticDecisionEnvelope`
3. 把 `ReviewSummary` 从“宿主输出侧”抽回结果链内部产物

### P2-05 `taskConversationPolicy=chat_open` 缺 formal owner（已关闭）
问题：
1. 顶层规范已声明默认 `taskConversationPolicy=chat_open`
2. 模板文档也把它写成开发约束
3. 但 `核心设计/` 中没有对应合同，也没有明确声明它只是非持久化标量默认值

根因：
1. 这条行为默认值从研究/模板进入了顶层主线
2. 但没有完成 owner 决策

影响范围：
1. 实现层将来如果要持久化、暴露 API、或写入 runtime，可能会自由发明字段位置
2. 聊天并行策略容易长成第二套隐式规则

证据：
1. `产品规范/目标规格.md:235-237`
2. `运行与治理/架构决策索引.md:27`
3. `模板与附录/智能体规范模板.md.ext:78`

建议修复：
1. 二选一，不要继续悬空：
   - 明确声明它只是顶层默认值，不是 formal runtime object
   - 或补 dedicated contract
2. 若第一条真实 spike 不实现它的持久化，就应选择第一种

当前结果：
1. 已选择第一种
2. `taskConversationPolicy / chat_open` 现在明确作为顶层标量默认值存在
3. owner 以 [目标规格.md](../产品规范/目标规格.md) 与 [正式对象总表.md](正式对象总表.md) 为准

### P3-07 治理测试没有覆盖关键语义漂移（已关闭）
问题：
1. `documentation-alignment.test.ts` 主要验证：
   - 目录结构
   - 阅读链
   - 旧路径清理
2. 它不验证：
   - `pendingUserDecision` 必须是 projection pointer
   - `pendingBoundaryConfirmation` 必须是窗口扩展对象
   - `ResultSummaryPayload` 顶层字段不能发明 `review_summary_excerpt`
   - `watchdog` 相关正式对象是否有唯一 owner

根因：
1. 当前治理测试仍然以迁移卫生检查为主

影响范围：
1. 文档语义漂移时，测试仍会继续绿
2. 会制造“测试通过 = 文档稳定”的错觉

证据：
1. `custom/open-source/openclaw-plugins/tests/governance/acceptance/documentation-alignment.test.ts:29-168`

建议修复：
1. 在治理测试里补语义断言
2. 至少覆盖：
   - 决策窗口家族层级
   - `ResultSummaryPayload` 字段路径
   - `watchdog` 缺失 owner 不允许进入顶层主线

当前结果：
1. [documentation-alignment.test.ts](../../custom/open-source/openclaw-plugins/tests/governance/acceptance/documentation-alignment.test.ts) 已补入主链对象层级与字段路径断言
2. 这条问题现在不再作为当前开放阻断项存在
