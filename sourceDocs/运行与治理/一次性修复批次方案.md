# 一次性修复批次方案

更新时间：2026-03-11

---

状态：历史修复批次记录（已执行）

## 1. 目标
这份方案只回答一个问题：

**怎样一次把当时阻断开发的文档缺口收掉，而不是继续做零散补丁。**

当前说明：
1. 这份文档保留的是历史修复批次设计与执行逻辑。
2. 它不再作为当前待执行主计划。
3. 后续围绕 8 个治理深化点的现行设计，统一以正式 owner 文档和实施/测试主链为准。
4. 当前接手实现请优先回到：
   - [产品实施总方案.md](../实施与验收/产品实施总方案.md)
   - [测试案例.md](../实施与验收/测试案例.md)
   - [正式对象总表.md](正式对象总表.md)

固定原则：
1. 先修顶层 owner 与层级错误
2. 再补真实缺失 contract
3. 再回填实施与验收
4. 同步补治理测试，避免问题回流

---

## 2. 批次 A：收紧顶层对象定义
目标：
1. 让顶层对象表、术语和状态机与 `决策窗口合同` 完全一致

必须修改：
1. `产品规范/治理架构.md`
2. `产品规范/状态机附录.md`
3. `产品规范/概念模型.md`
4. `运行与治理/运行时基线.md`
5. `总览/术语表.md`

固定改法：
1. `pendingUserDecision`
   - 改成“当前任务上指向 open `PendingDecisionWindow` 的 task projection pointer”
2. `pendingBoundaryConfirmation`
   - 改成“`BoundaryConfirmation` 窗口的边界扩展子实体”
3. `decision_token`
   - 改成“由 `PendingDecisionWindow` 持有生命周期的消费令牌”
4. 所有 payload 词条都补清：
   - `StartCardPayload / BoundaryCardPayload / ApprovalRequestPayload`
   - 只属于用户可见 projection
   - 不是正式入口本体

完成标准：
1. 顶层文档里不再出现“`pendingUserDecision` 是决策窗口本体”的表述
2. 顶层文档里不再把 `pendingBoundaryConfirmation` 写成泛化边界窗口本体

---

## 3. 批次 B：补齐 `watchdog` 缺失 contract
目标：
1. 让 `watchdog` 链不再依赖研究概念或泛化名词

固定决策：
1. 不新增泛化 `EscalationPolicy` 合同
2. 顶层主线统一显式引用：
   - `WipPolicy.escalation_policy`
   - `ReworkPolicy.escalation_policy`
   - `AcceptancePolicy.escalation_policy`
3. `watchdogNoticeState` 确认为真实缺口，补 dedicated contract

建议新合同：
1. `核心设计/治理策略/watchdog通知状态合同.md`

这份合同至少定义：
1. `watchdogNoticeState` 是什么
2. 它记录哪些 notice 已发、哪些还待发、哪些需要去重
3. 它和 `TaskEventModel`、`WipPolicy`、`ReworkPolicy`、`AcceptancePolicy` 的关系
4. 它不是宿主 UI 状态，也不是 compatibility projection

同步修改：
1. `产品规范/治理架构.md`
2. `总览/术语表.md`
3. `运行与治理/运行时基线.md`
4. `核心设计/目录说明.md`

完成标准：
1. 顶层主线不再出现无 owner 的 `EscalationPolicy`
2. `watchdogNoticeState` 在 `核心设计/` 中有唯一 owner

## 4. 批次 C：收口结果摘要字段形状
目标：
1. 让 `ResultSummaryPayload` 只有一套字段形状

必须修改：
1. `产品规范/治理链路图.md`
2. 如有必要，同步抽查：
   - `产品规范/运行时语义.md`
   - `实施与验收/验收规范.md`
   - `实施与验收/测试案例.md`

固定改法：
1. 删掉顶层伪字段 `review_summary_excerpt`
2. 统一改成正式路径：
   - `proof_of_work_excerpt.review_summary`
3. 明写：
   - `ResultSummaryPayload` 只是 `ProofOfWorkBundle` 的摘录
   - 不能另起第二套结果字段体系

完成标准：
1. `sourceDocs/` 中不再出现 `review_summary_excerpt`
2. 顶层、合同、案例的 `ResultSummaryPayload` 字段路径完全一致

## 5. 批次 D：收紧实施主文档与治理测试
目标：
1. 让开工文档和自动检查真正跟上 formal contract

必须修改的实施文档：
1. `实施与验收/产品实施总方案.md`
2. 如有必要，`实施与验收/接手速启.md`

固定改法：
1. 把“固定输入输出”拆成：
   - host transport input
   - kernel formal input
   - domain product
   - outbound payload
   - delivery object
2. 输入侧明确补上 `SemanticDecisionEnvelope`
3. 输出侧不再把 `ReviewSummary` 和宿主 payload 混成一组

必须补的治理测试：
1. `documentation-alignment.test.ts`

至少新增断言：
1. 顶层不得把 `pendingUserDecision` 写成窗口本体
2. 顶层不得把 `pendingBoundaryConfirmation` 写成泛化边界窗口
3. 顶层不得出现 `review_summary_excerpt`
4. 顶层 `watchdog` 文档不得使用无 owner 的 `EscalationPolicy`

可选补充：
1. 新增一份 sourceDocs 语义一致性测试
2. 只做文档字符串与阅读链断言，不绑定代码实现

完成标准：
1. 实施文档切片不会再混 transport / domain / payload / delivery
2. 治理测试能抓住这轮已经发现的语义漂移

## 6. 执行顺序
固定顺序不要改：
1. 批次 A
2. 批次 B
3. 批次 C
4. 批次 D

原因：
1. 先修顶层定义，才能避免新 contract 又被旧术语冲掉
2. 先补 `watchdog` owner，后面实施文档和测试才能写得准
3. 结果摘要字段形状收口后，再补测试最稳
