# 规格与工作证明吸收说明

状态：intake draft  
定位：吸收外部方案到 Loom 的正式借鉴稿  
更新时间：2026-03-10

---

## 1. 目的
这份文档回答 3 个问题：
1. Loom 从 “Spec 模式” 吸收什么
2. Loom 从 “Symphony / Proof of Work” 吸收什么
3. Loom 明确不吸收什么

更深层上，它解决的是：

**把外部文章中的好想法收进 Loom 的正式对象，而不是停留在口头灵感。**

---

## 2. 外部输入
### 2.1 Spec 模式文章
吸收点来自：
1. `scope / tasks / checklist` 三件套
2. 活文档可修订
3. 文档是执行锚点，不是附件

### 2.2 Symphony 文章
吸收点来自：
1. 单任务隔离 run
2. `Proof of Work`
3. 审批看证据，不靠感觉
4. 协议优先，不锁死实现语言

---

## 3. Loom 吸收什么
### 3.1 吸收成 `SpecBundle`
Loom 正式吸收：
1. `scope_doc`
2. `plan_doc`
3. `verification_doc`

它们统一收进：
1. `SpecBundle`

取舍：
1. 不把 Spec 只当 coding 技巧
2. 直接把它提升成 Loom 通用核心能力

### 3.2 吸收成 `ProofOfWorkBundle`
Loom 正式吸收：
1. `artifact_manifest`
2. `review_summary`
3. `acceptance_verdict`
4. `evidence_refs`
5. `run_summary`

它们统一收进：
1. `ProofOfWorkBundle`

取舍：
1. 不把交付只做成 summary
2. 必须把交付物、验收和证据绑在一起

### 3.3 吸收成 `IsolatedTaskRun`
Loom 正式吸收：
1. 一个 `managedTaskRef` 在执行期绑定一个 run
2. run 有自己的 id、事件、产物和完成态
3. 一个 run 的失败不能污染另一个任务

它统一收进：
1. `IsolatedTaskRun`

取舍：
1. v0 不承诺完整 container / OS 沙箱
2. 但必须承认 run 是正式运行时边界

---

## 4. Loom 明确不吸收什么
### 4.1 不吸收 `/spec` 产品壳
Loom 不做：
1. 一个以 `/spec` 命令为核心入口的 coding 产品

原因：
1. Loom 是治理内核
2. `SpecBundle` 是 Loom 核心对象，不是命令产品本体

### 4.2 不吸收 coding-only 视角
Loom 不做：
1. 只服务 PR / CI / Linear 的工程团队工具

原因：
1. Loom 面向个人 AI 终端
2. coding 只是第一批 pack

### 4.3 不吸收“Proof of Work = 附几条日志”
Loom 不做：
1. 用零散日志冒充证据包

原因：
1. `ProofOfWorkBundle` 必须是正式结构
2. 它要能被 `ResultContract`、review 和验收稳定消费

### 4.4 不夸大 v0 的隔离能力
Loom 不做：
1. 把 `IsolatedTaskRun` 写成已经具备完整沙箱

原因：
1. v0 先定义 run 边界
2. 真正隔离强度留给后续实现演进

---

## 5. 和现有 Loom 对象怎么对齐
1. `TaskScopeSnapshot`
   - 回答当前正式范围是什么
2. `SpecBundle`
   - 回答怎么把范围、计划、验证锚在可读文档里
3. `PhasePlan`
   - 回答这次真正怎么推进
4. `AcceptancePolicy`
   - 回答什么算完成
5. `ReviewResult`
   - 回答这次 review 的正式结论
6. `ProofOfWorkBundle`
   - 回答交付时有哪些正式证据
7. `ResultContract`
   - 回答最终怎么把范围、结果和证据交付给用户
8. `IsolatedTaskRun`
   - 回答执行期的正式运行边界

---

## 6. 对 Loom 的直接建议
1. `SpecBundle` 升为 Loom 通用能力，不只挂在 `coding_pack`
2. `ProofOfWorkBundle` 升为结果链核心对象，不再只是 `review_summary + artifact_manifest` 的松散组合
3. `IsolatedTaskRun` 升为运行时边界对象，不再把 run 语义散在 `ExecutionAuthorization` 和 event 里
4. `coding_pack / COMPLEX` 作为第一条最完整样板
5. `research_pack / COMPLEX` 作为第二个样板，证明这不是 coding-only 设计

---

## 7. 当前结论
Loom 这次吸收外部方案，不是为了模仿产品外壳，而是为了把这 3 个核心能力收成正式对象：
1. `SpecBundle`
2. `ProofOfWorkBundle`
3. `IsolatedTaskRun`
