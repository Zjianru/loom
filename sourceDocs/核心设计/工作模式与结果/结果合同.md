# 结果合同

状态：contract draft  
定位：Loom 最终结果包正式规格  
更新时间：2026-03-11

---

## 1. 目的
`ResultContract` 定义 managed task 最终对用户交付什么结果包。

它回答：
1. 任务结束时交付哪些正式内容
2. 用户默认先看到什么
3. 动态变更后最终按哪版范围交付
4. review、验收和证据怎样进入结果链

更深层上，它解决的是：

**让任务结束后的输出不是一段随意总结，而是一个带正式范围、正式结论、正式证据的可交付结果包。**

---

## 2. 设计边界
### 2.1 这不是 `AcceptancePolicy`
1. `AcceptancePolicy`
   - 定义什么算完成
2. `ResultContract`
   - 定义完成后最终怎么交付

### 2.2 这不是 recorder 的全量笔记
1. recorder 会产出大量阶段纪要和沉淀
2. `ResultContract` 只吸收其中用于交付的部分

### 2.3 这不是 `TaskEventModel`
1. `TaskEventModel`
   - 记录发生过什么
2. `ResultContract`
   - 提炼最终交付什么、为什么成立

### 2.4 结构化结果只在协议层存在
1. 结果包必须有结构化协议版
2. 用户看到的是 adapter 文本渲染版
3. 结构化结果不能直接倒给用户

---

## 3. v0 固定规则
1. 所有终态都必须产出正式结果包：
   - `completed`
   - `completed_with_notes`
   - `partial`
   - `blocked`
   - `cancelled`
   - `rework_exhausted`
2. 结果包采用统一外壳 + pack-specific section
3. 默认先给用户短总结，再允许展开完整结果包
4. 结果包必须反映动态任务变更
5. 结果包必须包含正式证据 section：`ProofOfWorkBundle`
6. 单次 review 的 `rework_required / EscalateToUser`
   - 不直接等于应当立即编译最终 `ResultContract`
   - 前者先进入返工链，后者先进入审批窗口

---

## 4. 动态变更下的结果语义
### 4.1 `managedTaskRef` 稳定，scope 可以修订
1. `managedTaskRef`
   - 任务 owner
2. `TaskScopeSnapshot`
   - 当前正式范围真相

所以：
1. 一个任务可以持续变更 scope
2. 不必因为每次补充要求就重建新 task

### 4.2 最终交付以最新接受范围为准
1. 已发生的阶段、review、返工是历史事实
2. 最终是否算完成，要以最后接受的范围为准

结果包必须同时说明：
1. 发生过什么
2. 最后按什么范围交付

---

## 5. 正式对象定义
```rust
pub struct ResultContract {
    pub result_id: ResultId,
    pub managed_task_ref: ManagedTaskRef,
    pub pack_ref: PackRef,
    pub outcome: ResultOutcome,
    pub acceptance_verdict: AcceptanceResult,
    pub final_scope_snapshot: FinalScopeSnapshot,
    pub scope_revision_summary: ScopeRevisionSummary,
    pub summary: UserSummary,
    pub key_outcomes: Vec<KeyOutcomeItem>,
    pub proof_of_work: ProofOfWorkBundle,
    pub next_actions: Vec<NextActionItem>,
    pub pack_sections: Vec<PackResultSection>,
    pub internal_appendix: InternalResultAppendix,
    pub render_policy: ResultRenderPolicy,
}
```

---

## 6. 核心字段解释
### 6.1 `outcome`
任务最终终态，建议值：
1. `completed`
2. `completed_with_notes`
3. `partial`
4. `blocked`
5. `cancelled`
6. `rework_exhausted`

### 6.2 `acceptance_verdict`
正式验收结论。  
它必须和 `AcceptancePolicy` 对齐，并进入最终交付。

固定边界：
1. 当 `acceptance_verdict = EscalateToUser`
2. 当前任务仍未完成最终交付收口
3. v0 必须先打开 `PendingDecisionWindow(kind=ApprovalRequest)`，再投影 `pendingUserDecision`，再发 `ApprovalRequestPayload`
4. `ApprovalRequestPayload` 只是审批窗口的用户可见 projection，不是正式入口本体

### 6.3 `final_scope_snapshot`
最终被接受的范围快照。  
它回答“这次到底按什么范围交付”。

### 6.4 `scope_revision_summary`
执行过程中发生过哪些关键范围修订。  
建议最小字段：
1. `original_scope_summary`
2. `accepted_changes`
3. `superseded_items`
4. `deferred_items`
5. `latest_scope_version`

### 6.5 `summary`
默认首先展示给用户的短总结。

### 6.6 `key_outcomes`
这次任务最重要的结果点列表。

### 6.7 `proof_of_work`
正式证据 section。  
它至少包含：
1. `artifact_manifest`
2. `review_summary`
3. `acceptance_verdict`
4. `evidence_refs`
5. `run_summary`
6. 第一版整体结果接受快照：
   - `accepted_by_ref`
   - `accepted_at`
   - `accepted_scope_version`

### 6.8 `next_actions`
后续建议动作或继续委派的建议。

### 6.9 `pack_sections`
pack-specific 的专属结果 section。

### 6.10 `internal_appendix`
不直接展示给用户的内部附录。  
建议包括：
1. reviewer 详细评论
2. raw diagnostics
3. recorder 全量沉淀引用
4. event trace refs

---

## 7. `ProofOfWorkBundle` 在结果链里的位置
结果链固定关系：
1. `ReviewResult`
   - 单次 review 事实
2. `AcceptancePolicy`
   - 完成判定规则
3. `ProofOfWorkBundle`
   - 汇总 review、验证、工件、关键证据
4. `ResultContract`
   - 最终把范围、结论、证据一起交付
5. 第一版不引入独立 `AcceptanceLedger`
6. 若需要正式接受记录，只允许通过 `ProofOfWorkBundle` 的整体接受快照进入结果链

取舍：
1. `review_summary` 不再只是一个松散字段
2. `artifact_manifest` 不再只是清单
3. 二者都收进 `ProofOfWorkBundle`

---

## 8. user-visible 与 internal 的边界
### 8.1 `user_visible`
建议包括：
1. `summary`
2. `acceptance_verdict`
3. `key_outcomes`
4. `final_scope_snapshot` 的简版
5. `scope_revision_summary` 的简版
6. `proof_of_work` 的摘要：
   - `artifact_manifest`
   - `review_summary`
   - `acceptance_basis`
   - 整体结果接受快照
7. `next_actions`

### 8.2 `internal_only`
建议包括：
1. reviewer 全量评论
2. raw diagnostics
3. recorder 全量沉淀引用
4. event trace refs

---

## 9. 和实例集的关系
这份 contract 只定义字段，还不够约束实现。  
正式实例统一以：
1. [结果样例.md](结果样例.md)

为准。

它的作用是：
1. 给 `loom-openclaw` 的文本渲染提供正式实例
2. 给代码 spike 和测试夹具提供结果样板
3. 保证 `final_scope_snapshot / scope_revision_summary / proof_of_work` 不会在实现层被缩没

---

## 10. 我的建议
1. 结果包不能只交“做了什么”，还要交“最后按什么范围做完”
2. `ProofOfWorkBundle` 应成为结果链核心对象，而不是附属说明
3. recorder 产物只部分进入结果包，其余继续留在 internal appendix
