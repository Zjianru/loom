# 编码工作模式预设

状态：preset draft  
定位：`coding_pack` 的默认产品形状  
更新时间：2026-03-10

---

## 1. 目的
这份文档不再定义抽象 `PackContract`，而是把 `coding_pack` 写成 Loom 可以直接落地的默认工作模式。

它回答：
1. coding 场景默认怎么生成 `PhasePlan`
2. coding 场景默认需要哪些角色
3. coding 场景默认套哪套验收、预算、WIP、delegation 骨架
4. 用户能改什么，不能改什么

更深层的取舍是：
1. `PackContract`
   - 回答“pack 这个对象长什么样”
2. `coding_pack preset`
   - 回答“coding 这个场景默认到底怎么跑”

如果没有这份 preset，后面的 `Harness` 很容易把默认阶段、默认角色、默认 review 强度偷偷写进代码。

---

## 2. 核心变量解释
### 2.1 `default_stage_sequence`
它是什么：
1. coding 场景默认按什么阶段骨架生成本次 `PhasePlan`

它代表：
1. 默认流程骨架
2. `Harness` 生成具体 `PhasePlan` 的起点

它不代表：
1. 最终执行图
2. 不可变阶段列表

### 2.2 `default_roster_policy`
它是什么：
1. coding 场景默认启用哪些角色、如何选人

它代表：
1. 角色骨架
2. 选人策略

它不代表：
1. 具体 `AgentBinding`
2. 固定 agent id

### 2.3 `default_acceptance_policy_ref`
它是什么：
1. 默认引用哪套验收规则

它代表：
1. `AcceptancePolicy` 的基线
2. coding 场景下“完成”至少要看哪些维度

### 2.4 `default_spec_policy_ref`
它是什么：
1. 默认生成哪种 `SpecBundle`

### 2.5 `default_proof_of_work_policy_ref`
它是什么：
1. 默认要求哪种 `ProofOfWorkBundle`

### 2.6 `default_budget_policy_ref`
它是什么：
1. 默认预算约束

它代表：
1. 这类任务默认愿意花多少 token / 模型额度 / 返工轮次

### 2.7 `default_wip_policy_ref`
它是什么：
1. 默认流动治理约束

它代表：
1. 仍然只有一个 `activeManagedTaskRef`
2. 默认一个 active stage
3. 当前 coding 任务内部不引入复杂并行图

### 2.8 `default_delegation_profile_ref`
它是什么：
1. 默认授权边界档位

它代表：
1. 哪些读操作默认允许
2. 哪些写操作必须走 `approval-gate`
3. 哪些高风险动作永远不能绕过用户

## 3. `coding_pack` 的正式定位
### 3.1 对外故事
`coding_pack` 对外承包的是：
1. 把一个编码任务正式委派出去
2. 在执行过程中保留阶段治理和 review
3. 在任务结束时给用户一个可以交付的结果包，而不是一段散乱执行日志

### 3.2 默认支持的 `managedTaskClass`
建议固定为：
1. `COMPLEX`
   - v0 写透
2. `HUGE`
   - 先保留 skeleton
3. `MAX`
   - 先保留 skeleton

### 3.3 默认 `managedTaskClass`
建议固定为：
1. `COMPLEX`

原因：
1. `COMPLEX` 是第一条真实代码 spike 的最小闭环
2. 它已经能验证：
   - 任务委派
   - recorder 参与
   - review 阶段
   - 结果包交付

### 3.4 `recommended_work_horizon`
建议默认 hint：
1. `extension`

取舍：
1. coding 场景很多任务本质上是在现有项目或代码基线上继续扩展
2. 这只是 hint
3. 最终 `WorkHorizon` 仍由宿主语义层判断

## 4. `COMPLEX` 的完整 preset
### 4.1 进入条件
建议最小 `entry_requirements`：
1. 存在可工作的目标上下文
   - 例如 repo、workspace、文件集、需求描述
2. 用户给出的任务至少能形成一个受管目标
3. 宿主语义层已给出：
   - `interactionLane=managed_task_candidate`
   - `managedTaskClass=COMPLEX`
   - `taskActivationReason`

### 4.2 默认阶段骨架
建议固定：
1. `clarify`
2. `execute`
3. `review`
4. `deliver`

这里的取舍是：
1. 先保持和 `loom-openclaw` 最小 spike 一致
2. 不在 v0 的 `COMPLEX` 里再引入额外并行阶段
3. `verification expectation` 不单独拆成独立 stage，而是写进：
   - `execute` 的产出要求
   - `review` 的验收证据要求
4. 这里的 `clarify / execute / review / deliver`
   - 都是 `default_stage_sequence` 里的 `StagePackageId`
   - 不是顶层 `workflowStage` 扩展枚举

### 4.3 默认角色骨架
建议固定：
1. `net`
   - 宿主主 assistant，保持和用户继续聊天
2. `recorder`
   - 正式记录角色，负责阶段纪要和结果摘要素材
3. `worker`
   - 1 个真实编码 agent

说明：
1. `review_group` 在 v0 的 `COMPLEX` preset 里先作为轻量 review 规则存在
2. 不要求一开始就是真实多 reviewer agent 组
3. 但 `review` 阶段必须存在

### 4.4 默认验收基线
`default_acceptance_policy_ref`
- 建议引用 coding 专用 acceptance profile，其最低检查项应包含：
1. `requirements_satisfied`
2. `tests_or_verification_passed`
3. `risk_within_threshold`
4. `review_approved`

取舍：
1. “测试通过”不是唯一条件
2. 但 coding 场景里它必须是硬门槛之一
3. 最终完成是“与关系”，不是“或关系”

### 4.5 默认 `SpecBundle`
`default_spec_policy_ref`
- 建议固定为稳定模板 id `coding.spec.full.v0`：
1. 生成 `scope_doc`
2. 生成 `plan_doc`
3. 生成 `verification_doc`

固定语义：
1. `default_spec_policy_ref`
   - 在 v0 不是外部 registry 引用
   - 而是 Loom 固定命名空间下的稳定模板 id
2. 它只用于：
   - 生成
   - 追溯
   - 渲染
   - 测试夹具选择

### 4.6 默认 `ProofOfWorkBundle`
`default_proof_of_work_policy_ref`
- 建议固定为稳定模板 id `coding.pow.full.v0`：
1. `artifact_manifest`
2. `review_summary`
3. `acceptance_verdict`
4. `evidence_refs`
5. `run_summary`

固定语义：
1. `default_proof_of_work_policy_ref`
   - 在 v0 不是外部 registry 引用
   - 而是 Loom 固定命名空间下的稳定模板 id
2. 它只用于：
   - 生成
   - 追溯
   - 渲染
   - 测试夹具选择

### 4.7 默认预算基线
`default_budget_policy_ref`
- 建议定位为中等 appetite：
1. 默认允许 1 个 worker
2. 默认允许 1 轮主 review + 最多 3 轮返工
3. 超预算时优先走 `watchdog` 升级，不静默自转

### 4.8 默认 WIP 基线
`default_wip_policy_ref`
- 建议固定：
1. `max_active_tasks = 1`
2. `max_active_stages = 1`
3. `COMPLEX` 内部不做多 worker 并行调度

### 4.9 默认 delegation 基线
`default_delegation_profile_ref`
- 建议固定：
1. 读类操作默认允许
2. 写类操作必须过 `approval-gate`
3. 危险发布、删除、外发类动作默认不可绕过用户

### 4.10 默认 explanation template
建议用户可见解释固定包含：
1. 这是一项 coding 工作模式
2. 系统会启用：
   - `worker`
   - `recorder`
   - `review`
3. 你会先看到 start card
4. 任务开始后会生成 `SpecBundle`
5. 结果会以摘要 + `ProofOfWorkBundle` 摘要形式返回

## 5. `HUGE / MAX` skeleton
### 5.1 `HUGE`
这轮只保留骨架：
1. 默认支持多阶段
2. 默认允许多 worker
3. 默认 review 严格度高于 `COMPLEX`

为什么这轮不写透：
1. 第一条真实 spike 还不验证 `1+n`
2. 先不让 preset 细节跑在实现前面

### 5.2 `MAX`
这轮只保留骨架：
1. 默认支持更深层级协作
2. 默认有更强的 review / budget / escalation
3. 默认需要更严的 capability 与 delegation 管控

为什么只保留保留位：
1. `MAX` 更像项目级编排
2. v0 先不在 preset 层发明完整多层治理细节

---

## 6. override 边界
### 6.1 `selectable`
建议允许用户直接选：
1. `managedTaskClass`
2. 预算档位
3. review 严格度
4. 通知频率
5. 可选验证强度

### 6.2 `bounded_tunable`
建议允许用户在边界内调：
1. 某些 acceptance 权重
2. 返工上限
3. 某些 delegation band

### 6.3 `extensible`
建议允许用户追加：
1. 专业 worker
2. 可选 review / validate 角色
3. 可选阶段

### 6.4 `locked`
必须锁死：
1. `recorder` 必须存在
2. `review` 阶段必须存在
3. `approval-gate` 底线不可绕过
4. `max_active_tasks = 1`

原因：
1. 这些是 Loom 的治理骨架
2. 如果允许用户把它们拆掉，pack 就会退化成自由 workflow builder

---

## 7. 对后续实现的约束
### 7.1 对 `Harness`
规则：
1. `Harness` 不能自己拍脑袋生成 coding 默认阶段
2. 第一条 `COMPLEX` 代码 spike 的默认阶段骨架必须来自这里

### 7.2 对 `loom-openclaw`
规则：
1. start card 文本解释必须和这里的 `explanation_template` 一致
2. 不允许 adapter 为了 demo 更顺手，偷偷删掉 recorder 或 review

---

## 8. 我的建议
1. `coding_pack` 的 v0 核心不是“做很多编码模式”，而是把：
   - 委派
   - 记录
   - review
   - 结果交付
   这一条治理链写透
2. `COMPLEX` 要和第一条 spike 完全同形，不要文档一套、实现一套
3. `HUGE / MAX` 现在只做 skeleton 是对的；这不是偷懒，而是避免默认行为先被写歪
