# 评审结果合同

状态：contract draft  
定位：review 阶段正式产物规格  
更新时间：2026-03-10

---

## 1. 目的
这份 contract 定义 review 阶段到底产出什么正式结果。

它回答：
1. 单次 review 的结构化结论长什么样
2. `review_verdict`、`ReviewFinding`、`ReviewSummary` 各自代表什么
3. review 如何喂给 `AcceptancePolicy` 和 `ResultContract`
4. 为什么 `review` 不能只是流程名词

更深层上，它解决的是：

**让 `review` 阶段从“流程里有这个步骤”升级成“能被 Loom、adapter、result contract 正式消费的治理证据”。**

---

## 2. 设计边界
### 2.1 这不是 `AcceptancePolicy`
边界：
1. `AcceptancePolicy`
   - 回答“怎样才算完成”
2. `ReviewResult`
   - 回答“这次 review 实际给出了什么结论”

### 2.2 这不是 recorder 的自由笔记
边界：
1. recorder 可以消费 review 结果并沉淀纪要
2. 但 review 的正式事实必须由 `ReviewResult` 持有

### 2.3 这不是最终结果包
边界：
1. `ReviewSummary`
   - 是 `ReviewResult` 内嵌的简版摘要值对象
2. `ResultContract`
   - 是整次任务的最终交付包

---

## 3. v0 固定规则
### 3.1 `review` 阶段至少必须产出一个 `ReviewResult`
固定规则：
1. 即使 v0 里的 reviewer group 还是轻量实现
2. `review` 阶段也不能只有文字说明
3. 必须落一个正式 `ReviewResult`

### 3.2 `ReviewResult` 负责 review 这一维的事实
固定规则：
1. `AcceptancePolicy` 继续定义规则
2. `ReviewResult` 提供 review 这一维的正式证据

### 3.3 `ResultContract` 必须能引用最近一次有效 `ReviewSummary`
固定规则：
1. 最终结果包必须能说明最近一次有效 review 怎么判
2. 不允许 adapter 或 result compiler 自己重新发明一套 review 文案

---

## 4. 正式对象定义
### 4.1 Rust 形态建议
```rust
pub enum ReviewVerdict {
    Approved,
    ApprovedWithNotes,
    ReworkRequired,
    Blocked,
    EscalateToUser,
}

pub struct ReviewFinding {
    pub finding_id: ReviewFindingId,
    pub severity: ReviewSeverity,
    pub summary: String,
    pub evidence_ref: Option<EvidenceRef>,
}

pub enum ReviewSeverity {
    Info,
    Warning,
    Major,
    Critical,
}

pub struct ReviewResult {
    pub review_result_id: ReviewResultId,
    pub managed_task_ref: ManagedTaskRef,
    pub stage_run_ref: StageRunRef,
    pub reviewer_group_ref: Option<ReviewGroupRef>,
    pub review_verdict: ReviewVerdict,
    pub findings: Vec<ReviewFinding>,
    pub review_artifacts: Vec<ReviewArtifactRef>,
    pub summary: ReviewSummary,
    pub reviewed_at: Timestamp,
}

pub struct ReviewSummary {
    pub review_verdict: ReviewVerdict,
    pub summary: String,
    pub key_findings: Vec<String>,
    pub follow_up_required: bool,
}
```

### 4.2 关键变量分别代表什么
1. `review_verdict`
   - 这次 review 的正式结论
2. `ReviewFinding`
   - 单条问题、风险或观察
3. `ReviewSummary`
   - `ReviewResult` 内嵌的结构化摘要值对象
   - 供 `ResultContract` 和宿主文本层引用
4. `review_artifacts`
   - review 过程产生的附件或证据引用

## 5. verdict 语义
### 5.1 建议值
1. `Approved`
2. `ApprovedWithNotes`
3. `ReworkRequired`
4. `Blocked`
5. `EscalateToUser`

### 5.2 取舍
1. `Approved`
   - 可继续进入 deliver / acceptance 收口
2. `ApprovedWithNotes`
   - 可以继续收口，但 notes 需进入结果包
3. `ReworkRequired`
   - 必须进入返工链
4. `Blocked`
   - 当前阶段无法继续，需等待外部条件或更高层处理
5. `EscalateToUser`
   - review 已经说明系统无法自行裁决，必须回用户
   - 它必须打开正式审批窗口，而不是直接伪装成最终结果

## 6. 和其它对象的边界
### 6.1 和 `AcceptancePolicy`
固定关系：
1. `AcceptancePolicy.review_group`
   - 回答谁来 review、最低人数、裁决规则
2. `ReviewResult`
   - 回答这次 review 实际输出了什么

### 6.2 和 `ResultContract`
固定关系：
1. `ResultContract`
   - 应引用最近一次有效 `ReviewSummary`
2. `ReviewResult`
   - 保留更完整 review 事实

### 6.3 和 `ProofOfWorkBundle`
固定关系：
1. `ProofOfWorkBundle`
   - 是 Loom 结果链里的正式证据包
2. `ReviewResult`
   - 提供 review 这一维的正式事实
3. `ReviewSummary`
   - 作为 `ProofOfWorkBundle.review_summary` 的直接来源

取舍：
1. `ProofOfWorkBundle` 不自己重新发明 review 结论
2. 它只能引用最近一次有效 `ReviewSummary`
3. review 事实仍然由 `ReviewResult` 持有

### 6.4 和 `ChangeImpactAssessment`
如果某次 change assessment 结论要求：
1. reopen stage review
2. reopen task acceptance

最终都必须回到新的 `ReviewResult`。

### 6.5 和 `ApprovalRequestPayload`
固定关系：
1. 当 `review_verdict = EscalateToUser`
2. Loom 必须先打开 `PendingDecisionWindow(kind=ApprovalRequest)`
3. 再把 `pendingUserDecision` 投影到该 open window
4. 再输出 `ApprovalRequestPayload`
5. 在用户决策前，不得直接编译最终 `ResultContract`

### 6.6 和 recorder
recorder 只消费：
1. `ReviewResult`
2. `TaskEvent`

recorder 不再自己发明一套 review 真相。

### 6.7 和 `TaskEventModel`
建议最少写这些事件：
1. `review_requested`
2. `review_completed`
3. `review_result_recorded`
4. `rework_started`

---

## 7. 我的建议
1. `review` 阶段第一天就要有 `ReviewResult`
2. `ReviewSummary` 应成为 `ResultContract` 和宿主文本层的标准引用
3. `ReviewFinding` 先保持轻量，但要有 severity 和 evidence ref
4. 后面即使把 reviewer group 池化成多 agent，`ReviewResult` 这层也不需要重命名
5. `ProofOfWorkBundle` 必须显式消费 `ReviewSummary`，否则 review 仍然只是流程名词
