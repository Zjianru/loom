# 工作证明合同

状态：contract draft  
定位：Loom 的正式交付证据包规格  
更新时间：2026-03-11

---

## 1. 目的
`ProofOfWorkBundle` 定义任务交付时的正式证据包。

它回答：
1. 最终交付凭什么成立
2. 哪些 review、验证、产物、证据被打包成正式交付依据
3. 它和 `ResultContract`、`ReviewResult`、`AcceptancePolicy` 的边界是什么

---

## 2. 设计边界
### 2.1 这不是 `ResultContract`
1. `ResultContract`
   - 最终结果外壳
2. `ProofOfWorkBundle`
   - 结果包中的正式证据 section

### 2.2 这不是 `ReviewResult`
1. `ReviewResult`
   - 单次 review 事实
2. `ProofOfWorkBundle`
   - 交付时汇总 review、验证、产物、证据的正式包

### 2.3 这不是随手附几条日志
1. 日志可以是 evidence ref 的来源
2. 但 `ProofOfWorkBundle` 必须是正式结构化对象

---

## 3. v0 固定规则
1. 每个进入 `result` 的 managed task 都必须生成 `ProofOfWorkBundle`
2. `ProofOfWorkBundle` 至少包含：
   - `artifact_manifest`
   - `review_summary`
   - `acceptance_verdict`
   - `evidence_refs`
   - `run_summary`
3. 第一版不引入独立 `AcceptanceLedger`
4. 如果需要正式接受记录，只允许把整体结果接受快照直接收进 `ProofOfWorkBundle`
5. `ResultSummaryPayload` 只能摘录它，不能取代它

---

## 4. 正式对象定义
```rust
pub struct ProofOfWorkBundle {
    pub proof_of_work_id: ProofOfWorkId,
    pub managed_task_ref: ManagedTaskRef,
    pub run_ref: IsolatedTaskRunRef,
    pub acceptance_verdict: AcceptanceResult,
    pub acceptance_basis: Vec<AcceptanceBasisItem>,
    pub accepted_by_ref: Option<String>,
    pub accepted_at: Option<Timestamp>,
    pub accepted_scope_version: u32,
    pub review_summary: Option<ReviewSummary>,
    pub artifact_manifest: ArtifactManifest,
    pub evidence_refs: Vec<EvidenceRef>,
    pub run_summary: RunSummary,
}

pub struct AcceptanceBasisItem {
    pub basis_kind: AcceptanceBasisKind,
    pub summary: String,
    pub passed: bool,
}

pub enum AcceptanceBasisKind {
    RequirementsSatisfied,
    VerificationPassed,
    ReviewApproved,
    RiskWithinThreshold,
}

pub struct RunSummary {
    pub run_id: IsolatedTaskRunId,
    pub run_outcome: IsolatedTaskRunOutcome,
    pub summary: String,
    pub stage_path: Vec<StagePackageId>,
    pub started_at: Timestamp,
    pub finished_at: Option<Timestamp>,
}
```

v0 固定取舍：
1. `ProofOfWorkPolicyRef`
   - 不是外部 registry lookup
   - 而是 Loom 固定命名空间下的稳定模板 id
2. `stage_path`
   - 记录的是本次 run 真实走过的细粒度 `StagePackageId`
   - 不是粗粒度 `workflowStage` 枚举

---

## 5. 字段解释
1. `proof_of_work_id`
   - 证据包稳定 id
2. `managed_task_ref`
   - 证据包归属哪个任务
3. `run_ref`
   - 证据包归属哪个 run
4. `acceptance_verdict`
   - 正式验收结论
5. `acceptance_basis`
   - 验收依据清单
6. `accepted_by_ref / accepted_at / accepted_scope_version`
   - 第一版整体结果接受快照
   - 只记录是谁在什么时候接受了哪版最终范围
   - 不扩展成 claim 级 ledger
7. `review_summary`
   - 最近一次有效 review 摘要
8. `artifact_manifest`
   - 产物清单
9. `evidence_refs`
   - 关键证据引用
10. `run_summary`
   - 这次 run 的简明摘要

这里的阶段层级必须分开：
1. `workflowStage`
   - 只表示 `candidate / execute / review / result / closed`
2. `run_summary.stage_path`
   - 只表示 run 真实经过的 `PhasePlan` / `StagePackageId` 路径

---

## 6. 和结果链的关系
1. `ReviewResult`
   - 提供单次 review 事实
2. `AcceptancePolicy`
   - 提供完成判定规则
3. `ProofOfWorkBundle`
   - 汇总正式证据
4. `ResultContract`
   - 把最终范围、结论、证据一起交付
5. 第一版整体接受快照也由 `ProofOfWorkBundle` 持有
6. 不再引入平行 `AcceptanceLedger` 抢结果 owner

---

## 7. 和 pack 的关系
1. `PackContract` 只定义：
   - `default_proof_of_work_policy_ref`
2. 这里的 `default_proof_of_work_policy_ref`
   - 在 v0 只表示稳定模板 id
   - 不要求外部 policy registry
3. 具体任务进入结果期时
4. Loom 依据 pack、review、acceptance、run artifacts 生成 `ProofOfWorkBundle`

---

## 8. 我的建议
1. `ProofOfWorkBundle` 应成为 Loom 结果链核心对象
2. `artifact_manifest` 和 `review_summary` 不应再只是松散字段
3. `ResultSummaryPayload` 只能提取摘要，不能替代完整证据包
4. 第一版如需记录“谁接受了结果”，先直接写进 `ProofOfWorkBundle`
