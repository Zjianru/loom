# 结果样例

状态：example catalog  
定位：`ResultContract` 的正式实例集  
更新时间：2026-03-10

---

## 1. 目的
这份文档不是再定义 `ResultContract` 字段，而是给实现、测试、文本渲染提供正式实例。

它回答：
1. `ResultContract` 在真实场景里长什么样
2. 用户最后会收到怎样的结果摘要
3. `proof_of_work`、`final_scope_snapshot`、`scope_revision_summary` 在实例里怎么出现

更深层的取舍是：
1. 没有 example，结果包字段很容易在实现时被缩成“随便吐个 summary”
2. 有了 example，`loom-openclaw` 的文本渲染和 `Harness` 的结果编译才有硬参照

---

## 2. 固定阅读方式
每个 example 都固定写 5 段：
1. 场景与输入摘要
2. pack 与 class 选择
3. 结构化结果包轮廓
4. 宿主用户可见文本摘要轮廓
5. 为什么这么判定 / 哪些字段最关键

---

## 3. Example A: `coding_pack / COMPLEX / completed`
### 3.1 场景与输入摘要
用户输入：
1. “在这个仓库里补一个命令行参数校验，并把测试一起补上。”

### 3.2 pack 与 class 选择
1. `pack_ref = coding_pack`
2. `managedTaskClass = COMPLEX`

### 3.3 结构化结果包轮廓
```yaml
outcome: completed
acceptance_verdict: accepted
final_scope_snapshot:
  version: 1
  scope_summary: "为现有 CLI 增加参数校验，并补齐对应测试。"
scope_revision_summary:
  latest_scope_version: 1
  accepted_changes: []
  superseded_items: []
  deferred_items: []
proof_of_work:
  run_summary:
    run_id: "run-coding-001"
    run_outcome: "completed"
    stage_path: ["clarify", "execute", "review", "deliver"]
    started_at: "2026-03-10T09:00:00+08:00"
    finished_at: "2026-03-10T09:26:00+08:00"
    summary: "worker 完成实现，recorder 整理了交付证据。"
  evidence_refs:
    - "event://review_result/rvw-001"
    - "artifact://tests/cli-validation"
  review_summary:
    review_verdict: approved
    summary: "实现满足目标，测试与 review 通过，没有剩余阻塞风险。"
    key_findings:
      - "参数校验逻辑与测试范围已闭环。"
    follow_up_required: false
  artifact_manifest:
    - artifact_name: "CLI validation patch"
      artifact_type: "code_change"
    - artifact_name: "CLI validation tests"
      artifact_type: "test_result"
  acceptance_basis:
    - basis_kind: requirements_satisfied
      summary: "CLI 参数校验与对应测试已覆盖当前范围。"
      passed: true
    - basis_kind: review_approved
      summary: "最新一次 review 判定为 approved。"
      passed: true
    - basis_kind: verification_passed
      summary: "参数校验相关测试已通过。"
      passed: true
summary:
  "CLI 参数校验已补齐，相关测试已通过。"
```

### 3.4 宿主用户可见文本摘要轮廓
1. 已完成本次 coding 任务
2. 已补充参数校验与对应测试
3. review 通过
4. 交付物：
   - 代码变更
   - 测试结果

### 3.5 为什么这么判定
关键字段：
1. `acceptance_verdict=accepted`
2. `proof_of_work.review_summary.review_verdict=approved`
3. `proof_of_work.artifact_manifest`
   - 说明最终交付了什么

这个例子验证：
1. coding 场景下，完成态必须同时体现代码、测试、review 和正式证据包

## 4. Example B: `coding_pack / COMPLEX / rework_exhausted`
### 4.1 场景与输入摘要
用户输入：
1. “修复这个接口的超时处理，并确保回归测试覆盖超时重试路径。”

### 4.2 pack 与 class 选择
1. `pack_ref = coding_pack`
2. `managedTaskClass = COMPLEX`

### 4.3 结构化结果包轮廓
```yaml
outcome: rework_exhausted
acceptance_verdict: rework_required
final_scope_snapshot:
  version: 1
  scope_summary: "修复接口超时处理并补齐超时重试测试。"
scope_revision_summary:
  latest_scope_version: 1
  accepted_changes: []
  superseded_items: []
  deferred_items: []
proof_of_work:
  run_summary:
    run_id: "run-coding-002"
    run_outcome: "partial"
    stage_path: ["clarify", "execute", "review"]
    started_at: "2026-03-10T09:40:00+08:00"
    finished_at: "2026-03-10T10:03:00+08:00"
    summary: "多轮返工后仍未满足交付标准，本次任务在返工上限处终止。"
  evidence_refs:
    - "event://review_result/rvw-002"
  review_summary:
    review_verdict: rework_required
    summary: "多轮返工后，超时重试回归测试覆盖仍不足，任务停止并保留部分产出。"
    key_findings:
      - "超时重试路径的回归测试覆盖不足。"
    follow_up_required: true
  artifact_manifest:
    - artifact_name: "Timeout handling patch"
      artifact_type: "code_change"
  acceptance_basis:
    - basis_kind: requirements_satisfied
      summary: "超时处理主修复已完成，但完整交付要求尚未满足。"
      passed: false
    - basis_kind: review_approved
      summary: "最新一次 review 要求返工，当前不能进入完成态。"
      passed: false
summary:
  "任务在返工上限处终止：主修复已有部分产出，但未达到可交付标准。"
```

### 4.4 宿主用户可见文本摘要轮廓
1. 当前任务未完成，并已达到返工上限
2. 主修复已有部分产出
3. review 仍要求返工
4. 当前卡点是测试覆盖不足

### 4.5 为什么这么判定
关键字段：
1. `acceptance_verdict=rework_required`
2. `proof_of_work.review_summary.review_verdict=rework_required`
3. `outcome=rework_exhausted`

这个例子验证：
1. 只有在返工链正式终止后，`rework_required` 才能进入最终结果包

## 5. Example C: `coding_pack / COMPLEX / blocked`
### 5.1 场景与输入摘要
用户输入：
1. “把这个本地功能改成调用线上服务接口。”

```yaml
outcome: blocked

final_scope_snapshot:

  accepted_changes: []
  superseded_items: []
  deferred_items: []

      summary: "目标服务接口边界未明，当前范围无法被安全满足。"

    - basis_kind: risk_within_threshold
      summary: "外部服务边界缺失导致风险超出可接受阈值。"
      passed: false

```

### 5.4 宿主用户可见文本摘要轮廓
1. 当前任务已阻塞
2. 尚未形成可交付代码变更
3. 需要补充接口说明或边界确认

### 5.5 为什么这么判定
关键字段：
1. `proof_of_work.artifact_manifest` 为空
2. `proof_of_work.review_summary.review_verdict=blocked`
3. `acceptance_verdict=blocked`

这个例子验证：
1. 失败或阻塞也必须有正式结果包
2. 用户必须知道“为什么停下”

## 6. Example D: `coding_pack / COMPLEX / scope_changed_then_completed`
### 6.1 场景与输入摘要
初始用户输入：
1. “给 CLI 新增批量导入命令。”

执行过程中用户补充：
1. “先不要做完整导入，只做 CSV 解析和 dry-run 验证。”

### 6.2 pack 与 class 选择
1. `pack_ref = coding_pack`
2. `managedTaskClass = COMPLEX`

### 6.3 结构化结果包轮廓
```yaml
outcome: completed_with_notes
acceptance_verdict: accepted_with_notes
final_scope_snapshot:
  version: 2
  scope_summary: "实现 CSV 解析与 dry-run 验证，不含正式导入写入。"
scope_revision_summary:
  latest_scope_version: 2
  accepted_changes:
    - "正式交付范围缩减为 CSV 解析与 dry-run。"
  superseded_items:
    - "完整批量导入写入流程"
  deferred_items:
    - "正式写入与落库"
proof_of_work:
  run_summary:
    run_id: "run-coding-004"
    run_outcome: "completed"
    stage_path: ["clarify", "execute", "review", "deliver"]
    started_at: "2026-03-10T10:50:00+08:00"
    finished_at: "2026-03-10T11:18:00+08:00"
    summary: "任务按第二版范围完成，旧范围中的写入流程已延期。"
  evidence_refs:
    - "event://scope_revision/scope-002"
    - "event://review_result/rvw-004"
  review_summary:
    review_verdict: approved_with_notes
    summary: "按修订后的范围交付成立，但完整导入流程已明确延期。"
    key_findings:
      - "CSV 解析与 dry-run 已闭环。"
      - "完整导入写入已明确延期到后续范围。"
    follow_up_required: true
  artifact_manifest:
    - artifact_name: "CSV parser"
      artifact_type: "code_change"
    - artifact_name: "Dry-run validation"
      artifact_type: "verification"
  acceptance_basis:
    - basis_kind: requirements_satisfied
      summary: "最新范围中的 CSV 解析与 dry-run 目标已满足。"
      passed: true
    - basis_kind: review_approved
      summary: "最新一次 review 判定为 approved_with_notes，延期项已显式记录。"
      passed: true
summary:
  "任务已按修订后的范围完成：交付 CSV 解析与 dry-run，正式导入已延期。"
```

### 6.4 宿主用户可见文本摘要轮廓
1. 任务已完成，但按修订后的范围交付
2. 已交付：
   - CSV 解析
   - dry-run 验证
3. 已延期：
   - 正式导入写入

### 6.5 为什么这么判定
关键字段：
1. `final_scope_snapshot`
   - 表示最终按哪版范围交付
2. `scope_revision_summary`
   - 说明哪些被吸收、覆盖、延期
3. `acceptance_verdict=accepted_with_notes`

这个例子验证：
1. 动态改任务后，结果包必须交代范围演化

1. “调研 3 种本地向量数据库方案，给我一份比较和建议。”

### 7.2 pack 与 class 选择
1. `pack_ref = research_pack`
2. `managedTaskClass = COMPLEX`

```yaml
outcome: completed
acceptance_verdict: accepted
final_scope_snapshot:
  version: 1
  scope_summary: "比较 3 种本地向量数据库方案并给出建议。"

    stage_path: ["clarify", "source_collection", "synthesis", "review", "deliver"]
    started_at: "2026-03-10T11:30:00+08:00"
    finished_at: "2026-03-10T12:05:00+08:00"
    summary: "worker 完成来源搜集与综合，recorder 形成正式简报证据。"
  evidence_refs:
    - "artifact://research/source-comparison-notes"
    - "event://review_result/rvw-research-001"

    follow_up_required: false
  artifact_manifest:
    - artifact_name: "Research summary"
      artifact_type: "report"
    - artifact_name: "Source comparison notes"
      artifact_type: "evidence_summary"
  acceptance_basis:
    - basis_kind: requirements_satisfied

      passed: true
    - basis_kind: verification_passed
```

### 7.4 宿主用户可见文本摘要轮廓
1. 调研已完成
2. 已形成比较结论和推荐方案
3. review 通过
4. 可查看来源与比较摘要

### 7.5 为什么这么判定
关键字段：
1. `proof_of_work.artifact_manifest`
   - 明确报告与证据摘要是正式交付物
2. `proof_of_work.review_summary`
   - 证明 research 不是“搜了一堆东西”，而是经过 review 的结论

## 8. 我的建议
1. 这 5 个例子不是演示文案，而是后续代码 spike 的渲染和测试夹具参考
2. `coding_pack / COMPLEX` 要先写透，因为它会直接约束第一条真实代码 spike
3. `research_pack / COMPLEX` 作为第二个正式对照样本，能防止 Loom 的结果链只适配编码任务
4. `proof_of_work` 是结果链正式 section，不再是 `review_summary + artifact_manifest` 的松散组合
