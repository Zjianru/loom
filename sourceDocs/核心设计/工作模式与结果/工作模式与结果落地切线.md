# 工作模式与结果落地切线

状态：implementation cutline draft  
定位：preset / result example 对代码 spike 的约束边界  
更新时间：2026-03-10

---

## 1. 目的
这份文档专门回答一个问题：

**新增的 pack preset 和 result example，后面到底怎样约束 `Harness` 与 `loom-openclaw` 的真实代码 spike。**

如果不把这条 cutline 写清，代码实现时最容易出现两种回退：
1. `Harness` 在代码里硬编码默认阶段和默认角色
2. `loom-openclaw` 在代码里临时拼接结果文本结构
3. 实现层重新把 `default_spec_policy_ref / default_proof_of_work_policy_ref` 当成自由 registry lookup

---

## 2. 关键边界
### 2.1 `PackContract` 不是默认实现
`PackContract`
- 表示 pack 这个对象的通用 schema

它回答：
1. 一个 pack 至少应该有哪类字段

它不回答：
1. `coding_pack` 默认到底怎么跑
2. `research_pack` 默认到底怎么跑

这些必须由 preset 承担。

### 2.2 `PackPreset` 才是默认产品形状
`编码工作模式预设` 和 `研究工作模式预设` 代表：
1. 默认阶段骨架
2. 默认角色骨架
3. 默认 acceptance / budget / WIP / delegation 引用
4. 默认 `SpecBundle / ProofOfWorkBundle` 形态
5. override 开放边界

这里的取舍固定为：
1. `default_spec_policy_ref`
   - 在 v0 只按稳定模板 id 解释
2. `default_proof_of_work_policy_ref`
   - 在 v0 只按稳定模板 id 解释
3. 实现层不得自己再创造第二套 policy registry

所以对实现层来说：
1. preset 是默认装配规则来源
2. 不是可以自由替换的说明文档

### 2.3 `ResultContract` 不是渲染模板
`ResultContract`
- 定义结果包字段

它回答：
1. 结果包最少包含哪些正式对象

它不回答：
1. coding / research 场景里这些字段实际怎么组合出现
2. 用户最终看到的摘要轮廓长什么样

这些必须由 `结果样例` 约束。

### 2.4 `ResultExample` 不是附录
`结果样例`
- 代表正式实例集

它回答：
1. 后续代码 spike 至少要产出到什么程度
2. adapter 文本渲染最低要覆盖哪些字段

---

## 3. 对 `Harness` 的明确约束
### 3.1 不能在代码里内嵌默认阶段骨架
规则：
1. `Harness` 不能在实现层自己发明：
   - coding 默认阶段
   - research 默认阶段
2. 这些默认骨架必须来自对应 pack preset

变量解释：
1. `default_stage_sequence`
   - 表示默认如何生成 `PhasePlan`
2. `PhasePlan`
   - 表示这次任务最终采用的阶段方案

取舍：
1. `PhasePlan` 允许被 change / boundary / system insert 修改
2. 但初始骨架必须来自 preset，而不是来自代码里隐藏常量

### 3.2 不能在代码里偷换默认角色骨架
规则：
1. `default_roster_policy` 必须来自 pack preset
2. `Harness` 只能把它解析成具体 `AgentBinding`
3. 不允许为了 demo 省事而偷偷删掉 recorder 或 review
4. 不允许在代码里自己发明 `SpecBundle` 结构或 `ProofOfWorkBundle` 最小形状

### 3.3 第一条真实 spike 的唯一默认 pack
规则：
1. `coding_pack` 是第一条 `COMPLEX` 代码 spike 的唯一默认 pack
2. `research_pack` 是第二个正式对照样本
3. 它不先进入第一条代码 spike 主闭环

原因：
1. 先把 coding 场景跑通，最容易验证：
   - 委派
   - approval
   - review
   - result chain

---

## 4. 对 `loom-openclaw` 的明确约束
### 4.1 不能临时发明结果文本结构
规则：
1. `loom-openclaw` 只能把 `ResultContract` 渲染成文本
2. 文本结构必须以 `结果样例` 为依据
3. 不允许：
   - 省略 `proof_of_work.review_summary`
   - 省略 `proof_of_work.artifact_manifest`
   - 省略 `proof_of_work.acceptance_basis`
   - 省略 `scope_revision_summary`
   只输出一个随意 summary

### 4.2 `review` 阶段不能只是流程占位
规则：
1. `review` 阶段至少要按：
   - `ReviewResult`
   - `ReviewSummary`
   的最小形状落地
2. `ResultSummaryPayload` 必须能引用 `ProofOfWorkBundle` 的正式摘录，而不是临时口头说“review 通过了”

### 4.3 start/result 渲染不能反向改 contract
规则：
1. adapter 的文本化只解决展示
2. 不能因为 UI 想简化，就反向把 contract 中关键字段删掉或悬空

---

## 5. 代码 spike 的最小落地顺序
### 5.1 先吃 preset
建议顺序：
1. `编码工作模式预设`
2. `结果样例`
3. `规格包合同`
4. `工作证明合同`
5. 再进 `loom-openclaw` 代码 spike

### 5.2 第一条代码闭环
建议固定：
1. `pack_ref = coding_pack`
2. `managedTaskClass = COMPLEX`
3. `default_stage_sequence = clarify -> execute -> review -> deliver`
4. 默认角色骨架：
   - `net`
   - `recorder`
   - `worker`

### 5.3 第二个对照样本
在第一条闭环之后，再用：
1. `research_pack / COMPLEX`

验证：
1. Loom 不是只会做编码类 pack
2. 结果包和 review 语义不是 coding 私有

---

## 6. 剩余仍然没有进入这轮实现 cutline 的内容
这轮明确不要求：
1. `operations_pack`
2. `HUGE / MAX` 的完整 preset
3. pack marketplace
4. 自定义 UI 卡片样式

原因：
1. 这些会扩大表面覆盖面
2. 但不会比 preset/example 更直接降低第一条代码 spike 的实现歧义

---

## 7. 我的建议
1. preset 和 result example 这两类文档，本质上是在替代码实现拿回默认行为的 owner
2. 如果不先写这层，`Harness` 和 `loom-openclaw` 一定会在实现时各自长出一套“方便的默认值”
3. 第一条真实代码 spike 应严格以 `coding_pack / COMPLEX` 为唯一默认主线，不再继续发散场景
