# 风险评估合同

状态：contract draft  
定位：`RiskAssessment` 正式规格  
更新时间：2026-03-11

---

## 1. 目的
`RiskAssessment` 定义 Loom 对任务或动作风险做出的正式评估结果。

它回答：
1. 当前风险总体落在哪个正式档位
2. 风险到底来自哪些维度
3. 这次风险评估评的是任务基线，还是某个动作覆盖风险
4. 风险结论会怎样影响治理模块挂载、执行授权和审批路径

更深层上，它解决的是：

**让“这次为什么要收紧治理”不再停留在口头判断，而是进入 Loom 正式评估对象族。**

---

## 2. 设计边界
### 2.1 这不是宿主语义判断
边界：
1. `HostSemanticBundle`
   - 可以提供风险 hint
2. `RiskAssessment`
   - 持有 Loom 最终消费的正式风险评估结果

固定规则：
1. 宿主 hint 不得直接替代内核风险结论
2. Loom 可以基于 pack、scope、动作类型和既有证据重算最终风险

### 2.2 这不是 `ChangeImpactAssessment`
边界：
1. `ChangeImpactAssessment`
   - 回答一次 change 会影响哪些治理对象
2. `RiskAssessment`
   - 回答当前风险有多高，以及会触发什么风险后果

### 2.3 这不是 `CapabilityDriftAssessment`
边界：
1. `CapabilityDriftAssessment`
   - 回答宿主现实能力变化后怎样重授权
2. `RiskAssessment`
   - 回答当前任务或动作的风险级别

### 2.4 这不是 `ExecutionAuthorization`
边界：
1. `RiskAssessment`
   - 回答为什么需要收紧治理
2. `ExecutionAuthorization`
   - 回答当前 run 最终被发放了什么 lease

### 2.5 这不是独立 `risk_profile` aggregate
边界：
1. `risk_profile`
   - 只作为 `RiskAssessment` 的结果视角与读取口径存在
2. `RiskAssessment`
   - 才是风险正式 owner

---

## 3. v0 固定规则
1. 每个进入 `execute` 的 managed task，至少要形成一份 `TaskBaseline` 风险评估。
2. 高风险动作、review 升级或范围发生重大变化时，可以形成新的 `ActionOverride` 风险评估。
3. `TaskBaseline` 与 `ActionOverride` 固定共存于同一风险对象族里，后者不得替代前者作为任务基线 owner。
4. 总体风险档位第一版固定为 `Low / Medium / High / Critical`。
5. `critical` 风险不得继续 silent execution。
6. 用户可以显式上调风险，但不能把风险压到系统底线以下。
7. pack 可以给出默认风险地板或推荐风险档位，但 runtime owner 仍然是 `RiskAssessment`。

### 3.1 第一层当前推进状态
当前阶段固定为：
1. 第一层风险治理
   - 正式推进
2. 当前实现主线
   - 直接消费 `RiskAssessment`
3. 这轮不做
   - 独立 `risk_profile` aggregate
   - 平行 `PolicyEvaluationRecord / DecisionBasis`

## 4. 正式对象定义
```rust
pub struct RiskAssessment {
    pub assessment_id: RiskAssessmentId,
    pub managed_task_ref: ManagedTaskRef,
    pub subject_kind: RiskSubjectKind,
    pub capability_snapshot_ref: Option<HostCapabilitySnapshotRef>,
    pub stage_run_ref: Option<StageRunRef>,
    pub decision_area: Option<DecisionArea>,
    pub overall_risk_band: RiskBand,
    pub risk_dimensions: RiskDimensions,
    pub trigger_reason: String,
    pub evidence_refs: Vec<EvidenceRef>,
    pub derived_consequences: Vec<RiskConsequence>,
    pub assessed_at: Timestamp,
    pub supersedes: Option<RiskAssessmentRef>,
}

pub enum RiskSubjectKind {
    TaskBaseline,
    ActionOverride,
}

pub enum RiskBand {
    Low,
    Medium,
    High,
    Critical,
}

pub struct RiskDimensions {
    pub write_impact: RiskBand,
    pub external_side_effect: RiskBand,
    pub secret_exposure: RiskBand,
    pub irreversibility: RiskBand,
    pub budget_burn: RiskBand,
    pub uncertainty: RiskBand,
}

pub enum RiskConsequence {
    MountAdditionalGovernance,
    NarrowExecutionAuthorization,
    RequireApprovalWindow,
    RequireAdditionalReview,
    StopSilentExecution,
}
```

---

## 5. 关键字段分别代表什么
1. `subject_kind`
   - 这次评估评的是任务基线风险，还是某个动作覆盖风险
2. `stage_run_ref`
   - 如果风险和某个阶段执行面强相关，就绑定当前 stage run
3. `capability_snapshot_ref`
   - 如果这次风险评估显式读取了当前宿主现实能力，就保留该引用，避免 drift 后丢失因果链
4. `decision_area`
   - 如果风险是围绕某个具体决策域生成的动作覆盖风险，就显式绑定该域
5. `overall_risk_band`
   - Loom 当前消费的总体风险档位
6. `risk_dimensions`
   - 风险分别来自哪些维度
7. `trigger_reason`
   - 为什么这次要重评风险
8. `derived_consequences`
   - 这次评估会直接推动哪些治理动作
9. `supersedes`
   - 这次评估是否替换了旧评估

## 6. 和其它对象的关系
### 6.1 和 `PackContract`
固定关系：
1. pack 可以给默认风险地板或推荐风险档位
2. 但 runtime 真正生效的风险结论仍由 `RiskAssessment` 持有

### 6.2 和 `TaskScopeSnapshot`
固定关系：
1. 任务基线风险必须读取当前有效 `TaskScopeSnapshot`
2. engineering context、范围约束和 secret class 都会影响风险结论

### 6.3 和 `ExecutionAuthorization`
固定关系：
1. `RiskAssessment` 可以要求收紧授权
2. 但它不直接替代 `ExecutionAuthorization`
3. 当前 run 最终被租给什么，仍由授权合同持有 owner

### 6.4 和 `PendingDecisionWindow`
固定关系：
1. 如果某次风险结论要求用户拍板
2. Loom 必须显式打开对应 `PendingDecisionWindow`
3. 不能把 `critical` 风险只写在日志里继续跑

### 6.5 和 `ChangeImpactAssessment`
固定关系：
1. 结构性 change 可能触发新的风险评估
2. 但变更影响和风险评估仍然是两份不同的 assessment

### 6.6 和 `CapabilityDriftAssessment`
固定关系：
1. drift assessment 可以触发新的风险评估
2. 例如宿主失去关键控制能力后，任务总体风险会被上调
3. 如果当前风险结论依赖某版 `HostCapabilitySnapshot`
4. 建议把该引用保留到 `capability_snapshot_ref`

---

## 7. 我的建议
1. 先把风险建模收敛到 `Assessment` 家族，不另开记录对象族。
2. 第一版只固定：
   - `TaskBaseline`
   - `ActionOverride`
3. 第一版风险维度固定为：
   - `write_impact`
   - `external_side_effect`
   - `secret_exposure`
   - `irreversibility`
   - `budget_burn`
   - `uncertainty`
4. `critical` 风险必须显式阻断 silent execution。

## 8. 第一层触发矩阵
### 8.1 `TaskBaseline`
必须生成于：
1. `approve_start` 后，`TaskScopeSnapshot(scope_version=1)` 冻结完成时
2. 任意正式接受、且导致 `scope_version` 递增的范围变更后
3. 改变总体风险的 capability drift 之后

固定约束：
1. 首份 `TaskBaseline` 必须早于首份 `ExecutionAuthorization`
2. 新 baseline 必须通过 `supersedes` 指向旧 baseline

### 8.2 `ActionOverride`
可以生成于：
1. 高风险工具或动作提案
2. 外部副作用、secret 使用、不可逆动作、预算超阈值动作
3. review 升级到 `escalate_to_user` 或等价审批链
4. 用户显式上调风险

固定约束：
1. `ActionOverride` 只绑定当前动作或当前审批区间
2. 它不替代 `TaskBaseline`
3. 它可以要求：
   - `NarrowExecutionAuthorization`
   - `RequireApprovalWindow`
   - `RequireAdditionalReview`
   - `StopSilentExecution`

## 9. 结构化日志义务
第一层至少必须发出：
1. `risk_assessment.created`
2. `risk_assessment.superseded`
3. `execution_authorization.issued`
4. `execution_authorization.narrowed`
5. `execution_authorization.reissued`

每条日志至少带：
1. `managed_task_ref`
2. `assessment_id` 或 `authorization_id`
3. `overall_risk_band`
4. `scope_version`
5. `derived_consequences`
6. `supersedes`
7. `capability_snapshot_ref`

## 10. WebUI 用户可见义务
1. 普通 start 不暴露底层对象名
2. 一旦发生 `ActionOverride` 或 `critical`，用户必须看到风险升级原因与后续动作
3. 范围变化后，用户必须看到“范围更新 -> 风险重评 -> 授权重发/收紧”的连续反馈

## 11. 非目标
1. 不把 `RiskAssessment` 做成第二套主状态机
2. 不引入 claim 级风险账本
3. 不把宿主 risk hint 直接当成最终风险 owner
