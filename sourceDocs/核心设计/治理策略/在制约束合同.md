# 在制约束合同

状态：contract draft  
定位：`WipPolicy` 正式规格  
更新时间：2026-03-10

---

## 1. 目的
`WipPolicy` 是 Loom 的流动治理对象。

它回答：
1. 同时允许多少活跃工作
2. 什么时候允许新的工作进入
3. 工作卡住多久算异常
4. 异常到什么程度必须升级

更深层上，它解决的是：

**让系统不是“能跑多少跑多少”，而是按容量、老化和阻塞来受控推进。**

---

## 2. v0 固定规则
### 2.1 同时只允许一个 `activeManagedTaskRef`
这次已锁定：
1. 同一时刻只允许一个 active managed task

这条规则是：
1. 产品治理原则
2. 也是流动治理原则

### 2.2 一个 active task 内只允许一个 active stage
这次已锁定：
1. v0 阶段串行
2. 因此一个 active task 里也只允许一个 active stage

### 2.3 一个 stage 内允许多个 agent 参与，但不引入复杂并行语义
这次已锁定：
1. 一个 stage 内可以有多个 agent 参与
2. 但 v0 不把它建模成复杂并行 stage graph

更深层取舍：
1. 允许多人协作
2. 不允许 runtime 先变成并行调度器

### 2.4 新 managed task 候选一律先走边界确认
这次已锁定：
1. 当已有 active managed task，又来了新 managed task 候选时
2. v0 不静默排队
3. 也不直接拒绝
4. 一律先走 `pendingBoundaryConfirmation`

### 2.5 WIP 不只管数量，也管老化
这次已锁定：
1. `WipPolicy` 不只是数量限制
2. 还要正式管 aging

原因：
1. 一个任务虽然只有 1 个，但可能已经卡死很久

## 3. 正式对象定义
### 3.1 Rust 形态建议
```rust
pub struct WipPolicy {
    pub policy_id: WipPolicyId,
    pub max_active_tasks: u8,
    pub max_active_stages_per_task: u8,
    pub max_active_agents_per_stage: u8,
    pub replenishment_policy: ReplenishmentPolicy,
    pub aging_policy: AgingPolicy,
    pub escalation_policy: WipEscalationPolicy,
}
```

### 3.2 相关对象建议
```rust
pub enum WorkItemState {

}
```

## 4. 字段 contract
### 4.1 `max_active_tasks`
它是什么：
1. 同时允许的 active managed task 数量

v0 固定值：
1. `1`

### 4.2 `max_active_stages_per_task`
它是什么：
1. 单个 task 同时允许多少 active stage

v0 固定值：
1. `1`

### 4.3 `max_active_agents_per_stage`
它是什么：
1. 单个 stage 同时允许多少 agent 参与

注意：
1. 这不是并行 stage 数量
2. 它只是当前阶段的人手上限

### 4.4 `replenishment_policy`
它是什么：
1. 新工作何时允许进入 managed lane

v0 建议规则：
1. active task 存在时，新的 managed task 候选先触发边界确认
2. 不做静默排队

### 4.5 `aging_policy`
它是什么：
1. 工作多久无推进视为 aging

建议最小字段：
1. `warn_after`
2. `escalate_after`
3. `block_after`

### 4.6 `escalation_policy`
它是什么：
1. 哪些 WIP 异常需要 `watchdog` 升级通知

典型触发：
1. 长时间无推进
2. review 多轮打回
3. 等待用户确认过久
4. active task 超预算

---

## 5. `watchdog` 与 `WipPolicy` 的关系
`watchdog` 不应该只是通知器。  
它应该是：

**流动治理观察者。**

`WipPolicy` 就是它最重要的语义输入之一。

`watchdog` 至少要读取：
1. `WorkItemState`
2. `aging_policy`
3. `escalation_policy`
4. `replenishment_policy`

---

## 6. 与其它对象的边界
### 6.1 和 `PhasePlan`
`PhasePlan` 解决：
1. 阶段顺序与回退

`WipPolicy` 解决：
1. 当前系统允许多少工作同时活跃、多久算卡住

### 6.2 和 `AcceptancePolicy`
`AcceptancePolicy` 解决：
1. 什么算通过

`WipPolicy` 解决：
1. 什么算拥堵、阻塞、老化

### 6.3 和 `pendingBoundaryConfirmation`
`pendingBoundaryConfirmation` 是状态对象。  
`WipPolicy` 是触发它的治理规则之一。

---

## 7. 我的建议
`WipPolicy` 在你的系统里非常重要，因为你强调的是：
1. 扔出去就继续做别的事
2. 但不能放任系统任务泛滥

我的建议：
1. v0 就把 `max_active_tasks = 1` 写成正式规则
2. 把 aging 和 escalation 一起写成强 contract
3. 把 `watchdog` 做成真正的 flow governor，而不是纯通知器
