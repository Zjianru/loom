# 委派级别合同

状态：contract draft  
定位：`DelegationLevel` 正式规格  
更新时间：2026-03-10

---

## 1. 目的
`DelegationLevel` 是 Loom 的授权语义对象。

它回答：
1. 某类动作系统能不能自己做
2. 是直接做，先建议，还是必须等用户确认
3. 哪些动作永远禁止

更深层上，它解决的是：

**把“允许/拒绝”升级成更细粒度、可解释的授权层级。**

---

## 2. 来源与边界
这个对象主要吸收自 Management 3.0 的 Delegation Poker / Delegation Board。  
它不是最终运行时授权结果，而是：

1. 授权语义
2. 决策区的默认授权层

边界：
1. `DelegationLevel`
   - 描述授权层级
2. `ExecutionAuthorization`
   - 描述本次运行的实际授权结果
3. `approval-gate`
   - 负责最终执法

---

## 3. v0 取舍
### 3.1 内核保留完整语义，产品层用压缩表达
这次已锁定：
1. 内核层可以保留完整 7 级授权思想
2. 产品层 v0 先用压缩版表达

原因：
1. 完整语义有利于内核长期演进
2. 压缩表达更利于普通用户理解

### 3.2 任务级默认，阶段级可收紧
这次已锁定：
1. `DelegationLevel` 默认任务级
2. 阶段级可以局部收紧

原因：
1. 整个任务先有一个授权基线
2. 某些高风险阶段要更保守

### 3.3 用户可以调，但不能绕过底线
这次已锁定：
1. 用户可以在安全边界内调整 delegation
2. 不能绕过系统底线

## 4. 正式对象定义
### 4.1 内核层建议值
```rust
pub enum DelegationLevel {
    Tell,
    Sell,
    Consult,
    Agree,
    Advise,
    Inquire,
    Delegate,
}
```

### 4.2 产品层压缩表达建议
```rust
pub enum DelegationBand {
    SystemCanDo,
    SystemCanPropose,
    RequiresUserApproval,
    Forbidden,
}
```

取舍：
1. 内核层保留细语义
2. 用户层先看简单表达

### 4.3 `DecisionArea`
建议至少有这些区：
1. `read_local`
2. `write_local`
3. `spawn_agent`
4. `modify_phase_plan`
5. `call_external_service`
6. `publish_or_send`
7. `close_task`

### 4.4 Rust 结构建议
```rust
pub struct DelegationMatrix {
    pub scope: DelegationScope,
    pub rules: Vec<DelegationRule>,
}

pub struct DelegationRule {
    pub decision_area: DecisionArea,
    pub level: DelegationLevel,
    pub band: DelegationBand,
    pub override_allowed: bool,
}
```

---

## 5. 字段 contract
### 5.1 `scope`
它是什么：
1. 授权矩阵作用于任务还是阶段

建议值：
1. `task`
2. `stage`

### 5.2 `decision_area`
它是什么：
1. 当前授权针对哪类决策区

这非常关键，因为：
1. 不同动作风险不同
2. 不能用一个总开关统一覆盖所有行为

### 5.3 `level`
它是什么：
1. 内核层完整授权语义

### 5.4 `band`
它是什么：
1. 对用户和高层治理更易理解的压缩表达

### 5.5 `override_allowed`
它是什么：
1. 用户是否可以在安全边界内调整该规则

---

## 6. 为什么不能直接只用布尔值
如果只有：
1. allow
2. deny

会丢掉三个关键语义：
1. 系统能不能先建议
2. 系统能不能自主决定但必须回报
3. 系统是不是必须先征求同意

而这些正是你的产品里最重要的治理边界。

---

## 7. 与其它对象的边界
### 7.1 和 `ExecutionAuthorization`
`DelegationLevel` 解决：
1. 抽象语义和默认授权层

`ExecutionAuthorization` 解决：
1. 这次运行时最终允许什么

### 7.2 和 `approval-gate`
`DelegationLevel` 不是执法器。  
`approval-gate` 才是。

也就是说：
1. `DelegationLevel` 提供规则语义
2. `approval-gate` 根据当前 `ExecutionAuthorization` 真正放行或阻断

### 7.3 和 `WorkHorizon`
`WorkHorizon` 会影响默认 delegation 倾向。  
例如：
1. `maintenance`
   - 默认更保守
2. `disruption`
   - 可以更探索，但必须配合更强 kill-switch

---

## 8. 我的建议
`DelegationLevel` 是 `approval-gate` 从“硬拦截器”升级成“可解释治理器”的关键。

我的建议：
1. v0 就把 `DecisionArea` 明确下来
2. 内核保留 7 级语义
3. 用户层先用压缩 band
4. 默认任务级，阶段级允许收紧
