# 运行中断与取消合同

状态：contract draft  
定位：active managed task 下 worker 中断、暂停、恢复、取消的正式规格  
更新时间：2026-03-10

---

## 1. 目的
这份 contract 定义 active managed task 运行期间，Loom 如何正式处理中断和取消。

它回答：
1. `pause_task / resume_task / cancel_task` 对 worker 到底意味着什么
2. 这些动作如何影响 `ExecutionAuthorization`
3. 宿主不支持硬中断时，系统该怎样降级
4. 哪些事件、状态和结果必须留痕

更深层上，它解决的是：

**让控制面不只是“收到动作”，而是真能影响正在跑的 worker 与阶段推进。**

---

## 2. 为什么必须单独成文
如果只定义控制动作，不定义执行面后果，会出现 4 个问题：
1. `pause_task` 只是 UI 上暂停，worker 实际还在跑
2. `cancel_task` 发出后，旧 `ExecutionAuthorization` 仍然能继续写
3. 宿主不支持中断时，系统不知道该回什么状态
4. `ResultContract` 和 `TaskEventModel` 无法说明任务是怎样被暂停或取消的

它直接涉及这些对象：
1. `ExecutionAuthorization`
   - 表示本轮 worker 真正拿到的能力租约
2. `AgentBinding`
   - 表示当前 task/stage 实际绑定了哪些 agent
3. `PhasePlan`
   - 表示任务阶段蓝图
4. `ControlAction`
   - 表示用户正式治理动作

## 3. 设计边界
### 3.1 这不是宿主语义判断
宿主语义层负责：
1. 把用户输入结构化成 `pause_task / resume_task / cancel_task`

这份 contract 负责：
1. Loom 收到这些动作后，怎样影响 worker 与阶段

### 3.2 这不是 host process kill API
这份 contract 不假设宿主一定支持：
1. 强制 kill 子进程
2. 强制暂停运行中的 agent

它只要求：
1. adapter 必须声明真实 `worker_control_capabilities`
2. Loom 必须按宿主真实能力决定是 cooperative interrupt、best-effort cancel，还是 fail closed

### 3.3 这不是 `AcceptancePolicy`
`AcceptancePolicy`
1. 回答“完成是否合格”

这份 contract
1. 回答“运行中的 worker 如何被停下、挂起、恢复或结束”

---

## 4. v0 固定规则
### 4.1 `pause_task` 必须阻止新工作继续进入执行面
一旦 `pause_task` 被接受：
1. 当前 stage 不得继续分派新的 worker work item
2. 不得继续发放新的 `ExecutionAuthorization`
3. 正在运行的 worker 必须进入 pause/interruption 路径

### 4.2 `cancel_task` 必须收回执行授权
一旦 `cancel_task` 被接受：
1. 当前 task 的 active execution lease 必须被 revoke
2. 不得继续允许新的工具调用和写操作
3. 当前 worker 必须进入 cancel/interruption 路径

### 4.3 `resume_task` 不是恢复旧 lease，而是重新进入受控执行
固定规则：
1. `resume_task` 只能作用于 paused task
2. 恢复后可以：
   - 继续当前 stage
   - 或重新生成新的 `ExecutionAuthorization`
3. 不允许假定旧 lease 仍然有效

### 4.4 v0 先采用 cooperative-first
我的建议：
1. 先尝试 cooperative pause/cancel
2. 如果宿主支持 hard interrupt，再作为更强能力使用
3. 宿主不支持时，系统必须显式暴露“requested but pending”

原因：
1. OpenClaw 和未来宿主未必都给强 kill 能力
2. 如果现在把强 kill 写成默认，contract 会先脱离现实

## 5. 正式对象定义
### 5.1 Rust 形态建议
```rust
pub struct WorkerRunRef {
    pub worker_run_id: WorkerRunId,
    pub managed_task_ref: ManagedTaskRef,
    pub stage_run_ref: StageRunRef,
    pub agent_ref: AgentRef,
}

pub enum WorkerRunStatus {
    Pending,
    Running,
    Pausing,
    Paused,
    Cancelling,
    Cancelled,
    Completed,
    Failed,
}

pub enum WorkerInterruptionKind {
    Pause,
    Resume,
    Cancel,
}

pub enum WorkerInterruptionOutcome {
    Accepted,
    InProgress,
    Completed,
    NotSupported,
    TimedOut,
    Failed,
}

pub struct WorkerInterruptionRequest {
    pub request_id: WorkerInterruptionRequestId,
    pub managed_task_ref: ManagedTaskRef,
    pub worker_run_ref: WorkerRunRef,
    pub source_action_id: ControlActionId,
    pub kind: WorkerInterruptionKind,
    pub issued_at: Timestamp,
    pub reason: String,
}

pub struct WorkerInterruptionResult {
    pub result_id: WorkerInterruptionResultId,
    pub request_id: WorkerInterruptionRequestId,
    pub worker_run_ref: WorkerRunRef,
    pub outcome: WorkerInterruptionOutcome,
    pub resulting_status: WorkerRunStatus,
    pub observed_at: Timestamp,
    pub detail: Option<String>,
}
```

### 5.2 关键变量分别代表什么
1. `WorkerRunRef`
   - 当前被控制的是哪个具体 worker run
   - 它解决“暂停/取消到底打到谁”
2. `WorkerRunStatus`
   - 当前 worker run 处于什么执行状态
   - 它解决“请求已经发出，还是已经停下”
3. `WorkerInterruptionRequest`
   - 一次正式的暂停/恢复/取消请求
   - 它解决“用户动作如何进入执行面”
4. `WorkerInterruptionResult`
   - 该请求最终观察到的结果
   - 它解决“宿主支持不足或失败时，Loom 如何留痕”

## 6. Host 能力映射
### 6.1 `HostWorkerControlCapabilities`
正式字段 contract 统一以：
1. [宿主能力快照合同.md](../宿主接入/宿主能力快照合同.md)
   为准。

### 6.2 它和 `HostCapabilitySnapshot` 的关系
`HostCapabilitySnapshot`
1. 回答“这个宿主总体有什么能力”

`HostWorkerControlCapabilities`
1. 回答“正在跑的 worker 到底能不能被停、被挂起、被恢复”

这层不显式建模，`pause_task / cancel_task` 就会停留在口头能力。

## 7. 动作语义
### 7.1 `pause_task`
建议语义：
1. 接受 `ControlAction::PauseTask`
2. task 进入 paused-pending 或 paused
3. 当前 stage 停止派发新工作
4. 对 active worker 发出 `WorkerInterruptionRequest(kind=Pause)`
5. 将对应 `ExecutionAuthorization` 标记为 suspended

如果宿主不支持 pause：
1. 返回 `NotSupported`
2. 不得伪装成已暂停
3. 由 Loom 决定是否升级成 `cancel_task` 建议或用户告警

### 7.2 `resume_task`
建议语义：
1. 只允许作用于 paused task
2. 重新评估当前 `HostCapabilitySnapshot`
3. 重新发放新的 `ExecutionAuthorization`
4. 允许：
   - 恢复当前 worker
   - 或重建 worker run

取舍：
1. v0 不要求宿主一定支持“恢复同一运行单元”
2. 但必须支持“恢复同一任务/阶段治理”

### 7.3 `cancel_task`
建议语义：
1. 接受 `ControlAction::CancelTask`
2. 当前 task 进入 cancelling 或 cancelled
3. 立刻 revoke 当前 `ExecutionAuthorization`
4. 对所有 active worker 发出 `WorkerInterruptionRequest(kind=Cancel)`
5. 不再允许任何新 work item 进入执行面

如果宿主只支持 soft interrupt：
1. 状态应先进入 `Cancelling`
2. 等待 worker 自行收束
3. 超时后记为 `TimedOut` 或 `Failed`

## 8. 与其它对象的边界
### 8.1 和 `ExecutionAuthorization`
这份 contract 最关键的一条边界是：
1. `pause_task`
   - suspend active authorization
2. `cancel_task`
   - revoke active authorization
3. `resume_task`
   - 重新建立新的 authorization

如果不这样切：
1. worker 停了，写权限却还活着
2. 或 task 恢复了，却还拿着旧 lease 继续跑

### 8.2 和 `PhasePlan`
`PhasePlan`
1. 回答任务阶段怎么走

这份 contract
1. 回答当前 stage 里的 worker 如何被停住或结束

`cancel_task`
1. 可以终止整个 plan
`pause_task`
1. 只冻结推进，不改写历史 plan

### 8.3 和 `TaskEventModel`
最少要产生这些事件：
1. `control_action_received`
2. `worker_interruption_requested`
3. `worker_interruption_completed`
4. `execution_authorization_suspended`
5. `execution_authorization_revoked`
6. `task_paused`
7. `task_resumed`
8. `task_cancelled`

### 8.4 和 `ResultContract`
如果任务因 pause/cancel 终止或部分完成：
1. `ResultContract.outcome`
   - 必须能解释这次 pause/cancel 的收口结果

### 8.5 和 capability drift
如果宿主能力变化导致：
1. 当前 `ExecutionAuthorization` 不再安全或不再可执行
2. `HostWorkerControlCapabilities` 发生关键变化

那么这份 contract 的执行路径必须服从：
1. [能力漂移与重授权合同.md](能力漂移与重授权合同.md)

也就是说：
1. worker interruption 负责“怎么停”
2. capability drift 负责“为什么现在必须停、缩权或重发 lease”
   - 可能是 `partial / cancelled / blocked`
2. 结果包必须说明：
   - 哪些工作已完成
   - 哪些 work item 因中断未完成

## 9. FailurePolicy
### 9.1 宿主不支持中断
固定规则：
1. 不得把“不支持”伪装成“成功”
2. 必须把结果写成：
   - `NotSupported`
   - 或 `Failed`
3. 必须升级给 `watchdog` 和用户摘要

### 9.2 worker 无响应
建议：
1. 先 cooperative interrupt
2. 等待 bounded timeout
3. 超时后把结果写成 `TimedOut`
4. 由 Loom 决定：
   - 保持 cancelling
   - 或终止并给出 partial result

### 9.3 bridge 或宿主重启
建议：
1. 恢复时重新读取 active `WorkerRunRef`
2. 重新同步 `HostCapabilitySnapshot`
3. 不假定旧 worker 仍可恢复
4. 必要时进入 `request_review` 或 `stop_and_report`

---

## 10. 我的建议
### 10.1 v0 最稳的取舍
我建议 v0 固定成：
1. 控制动作先进 Loom
2. Loom 再发 worker interruption request
3. pause = suspend authorization
4. cancel = revoke authorization
5. resume = 重新发 authorization，而不是复活旧 lease

### 10.2 为什么现在就要补
因为不补这份 contract，会出现一个很典型的伪完成：
1. 控制面看上去已经有 `pause/cancel`
2. 但真实 worker 仍在跑
3. 最后用户对系统治理能力的信任会直接坍塌

### 10.3 我的建议
1. 这份 contract 先把执行面边界定住
2. 真正写代码时再根据 OpenClaw 实际可中断能力做最小实现
3. 先不要承诺超出宿主现实的 hard kill 能力
