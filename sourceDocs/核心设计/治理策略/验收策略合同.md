# 验收策略合同

状态：contract draft  
定位：`AcceptancePolicy` 正式规格  
更新时间：2026-03-10

---

## 1. 目的
`AcceptancePolicy` 是 Loom 中最关键的治理对象之一。

它回答：
1. 什么叫完成
2. 谁有资格验收
3. 失败后怎么返工
4. 最多返工几轮
5. 什么情况下必须升级给用户

更深层上，它解决的是：

**把“我感觉差不多好了”升级成结构化的完成判定。**

---

## 2. 设计边界
### 2.1 `AcceptancePolicy` 和 `approval-gate` 不是一回事
边界必须先说清：
1. `approval-gate`
   - 回答“能不能做”
2. `AcceptancePolicy`
   - 回答“算不算完成”

如果不拆开，会出现两个冲突：
1. 把写权限误当成完成标准
2. 把完成标准误当成执行授权

### 2.2 `AcceptancePolicy` 不是纯任务级对象
这次已经锁定：
1. 阶段级和任务级验收都要有

所以它必须支持两级作用域：
1. `task`
2. `stage`

### 2.3 所有 managed task 都强制有正式验收
这次也已锁定：
1. `COMPLEX`
2. `HUGE`
3. `MAX`

全部必须有正式验收策略。

取舍：
1. `COMPLEX` 可以轻量
2. `HUGE / MAX` 必须更严格

---

## 3. 核心原则
### 3.1 “完成”是与关系，不是或关系
这次已经明确：
1. 测试结果
2. 需求满足
3. 风险约束
4. review 通过
5. 其他要求

应该是“与”关系，而不是“或”关系。

这意味着：
1. 某项单独通过，不代表整体完成
2. 最终完成是多维约束共同满足

### 3.2 用户可以调权重或阈值
虽然总体是“与”关系，但用户可以调：
1. 某项要求的严格程度
2. 某项要求是否启用
3. 某项要求的阈值

例如：
1. 显式降低合规审查要求
2. 显式提高安全审查要求

这意味着：
1. `AcceptancePolicy` 既是正式规则
2. 也是可配置规则

### 3.3 默认返工上限为 3
这次已锁定：
1. 默认返工上限为 `3`
2. 未来允许用户配置

这很重要，原因是：
1. 没有返工上限，系统会自转
2. 有上限后，任务才有止损和升级机制

## 4. 正式对象定义
### 4.1 Rust 形态建议
```rust
pub struct AcceptancePolicy {
    pub policy_id: AcceptancePolicyId,
    pub scope: AcceptanceScope,
    pub criteria: Vec<AcceptanceCriterion>,
    pub review_group: ReviewGroupPolicy,
    pub max_rework_rounds: u8,
    pub escalation_policy: AcceptanceEscalationPolicy,
    pub override_policy: AcceptanceOverridePolicy,
    pub metadata: AcceptanceMetadata,
}
```

### 4.2 相关对象建议
```rust
pub enum AcceptanceScope {
    Task,
    Stage(StagePackageId),
}

pub enum AcceptanceResult {
    Accepted,
    AcceptedWithNotes,
    ReworkRequired,
    Blocked,
    EscalateToUser,
}
```

这次已锁定：
1. 结果状态先按这组来
2. 后续可以扩，但 v0 先收稳
3. `EscalateToUser`
   - 表示当前判定无法自行收口
   - 必须先打开用户决策窗口
   - 不直接等于最终结果包已经生成

## 5. 字段 contract
### 5.1 `policy_id`
它是什么：
1. 验收策略稳定标识

### 5.2 `scope`
它是什么：
1. 策略作用域

建议值：
1. `task`
2. `stage`

建议规则：
1. 默认存在一个任务级策略
2. 允许某些 `StagePackage` 局部覆盖

这是我建议补的一个小点，现在直接收进 contract：
1. 默认任务级
2. 阶段级允许局部覆盖

### 5.3 `criteria`
它是什么：
1. 验收维度列表

建议最小 criteria：
1. `tests_passed`
2. `requirements_satisfied`
3. `risk_within_threshold`
4. `review_approved`
5. `artifact_complete`

可选扩展：
1. `compliance_passed`
2. `security_review_passed`
3. `documentation_complete`

#### `AcceptanceCriterion`
建议字段：
1. `criterion_id`
2. `enabled`
3. `required`
4. `threshold`
5. `weight_hint`
6. `evidence_sources`

取舍：
1. `required=true` 表示默认必须满足
2. `weight_hint` 允许用户调优 strictness
3. 但整体完成仍然遵守“与关系”

### 5.4 `review_group`
它是什么：
1. 当前策略下负责 review/validate 的验收组

这次已锁定：
1. reviewer 可以池化
2. 甚至可以形成 review group / validate group

但为了 v0 收敛，我的建议是：
1. v0 先统一成 `review_group`
2. 后续再拆成更细的 validate group

建议字段：
1. `min_reviewers`
2. `preferred_roles`
3. `quorum_policy`
4. `disagreement_policy`

默认建议：
1. `COMPLEX`
   - `min_reviewers = 1`
2. `HUGE / MAX`
   - `min_reviewers = 2`

再补一条 review 事实边界：
1. `review_group`
   - 只回答谁来 review、最低人数和裁决规则
2. 单次 review 的正式输出
   - 统一由 [评审结果合同.md](../工作模式与结果/评审结果合同.md)
     定义的 `ReviewResult` 持有

### 5.5 `max_rework_rounds`
它是什么：
1. 最大返工轮次

默认值：
1. `3`

### 5.6 `escalation_policy`
它是什么：
1. 验收失败或冲突时的升级规则

典型触发：
1. reviewer 结论冲突
2. 多轮返工后仍失败
3. 风险高于阈值
4. 测试通过但需求明显不满足

### 5.7 `override_policy`
它是什么：
1. 用户如何调整或覆盖默认验收权重和阈值

它表示：
1. 哪些 criterion 允许用户调
2. 哪些不能被降低到底线以下

更深层的价值：
1. 让用户可以针对不同任务调节严格程度
2. 但同时保留系统底线

## 6. 阶段级与任务级的关系
### 6.1 任务级策略
任务级策略回答：
1. 整个 managed task 什么时候算完成

### 6.2 阶段级策略
阶段级策略回答：
1. 当前阶段什么时候允许收口并进入下一阶段

### 6.3 覆盖规则建议
建议采用：
1. 任务级策略是默认基线
2. 阶段级策略可以追加要求或局部覆盖

不要采用：
1. 每个阶段完全自定义一套无关策略

原因：
1. 否则任务整体的完成语义会碎裂

### 6.4 和 `ReviewResult` 的关系
`AcceptancePolicy` 最容易被误解的一点是：
1. 它定义的是规则
2. 不是 review 本身的产物

所以 v0 固定关系应是：
1. `AcceptancePolicy`
   - 定义 `review_approved` 这种 criterion 如何判断
2. `ReviewResult`
   - 提供 review 这一维的正式事实
3. `AcceptanceResult`
   - 最终再把多维 criterion 聚合成是否完成

## 7. 编码类任务的特殊规则
### 7.1 测试先行必须进入 coding package
你已经明确：
1. 编码类任务要纳入 TDD / 测试先行

因此 coding package 的 `AcceptancePolicy` 必须额外强调：
1. `tests_passed` 是硬门槛
2. `requirements_satisfied` 也是硬门槛
3. `risk_within_threshold` 也是硬门槛
4. `review_approved` 也是硬门槛

也就是说：
1. 不能把“测试通过”当成“已经完成”

### 7.2 `AcceptanceExamples`
建议在 coding package 中补一个对象：
1. `AcceptanceExamples`

它代表：
1. 需求被转成的可验证样例

这能帮助：
1. worker 更准确实现
2. reviewer 更准确判断需求是否被满足

---

## 8. 和其它对象的边界
### 8.1 和 `StagePackage`
`StagePackage` 解决：
1. 当前阶段需不需要 review / gate / output

`AcceptancePolicy` 解决：
1. review / gate / output 满足到什么程度才算通过

### 8.2 和 `watchdog`
`AcceptancePolicy` 解决：
1. 验收失败如何定义

`watchdog` 解决：
1. 验收失败到什么程度需要提醒、何时升级

### 8.3 和 `approval-gate`
`approval-gate` 解决：
1. 能不能写、能不能执行某动作

`AcceptancePolicy` 解决：
1. 事情做完后算不算达到交付标准

---

## 9. 我的建议
`AcceptancePolicy` 最容易被低估，但它其实是你这个产品最核心的护城河之一。

原因：
1. 市面上很多系统擅长“跑起来”
2. 但不擅长“正式收口”

我的建议是：
1. v0 就把它写成强 contract
2. 先明确结果状态、criteria、review_group、rework 上限、用户 override
3. 不要把验收留到“以后再补”
