# 控制动作合同

状态：contract draft  
定位：`ControlAction` 正式规格  
更新时间：2026-03-10

---

## 1. 目的
`ControlAction` 定义用户在任务运行期间可以正式行使哪些治理动作。

它回答：
1. 用户如何启动任务
2. 用户如何修改候选任务
3. 用户如何暂停、恢复、取消 active task
4. 用户如何申请重判、要求 review、处理边界确认

更深层上，它解决的是：

**让“用户退出微操，但保留治理权”变成正式控制面，而不是靠自然语言临时猜。**

---

## 2. 为什么必须有正式控制面
如果没有正式控制面，系统会出现两个极端：
1. 要么什么都自动，用户没有正式治理入口
2. 要么什么都靠自然语言猜，系统不稳定

`ControlAction` 的作用是：
1. 把关键用户治理动作结构化
2. 让 adapter、watchdog、Harness、approval-gate 都能对齐同一套动作语义

---

## 3. v0 正式动作
### 3.1 候选任务动作
1. `approve_start`
2. `modify_candidate`
3. `cancel_candidate`

### 3.2 active task 边界动作
1. `keep_current_task`
2. `replace_active`

### 3.3 审批请求动作
1. `approve_request`
2. `reject_request`

### 3.4 active task 治理动作
1. `pause_task`
2. `resume_task`
3. `cancel_task`
4. `request_review`
5. `request_horizon_reconsideration`
6. `request_task_change`

### 3.5 token 约束
以下动作在 v0 必须带有效 `decision_token`：
1. `approve_start`
2. `modify_candidate`
3. `cancel_candidate`
4. `keep_current_task`
5. `replace_active`
6. `approve_request`
7. `reject_request`

原因：
1. 它们都在消费一个明确的 pending decision window
2. 不带 token 会让迟到回复有机会错误命中新窗口

## 4. 正式对象定义
```rust
pub enum ControlActionKind {
    ApproveStart,
    ModifyCandidate,
    CancelCandidate,
    KeepCurrentTask,
    ReplaceActive,
    ApproveRequest,
    RejectRequest,
    PauseTask,
    ResumeTask,
    CancelTask,
    RequestReview,
    RequestHorizonReconsideration,
    RequestTaskChange,
}

pub struct ControlAction {
    pub action_id: ControlActionId,
    pub managed_task_ref: Option<ManagedTaskRef>,
    pub kind: ControlActionKind,
    pub actor: ControlActorRef,
    pub payload: ControlActionPayload,
    pub source_decision_ref: Option<HostSemanticDecisionRef>,
    pub decision_token: Option<DecisionToken>,
}
```

这里新增的 `source_decision_ref` 表示：
1. 这条控制动作可以追溯到宿主语义层给出的哪条结构化 judgment
2. adapter 不需要再从自然语言自己猜“这是批准、取消还是变更”

这里新增的 `decision_token` 表示：
1. 这条控制动作针对的是哪一个仍然有效的 pending decision window
2. 它防止旧 start card、旧 boundary card、旧 review request 被迟到回复误消费
3. 它和 `managed_task_ref` 一起，解决“同一任务下多次窗口重开”时的串线问题

再补一条边界窗口约束：
1. `keep_current_task / replace_active`
   - 不是自由命中某个 candidate task
2. 这两条动作必须命中一个仍然有效的 `pendingBoundaryConfirmation`
3. 该窗口至少要绑定：
   - 当前 active task owner
   - 当前 boundary candidate owner
   - 当前 `decision_token`
4. kernel 只允许依据该窗口解析“保留谁、替换谁”，adapter 不得自行猜测第二项任务的目标对象

## 5. `request_horizon_reconsideration`
### 5.1 它是什么
这是这轮新补的关键动作。

它表示：
1. 用户认为当前 `WorkHorizon` 判定不准确
2. 要求宿主语义层重新判断并给出新解释

### 5.2 它不是什么
它不表示：
1. 用户直接把 `maintenance` 改成 `disruption`

原因：
1. 你已经明确不允许用户直接编辑 `WorkHorizon`
2. 但用户必须有纠偏入口

### 5.3 与事件模型的关系
它至少应产生两类事件：
1. `control_action_received`
2. 宿主语义层重新产出新的 `HostSemanticBundle`
3. adapter 归一化出新的 `SemanticDecisionEnvelope::WorkHorizon`
4. 如果判断改变，则 `work_horizon_revised`

---

## 6. `request_task_change`
### 6.1 它是什么
这是 active managed task 运行期的正式变更入口。

它表示：
1. 用户要修改任务范围、细节、补充要求或纠正目标

### 6.2 它为什么必须存在
你已经明确：
1. 用户可以在执行过程中动态修改任务
2. 可以修改未执行到的细节
3. 也可以对已经部分执行的方向提出新要求和补充

如果没有正式动作，系统就会回到：
1. 全靠自然语言猜
2. 变更是否被吸收没有正式留痕
3. 最终结果包无法说明“最后到底按什么范围交付”

### 6.3 它不等于直接改写历史事实
这条边界很重要。

`request_task_change` 表示：
1. 用户发起变更请求

它不表示：
1. 已经完成的阶段历史会被抹掉
2. 系统立即无条件接受所有变更

正确流程应该是：
1. 收到变更请求
2. 做 impact assessment
3. 需要时修订 scope / phase plan / acceptance strictness
4. 把结果写进 task event 和最终结果包

### 6.4 这条动作如何进入 Loom
这轮边界要写死：
1. 用户自然语言回复不会被 adapter 直接猜成 `request_task_change`
2. 宿主语义层必须先给出结构化 `control_action` judgment
3. adapter 只把该 judgment 映射成正式 `ControlAction`

## 7. 与其它对象的边界
### 7.1 和 `AdapterContract`
`AdapterContract` 负责：
1. 宿主如何把这些动作回传给 kernel
2. 如何把宿主结构化控制 judgment 映射成 `ControlAction`

`ControlAction` 负责：
1. 这些动作本身的正式语义

### 7.2 和 `TaskEventModel`
每个控制动作都必须写成统一事件。  
否则后续 recorder 与 watchdog 无法正确消费。

### 7.3 和 `AcceptancePolicy`
`request_review` 只是触发控制动作。  
真正 review 怎么判，仍由 `AcceptancePolicy` 决定。

### 7.4 和 `ResultContract`
`request_task_change` 会直接影响：
1. `final_scope_snapshot`
2. `scope_revision_summary`

### 7.5 和 `ApprovalRequestPayload`
`ApprovalRequestPayload`
1. 定义当前到底在请求用户批准什么

`approve_request / reject_request`
1. 负责消费该窗口
2. 不允许 adapter 自己把自由文本猜成批准或拒绝

---

## 8. 我的建议
`ControlAction` 是你“人退出微操，但保留治理权”的核心 contract。

我的建议：
1. v0 就把这些动作固定下来
2. 尤其把 `request_horizon_reconsideration` 收成正式动作
3. 同时把 `request_task_change` 收成 active task 的正式变更入口
4. 控制动作和普通语义判断一样，都必须先由宿主语义层结构化
5. 后续 crate 和 adapter 都围绕它对齐
