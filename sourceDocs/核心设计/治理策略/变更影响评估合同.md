# 变更影响评估合同

状态：contract draft  
定位：`ChangeImpactAssessment` 正式规格  
更新时间：2026-03-11

---

## 1. 目的
`ChangeImpactAssessment` 定义一次 active task 变更在进入正式 `request_task_change` 流程后，kernel 应如何映射成治理影响。

它回答：
1. 这次变更属于什么级别
2. 影响未来阶段、当前阶段，还是已完成工作
3. 要不要改 `PhasePlan`
4. 要不要触发返工
5. 要不要重算验收
6. 要不要升级成边界确认

更深层上，它解决的是：

**让“任务可以动态变化”不只是入口开放，而是有正式的影响映射链。**

---

## 2. 设计边界
### 2.1 `ChangeImpactAssessment` 不做语义判断
这次已锁定：
1. kernel 不判断这是 change 还是新任务
2. `ChangeImpactAssessment` 也不负责理解自然语言

它只消费：
1. `SemanticDecisionEnvelope`
2. 尤其是其中的 `TaskChangeDecision`
3. 已被正式推进的 `control_action(request_task_change)`

三层 owner 固定切开：
1. `task_change`
   - 回答这次 change 的治理判断
   - 也就是 `classification / execution_surface / boundary_recommendation`
2. `control_action(request_task_change)`
   - 回答这次 change 是否正式推进
   - 同时通过 `control_action.payload` 承载具体 patch 内容
3. `ChangeImpactAssessment`
   - 回答这次 change 会影响哪些治理面

### 2.2 `ChangeImpactAssessment` 不是 `TaskScopeSnapshot`
边界：
1. `TaskScopeSnapshot`
   - 当前正式范围是什么
2. `ChangeImpactAssessment`
   - 这次变更会如何影响当前范围和执行面

### 2.3 `ChangeImpactAssessment` 不是 `PhasePlan`
边界：
1. `PhasePlan`
   - 真实阶段蓝图
2. `ChangeImpactAssessment`
   - 告诉系统是否要修订这份蓝图

---

## 3. v0 固定规则
### 3.1 风格偏保守
这次已锁定：
1. 在“吸收为同一任务变更”还是“升级为新任务/边界冲突”之间
2. v0 偏保守

原因：
1. 你更重治理，不想静默吞掉第二项任务

### 3.2 只有未来未执行的小修订可以自动吸收
这次已锁定：
1. 只影响未来未执行阶段的小修订
   - 可以自动吸收
2. 影响当前 active stage
   - 通常要确认
3. 影响已完成工作
   - 必须触发更严格确认/返工评估

### 3.3 已完成阶段不会被“直接改写”
这条边界和 `TaskScopeSnapshot` 一样：
1. 已完成阶段是历史事实
2. 变更只能触发 rework / reopen
3. 不能假装之前没发生过

## 4. 正式对象定义
### 4.1 Rust 形态建议
```rust
pub struct ChangeImpactAssessment {
    pub assessment_id: ChangeImpactAssessmentId,
    pub managed_task_ref: ManagedTaskRef,
    pub input_decision_ref: SemanticDecisionEnvelopeId,
    pub scope_version_before: u32,
    pub classification: TaskChangeClassification,
    pub execution_surface: ChangeExecutionSurface,
    pub boundary_action: BoundaryAction,
    pub phase_plan_impact: PhasePlanImpact,
    pub rework_impact: ReworkImpact,
    pub acceptance_impact: AcceptanceImpact,
    pub budget_impact: BudgetImpact,
    pub confirmation_required: bool,
    pub rationale: String,
}
```

## 5. 核心枚举建议
### 5.1 `TaskChangeClassification`
建议值：
1. `same_task_minor`
2. `same_task_material`
3. `same_task_structural`
4. `boundary_conflict_candidate`

### 5.2 `ChangeExecutionSurface`
建议值：
1. `future_only`
2. `active_stage`
3. `completed_scope`

### 5.3 `BoundaryAction`
建议值：
1. `absorb_change`
2. `require_confirmation`
3. `open_boundary_confirmation`

### 5.4 `PhasePlanImpact`
建议值：
1. `none`
2. `rewrite_future_entries`
3. `append_optional_stage`
4. `reopen_rework_target`

### 5.5 `ReworkImpact`
建议值：
1. `none`
2. `same_stage_retry`
3. `return_to_rework_target`
4. `reassign_worker`

### 5.6 `AcceptanceImpact`
建议值：
1. `none`
2. `recompute_selected_criteria`
3. `reopen_stage_review`
4. `reopen_task_acceptance`

### 5.7 `BudgetImpact`
建议值：
1. `none`
2. `increase_within_limits`
3. `requires_user_budget_approval`

## 6. 影响映射建议
### 6.1 `future_only + same_task_minor`
默认建议：
1. `boundary_action = absorb_change`
2. `phase_plan_impact = rewrite_future_entries` 或 `none`
3. `rework_impact = none`
4. `acceptance_impact = recompute_selected_criteria`
5. `confirmation_required = false`

### 6.2 `active_stage + same_task_material`
默认建议：
1. `boundary_action = require_confirmation`
2. `phase_plan_impact = rewrite_future_entries`
3. `rework_impact = same_stage_retry` 或 `return_to_rework_target`
4. `acceptance_impact = reopen_stage_review`
5. `confirmation_required = true`

### 6.3 `completed_scope + same_task_structural`
默认建议：
1. `boundary_action = require_confirmation`
2. `phase_plan_impact = reopen_rework_target`
3. `rework_impact = return_to_rework_target`
4. `acceptance_impact = reopen_task_acceptance`
5. `confirmation_required = true`

### 6.4 `boundary_conflict_candidate`
默认建议：
1. `boundary_action = open_boundary_confirmation`
2. 其余 impact 只做草拟，不直接执行

## 7. 与其它对象的边界
### 7.1 和 `SemanticDecisionEnvelope`
`ChangeImpactAssessment` 的输入前提是：
1. 宿主已经给出结构化 change judgement

它不重新判断语义，只做治理映射。

### 7.2 和 `TaskScopeSnapshot`
`TaskScopeSnapshot` 回答：
1. 当前正式范围是什么

`ChangeImpactAssessment` 回答：
1. 这次 change 应如何改写那份范围

### 7.3 和 `PhasePlan`
`ChangeImpactAssessment` 回答：
1. 是否要修订阶段方案

真正的 `PhasePlan` 修订仍由 phase runtime 执行。

### 7.4 和 `AcceptancePolicy`
`ChangeImpactAssessment` 回答：
1. 哪些验收需要重算

`AcceptancePolicy` 继续定义：
1. 重算时具体怎么判

### 7.5 和 `pendingBoundaryConfirmation`
`ChangeImpactAssessment` 回答：
1. 是否应该升级成边界确认

真正的窗口 owner 仍是：
1. `PendingDecisionWindow(kind=BoundaryConfirmation)`
2. 与之绑定的边界扩展子实体是 `pendingBoundaryConfirmation`

## 8. 我的建议
### 8.1 这份 contract 是动态任务能力真正落地的关键
如果没有它，系统虽然有：
1. `request_task_change`
2. `TaskScopeSnapshot`
3. `ResultContract`

但中间仍然缺：
1. 这次变更到底影响了什么

### 8.2 我的建议
1. v0 保持保守
2. 只自动吸收未来未执行的小修订
3. 当前阶段与已完成工作的变更都走更强确认/返工映射
4. “新任务还是旧任务补充”由宿主大模型给结构化判断
5. kernel 只负责按 schema 映射治理后果
