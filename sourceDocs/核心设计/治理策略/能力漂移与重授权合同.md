# 能力漂移与重授权合同

状态：contract draft  
定位：宿主能力变化后 Loom 的重授权与治理后果正式规格  
更新时间：2026-03-12

---

## 1. 目的
这份 contract 定义宿主能力变化时，Loom 应如何重新评估当前执行面。

它回答：
1. `HostCapabilitySnapshot` 变化后怎样做 drift assessment
2. 当前 `ExecutionAuthorization` 是继续有效、收紧还是作废
3. `AgentBinding` 和 worker interruption 路径如何受影响
4. 哪些变化必须暂停、取消或要求用户重新决策

更深层上，它解决的是：

**让“宿主能力同步”不只是拿到一个新快照，而是真正改变 Loom 后续能不能继续执行。**

---

## 2. 设计边界
### 2.1 这不是单纯的 capability sync
边界：
1. `HostCapabilitySnapshot`
   - 表示宿主当前能力快照
2. `CapabilityDriftAssessment`
   - 表示新旧快照相比后，Loom 得出的治理影响

### 2.2 这不是 `WorkerInterruption` contract
边界：
1. worker interruption
   - 回答“暂停/取消当前运行中的 worker 怎么做”
2. capability drift
   - 回答“为什么现在必须暂停/取消/重发 authorization”

### 2.3 能力增强不等于当前 lease 自动扩权
固定边界：
1. 宿主能力增强后
2. 当前 active `ExecutionAuthorization` 不自动扩大
3. 只影响后续 reissue 或新的 stage/run

---

## 3. v0 固定规则
### 3.1 drift assessment 必须是正式对象
固定规则：
1. 每次 `HostCapabilitySnapshot` 变化后
2. Loom 都必须能形成一份正式 `CapabilityDriftAssessment`

### 3.2 能力缩减不能继续拿旧 lease 运行
固定规则：
1. 如果能力缩减碰到了当前 authorization 的关键边界
2. 必须：
   - 收紧
   - 或撤销并重发
   - 或暂停/取消
3. 不能继续拿旧 lease 执行

### 3.3 `spawn_capabilities / session_scope / worker_control_capabilities` 变化必须影响执行与控制路径
固定规则：
1. 宿主原来允许 `runtime=subagent`、现在不允许
2. 或当前会话原来有 `control_scope=children`、现在降成 `none`
3. 或 `spawn_capabilities[].host_agent_scope.mode`
   - 从 `All` 收紧到 `ExplicitList / None / Unknown`
4. 或 `session_scope.source`
   - 从 `Authoritative` 降成 `Derived / Unknown`
5. 或宿主原来支持 pause、现在不支持
6. 或原来只支持 soft interrupt、现在支持 hard interrupt
7. Loom 都必须重算：
   - `AgentBinding`
   - `ExecutionAuthorization`
   - control surface
   - interruption path

### 3.4 `Derived / Unknown` 事实不得触发静默扩权
固定规则：
1. 如果能力值本身没变，但事实来源从 `Authoritative` 退化为 `Derived / Unknown`
2. Loom 也必须重新评估：
   - 当前 lease 是否还能继续成立
   - 当前 control surface 是否还能继续放行同级动作
3. `Derived / Unknown`
   - 只能导致 keep with same-or-narrower boundary、reissue、pause 或 require user decision
   - 不得静默扩大 worker dispatch、resume 或 child control

### 3.5 workspace / secret / budget 边界变化必须影响 lease
固定规则：
1. `readable_roots / writable_roots / secret_classes / max_budget_band`
   任何一项变化
2. 都不得只记录一个“新快照已同步”
3. 必须进一步判断：
   - 当前 lease 是否仍然成立
   - 是否需要 narrow / revoke / reissue

## 4. 正式对象定义
### 4.1 Rust 形态建议
```rust
pub struct CapabilityDriftAssessment {
    pub assessment_id: CapabilityDriftAssessmentId,
    pub managed_task_ref: ManagedTaskRef,
    pub previous_snapshot_ref: HostCapabilitySnapshotRef,
    pub current_snapshot_ref: HostCapabilitySnapshotRef,
    pub drift_severity: DriftSeverity,
    pub impacted_bindings: Vec<AgentBindingRef>,
    pub impacted_authorizations: Vec<ExecutionAuthorizationRef>,
    pub reauthorization_decision: ReauthorizationDecision,
    pub rationale: String,
    pub assessed_at: Timestamp,
}

pub enum DriftSeverity {
    None,
    Minor,
    Material,
    Critical,
}

pub enum ReauthorizationDecision {
    Keep,
    Narrow,
    RevokeAndReissue,
    PauseTask,
    CancelTask,
    RequireUserDecision,
}
```

### 4.2 关键变量分别代表什么
1. `drift_severity`
   - 这次能力变化的治理严重度
2. `impacted_bindings`
   - 哪些 `AgentBinding` 被这次变化打到了
3. `impacted_authorizations`
   - 哪些 active `ExecutionAuthorization` 需要重算
4. `reauthorization_decision`
   - Loom 最终决定怎样处理

## 5. 典型评估规则
### 5.1 能力增强
示例：
1. 新增一个工具
2. 新增一个可用 worker agent
3. 宿主开始支持 hard interrupt
4. `runtime=acp` 开始支持 `resumeSessionId`
5. `spawn_capabilities[].host_agent_scope.mode`
   - 从 `ExplicitList` 放宽到 `All`
6. `session_scope.source`
   - 从 `Derived` 升成 `Authoritative`

默认规则：
1. 不自动扩大当前 authorization
2. 可以影响后续：
   - 新 stage
   - 新 worker run
   - resume 后的新 lease

### 5.2 能力缩减
示例：
1. 某个允许写的工具被撤掉
2. 某个绑定 worker 不再可用
3. 宿主失去 pause/cancel 能力
4. 当前宿主会话失去 children control scope
5. 当前宿主会话不再允许 `runtime=subagent` 或 `runtime=acp`
6. `writable_roots` 或 `secret_classes` 被收紧
7. `spawn_capabilities[].host_agent_scope.mode`
   - 从 `All` 收紧到 `ExplicitList / None`
8. `session_scope.source`
   - 从 `Authoritative` 降成 `Derived / Unknown`

默认规则：
1. 如果碰到 active authorization 的边界
2. 至少进入：
   - `Narrow`
   - `RevokeAndReissue`
   - `PauseTask`
   - `RequireUserDecision`

### 5.3 critical drift
示例：
1. 当前执行必需的 worker 消失
2. 当前授权必需的关键工具失效
3. 当前安全边界突然缩到无法继续当前执行

默认规则：
1. 不允许继续 silent execution
2. 必须暂停、取消或要求用户重新决策

## 6. 和其它对象的关系
### 6.1 和 `ExecutionAuthorization`
关系：
1. `ExecutionAuthorization`
   - 表示当前 worker 真正拿到的执行租约
2. drift contract
   - 回答当宿主现实能力变化时，这张租约还算不算有效

固定规则：
1. `Keep`
   - 继续当前 lease
2. `Narrow`
   - 收紧当前 lease 的能力边界
3. `RevokeAndReissue`
   - 当前 lease 废止，重新生成新的 lease

### 6.2 和 `AgentBinding`
固定规则：
1. `AgentBinding` 的角色骨架可以保留
2. 但具体实例、spawn runtime 和授权必须按最新能力重评估
3. `spawn_capabilities / session_scope`
   - 会直接影响 worker 还能不能继续以当前 runtime 和 authority 运行
4. `spawn_capabilities[].host_agent_scope`
   - 直接影响当前 binding 还能不能继续派到原先的 host agent 集
5. `session_scope.source`
   - 会影响 binding 是否还能把 role/control scope 当成可信前提

### 6.3 和 worker interruption
如果 drift 结论是：
1. `PauseTask`
2. `CancelTask`

那么必须进一步进入：
1. worker interruption / cancellation path

### 6.4 和 `TaskEventModel`
建议最少写这些事件：
1. `capability_snapshot_synced`
2. `capability_drift_assessed`
3. `execution_authorization_reissued`
4. `execution_authorization_revoked_due_to_drift`

---

## 7. 我的建议
1. 不要把 capability sync 当成“宿主传个新快照就完了”
2. Loom 必须正式建模 drift assessment
3. 能力增强不自动扩权，能力缩减不允许继续拿旧 lease 默默跑
4. `HostWorkerControlCapabilities` 的变化要直接进入 pause/resume/cancel 路径
5. `OpenClaw 3.11` 之后，runtime 能力和 authority scope 也必须进入 drift assessment
6. `host_agent_scope.mode` 和 `session_scope.source`
   - 是 3.11 之后必须进入 drift 的两个高影响字段
