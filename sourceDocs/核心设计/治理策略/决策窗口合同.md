# 决策窗口合同

状态：contract draft  
定位：`PendingDecisionWindow` 正式规格  
更新时间：2026-03-10

---

## 1. 目的
`PendingDecisionWindow` 定义 Loom 中所有“仍待用户结构化消费”的正式决策窗口。

它回答：
1. start card、review 升级审批、边界确认各自对应什么 authoritative state
2. `pendingUserDecision / pendingBoundaryConfirmation / decision_token` 如何形成一套闭环
3. 哪些 payload 只是窗口 projection，哪些动作才是真正消费窗口的正式入口
4. 窗口如何打开、被消费、过期、被 supersede

更深层上，它解决的是：

**把“等用户拍板”从临时字段和卡片文案里抽出来，收成 Loom 的正式状态对象。**

---

## 2. 设计边界
### 2.1 这不是出站 payload
边界：
1. `StartCardPayload / BoundaryCardPayload / ApprovalRequestPayload`
   - 是用户可见 projection
2. `PendingDecisionWindow`
   - 才是 Loom authoritative state

固定规则：
1. payload 必须引用当前窗口
2. payload 不得自成第二套真相源

### 2.2 这不是 `ControlAction`
边界：
1. `PendingDecisionWindow`
   - 表示当前到底在等用户决定什么
2. `ControlAction`
   - 表示用户如何消费这个窗口

固定规则：
1. 所有消费窗口的动作都必须命中一个仍然有效的窗口
2. `decision_token` 是这条命中关系的绑定令牌

### 2.3 这不是 `managedTaskRef`
边界：
1. `managedTaskRef`
   - 是任务 owner
2. `PendingDecisionWindow`
   - 是任务当前挂起的决策窗口

固定规则：
1. 一个任务可以在不同时间打开多个窗口
2. 这些窗口不能靠 `managedTaskRef` 单独区分
3. 必须通过 `decision_token` 和窗口状态闭环

### 2.4 `PendingBoundaryConfirmation` 是扩展子实体，不是第二个窗口 aggregate
固定边界：
1. `PendingDecisionWindow(kind=BoundaryConfirmation)`
   - 仍然是边界确认窗口的 authoritative root
2. `PendingBoundaryConfirmation`
   - 只是该窗口绑定 active/candidate/boundary_reason 的扩展子实体
3. 它不能脱离窗口单独存在

---

## 3. v0 固定规则
1. 所有待用户结构化消费的治理入口，都必须先形成 `PendingDecisionWindow`
2. v0 只收这三类窗口：
   - `StartCandidate`
   - `ApprovalRequest`
   - `BoundaryConfirmation`
3. 每个 open window 都必须生成新的 `decision_token`
4. 同一条控制动作只能消费一个仍然有效的窗口
5. `pendingUserDecision`
   - 是 task projection 上“当前 open 决策窗口”的通用指针
6. `pendingBoundaryConfirmation`
   - 是边界确认窗口的专门扩展子实体
   - 因为它还要额外绑定 active task owner、boundary candidate owner 和 `boundary_reason`
7. review 的 `escalate_to_user`
   - 必须先打开 `PendingDecisionWindow(kind=ApprovalRequest)`
   - 不能直接伪装成最终结果

## 4. 正式对象定义
```rust
pub struct PendingDecisionWindow {
    pub window_id: PendingDecisionWindowId,
    pub managed_task_ref: ManagedTaskRef,
    pub kind: PendingDecisionWindowKind,
    pub decision_token: DecisionToken,
    pub status: PendingDecisionWindowStatus,
    pub allowed_actions: Vec<ControlActionKind>,
    pub opened_reason: String,
    pub opened_at: Timestamp,
    pub expires_at: Option<Timestamp>,
    pub supersedes: Option<PendingDecisionWindowRef>,
}

pub enum PendingDecisionWindowKind {
    StartCandidate,
    ApprovalRequest,
    BoundaryConfirmation,
}

pub enum PendingDecisionWindowStatus {
    Open,
    Consumed,
    Superseded,
    Expired,
    Cancelled,
}

pub struct PendingBoundaryConfirmation {
    pub window_ref: PendingDecisionWindowRef,
    pub active_managed_task_ref: ManagedTaskRef,
    pub boundary_candidate_ref: ManagedTaskRef,
    pub boundary_reason: BoundaryReason,
}
```

---

## 5. 关键字段分别代表什么
1. `window_id`
   - 当前决策窗口的稳定 id
2. `managed_task_ref`
   - 这次窗口归属哪个任务 owner
3. `kind`
   - 这次到底在等待哪类正式决策
4. `decision_token`
   - 当前窗口绑定令牌
5. `allowed_actions`
   - 当前窗口只允许哪些正式 `ControlActionKind` 消费
6. `supersedes`
   - 当前窗口是否替换了旧窗口
7. `PendingBoundaryConfirmation.boundary_reason`
   - 为什么这次输入被升级成边界确认，而不是直接吸收或直接替换

## 6. 和其它对象的关系
### 6.1 和 `StartCardPayload`
固定关系：
1. candidate 创建后，Loom 先打开 `PendingDecisionWindow(kind=StartCandidate)`
2. 再输出 `StartCardPayload`
3. `approve_start / modify_candidate / cancel_candidate`
   - 只能消费这个窗口

### 6.2 和 `ApprovalRequestPayload`
固定关系：
1. review 升级或高风险动作需要用户决策时
2. Loom 先打开 `PendingDecisionWindow(kind=ApprovalRequest)`
3. 再输出 `ApprovalRequestPayload`
4. 只允许：
   - `approve_request`
   - `reject_request`
   消费

### 6.3 和 `pendingBoundaryConfirmation`
固定关系：
1. 第二项重任务到来时
2. Loom 先打开 `PendingDecisionWindow(kind=BoundaryConfirmation)`
3. 再建立 `PendingBoundaryConfirmation`
4. `BoundaryCardPayload`
   - 只是该对象的用户可见 projection

### 6.4 和 `decision_token`
固定关系：
1. `decision_token` 不单独拥有生命周期
2. 它的生命周期由 `PendingDecisionWindow` 持有
3. 旧 token 不得命中新窗口

---

## 7. 我的建议
1. `pendingUserDecision / pendingBoundaryConfirmation` 不应再以“临时挂在任务上的 JSON 片段”存在
2. v0 先把决策窗口统一收成一个 contract family，再让具体 payload 去做 projection
3. 任何新增审批或确认面，都必须先回答它属于哪一种 `PendingDecisionWindowKind`
