# 执行授权合同

状态：contract draft  
定位：`ExecutionAuthorization` 正式规格  
更新时间：2026-03-11

---

## 1. 目的
`ExecutionAuthorization` 定义某次 run 在当前宿主现实下真正拿到的执行租约。

它回答：
1. 当前 run 到底被允许做什么
2. 哪些能力只是抽象授权语义，哪些已经收敛成运行时可执法边界
3. `approval-gate` 应依据哪份正式对象放行或阻断
4. 能力漂移、暂停、取消后这张租约怎样变化

更深层上，它解决的是：

**把“默认授权倾向”收敛成当前 run 可被真正执法的一张正式 lease。**

---

## 2. 设计边界
### 2.1 这不是 `DelegationLevel`
边界：
1. `DelegationLevel`
   - 描述抽象授权语义和默认倾向
2. `ExecutionAuthorization`
   - 描述当前 run 最终被放行的实际执行边界

### 2.2 这不是 `HostCapabilitySnapshot`
边界：
1. `HostCapabilitySnapshot`
   - 描述宿主当前有哪些真实能力
2. `ExecutionAuthorization`
   - 描述在这些真实能力下，这次 run 最终被允许使用哪一部分

### 2.3 这不是 `IsolatedTaskRun`
边界：
1. `IsolatedTaskRun`
   - 是 run 本体
2. `ExecutionAuthorization`
   - 是该 run 当前生效的能力租约

### 2.4 这不是 `approval-gate`
边界：
1. `ExecutionAuthorization`
   - 定义当前 lease
2. `approval-gate`
   - 依据当前 lease、`DecisionArea` 与用户决策窗口真正执法

### 2.5 这不是 `TaskScopeSnapshot`
边界：
1. `TaskScopeSnapshot`
   - 定义任务级原则边界
2. `ExecutionAuthorization`
   - 定义当前 run 可被执法的实际子集

---

## 3. v0 固定规则
1. 每个 active `IsolatedTaskRun` 至少绑定一个当前生效的 `ExecutionAuthorization`
2. 当前 lease 必须同时受这些对象约束：
   - `HostCapabilitySnapshot`
   - `TaskScopeSnapshot`
   - `DelegationLevel / DelegationBand`
   - `DecisionArea`
   - 当前任务/阶段上下文
3. 高风险决策区不得因为宿主真实有能力就自动放行
4. capability drift、pause、cancel、resume 都必须显式影响 lease 状态
5. `resume_task` 不恢复旧 lease，而是重新评估并发放新 lease

## 4. 正式对象定义
```rust
pub struct ExecutionAuthorization {
    pub authorization_id: ExecutionAuthorizationId,
    pub managed_task_ref: ManagedTaskRef,
    pub run_ref: IsolatedTaskRunRef,
    pub stage_run_ref: Option<StageRunRef>,
    pub capability_snapshot_ref: HostCapabilitySnapshotRef,
    pub task_scope_ref: TaskScopeId,
    pub granted_areas: Vec<AuthorizedDecisionArea>,
    pub status: ExecutionAuthorizationStatus,
    pub issued_reason: String,
    pub issued_at: Timestamp,
    pub expires_at: Option<Timestamp>,
    pub supersedes: Option<ExecutionAuthorizationRef>,
}

pub struct AuthorizedDecisionArea {
    pub decision_area: DecisionArea,
    pub band: DelegationBand,
    pub allowed_tools: Vec<String>,
    pub readable_roots: Vec<String>,
    pub writable_roots: Vec<String>,
    pub allowed_secret_classes: Vec<String>,
    pub not_before: Option<Timestamp>,
    pub not_after: Option<Timestamp>,
    pub budget_band: AuthorizationBudgetBand,
    pub spawn_agent_allowed: bool,
    pub requires_user_approval: bool,
}

pub enum AuthorizationBudgetBand {
    Conservative,
    Standard,
    Elevated,
}

pub enum ExecutionAuthorizationStatus {
    Draft,
    Active,
    Suspended,
    Narrowed,
    Revoked,
    Expired,
    Superseded,
}
```

---

## 5. 关键字段分别代表什么
1. `authorization_id`
   - 这张 lease 的稳定 id
2. `run_ref`
   - 当前 lease 归属哪个 `IsolatedTaskRun`
3. `capability_snapshot_ref`
   - 这张 lease 是基于哪版宿主能力快照发放的
4. `task_scope_ref`
   - 这张 lease 受哪版 `TaskScopeSnapshot` 上限约束
5. `granted_areas`
   - 当前 run 在各个 `DecisionArea` 上真正拿到的授权边界
   - 至少明确工具、读路径、写路径、secret class、时间窗与预算带
6. `status`
   - 这张 lease 当前是否仍可执法
7. `supersedes`
   - 这张 lease 是否替换了旧 lease

## 6. 和其它对象的关系
### 6.1 和 `DelegationLevel`
固定关系：
1. `DelegationLevel`
   - 提供抽象授权语义
2. `ExecutionAuthorization`
   - 把这些语义收敛成当前 run 的可执法结果

### 6.2 和 `HostCapabilitySnapshot`
固定关系：
1. 宿主能力增强
   - 不自动扩大当前 lease
2. 宿主能力缩减
   - 可能触发 `Narrow / Revoked / Superseded`

### 6.3 和 `TaskScopeSnapshot`
固定关系：
1. `TaskScopeSnapshot`
   - 给出任务级原则边界
2. `ExecutionAuthorization`
   - 只给出当前 run 的租约子集
3. `TaskScopeSnapshot.allowed_roots`
   - 是 `ExecutionAuthorization.readable_roots / writable_roots` 的上限
4. `TaskScopeSnapshot.secret_classes`
   - 是 `ExecutionAuthorization.allowed_secret_classes` 的上限

统一公式：
1. 当前 run 的实际可执行范围 =
   `HostCapabilitySnapshot ∩ TaskScopeSnapshot ∩ ExecutionAuthorization`

### 6.4 和 `approval-gate`
固定关系：
1. `approval-gate` 必须读取当前 `ExecutionAuthorization`
2. 对 `requires_user_approval=true` 的决策区，必须显式打开审批窗口

### 6.5 和中断/重授权
固定关系：
1. `pause_task`
   - 当前 lease 至少进入 `Suspended`
2. `cancel_task`
   - 当前 lease 至少进入 `Revoked`
3. `resume_task`
   - 必须重新发放新 lease
4. capability drift
   - 可以把 lease 收紧、撤销或重发

---

## 7. 我的建议
1. `ExecutionAuthorization` 要被当成正式治理对象，而不是实现层临时拼出来的 allow/deny 集合
2. `approval-gate` 只执法，不自己发明 lease
3. 所有 capability drift、pause、cancel、resume 后果都应体现在 lease 状态上
4. `granted_areas` 继续作为授权细化点，不额外新建平行授权范围对象
5. 第一版读路径与 secret class 也必须进入 lease，避免任务范围与 run 授权在这些维度上失去闭环
