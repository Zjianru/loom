# 宿主语义协议

状态：schema draft  
定位：宿主语义层到 adapter 的正式结构化协议  
更新时间：2026-03-10

---

## 1. 目的
这份文档回答：
1. 宿主主模型或宿主语义层应该把什么样的结构化判断交给 adapter
2. 这些判断如何版本化、留痕、降级与失败关闭
3. 宿主 schema 和 kernel 内部 `SemanticDecisionEnvelope` 的边界是什么

更深层上，它解决的是：

**让“kernel 不做语义推断”从原则变成正式宿主协议。**

---

## 2. 为什么还需要一层 Host Semantic Schema
这层 schema 不是重复定义 [语义决策封套合同.md](语义决策封套合同.md)。

两者边界不同：
1. `HostSemanticSchema`
   - 是宿主语义层给 adapter 的综合判断包
   - 一个输入可以带多个 decision
   - 带宿主侧版本、来源、失败策略和补默认来源
2. `SemanticDecisionEnvelope`
   - 是 adapter 归一化后交给 kernel 的 bounded 结构
   - kernel 只消费它
3. `ControlAction`
   - 是 adapter 基于宿主结构化控制 judgment 生成的正式控制动作
   - 它同样不能由 adapter 从自然语言自行猜测

我的建议：
1. 宿主负责“理解用户在说什么”
2. adapter 负责“把宿主判断归一化”
3. kernel 负责“消费归一化结果并执行治理”

这样做的取舍是：
1. 不让 kernel 再长成第二个语义分类器
2. 也不让 adapter 只能拿零散字符串临时拼接

---

## 3. 核心立场
### 3.1 每条宿主输入对应一个综合 schema bundle
v0 的默认规则：
1. 一条宿主输入对应一个 `HostSemanticBundle`
2. 一个 bundle 内可以包含多个 decision
3. 不要求宿主分多次提交零散判断

原因：
1. 同一条输入上的 `interaction_lane / managed_task_class / work_horizon / task_change` 往往彼此相关
2. 拆成多包会让 `input_ref`、时间戳、来源和信心值更容易串线

### 3.2 decision 的置信度按 decision 维度单独记录
这次已锁定：
1. 置信度不是整包一个
2. 每个 decision 单独带 `confidence`

原因：
1. 宿主模型可能对 `interaction_lane` 很确定
2. 但对 `work_horizon` 或 `boundary_recommendation` 没那么确定

### 3.3 语义来源必须显式标注
每个 decision 都必须带 `DecisionSource`。

它回答：
1. 这是宿主模型的判断
2. 还是 pack 默认补值
3. 还是用户控制动作触发后的重判
4. 还是 adapter 的保守 fallback

如果没有这层来源，后面会出现更深层冲突：
1. 默认值会伪装成模型判断
2. 用户重判结果会和旧判断混成一条线

### 3.4 schema 必须版本化
这次已锁定：
1. 握手层声明支持的 schema version
2. 每个 bundle 也带 `schema_version`

原因：
1. 宿主 prompt 或结构化输出模板后面一定会迭代
2. 如果不版本化，adapter 与 kernel 会默默漂移

## 4. 正式对象定义
### 4.1 Rust 形态建议
```rust
pub struct HostSemanticSchemaVersion {
    pub major: u16,
    pub minor: u16,
}

pub struct HostSemanticHandshake {
    pub adapter_id: AdapterId,
    pub host_id: HostId,
    pub supported_schema_versions: Vec<HostSemanticSchemaVersion>,
}

pub struct HostSemanticBundle {
    pub schema_version: HostSemanticSchemaVersion,
    pub host_session_id: HostSessionId,
    pub host_message_ref: Option<HostMessageRef>,
    pub managed_task_ref: Option<ManagedTaskRef>,
    pub input_ref: HostInputRef,
    pub source_model_ref: HostModelRef,
    pub issued_at: Timestamp,
    pub decisions: Vec<HostSemanticDecision>,
    pub rationale_summary: Option<String>,
}

pub struct HostSemanticDecision {
    pub decision_kind: HostSemanticDecisionKind,
    pub decision_source: DecisionSource,
    pub confidence: ConfidenceScore,
    pub rationale: String,
    pub payload: HostSemanticDecisionPayload,
}
```

### 4.2 `DecisionSource`
建议正式值：
```rust
pub enum DecisionSource {
    HostModel,
    PackDefault,
    SystemReconsideration,
    UserControlAction,
    AdapterFallback,
}
```

它们分别代表：
1. `HostModel`
   - 宿主主模型或宿主语义层的直接判断
2. `PackDefault`
   - 来自 pack 的默认 hint，不是模型理解结果
3. `SystemReconsideration`
   - 用户申诉或系统重审后得到的新判断
4. `UserControlAction`
   - 来自明确控制动作的结构化结果，不是模型猜测
5. `AdapterFallback`
   - adapter 为 fail-closed 安全补的保守值

### 4.3 `HostSemanticDecisionKind`
建议 v0 正式值：
1. `interaction_lane`
2. `task_activation_reason`
3. `managed_task_class`
4. `work_horizon`
5. `task_change`
6. `control_action`

## 5. v0 必填与条件必填
### 5.1 总规则
v0 不要求每条输入都带所有 decision。  
但要区分：
1. 必填
2. 条件必填
3. 可选

### 5.2 必填 decision
以下 decision 在“用户自然语言输入进入治理前置层”时必须存在：
1. `interaction_lane`

原因：
1. 没有它，kernel 不知道当前输入是 `chat` 还是 managed lane
2. 这会直接破坏“插件默认静默”的总边界

### 5.3 条件必填 decision
当满足对应条件时，以下 decision 必须存在：
1. `task_activation_reason`
   - 当 `interaction_lane != chat`
2. `managed_task_class`
   - 当 `interaction_lane == managed_task_candidate`
3. `work_horizon`
   - 当 `interaction_lane == managed_task_candidate`
4. `task_change`
   - 当存在 `managed_task_ref` 且本轮输入被宿主理解为 active task 变更
   - 尤其当宿主要正式推进 `request_task_change` 时，必须提供
5. `control_action`
   - 当本轮输入被宿主理解为 candidate/active task 的正式控制回复

### 5.4 可选 decision
以下 decision 在 v0 可选：
1. 更细的 pack/host hint
   - 只要不直接驱动 kernel 分支即可

## 6. 典型 payload contract
### 6.1 `interaction_lane`
建议值：
1. `chat`
2. `managed_task_candidate`
3. `managed_task_active`

### 6.2 `task_activation_reason`
建议值：
1. `explicit_start_task`
2. `explicit_track_this`
3. `delegate_heavy_work`
4. `heavy_multi_stage_goal`

### 6.3 `managed_task_class`
建议值：
1. `complex`
2. `huge`
3. `max`

固定接缝：
1. 宿主 schema 的 wire 值统一用小写：
   - `complex / huge / max`
2. adapter 归一化后再映射成 Loom 内部的正式档位：
   - `COMPLEX / HUGE / MAX`
3. start card、结果包和顶层规范文档统一按 Loom 正式档位展示

### 6.4 `work_horizon`
建议值：
1. `maintenance`
2. `improvement`
3. `extension`
4. `disruption`

### 6.5 `task_change`
建议字段：
```rust
pub struct HostTaskChangeDecision {
    pub classification: TaskChangeClassification,
    pub execution_surface: ChangeExecutionSurface,
    pub boundary_recommendation: BoundaryRecommendation,
}
```

固定边界：
1. `task_change`
   - 只承载治理判断
2. 它不承载这次具体要改哪些 patch 内容
   - 例如 `summary / expected_outcome / requirement_items / allowed_roots / workspace_ref / repo_ref`
3. 如果宿主要正式推进 `request_task_change`
   - 同一条 `HostSemanticBundle` 里还必须再给一条
     `control_action(action_kind=request_task_change)`

建议值：
1. `TaskChangeClassification`
   - `same_task_minor`
   - `same_task_material`
   - `same_task_structural`
   - `boundary_conflict_candidate`
2. `ChangeExecutionSurface`
   - `future_only`
   - `active_stage`
   - `completed_scope`
3. `BoundaryRecommendation`
   - `absorb_change`
   - `require_confirmation`
   - `open_boundary_confirmation`

### 6.6 `control_action`
建议字段：
```rust
pub struct HostControlActionDecision {
    pub action_kind: ControlActionKind,
    pub managed_task_ref: Option<ManagedTaskRef>,
    pub decision_token: Option<DecisionToken>,
    pub payload: Option<HostControlActionPayload>,
}

pub struct HostControlActionPayload {
    pub title: Option<String>,
    pub summary: Option<String>,
    pub expected_outcome: Option<String>,
    pub requirement_items: Vec<RequirementItemDraft>,
    pub allowed_roots: Vec<String>,
    pub secret_classes: Vec<String>,
    pub workspace_ref: Option<String>,
    pub repo_ref: Option<String>,
    pub rationale: Option<String>,
}
```

建议值至少包含：
1. `approve_start`
2. `modify_candidate`
3. `cancel_candidate`
4. `keep_current_task`
5. `replace_active`
6. `pause_task`
7. `resume_task`
8. `cancel_task`
9. `request_review`
10. `request_horizon_reconsideration`
11. `request_task_change`

字段约束建议：
1. `managed_task_ref`
   - 对所有 task-scoped control action 都建议提供
   - 对 `approve_start / modify_candidate / cancel_candidate / keep_current_task / replace_active` 应视为条件必填
2. `decision_token`
   - 对所有消费 pending decision window 的动作应视为条件必填
   - adapter 不得在缺 token 时“猜测”该动作仍然有效
3. `keep_current_task / replace_active`
   - 这里的 `managed_task_ref` 指向当前 active managed task
   - 第二项 candidate task 必须由 `decision_token` 绑定的 `pendingBoundaryConfirmation` authoritative state 解析
   - 宿主语义层和 adapter 不得自己补猜“要替换成哪一个 candidate”
4. `request_task_change`
   - 必须与同 bundle 中的 `task_change` judgment 成对出现
   - `control_action.payload`
     才承载这次具体请求改写的 patch 内容
   - adapter 不得只看到 `task_change` 就自动脑补出 `request_task_change`
5. 如果显式 grammar 或宿主 parser 只能产出 patch，不能产出 `task_change`
   - 必须请求补判或 fail closed
   - 不得直接把 patch 落成正式 `request_task_change`

## 7. FailurePolicy
### 7.1 版本不兼容
规则：
1. major version 不兼容时直接 fail closed
2. adapter 不得把未知 major 版本偷偷降级成旧语义

我的建议：
1. 给宿主返回结构化 schema mismatch
2. 同时在用户层退化成保守文本说明，不推进治理

### 7.2 小版本新增字段
规则：
1. minor version 新增字段时，adapter 可以忽略未知字段
2. 但 adapter 应负责清洗后再转交 kernel

原因：
1. kernel 的 bounded schema 不该被宿主小升级直接冲破

### 7.3 缺 `interaction_lane`
规则：
1. 不激活 managed lane
2. 退回 `chat` 或请求澄清

我的建议：
1. `DecisionSource=AdapterFallback`
2. fallback 值只允许是 `chat`

### 7.4 已进入 managed lane，但缺 `managed_task_class / work_horizon`
规则：
1. adapter 允许发起一次自动重判
2. 如果重判后仍缺字段：
   - 不进入执行
   - 不生成看似完整的 start card
   - 退化成保守澄清或补判请求

原因：
1. candidate 可以保守存在
2. execution 不能建立在半残缺语义上

### 7.5 低置信度
这次已锁定：
1. 允许一次自动重判
2. 第二次仍低置信度则保守退化

建议规则：
1. `interaction_lane` 低置信度
   - 回到 `chat` 或请求澄清
2. `managed_task_class / work_horizon` 低置信度
   - 允许 candidate 保守停留
   - 不得直接进入 execution
3. `task_change` 治理判断低置信度
   - 走更保守确认流
   - 或请求宿主补判

## 8. adapter 归一化规则
### 8.1 一条 Host bundle 可以展开成多条 kernel envelope
这是最关键的边界之一。

我的建议：
1. 宿主只负责给出一条综合 bundle
2. adapter 再按 `decision_kind` 拆成：
   - 一条或多条 `SemanticDecisionEnvelope`
   - 或一条正式 `ControlAction`
3. kernel 继续只消费 bounded decision 与正式控制动作

固定说明：
1. `task_activation_reason`
   - 必须进入归一化后的 envelope 集合
   - 因为 candidate 创建和 `StartCardPayload` 都依赖它
2. `task_change`
   - 必须继续保持为治理 judgment
   - 不得被 adapter 当成 patch carrier
3. `ChangeImpactAssessment`
   - 不由宿主直接给出
   - 而是 Loom 消费 `task_change` 后生成的治理对象

原因：
1. 宿主最自然的是一次理解整条输入
2. kernel 最稳定的是消费小而清楚的 decision 单元

### 8.2 adapter 不得补充自由语义文本
adapter 允许：
1. 补 schema version
2. 做字段清洗
3. 做 `DecisionSource=AdapterFallback` 的保守默认

adapter 不允许：
1. 从自然语言自己猜新的 `managed_task_class`
2. 自己脑补新的 `work_horizon`
3. 自己决定“这其实是另一个任务”

### 8.3 `control_action` 不经文本猜测
新增规则：
1. 如果 bundle 中带 `control_action`
2. adapter 必须把它映射成正式 `ControlAction`
3. 不允许从用户可见文本里自行猜测批准、取消、重判或变更
4. 对 `request_task_change`
   - adapter 还必须检查同 bundle 中是否有 `task_change`
   - 缺失时必须 fail closed

## 9. 与其它对象的边界
### 9.1 和 `SemanticDecisionEnvelope`
`HostSemanticSchema` 负责：
1. 宿主综合语义包
2. schema version
3. per-decision confidence
4. decision source
5. host-side failure handling

`SemanticDecisionEnvelope` 负责：
1. adapter 已归一化好的 bounded decision
2. kernel 可直接消费的治理输入

### 9.2 和 `AdapterContract`
`AdapterContract` 负责：
1. 宿主能力与消息映射
2. schema 清洗与归一化

`HostSemanticSchema` 负责：
1. 语义判断到底要长什么样

### 9.3 和 `OpenClaw Adapter Spike`
Spike 文档负责：
1. 在 OpenClaw 里最小怎么把这份 schema 真接起来

本文件不负责：
1. 具体 hook 选哪个
2. 具体 transport 选哪个

---

## 10. 我的建议
### 10.1 v0 的关键不是“让宿主更聪明”
而是：
1. 让宿主把判断交得更正式
2. 让 adapter 把边界守得更硬
3. 让 kernel 不再长成第二个语义系统

### 10.2 v0 先收紧，再放开
我的建议：
1. 先把 schema 设计得严格
2. 先用 bounded enum 和 per-decision confidence
3. 后面再扩更多 decision kind

这样做更符合你前面定下来的方向：
1. 能力交给更强的 agent 和模型
2. 我们只做编排和治理
