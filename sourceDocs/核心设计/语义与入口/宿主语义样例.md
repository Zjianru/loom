# 宿主语义样例

状态：examples draft  
定位：`HostSemanticSchema` 与 OpenClaw adapter spike 的实例集  
更新时间：2026-03-12

---

## 1. 目的
这份文档不再解释 schema 定义本身。  
它回答的是：
1. 宿主面对真实用户输入时，应该产出什么 `HostSemanticBundle`
2. adapter 应如何把它归一化成一个或多个 `SemanticDecisionEnvelope`
3. kernel 收到后应该走哪条治理分支
4. 缺字段、低置信度和边界冲突时应如何 fail closed

更深层上，它是两份文档的实例附录：
1. [宿主语义协议.md](宿主语义协议.md)
2. [OpenClaw适配验证方案.md](../宿主接入/OpenClaw适配验证方案.md)

另外，这些 example 在真正进入 Loom outbound 时，还会落到：
1. [内核出站载荷合同.md](../宿主接入/内核出站载荷合同.md)

也就是说：
1. 这里回答“宿主应该怎样给出结构化判断”
2. payload contract 回答“Loom 最终怎样把治理结果发回宿主”

---

## 2. 阅读方式
每个 example 都固定包含 5 段：
1. 用户原话
2. `HostSemanticBundle`
3. adapter 归一化后的 `SemanticDecisionEnvelope[]`
4. kernel 预期动作
5. 退化与取舍

统一约束：
1. 用户原话用中文
2. enum 保持英文
3. `managed_task_class` 一律按 `complex / huge / max`
4. 结构化结果只用于协议层，不直接展示给用户

## 3. Example A: chat default
### 3.1 用户原话
```text
帮我解释一下你刚才为什么说这个架构会双真相源。
```

### 3.2 HostSemanticBundle
```json
{
  "schema_version": { "major": 0, "minor": 1 },
  "host_session_id": "sess-001",
  "host_message_ref": "msg-001",
  "input_ref": "msg-001",
  "source_model_ref": "host-main-model",
  "issued_at": "2026-03-10T10:00:00+08:00",
  "decisions": [
    {
      "decision_kind": "interaction_lane",
      "decision_source": "host_model",
      "confidence": 0.96,
      "rationale": "这是解释性追问，不要求正式委派或任务治理。",
      "payload": { "interaction_lane": "chat" }
    }
  ]
}
```

### 3.3 Adapter 归一化
```json
[
  {
    "decision_kind": "interaction_lane",
    "decision_payload": { "interaction_lane": "chat" },
    "confidence": 0.96
  }
]
```

### 3.4 Kernel 预期动作
1. 不创建 `managedTaskRef`
2. 不创建 `pendingUserDecision`
3. 不创建 `pendingBoundaryConfirmation`
4. 保持主聊天直接回复

### 3.5 退化与取舍
1. 如果这条输入缺 `interaction_lane`，adapter 应 fail-open to chat
2. 这里不能因为用户问的是“架构问题”就自动升级成 managed lane

## 4. Example B: explicit COMPLEX activation
### 4.1 用户原话
```text
开始一个任务：把这段数据库迁移方案交给擅长 SQL 的 agent 审一遍，整理成一页结论给我。
```

### 4.2 HostSemanticBundle
```json
{
  "schema_version": { "major": 0, "minor": 1 },
  "host_session_id": "sess-001",
  "host_message_ref": "msg-010",
  "input_ref": "msg-010",
  "source_model_ref": "host-main-model",
  "issued_at": "2026-03-10T10:05:00+08:00",
  "decisions": [
    {
      "decision_kind": "interaction_lane",
      "decision_source": "host_model",
      "confidence": 0.94,
      "rationale": "用户显式说“开始一个任务”，并要求正式委派给专业 agent。",
      "payload": { "interaction_lane": "managed_task_candidate" }
    },
    {
      "decision_kind": "task_activation_reason",
      "decision_source": "host_model",
      "confidence": 0.93,
      "rationale": "这是显式启动并委派专业工作的受管任务。",
      "payload": { "task_activation_reason": "explicit_start_task" }
    },
    {
      "decision_kind": "managed_task_class",
      "decision_source": "host_model",
      "confidence": 0.87,
      "rationale": "这是单 worker 的专业委派，适合 1+1 的 complex 档位。",
      "payload": { "managed_task_class": "complex" }
    },
    {
      "decision_kind": "work_horizon",
      "decision_source": "host_model",
      "confidence": 0.81,
      "rationale": "这是对既有迁移方案的正式审查，更像 improvement。",
      "payload": { "work_horizon": "improvement" }
    }
  ],
  "rationale_summary": "显式启动 + 单 worker 专业委派 + 有正式交付要求。"
}
```

### 4.3 Adapter 归一化
```json
[
  {
    "decision_kind": "interaction_lane",
    "decision_payload": { "interaction_lane": "managed_task_candidate" }
  },
  {
    "decision_kind": "task_activation_reason",
    "decision_payload": { "task_activation_reason": "explicit_start_task" }
  },
  {
    "decision_kind": "managed_task_class",
    "decision_payload": { "managed_task_class": "complex" }
  },
  {
    "decision_kind": "work_horizon",
    "decision_payload": { "work_horizon": "improvement" }
  }
]
```

### 4.4 Kernel 预期动作
1. 创建 candidate 级 `managedTaskRef`
2. 生成 `managed task start card`
3. start card 中的建议档位为 `COMPLEX`
4. 不直接进入 execute
5. 一旦用户 `approve_start`：
   - Loom 依据 `coding_pack` 生成 `SpecBundle`
   - Loom 为这个 `managedTaskRef` 创建 `IsolatedTaskRun`
   - 任务进入结果阶段后再编译 `ProofOfWorkBundle`

### 4.5 退化与取舍
1. `managed_task_class=complex` 代表正式受管任务最小档位，不应再回落成 chat lane
2. 如果 `work_horizon` 缺失，adapter 允许一次自动重判；仍缺则只保守停在 candidate，不进入执行
3. `SpecBundle / IsolatedTaskRun / ProofOfWorkBundle` 都由 Loom 依据 pack preset 和运行时状态生成，不由 adapter 自己猜


```json
{
  "schema_version": { "major": 0, "minor": 1 },
  "host_session_id": "sess-001",
  "host_message_ref": "msg-020",
  "input_ref": "msg-020",
  "source_model_ref": "host-main-model",
  "issued_at": "2026-03-10T10:10:00+08:00",
  "decisions": [
    {
      "decision_kind": "interaction_lane",
      "decision_source": "host_model",
      "confidence": 0.97,
      "rationale": "用户显式启动并要求阶段化推进。",
      "payload": { "interaction_lane": "managed_task_candidate" }
    },
    {
      "decision_kind": "task_activation_reason",
      "decision_source": "host_model",
      "confidence": 0.95,
      "rationale": "这是显式启动的多阶段受管任务。",
      "payload": { "task_activation_reason": "heavy_multi_stage_goal" }
    },
    {
      "decision_kind": "managed_task_class",
      "decision_source": "host_model",
      "confidence": 0.91,
      "rationale": "需要多个专业 worker 分阶段协作，属于 huge。",
      "payload": { "managed_task_class": "huge" }
    },
    {
      "decision_kind": "work_horizon",
      "decision_source": "host_model",
      "confidence": 0.84,
      "rationale": "这是对现有系统的结构性扩展与重构，更像 extension。",
      "payload": { "work_horizon": "extension" }
    }
  ]
}
```

### 5.3 Adapter 归一化
与 Example B 相同，只是 `managed_task_class=huge`、`work_horizon=extension`。

### 5.4 Kernel 预期动作
1. 创建 candidate 级 `managedTaskRef`
2. 生成建议档位为 `HUGE` 的 start card
3. 后续允许 `PhasePlan` 采用多阶段骨架
4. `watchdog` 在 spike 中只需要证明最小 `StatusNotice` 可接

### 5.5 退化与取舍
1. `huge` 代表 `1 + n` 的阶段化协作，不等于“MAX-lite”
2. 如果类和 horizon 都高置信度，kernel 不需要再做任何语义解释

## 7. Example E: active task minor future-only change
### 7.1 用户原话
```text
刚才那个任务再补一条：最后的总结里加一个风险清单就行，其他先按原计划推进。
```

### 7.2 HostSemanticBundle
```json
{
  "schema_version": { "major": 0, "minor": 1 },
  "host_session_id": "sess-001",
  "host_message_ref": "msg-040",
  "managed_task_ref": "task-001",
  "input_ref": "msg-040",
  "source_model_ref": "host-main-model",
  "issued_at": "2026-03-10T10:20:00+08:00",
  "decisions": [
    {
      "decision_kind": "interaction_lane",
      "decision_source": "host_model",
      "confidence": 0.90,
      "rationale": "这是当前 active task 的后续补充，不是普通闲聊。",
      "payload": {
        "interaction_lane": "managed_task_active",
        "managed_task_ref": "task-001"
      }
    },
    {
      "decision_kind": "task_change",
      "decision_source": "host_model",
      "confidence": 0.87,
      "rationale": "只影响未来交付内容，不要求重做已完成阶段。",
      "payload": {
        "classification": "same_task_minor",
        "execution_surface": "future_only",
        "boundary_recommendation": "absorb_change"
      }
    },
    {
      "decision_kind": "control_action",
      "decision_source": "user_control_action",
      "confidence": 0.99,
      "rationale": "用户明确要求把这次补充正式并入当前 active task。",
      "payload": {
        "action_kind": "request_task_change",
        "managed_task_ref": "task-001",
        "payload": {
          "summary": "把导航重构范围补充到核心设计目录，并同步修改目录说明。",
          "expected_outcome": "最终结果按修订后的导航与目录说明一并交付。"
        }
      }
    }
  ]
}
```

### 7.3 Kernel 预期动作
1. 生成 `task_change_requested`
2. 生成 `ChangeImpactAssessment`
3. 自动吸收变更
4. 必要时改写未来 `PhasePlan` 条目
5. 最终结果包写入 `scope_revision_summary`

### 7.4 退化与取舍
1. 这类变更不需要边界确认
2. 但它仍然不能靠 kernel 从自然语言自己猜

## 9. Example G: boundary candidate
### 9.1 用户原话
```text
另外再开始一个任务：把控制台的通知系统也重做一遍。
```

### 9.2 HostSemanticBundle
```json
{
  "schema_version": { "major": 0, "minor": 1 },
  "host_session_id": "sess-001",
  "host_message_ref": "msg-060",
  "managed_task_ref": "task-001",
  "input_ref": "msg-060",
  "source_model_ref": "host-main-model",
  "issued_at": "2026-03-10T10:30:00+08:00",
  "decisions": [
    {
      "decision_kind": "interaction_lane",
      "decision_source": "host_model",
      "confidence": 0.91,
      "rationale": "当前已有 active task，这条输入被识别成第二项新的 managed task 候选。",
      "payload": {
        "interaction_lane": "managed_task_active",
        "managed_task_ref": "task-001"
      }
    },
    {
      "decision_kind": "task_change",
      "decision_source": "host_model",
      "confidence": 0.84,
      "rationale": "这不是原任务补充，而是新的受管任务边界候选。",
      "payload": {
        "classification": "boundary_conflict_candidate",
        "execution_surface": "future_only",
        "boundary_recommendation": "open_boundary_confirmation"
      }
    }
  ]
}
```

### 9.3 Kernel 预期动作
1. 不吸收成 `request_task_change`
2. adapter 不得从这条 `task_change` judgment 自动脑补 `request_task_change`
3. 先打开 `PendingDecisionWindow(kind=BoundaryConfirmation)`
4. 再创建 `pendingBoundaryConfirmation`
5. 第一条用户可见输出必须是边界确认卡

### 9.4 退化与取舍
1. 这里最容易出错的是把第二个任务静默吞进当前 scope
2. v0 应偏保守，宁可多一次确认，也不要把第二任务吞进去

## 10. Example H: low-confidence fail-closed
### 10.1 用户原话
```text
这个事情你看着安排吧，顺便帮我推进一下。
```

### 10.2 HostSemanticBundle
```json
{
  "schema_version": { "major": 0, "minor": 1 },
  "host_session_id": "sess-001",
  "host_message_ref": "msg-070",
  "input_ref": "msg-070",
  "source_model_ref": "host-main-model",
  "issued_at": "2026-03-10T10:35:00+08:00",
  "decisions": [
    {
      "decision_kind": "interaction_lane",
      "decision_source": "host_model",
      "confidence": 0.42,
      "rationale": "语义过于模糊，无法稳定区分闲聊、轻量委托还是正式任务。",
      "payload": { "interaction_lane": "managed_task_candidate" }
    }
  ]
}
```

### 10.3 Kernel 预期动作
1. 不推进 managed lane
2. adapter 允许一次自动重判
3. 第二次仍低置信度则 fail closed
4. 用户层退化成保守澄清文本

### 10.4 退化与取舍
1. 这里不能因为“看着安排吧”像任务，就硬进 candidate
2. 低置信度场景下，`chat` 与澄清优先级高于任务治理

## 11. Example I: horizon reconsideration
### 11.1 用户原话
```text
你把这个判成 disruption 不对，它其实只是对现有流程做稳定扩展，请重新判断。
```

### 11.2 HostSemanticBundle
这类输入前提是：
1. 当前已有 `managed_task_ref`
2. 用户显式发起 `request_horizon_reconsideration`

当前已经冻结的边界只有三条：
1. 用户入口先形成正式 `control_action(action_kind=request_horizon_reconsideration)`
2. 宿主语义层随后重新产出新的 `work_horizon` judgment
3. 如果判断改变，kernel 留下 `control_action_received -> work_horizon_revised`

当前切片尚未冻结 `request_horizon_reconsideration` 的正式 paired bundle 形状，  
因此这里不提供一个可直接照抄的完整 `HostSemanticBundle`，  
只展示“重新判断后的 `work_horizon` decision fragment”。

```json
{
  "decision_kind": "work_horizon",
  "decision_source": "system_reconsideration",
  "confidence": 0.83,
  "rationale": "经重新审视，这更像 extension，而不是 disruption。",
  "payload": { "work_horizon": "extension" }
}
```

### 11.3 Kernel 预期动作
1. 更新 `work_horizon`
2. 记录 `work_horizon_revised`
3. 若影响预算/风险/验收，通知用户

### 11.4 退化与取舍
1. 用户不是直接改值
2. `control_action` 是“要求重审”的正式动作 owner
3. `work_horizon` 是“重审结果是什么”的语义 owner
4. 在 paired bundle 未冻结前，文档不应给出看似可直接复用的完整 wire 样例

## 12. Example J: approve_start as host control judgment
### 12.1 用户原话
```text
继续，按这个方案开始做。
```

### 12.2 HostSemanticBundle
```json
{
  "schema_version": { "major": 0, "minor": 1 },
  "host_session_id": "sess-001",
  "host_message_ref": "msg-090",
  "managed_task_ref": "task-001",
  "input_ref": "msg-090",
  "source_model_ref": "host-main-model",
  "issued_at": "2026-03-10T10:45:00+08:00",
  "decisions": [
    {
      "decision_kind": "control_action",
      "decision_source": "host_model",
      "confidence": 0.95,
      "rationale": "这条回复是在 candidate 上明确批准开始。",
      "payload": {
        "action_kind": "approve_start",
        "managed_task_ref": "task-001",
        "decision_token": "start-win-001"
      }
    }
  ]
}
```

### 12.3 Adapter 归一化
1. 不生成新的 `SemanticDecisionEnvelope`
2. 直接映射成 `ControlAction::ApproveStart`
3. 保留 `decision_token=start-win-001`

### 12.4 Kernel 预期动作
1. 消费正式 `ControlAction`
2. 从 candidate 进入 active
3. 不要求 adapter 从自然语言文本里猜“继续”是什么意思
4. token 失效或不匹配时，必须拒绝消费

### 12.5 退化与取舍
1. 这里如果缺 `control_action` judgement，adapter 不得自行把“继续”猜成 `approve_start`
2. 这里如果缺 `decision_token`，adapter 也不得继续把它当成有效批准
3. 应先请求宿主补判，或保守退化
