# 当前轮输入合同

状态：contract draft  
定位：`CurrentTurnEnvelope` 正式规格  
更新时间：2026-03-10

---

## 1. 目的
`CurrentTurnEnvelope` 定义 adapter 归一化后交给 Loom 的正式入站事实对象。

它回答：
1. 当前到底是哪条宿主输入进入了 Loom
2. 这条输入来自哪个宿主会话、哪个宿主消息、哪个 actor
3. 当前输入和后续 `HostSemanticBundle / SemanticDecisionEnvelope` 的追溯关系如何绑定
4. 哪些信息属于宿主 transport，哪些才属于 Loom 正式领域对象

更深层上，它解决的是：

**把“宿主原始消息事实”从 host hook transport 中抽出来，收成 Loom 可稳定消费的一层正式入站对象。**

---

## 2. 设计边界
### 2.1 这不是 `HostInboundTurn`
边界：
1. `HostInboundTurn`
   - 是具体宿主 adapter 在 hook 边界捕获到的 transport 载体
   - 它带 `IngressMeta`
   - 它首先服务幂等、重放和宿主事实采集
2. `CurrentTurnEnvelope`
   - 是 adapter 归一化后交给 Loom 的正式入站对象
   - 它回答“当前输入是谁”

固定规则：
1. `HostInboundTurn` 不得和 `CurrentTurnEnvelope` 并列成为两份正式 owner
2. Loom 正式消费的是 `CurrentTurnEnvelope`
3. `HostInboundTurn` 只允许停留在宿主接入与 bridge transport 层

### 2.2 这不是 `HostSemanticBundle`
边界：
1. `CurrentTurnEnvelope`
   - 描述“发生了哪条宿主输入事实”
2. `HostSemanticBundle`
   - 描述“宿主怎样理解这条输入”

固定规则：
1. 一条 `CurrentTurnEnvelope`
   - 可以对应零条、一条或多条后续语义判断
2. 语义判断缺失时
   - 当前输入事实仍然成立

### 2.3 这不是 `managedTaskRef`
边界：
1. `CurrentTurnEnvelope`
   - 是输入 owner
2. `managedTaskRef`
   - 是受管任务 owner

固定规则：
1. `host_session_id` 和 `managedTaskRef` 必须继续分离
2. 某条输入是否命中某个任务
   - 由后续 `SemanticDecisionEnvelope / ControlAction`
     绑定
   - 不能由 `CurrentTurnEnvelope` 自己越界决定

---

## 3. v0 固定规则
1. 每条进入 Loom 主链的宿主消息，都必须先收成一条 `CurrentTurnEnvelope`
2. adapter 必须先完成幂等、去重、重放保护，再归一化出 `CurrentTurnEnvelope`
3. `CurrentTurnEnvelope`
   - 只记录输入事实
   - 不携带 lane/class/change 等治理判断
4. 同一条 `CurrentTurnEnvelope`
   - 可以触发：
   - `SemanticDecisionEnvelope`
   - `ControlAction`
   - 或仅停留在聊天事实记录
5. `content`
   - 是可追溯输入原文
   - 不是 kernel 直接拿来做语义推断的依据

---

## 4. 正式对象定义
```rust
pub struct CurrentTurnEnvelope {
    pub turn_id: CurrentTurnId,
    pub host_session_id: HostSessionId,
    pub host_message_id: HostMessageId,
    pub host_message_ref: Option<HostMessageRef>,
    pub actor_ref: HostActorRef,
    pub content: String,
    pub message_kind: HostMessageKind,
    pub host_context: HostContextSnapshot,
    pub received_at: Timestamp,
}
```

---

## 5. 关键字段分别代表什么
1. `turn_id`
   - 当前这条正式入站事实的稳定 id
2. `host_session_id`
   - 这条输入来自哪个宿主聊天容器
3. `host_message_id`
   - 宿主侧原始消息 id
4. `host_message_ref`
   - 用于跨对象追溯时的稳定引用
5. `actor_ref`
   - 当前输入由谁发出
6. `content`
   - 原始输入内容
7. `message_kind`
   - 用户消息、系统消息或其它宿主消息种类
8. `host_context`
   - 当前宿主上下文快照

---

## 6. 和其它对象的关系
### 6.1 和 `HostInboundTurn`
固定关系：
1. OpenClaw 等宿主先产出 `HostInboundTurn`
2. adapter 在去重、追踪和字段对齐后
   - 归一化成 `CurrentTurnEnvelope`
3. Loom 不直接把 `HostInboundTurn` 当正式领域对象消费

### 6.2 和 `HostSemanticBundle`
固定关系：
1. `CurrentTurnEnvelope`
   - 先落输入事实
2. `HostSemanticBundle`
   - 再补宿主理解结果
3. 两者都可以通过 `host_message_ref`
   - 追溯回同一条宿主消息

### 6.3 和 `SemanticDecisionEnvelope`
固定关系：
1. `SemanticDecisionEnvelope`
   - 必须能追溯到它针对的是哪条输入
2. `CurrentTurnEnvelope`
   - 不负责 lane/class/change 的正式判断

---

## 7. 我的建议
1. `CurrentTurnEnvelope` 应成为 adapter 到 Loom 的正式入站对象，不再继续使用 `AdapterInboundMessage` 之类并行名字
2. `HostInboundTurn` 只保留宿主 transport 角色，不再上升成 kernel 正式输入对象
3. 实现时先记录 `CurrentTurnEnvelope`，再消费语义判断，才能保证 event chain 和任务治理链的追溯稳定
