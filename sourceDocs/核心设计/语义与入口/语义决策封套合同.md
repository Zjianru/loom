# 语义决策封套合同

状态：contract draft  
定位：`SemanticDecisionEnvelope` 正式规格  
更新时间：2026-03-11

---

## 1. 目的
`SemanticDecisionEnvelope` 定义 adapter 归一化后向 Loom 提交的结构化语义判断。

它回答：
1. 哪些语义判断由宿主大模型负责
2. adapter 应把宿主综合语义包归一化成什么结构再提交给 kernel
3. kernel 应该如何消费这些判断
4. 缺字段、低置信度或超出 schema 的判断如何处理

更深层上，它解决的是：

**让内核不再猜自然语言，只消费宿主已经做好的结构化语义结果。**

---

## 2. 核心立场
### 2.1 kernel 不做语义推断
这次已经锁定：
1. 任何关于语义的推断
2. 例如：
   - 这是新任务还是旧任务补充
   - 这是 chat 还是 managed task candidate
   - 这是 `COMPLEX / HUGE / MAX`
   - 这是 `maintenance / extension`
   - 这是普通 change 还是边界冲突
3. 都不由 kernel 自己判断

kernel 只做：
1. 结构校验
2. 枚举校验
3. 治理执行
4. 留痕与投影

### 2.2 宿主语义层负责解释自然语言
宿主语义层可以是：
1. OpenClaw 主 assistant 的大模型
2. 未来其它宿主的语义前置层

它负责把自然语言、上下文和当前任务状态理解成正式结构化判断。

### 2.3 kernel 只接受 schema 限定值，不接受自由语义文本
这条边界非常关键。

允许：
1. bounded enum
2. 结构化 payload
3. rationale
4. confidence

不允许：
1. “大概是类似 extension 的事情”这类自由描述直接驱动治理分支

---

## 3. 为什么必须先补
如果不把这份 contract 写清楚，会出现两个高风险冲突：
1. 宿主模型判断一套
2. kernel 再自己猜一套

最终就会形成：
1. 双语义真相源
2. 双治理分支
3. UI 和 runtime 状态不一致

---

## 4. v0 固定规则
### 4.1 语义判断由宿主生成，内核不生成
v0 固定规则：
1. kernel 不直接从原始自然语言生成治理分类
2. 宿主先提交 `HostSemanticBundle`
3. adapter 再把它归一化成：
   - `SemanticDecisionEnvelope`
   - 或 `ControlAction`

### 4.2 缺关键字段时 fail closed
我的建议：
1. 缺 `decision_kind`
2. 缺 `decision_payload`
3. 缺 `confidence`
4. 缺 `source_model_ref`
5. 或 payload 超出 schema

都不允许推进关键治理动作。

### 4.3 低置信度时走保守路径
我的建议：
1. 高置信度
   - 正常消费
2. 中等置信度
   - 允许保守消费
3. 低置信度
   - 不做关键治理推进
   - 请求重判、澄清或进入更保守的确认流

### 4.4 每个语义判断都必须能追溯到输入引用
原因：
1. 后面 `TaskScopeSnapshot`、`ChangeImpactAssessment`、`ResultContract` 都会依赖它

## 5. 正式对象定义
### 5.1 Rust 形态建议
```rust
pub struct SemanticDecisionEnvelope {
    pub envelope_id: SemanticDecisionEnvelopeId,
    pub host_session_id: HostSessionId,
    pub host_message_ref: Option<HostMessageRef>,
    pub managed_task_ref: Option<ManagedTaskRef>,
    pub decision_kind: SemanticDecisionKind,
    pub decision_payload: SemanticDecisionPayload,
    pub rationale: String,
    pub confidence: ConfidenceScore,
    pub source_model_ref: HostModelRef,
    pub issued_at: Timestamp,
}
```

### 5.2 `SemanticDecisionKind`
建议值：
1. `interaction_lane`
2. `task_activation_reason`
3. `managed_task_class`
4. `work_horizon`
5. `task_change`

### 5.3 `SemanticDecisionPayload`
建议形态：
```rust
pub enum SemanticDecisionPayload {
    InteractionLane(InteractionLaneDecision),
    TaskActivationReason(TaskActivationReasonDecision),
    ManagedTaskClass(ManagedTaskClassDecision),
    WorkHorizon(WorkHorizonDecision),
    TaskChange(TaskChangeDecision),
}
```

## 6. 典型 payload contract
### 6.1 `InteractionLaneDecision`
它是什么：
1. 当前输入应留在 chat，还是进入 managed task 路径

建议值：
1. `chat`
2. `managed_task_candidate`
3. `managed_task_active`

### 6.2 `TaskActivationReasonDecision`
它是什么：
1. 这条输入为什么要从 `chat lane` 升级成 managed lane

建议值：
1. `explicit_start_task`
2. `explicit_track_this`
3. `delegate_heavy_work`
4. `heavy_multi_stage_goal`

### 6.3 `ManagedTaskClassDecision`
它是什么：
1. 当前受管任务应采用什么协作规模

建议值：
1. `complex`
2. `huge`
3. `max`

固定接缝：
1. envelope 的 wire 值继续使用小写：
   - `complex / huge / max`
2. kernel 内部 domain enum 与顶层规范继续使用 Loom 正式档位：
   - `COMPLEX / HUGE / MAX`

### 6.4 `WorkHorizonDecision`
它是什么：
1. 当前任务或阶段的投资意图

建议值：
1. `maintenance`
2. `improvement`
3. `extension`
4. `disruption`

### 6.5 `TaskChangeDecision`
它是什么：
1. 当前变更请求在语义上属于哪类 change
2. 影响到了哪一类执行面
3. 宿主建议内核如何走治理分支

固定边界：
1. `TaskChangeDecision`
   - 只承载治理判断
2. 它不承载这次变更的 patch 内容
3. `summary / expected_outcome / requirement_items / allowed_roots / workspace_ref / repo_ref`
   - 不属于这份 envelope

建议字段：
```rust
pub struct TaskChangeDecision {
    pub classification: TaskChangeClassification,
    pub execution_surface: ChangeExecutionSurface,
    pub boundary_recommendation: BoundaryRecommendation,
}
```

其中：
1. `TaskChangeClassification`
   - `same_task_minor`
   - `same_task_material`
   - `same_task_structural`
   - `boundary_conflict_candidate`
2. `ChangeExecutionSurface`
   - `future_only`
   - `active_stage`
   - `completed_scope`
3. `BoundaryRecommendation`
   - `absorb_change`
   - `require_confirmation`
   - `open_boundary_confirmation`

### 6.6 和 `request_task_change` 的固定关系
1. `TaskChangeDecision`
   - 只回答“怎么判这次变更”
2. 正式推进 `request_task_change`
   - 还必须有同 bundle 的 `control_action(request_task_change)`
3. 这次具体请求修改的 patch 内容
   - 留在 `control_action.payload`
4. adapter 不得从 `TaskChangeDecision` 自己脑补正式 `request_task_change`

## 7. 和其它对象的边界
### 7.1 和 `AdapterContract`
`AdapterContract` 负责：
1. 把宿主语义判断结果传给 kernel

`SemanticDecisionEnvelope` 负责：
1. 这些判断结果的正式结构

补充边界：
1. `control_action` judgment 不进入 `SemanticDecisionEnvelope`
2. 它应由 adapter 直接转成正式 `ControlAction`
3. `request_task_change`
   - 不能只靠 `TaskChangeDecision` 单独推进

### 7.2 和 `TaskScopeContract`
`TaskScopeContract` 不自己理解自然语言。  
它只消费已经结构化好的 change / scope 判断。

### 7.3 和 `ChangeImpactAssessmentContract`
`ChangeImpactAssessmentContract` 不负责理解这是不是“同一个任务”。  
它只负责根据已给出的结构化 change judgement 进行治理影响映射。

固定边界：
1. `ChangeImpactAssessment`
   - 是 Loom 在消费 `TaskChangeDecision` 后生成的治理对象
2. 它不是宿主直接提供的 `SemanticDecisionEnvelope`
3. adapter 不得伪造 `change_impact_classification` 这类内核治理结论

---

## 8. 我的建议
### 8.1 语义判断与治理执行必须分层
这是这次最重要的设计收口。

我的建议：
1. 宿主负责理解
2. kernel 负责治理
3. 中间只通过 `SemanticDecisionEnvelope` 传递 bounded schema

### 8.2 kernel 不是语言模型替身
kernel 的价值不在于再做一层语义猜测。  
它的价值在于：
1. 状态
2. 编排
3. 审计
4. 边界
5. 验收
6. 通知

所以我的建议：
1. v0 就把这条边界写死
2. 后续所有 runtime 分支都围绕它实现
