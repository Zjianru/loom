# 技术选型

状态：architecture selection draft  
定位：Loom v0 的正式技术选型草案  
更新时间：2026-03-10

---

## 1. 目的
这份文档专门回答 3 类问题：
1. Loom v0 的进程形态、部署边界和宿主接入方式怎么选
2. Loom v0 的状态、事件、投影、runtime 根如何设计
3. 为什么这些选型比其他候选更适合当前阶段

它不是产品愿景，也不是详细接口说明。  
它是：

**把“Loom 应该怎么实现”收成一套可评审、可开工的技术选型决议。**

---

## 2. 当前背景
前面的设计已经锁定了几个不会再退回去的前提：
1. Loom 不做自然语言语义推断
2. 宿主主模型或宿主语义层先产出 `HostSemanticBundle`
3. `loom-openclaw` 只做宿主映射、归一化、文本渲染和执行桥接
4. Loom 自己持有：
   - `managedTaskRef`
   - `activeManagedTaskRef`
   - `PhasePlan`
   - `ExecutionAuthorization`
   - `AcceptancePolicy`
   - `TaskEventModel`
5. `runtime/loom/` 是 authoritative truth
6. 宿主兼容投影只保留 compatibility projection

这意味着这轮技术选型真正要解决的，不再是“产品要不要这样做”，而是：

**怎么让 Loom 在 OpenClaw 上先落地，同时不被旧运行时投影重新绑回去。**

这里还要补一条约束：
1. pack 默认行为与结果渲染不由实现层自由发明
2. 它们必须分别受：
   - [编码工作模式预设.md](../工作模式与结果/编码工作模式预设.md)
   - [研究工作模式预设.md](../工作模式与结果/研究工作模式预设.md)
   - [结果样例.md](../工作模式与结果/结果样例.md)
   约束

这里同时吸收 [设计对齐纪要.md](../../研究与讨论/设计对齐纪要.md) 的正式结论：
1. 我们卖的是治理能力层，不是模型平台
2. 技术选型必须服务任务治理 owner 清晰，而不是服务宿主 glue 层方便

---

## 3. 选型总览
### 3.1 v0 推荐总方案
我建议 v0 采用这一组组合：
1. Loom 内核语言：`Rust`
2. 宿主 adapter：`TypeScript`
3. 进程形态：**逻辑上 sidecar daemon-ready，工程上 bridge-first**
4. 内核存储：`SQLite + append-only task events + runtime projections`
5. 宿主桥协议：**结构化协议优先，不解析自由文本**
6. OpenClaw 语义入口：**专门结构化 tool/result 优先**
7. 第一条代码 spike：`COMPLEX` 最小闭环
8. 代码 spike 的具体 bridge：`LocalHttpBridge`

### 3.2 为什么是这一组
因为它同时满足 4 个目标：
1. 不把 Loom 重新做回 OpenClaw 插件内部状态机
2. 不为了 daemon 理想形态，过早把 v0 拖进复杂部署
3. 不为了快接入而牺牲 authoritative truth
4. 不让宿主 glue 层重新长出治理引擎

## 4. 进程与部署选型
### 4.1 候选方案
#### A. 完全嵌入 OpenClaw 插件
优点：
1. 最快落地
2. 不需要额外 bridge

缺点：
1. Loom 很容易重新变成 OpenClaw 私有 runtime
2. `managedTaskRef`、`PhasePlan`、`ExecutionAuthorization` 会再次被宿主会话上下文污染
3. 以后接第二个宿主时要重拆一次

#### B. 从第一天就是独立 daemon
优点：
1. 架构最干净
2. 最符合“通用内核 + adapter”

缺点：
1. v0 成本高
2. transport、生命周期、恢复、日志、调试都要一起解决

#### C. bridge-first，逻辑按 daemon-ready 设计
优点：
1. owner 边界一开始就对
2. v0 不被部署细节拖死
3. 后续平滑升级到 sidecar daemon

缺点：
1. 需要 discipline，防止桥接层重新偷拿 owner

### 4.2 推荐结论
推荐 `C`。

这句话的真实含义是：
1. Loom 在架构上按独立内核设计
2. `loom-openclaw` 只按宿主桥接层设计
3. 物理部署上，v0 允许先用最容易调试和最容易开工的 bridge

取舍：
1. 不先拍死 HTTP / socket / in-process
2. 但必须先把 owner 写死
3. 代码 spike 虽然先选 `LocalHttpBridge`
4. 但它必须至少满足：
   - 只绑定 loopback
   - 使用 adapter<->Loom 的 shared bridge secret
   - 不接受匿名本地请求

## 5. 语言与模块选型
### 5.1 为什么 Loom 用 Rust
Rust 在这里不是为了追求“更快”，而是为了支撑 Loom 的几类正式对象：
1. `managedTaskRef`
   - 单个受管任务聚合根
2. `PhasePlan`
   - 正式阶段计划
3. `ExecutionAuthorization`
   - 本轮真实授权租约
4. `TaskEventModel`
   - 全运行线共享事件链

这些对象都要求：
1. 强状态建模
2. 并发安全
3. 长生命周期稳定性
4. 清晰的模块边界

Rust 正适合：
1. 把 lane/class/policy/action 表达成强类型 enum
2. 把 store / events / bridge / approval 拆成稳定模块
3. 避免长期运行 runtime 的低级内存问题

### 5.2 为什么 adapter 继续留 TypeScript
因为 `loom-openclaw` 的核心职责是：
1. 接 OpenClaw hook
2. 读宿主能力
3. 捕获 `HostSemanticBundle`
4. 文本渲染
5. 工具和 subagent 执行桥接

这些都更贴 OpenClaw 当前生态。  
所以 v0 最稳的切法是：
1. Loom：Rust
2. `loom-openclaw`：TypeScript

取舍：
1. 不追求第一天全 Rust
2. 先把治理真相和宿主 glue 分开

## 6. 状态与存储选型
### 6.1 候选方案
#### A. 全内存 + 文件快照
优点：
1. 最简单

缺点：
1. 恢复差
2. 事件回放弱
3. 后续 recorder/watchdog/audit 不稳定

#### B. 纯 event sourcing
优点：
1. 历史最完整

缺点：
1. v0 太重
2. projection、调试、查询复杂

#### C. SQLite + append-only events + projection
优点：
1. 足够强
2. 易调试
3. 易恢复
4. 适合本地 sidecar/daemon 形态

缺点：
1. 仍需要认真设计 schema 和 projection

### 6.2 推荐结论
推荐 `C`。

建议组合：
1. `runtime/loom/events/`
   - 从 authoritative event store 导出的调试/导出投影
2. SQLite
   - authoritative event/store/query persistence
3. `runtime/loom/projections/`
   - UI、debug、compat 查询友好的读模型
4. `runtime/loom/tasks/`
   - 任务聚合态只读投影
5. `runtime/loom/runs/`
   - `IsolatedTaskRun` 的只读投影
6. durable outbound outbox
   - authoritative store 内的宿主投递队列

对应正式 contract：
1. outbound payload 结构：
   [内核出站载荷合同.md](../宿主接入/内核出站载荷合同.md)
2. durable outbox 生命周期：
   [出站投递生命周期合同.md](../宿主接入/出站投递生命周期合同.md)
3. `SpecBundle`
   - [规格包合同.md](../任务与协作/规格包合同.md)
4. `ProofOfWorkBundle`
   - [工作证明合同.md](../工作模式与结果/工作证明合同.md)
5. `IsolatedTaskRun`
   - [隔离任务运行合同.md](../任务与协作/隔离任务运行合同.md)

更深层的取舍：
1. authoritative truth 不放在 projection 文件里
2. projection 文件是为了让人和宿主更容易读
3. 真正的“这次任务当前状态是什么”，由 Loom store 决定
4. outbound delivery queue 也应由同一 authoritative store 持有，不单独再造第二套文件真相源

## 7. 事件与投影模型
### 7.1 正式建议
采用：
1. 单一 `TaskEventModel`
2. 任务聚合态 projection
3. notice projection
4. host-bridge projection
5. durable outbound outbox

### 7.2 为什么必须统一事件链
因为这 4 条运行线都要读同一条事实：
1. Harness
2. watchdog
3. approval-gate
4. recorder

如果它们各读各的 store，会重新出现旧运行时分裂问题：
1. follower 插件读 projection 反向把 projection 当真相源
2. session 语义和 task 语义重新缠在一起
3. outbound 先发出、状态后提交，导致宿主看到不存在于 authoritative truth 的卡片

这里再补一条：
1. review 不能只停留在阶段名
2. 最小 spike 里的 review 产物统一以
   [评审结果合同.md](../工作模式与结果/评审结果合同.md)
   为准
3. 结果层里的正式证据包统一以
   [工作证明合同.md](../工作模式与结果/工作证明合同.md)
   为准

---

## 8. 宿主语义入口选型
### 8.1 候选方案
#### A. 自由文本 + adapter 解析
优点：
1. 快

缺点：
1. 直接违反“Loom 不做语义推断”
2. adapter 会重新长出语义引擎

#### B. 消息 metadata / sidecar payload
优点：
1. 比文本解析稳

缺点：
1. 宿主实现可能漂移
2. 隐式性较强

#### C. 专门结构化 tool/result
优点：
1. 边界最清楚
2. fail-closed 最稳
3. 最适合变成测试夹具和协议契约

缺点：
1. 宿主要显式配合产出

### 8.2 推荐结论
推荐 `C`，`B` 仅作桥接备选。

这里几个对象的关系是：
1. `HostSemanticBundle`
   - 宿主主模型输出的综合判断包
2. `SemanticDecisionEnvelope`
   - `loom-openclaw` 归一化后的 bounded judgment
3. `DecisionSource`
   - 表示判断来自：
     - `host_model`
     - `pack_default`
     - `system_reconsideration`
     - `user_control_action`
     - `adapter_fallback`
4. `schema_version`
   - 表示宿主语义协议版本

推荐策略：
1. 必填 decision 缺失 => fail closed
2. 低置信度 => 一次自动重判
3. 仍不够 => 保守退化

这条语义入口下游再补两条：
1. 宿主能力变化后的重授权，统一以
   [能力漂移与重授权合同.md](../治理策略/能力漂移与重授权合同.md)
   为准
2. 宿主出站卡片和结果摘要的 payload，统一以
   [内核出站载荷合同.md](../宿主接入/内核出站载荷合同.md)
   为准

## 9. OpenClaw adapter 选型
### 9.1 为什么叫 `loom-openclaw`
因为它的身份应该很短、很清楚：
1. 它是 Loom 的 OpenClaw 宿主插件
2. 它不是产品本体
3. 它不是治理真相源

### 9.2 `loom-openclaw` 的正式职责
它负责：
1. `message_received` ingress
2. `HostSemanticBundle` ingress
3. `ControlAction` ingress
4. `HostToolIntent / HostToolResult` bridge
5. `HostSubagentLifecycleEvent` bridge
6. Loom outbound 文本化渲染
7. legacy 宿主兼容投影

它不负责：
1. 决定 `interactionLane`
2. 决定 `managedTaskClass`
3. 决定 `PhasePlan`
4. 决定 `ExecutionAuthorization`
5. 解释 `AcceptancePolicy`

## 10. Loom 运行时模块选型
### 10.1 推荐模块
1. `loom-domain`
2. `loom-task`
3. `loom-phase`
4. `loom-harness`
5. `loom-policy`
6. `loom-approval`
7. `loom-watchdog`
8. `loom-events`
9. `loom-store`
10. `loom-bridge`

### 10.2 关键取舍
#### `loom-harness`
它承接 Loom 的任务治理主链，但边界更清楚：
1. 读取 `interactionLane`
2. 读取 `managedTaskClass`
3. 读取 `PhasePlan`
4. 建立 `AgentBinding`
5. 推进阶段与任务

它不该再：
1. 直接拿宿主 session 文件当真相源
2. 自己做自然语言判断

#### `loom-approval`
这里要分清两个变量：
1. `DelegationLevel`
   - 抽象授权语义
2. `ExecutionAuthorization`
   - 本轮真实授权结果

我的建议是：
1. 用户可配置和 pack 默认先落到 `DelegationLevel`
2. 真正执行前必须收敛成 `ExecutionAuthorization`
3. `approval-gate` 最终只执法后者

#### `loom-watchdog`
它不是普通通知器，而是流动治理器。  
它要读：
1. `WipPolicy`
2. `BudgetPolicy`
3. `ReworkPolicy`
4. `TaskEventModel`

这意味着它天然依赖统一事件链和统一 projection。

## 11. 第一条代码 spike 选型
### 11.1 为什么先做 `COMPLEX`
因为 `COMPLEX` 是 managed lane 的最小正式档位。  
它已经能验证：
1. `managedTaskRef`
2. `StartCard`
3. `ApproveStart`
4. Harness binding
5. 最小 `review` 阶段
6. recorder summary
7. ResultSummary 文本化

但又不会立刻把系统拖进：
1. 多阶段重排
2. 多 reviewer 组
3. 大规模 watchdog cadence

### 11.2 建议最小角色
1. `net`
2. `recorder`
3. `1 worker`

### 11.3 建议最小阶段
1. `clarify`
2. `execute`
3. `review`
4. `deliver`

## 12. 安全与失败策略
### 12.1 Fail closed 优先
这轮最重要的安全取舍不是某个加密或 ACL 细节，而是：

**缺结构化语义、缺关键字段、schema 不兼容、置信度过低时，不偷偷推进治理分支。**

### 12.2 为什么这比“先跑通再说”更重要
因为 Loom 的卖点不是“偶尔猜对”，而是：
1. 边界稳定
2. 任务和会话 owner 分离
3. 写边界可靠
4. 用户可以把复杂任务扔出去而不担心系统乱跑

---

## 13. v0 非目标
v0 不建议现在就做：
1. 多宿主并发支持
2. Postgres / 云端多租户存储
3. gRPC-first
4. pack marketplace
5. `HUGE / MAX` 的完整代码实现
6. 自定义卡片 UI

原因不是这些不重要，而是：
1. 这些都建立在 owner 已经切干净的前提上
2. 现在更值钱的是先证明 Loom 边界是对的

---

## 14. 我的建议
### 14.1 v0 现在就该定死的
1. Loom 用 Rust
2. `loom-openclaw` 用 TypeScript
3. `runtime/loom/` 是 authoritative truth
4. 宿主兼容投影只做 compatibility projection
5. `HostSemanticBundle` 走专门结构化通道
6. 所有 ingress 带幂等元数据
7. 第一条 spike 只做 `COMPLEX`
8. 代码 spike 先选 `LocalHttpBridge`
9. outbound 采用 authoritative outbox + ack，而不是瞬时内存发送
10. `LocalHttpBridge` 至少要求 loopback-only + shared bridge secret

### 14.2 v0 现在不要定死的
1. 最终一定是 HTTP 还是 Unix socket
2. 第一天一定是 daemon 还是 in-process bridge
3. `HUGE / MAX` 的完整执行拓扑

### 14.3 总结
最稳的技术路线不是：
1. 一上来就把 Loom 做成很大的平台

而是：
1. 先做对 owner
2. 先做对语义边界
3. 先做对 `runtime/loom/`
4. 先做对 `loom-openclaw`
5. 先用 `COMPLEX` 跑通最小真实闭环

这条路最慢的地方不是代码，而是边界。  
边界现在已经基本定住了，所以可以开始往真实代码压。
