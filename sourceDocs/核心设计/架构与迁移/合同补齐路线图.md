8. `HUGE / MAX` 的 pack-specific presets 与 result examples
9. `operations_pack` preset

---

## 4. 推荐顺序
我建议的顺序是：
1. 先用 `coding_pack / COMPLEX` 开始 Loom runtime 与 `loom-openclaw` 的真实代码 spike
2. 再把 `research_pack / COMPLEX` 作为第二个真实对照样本接进 spike
3. 在 spike 中同时落实：
   - `SpecBundle`
   - `ProofOfWorkBundle`
   - `IsolatedTaskRun`
4. 根据真实 spike 回写技术选型与 deployment 细节
5. 之后再细化 `HUGE / MAX` presets 与 result examples

---

## 5. 为什么现在可以转向代码 spike
原因不是“文档已经很多”，而是默认行为的 owner 已经被拿回来了。

现在运行时 owner、adapter 边界、migration cutline 已经写清。  
在 preset / example 已经落地之后，直接开始代码 spike 终于变得安全。之前最容易发生的两件事是：
1. 对象之间 owner 漂移
2. adapter 和 Harness 在实现层各自长出一套“方便的默认值”

现在这两层已经被这轮文档拿回来了：
1. 运行时最核心的治理 contract 已经收齐
2. pack preset 与 result example 已经写实
3. 下一步应该转向真实代码 spike，而不是继续补抽象对象
