# 宿主迁移切线

状态：migration draft  
定位：现有 OpenClaw TypeScript 插件到 Rust Loom 的 owner 切线  
更新时间：2026-03-10

---

## 1. 目的
这份文档回答：
1. 今天的 TypeScript runtime 哪些地方正在持有治理事实
2. v0 应该把哪些东西迁进 Rust kernel
3. 哪些东西暂时留在 OpenClaw adapter / TS glue 层
4. 哪些 runtime 文件只保留为兼容投影，不再充当真相源

更深层上，它解决的是：

**如何在不一次重写整个宿主执行层的前提下，切断双真相源。**

---

## 2. 当前 reality 的问题不在“没能力”，而在 owner 冲突
今天的现状不是系统不会治理，而是：
1. 旧治理插件
   - 持有当前任务与会话运行态
2. `approval-gate`
   - 通过 runtime session 文件读取授权真相
3. `watchdog`
   - 通过 runtime session 文件读取阶段真相

这造成两个更深层冲突：
1. 任务 truth 还是偏 session-scoped
2. follower 插件通过读 projection 文件反向把 projection 当真相源

这正好和我们现在的 to-be 相冲突：
1. task owner 应该是 `managedTaskRef`
2. session 只保留轻量 pointer

---

## 3. v0 cutline 原则
### 3.1 Rust Loom 负责治理真相
Rust Loom 要负责：
1. 任务对象
2. 阶段对象
3. 授权对象
4. 验收对象
5. 预算/WIP/返工对象
6. 统一事件链

### 3.2 `loom-openclaw` 负责宿主映射
`loom-openclaw` 要负责：
1. hook 收发
2. host identity
3. host capability discovery
4. 文本渲染
5. host tool/subagent execution bridge

### 3.3 compatibility projection 允许存在，但不再是 truth
v0 正式建议：
1. `runtime/loom/`
   - authoritative truth
2. 宿主兼容投影根
   - compatibility projection

其中 compatibility projection 仍可继续保留：
1. 宿主兼容任务投影
2. 宿主兼容输入投影
3. 宿主兼容内部 transcript 投影

但这些文件只能代表：
1. adapter / legacy follower 的兼容投影

不能再代表：
1. authoritative task truth

## 4. 当前对象到未来 owner 的映射
### 4.1 `currentTurnStore`
它是什么：
1. 当前 TS runtime 保存入站 turn 的结构化 store

今天的问题：
1. 它既像宿主 ingress journal
2. 又被部分治理逻辑拿来当事实源

v0 建议 owner：
1. adapter ingress journal

取舍：
1. kernel 只消费结构化 turn 输入
2. 不直接依赖 OpenClaw turn 文件格式

### 4.2 `sessionContextStore`
它是什么：
1. 当前 session-scoped runtime 上下文 store

今天的问题：
1. 它里面放了：
   - `workflowStage`
   - `pendingUserDecision`
   - `pendingBoundaryConfirmation`
   - `executionAuthorization`
2. 这些本来都属于 task-scoped 治理对象

v0 建议 owner：
1. kernel aggregate projection

兼容策略：
1. TS 侧只保留 projection writer
2. 不再保留最终解释权

### 4.3 `bindingStore`
它是什么：
1. 当前 alias / canonical / binding 恢复层

今天的问题：
1. host identity 和治理 identity 混在一起

v0 建议 owner 切分：
1. host alias / session 解析
   - 归 adapter
2. task-to-session pointer
   - 归 kernel projection

### 4.4 `taskRuntimeStore`
它是什么：
1. 当前 TS 里最接近任务 aggregate 的对象

v0 建议 owner：
1. 迁进 Rust `kernel-task`

原因：
1. 这层今天已经在承担 task close / state lifecycle
2. 它最不该继续留在 session-scoped TS runtime

### 4.5 `taskEventSink / compatProjectionWriter`
它是什么：
1. 当前 task event append 与治理事件文件写入入口

v0 建议 owner：
1. task event append 归 Rust `kernel-events`
2. TS 侧只保留兼容投影

### 4.6 `before_tool_call`
它是什么：
1. 当前工具调用治理前门

今天的问题：
1. 它既观察宿主工具意图
2. 又执行部分治理判断

v0 建议切法：
1. adapter
   - 观察 `HostToolIntent`
   - 转发给 kernel
2. kernel
   - 返回 `ToolDecisionPayload`
3. adapter
   - 再桥接真实 OpenClaw 工具调用

### 4.7 `before_message_write / message_sending`
它是什么：
1. 当前出站治理点

今天的问题：
1. 出站改写里混了 host transcript 处理和治理决策

v0 建议切法：
1. kernel
   - 产生结构化 outbound governance decision
2. adapter
   - 决定如何 suppress / render / persist compatibility projection

### 4.8 approval follower
它是什么：
1. approval-gate 当前通过 runtime session 文件读取授权状态的 follower

v0 建议：
1. 逐步废掉“从 session 文件读真相”的模式
2. 改成直接查询 kernel approval state / authorization lease

### 4.9 watchdog follower
它是什么：
1. watchdog 当前通过 runtime session 文件读取任务阶段

v0 建议：
1. 逐步废掉“从 session 文件追阶段”的模式
2. 改成直接消费 kernel event stream 和 watchdog projection

## 5. v0 分阶段迁移建议
### 5.1 Phase 0: Shadow Read
目标：
1. Rust kernel 先能消费 host ingress
2. 先不强执法
3. 先镜像 today reality

这阶段做什么：
1. 接 OpenClaw adapter
2. 接收结构化 inbound / tool / subagent lifecycle
3. 生成自己的 task aggregate 和 event log
4. 暂不替换现有 TS 执法逻辑

作用：
1. 验证 kernel 能不能正确重建 today reality

### 5.2 Phase 1: Truth-Source Cutover
目标：
1. kernel 成为治理真相源

这阶段做什么：
1. `managedTaskRef / PhasePlan / AcceptancePolicy / WipPolicy / ExecutionAuthorization`
   - 全部由 kernel 定
2. adapter 把 kernel 决策翻译回 OpenClaw hook 行为
3. TS runtime 文件只写 compatibility projection

这是 v0 最关键的一刀。

### 5.3 Phase 2: Follower Decoupling
目标：
1. 让 `watchdog` 和 `approval-gate` 不再读 legacy runtime session 文件

这阶段做什么：
1. watchdog 改读 kernel event/query
2. approval 改读 kernel authorization/query

### 5.4 Phase 3: Legacy Projection Optional
目标：
1. runtime session JSON 不再是系统必须组成部分

这阶段做什么：
1. 仅在 OpenClaw 兼容需要时才继续输出 projection

## 6. v0 不应该一次做完的东西
我不建议 v0 一步做完：
1. 全量 UI 重写
2. 全量宿主配置自动改写
3. 所有 TS 插件逻辑一次清空
4. 多宿主并行支持

原因：
1. 这会把“治理切线”问题淹没在“大迁移工程”里

---

## 7. 和你已定方向的关系
### 7.1 为什么这条 cutline 符合“只做治理，不做能力”
因为：
1. kernel 接管的是：
   - 阶段
   - agent 编排
   - 通知
   - 验收
   - 授权
2. 真正的能力执行仍然留给宿主 agent 和 worker agent

### 7.2 为什么这条 cutline 符合“OpenClaw 只是第一个 adapter”
因为：
1. host-specific 事实尽量留在 adapter
2. kernel 不直接吃 OpenClaw session file / config file 作为正式领域对象

---

## 8. 我的建议
### 8.1 v0 必须先切 owner，不是先切语言
很多团队会把迁移误解成：
1. 先把 TS 改成 Rust

但真正更深层的工作是：
1. 先把谁拥有真相源切清楚

不然只是换语言，不是换架构。

### 8.2 我的建议
1. v0 先完成 `Phase 0 + Phase 1`
2. 也就是：
   - kernel 能重建 today reality
   - kernel 开始做 authoritative governance
3. TS 侧暂时保留 adapter glue 和 compatibility projection

这样最符合你现在的目标：
1. 不重写整个世界
2. 但把治理真正从宿主插件里抽出来
