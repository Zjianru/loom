# 内核总设计

状态：kernel draft  
定位：Rust 内核草案，承接个人 AI 终端的协作治理能力  
更新时间：2026-03-10

---

## 1. 目标
这个内核不负责生产世界上最强的模型能力。  
它负责把现有 assistant 与更专业的 agent 组织成一个可持续工作的系统。

一句话目标：

**让个人 AI 终端获得可委派、可持续推进、可主动汇报、可受控写入、可验收、可沉淀的复杂任务治理能力。**

## 1.1 本轮吸收的固定结论
这份文档已经吸收 [设计对齐纪要.md](../../研究与讨论/设计对齐纪要.md) 的核心结论，固定为：
1. 缺的不是能力层，而是治理层
2. `Harness / watchdog / approval-gate` 是固定三条运行线
3. `StagePool / AgentPool` 是固定两类资源池
4. OpenClaw 只是第一宿主，不是唯一宿主

---

## 2. 为什么内核建议用 Rust
### 2.1 这是一个长生命周期、强状态、强并发的 runtime
这个系统天然会遇到：
1. 长时间运行
2. 多任务状态切换
3. 异步通知
4. 多 agent 并发
5. 授权边界
6. 持续事件流

这类系统不是简单脚本，更像 daemon / runtime / Harness。

### 2.2 Rust 的价值
Rust 在这里的价值不是“更快”优先，而是：
1. **内存安全**
   - 避免悬垂引用、越界、use-after-free 这类底层问题。
2. **并发安全**
   - 任务状态、通知、调度、持久化更适合用强类型方式表达。
3. **状态建模强**
   - `enum`、trait、module 边界很适合表达 lane、phase、policy、authorization。
4. **容易做 daemon**
   - 适合做长期运行的 kernel service。
5. **更容易做 capability-style boundary**
   - 可以把高风险操作权限限制在少量模块里。

### 2.3 取舍
Rust 不是没有代价：
1. 开发门槛比 TypeScript 高
2. 快速试错速度可能更慢
3. UI / adapter 侧往往还要配合别的语言

我的建议：
1. **kernel 用 Rust**
2. **adapter 可以用宿主生态常用语言**
3. **对外通过稳定协议通信，而不是把所有逻辑塞进宿主插件里**

## 3. 顶层分层
### 3.1 Kernel
它是什么：
1. 协作治理内核

它负责：
1. 消费宿主结构化语义判断并做治理分发
2. 任务对象
3. 阶段运行时
4. agent 编排
5. 通知治理
6. 边界授权
7. 验收与返工

### 3.2 Adapter
它是什么：
1. 宿主 assistant 的适配层

它负责：
1. 接收用户消息
2. 展示 start card / status / boundary card
3. 转发用户确认和变更
4. 接收任务进展和最终结果

OpenClaw 只是第一个 adapter，不应该是唯一 adapter。

### 3.3 Pack
它是什么：
1. 面向普通用户的预置工作模式包

它负责：
1. 提供阶段组合
2. 提供默认 agent roster
3. 提供验收策略
4. 提供通知策略

## 4. 内核的正式对象
### 4.1 `interactionLane`
它是什么：
1. 宿主语义层给出的顶层输入归类结果

它表示：
1. 当前输入继续按普通聊天处理
2. 还是进入 managed task 候选/活跃路径

建议实现：
1. Rust `enum`
2. adapter 把 `SemanticDecisionEnvelope::InteractionLane` 转交给 kernel
3. kernel 不自己从自然语言推断 `interactionLane`

### 4.2 `ManagedTaskRef`
它是什么：
1. 单个受管任务的稳定标识

它表示：
1. 任务卡、阶段、agent 绑定、通知、结果、关闭都归它

建议实现：
1. 独立 task aggregate id
2. 不复用宿主 session id

原因：
1. session 是聊天容器
2. task 是治理容器
3. 这两个 owner 必须分开

### 4.3 `ActiveManagedTaskRef`
它是什么：
1. 当前会话唯一活跃任务的指针

建议实现：
1. 它只是 pointer，不应该承担完整任务状态

原因：
1. active 约束和完整账本是两个层次

### 4.4 `ManagedTaskClass`
它是什么：
1. 宿主语义层给出的受管任务协作框架分类结果

正式值：
1. `COMPLEX`
2. `HUGE`
3. `MAX`

建议实现：
1. Rust `enum ManagedTaskClass`
2. 每个 class 绑定一套默认拓扑与策略
3. adapter 把 `SemanticDecisionEnvelope::ManagedTaskClass` 转交给 kernel

固定接缝：
1. 宿主 wire schema 可以继续使用：
   - `complex / huge / max`
2. adapter 负责把 wire 值归一化成 Loom 内部 domain enum
3. kernel 和顶层规范文档只使用：
   - `COMPLEX / HUGE / MAX`

### 4.5 `StagePackage`
它是什么：
1. 阶段模板

它表示：
1. 阶段做什么
2. 阶段需要什么输入
3. 阶段产出什么
4. 阶段能调用哪些 agent class
5. 阶段结束条件是什么

建议实现：
1. 数据驱动定义
2. 尽量少写死在代码里

### 4.6 `PhasePlan`
它是什么：
1. 某个任务选定的阶段组合

它表示：
1. 顺序
2. 依赖
3. 是否允许并行
4. 是否允许重试

### 4.7 `AgentCapabilityProfile`
它是什么：
1. agent 能力画像

它表示：
1. 专长
2. 模型
3. 工具权限
4. 成本
5. 风险
6. 适合哪些 stage

### 4.8 `AgentBinding`
它是什么：
1. 某次任务/阶段实际启用的 agent roster

它表示：
1. 谁是 worker
2. 谁是 reviewer
3. 谁是 recorder

### 4.9 `ExecutionAuthorization`
它是什么：
1. 当前任务/阶段被授予的执行权限集合

它表示：
1. 可以用哪些工具
2. 可以写哪里
3. 能否创建子 agent
4. 哪些动作必须人工确认

这条线应该继续由 `approval-gate` 负责守住。

### 4.10 `WatchdogNoticeState`
它是什么：
1. 异步通知状态

它表示：
1. 已通知事件
2. 待通知事件
3. 升级级别
4. 节流与去重信息

### 4.11 `AcceptancePolicy`
它是什么：
1. 验收策略

它表示：
1. 谁来 review
2. review 通过标准
3. 最多返工几轮
4. 何时升级给用户

这是“人退出微操，但保留治理权”的关键对象。

### 4.12 `SpecBundle`
它是什么：
1. 某个 `managedTaskRef` 当前正式执行锚点的文档组

它表示：
1. `scope_doc`
   - 当前任务范围、关键决策与边界
2. `plan_doc`
   - 阶段与任务拆解
3. `verification_doc`
   - 验证与验收清单

它的边界：
1. `TaskScopeSnapshot`
   - 回答“范围是什么”
2. `SpecBundle`
   - 回答“怎么把范围、计划、验证锚在活文档里”
3. `PhasePlan`
   - 回答“这次真正按什么阶段推进”

### 4.13 `ProofOfWorkBundle`
它是什么：
1. 任务交付时的正式证据包

它表示：
1. `artifact_manifest`
2. `review_summary`
3. `acceptance_verdict`
4. `evidence_refs`
5. `run_summary`

它的边界：
1. `ReviewResult`
   - 持有单次 review 事实
2. `AcceptancePolicy`
   - 持有完成判定规则
3. `ProofOfWorkBundle`
   - 把 review、验证、产物、关键证据打成正式证据包
4. `ResultContract`
   - 把最终范围、变更摘要、结果摘要、证据包一起交付

### 4.14 `IsolatedTaskRun`
它是什么：
1. 一个 `managedTaskRef` 在执行期对应的一次隔离受管 run

它表示：
1. 独立 run id
2. 独立 run boundary
3. 独立 run artifacts
4. 独立 run evidence
5. run completion / failure

它不等于：
1. `managedTaskRef`
   - task owner
2. `host_session_id`
   - 宿主聊天容器
3. `ExecutionAuthorization`
   - 当前 run 获得的能力租约
4. `OutboundDelivery`
   - 出站交互单元

## 5. 模块划分建议
### 5.1 `kernel-core`
负责：
1. 基础类型
2. task aggregate
3. lane / class / policy enum
4. 事件模型

### 5.2 `kernel-dispatch`
负责：
1. 消费 `SemanticDecisionEnvelope`
2. 基于结构化判断做治理分发
3. task 激活
4. 用户变更和控制入口

### 5.3 `kernel-stage-runtime`
负责：
1. 加载 `StagePackage`
2. 运行 `PhasePlan`
3. 推进阶段生命周期

### 5.4 `loom-harness`
负责：
1. 从 `AgentPool` 选人
2. 建立 `AgentBinding`
3. 调度 worker / recorder / reviewer
4. 生成和修订 `SpecBundle`
5. 创建和推进 `IsolatedTaskRun`
6. 汇总并编译 `ProofOfWorkBundle`

### 5.5 `kernel-approval`
负责：
1. `ExecutionAuthorization`
2. 写入和关键工具边界

### 5.6 `kernel-watchdog`
负责：
1. 主动通知
2. 升级策略
3. 去重和节流

### 5.7 `kernel-records`
负责：
1. task event log
2. stage summary
3. recorder 产物索引
4. 审计与回放基础

### 5.8 `kernel-adapter-api`
负责：
1. 为宿主提供稳定接口
2. 隔离 UI/host 变化

## 6. 数据流建议
### 6.1 任务启动
1. adapter 把用户输入发给 kernel
2. 宿主语义层先产出 `HostSemanticBundle`
3. adapter 把它归一化成一条或多条 `SemanticDecisionEnvelope`
4. `kernel-dispatch` 消费 `interactionLane / managedTaskClass / WorkHorizon`
5. 如果进入 managed lane，创建 `ManagedTaskRef`
6. 生成 start card
7. 用户确认后进入 active

### 6.2 阶段运行
1. `kernel-stage-runtime` 解析 `PhasePlan`
2. `loom-harness` 从 `AgentPool` 绑定 agent
3. worker 执行阶段任务
4. recorder 记录阶段过程与产出
5. `kernel-watchdog` 在关键节点主动通知
6. 如果发生写操作，由 `kernel-approval` 先做边界判断

### 6.3 验收与返工
1. review agents 根据 `AcceptancePolicy` 审核产出
2. 不通过则触发返工
3. 返工超限或高冲突时升级给用户
4. 通过后进入结果汇总

### 6.4 结果关闭
1. recorder 输出阶段沉淀与最终摘要
2. `net` 向用户做结果汇总
3. `ManagedTaskRef` 关闭
4. `ActiveManagedTaskRef` 释放

## 7. OpenClaw adapter 的现实意义
### 7.1 OpenClaw 为什么适合做第一个 adapter
因为它已经有：
1. chat-first terminal 形态
2. tool / hook / governance 基础
3. 多 agent 和 relay 基础
4. notification 与 approval 的现实土壤

### 7.2 但为什么不能把 kernel 写死在 OpenClaw 里
因为 OpenClaw 是宿主，不是未来全部产品。

如果把 kernel 写成：
1. OpenClaw 插件拼接物
2. 宿主 session 语义的附属物

会有两个风险：
1. 很难复用到别的个人 AI terminal
2. 很难保证 task owner 与 session owner 分离

我的建议：
1. kernel 独立
2. OpenClaw 通过 adapter 接入
3. 插件层逐步变成 adapter glue，而不是最终治理真相源

---

## 8. 为什么 recorder 要做成 agent，而不是插件
这是一个重要取舍。

### 8.1 如果 recorder 做插件
好处：
1. 更统一地拦截事件

问题：
1. 职责会和 watchdog、records、approval 发生重叠
2. 它会越来越像“又一个基础设施 plugin”
3. 很难表达 recorder 是一个工作角色

### 8.2 如果 recorder 做正式 agent role
好处：
1. 更贴近“把更具体的工作交给更有能力的 agent”
2. 天然适合进入 `AgentPool`
3. 可以按 stage 动态启用
4. 可以输出结构化文档和记忆资产

我的建议：
1. recorder 作为 `AgentPool` 中的 first-class role
2. kernel 负责在合适阶段自动绑定它

## 9. 安全与边界
### 9.1 基本原则
1. 编排能力和写入能力必须分离
2. 通知能力和记录能力必须分离
3. task owner 和 session owner 必须分离
4. agent roster 和实际执行授权必须分离

### 9.2 为什么 Rust 在这里重要
我们这里最怕的不是单一 bug，而是：
1. 长期运行后的状态污染
2. 并发通知和状态推进打架
3. 高风险执行路径越界
4. adapter 把宿主脏状态带进内核

Rust 不能自动解决治理问题，但它能帮助：
1. 把危险路径关进少量模块
2. 用类型系统约束 lane、policy、authorization
3. 用 ownership 把共享可变状态控制住

## 10. 建议的第一版落地顺序
### 10.1 Phase 0
1. 固定对象名
2. 固定 `StagePool` / `AgentPool`
3. 固定 `COMPLEX / HUGE / MAX`
4. 固定 start card / acceptance / watchdog policy

### 10.2 Phase 1
1. Rust kernel skeleton
2. OpenClaw adapter 最小闭环
3. `COMPLEX` 路径跑通
4. recorder 与 watchdog 最小闭环

### 10.3 Phase 2
1. `HUGE` 阶段化编排
2. review / rework 闭环
3. 用户变更、暂停、取消控制面

### 10.4 Phase 3
1. `MAX` 深层 worker tree
2. 更丰富的 stage package 和 pack 系统
3. 更多 personal AI terminal adapter
