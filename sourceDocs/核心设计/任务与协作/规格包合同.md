# 规格包合同

状态：contract draft  
定位：Loom 的执行锚点文档组规格  
更新时间：2026-03-10

---

## 1. 目的
`SpecBundle` 定义某个 `managedTaskRef` 当前正式执行锚点的文档组。

它回答：
1. 这个任务当前有哪些正式文档锚点
2. 范围、计划、验证分别如何落文档
3. 文档如何修订和留痕
4. 它和 `TaskScopeSnapshot / PhasePlan / AcceptancePolicy` 的边界是什么

---

## 2. 设计边界
### 2.1 这不是 `TaskScopeSnapshot`
1. `TaskScopeSnapshot`
   - 结构化范围真相
2. `SpecBundle`
   - 围绕范围生成的可读执行锚点文档组

### 2.2 这不是 `PhasePlan`
1. `PhasePlan`
   - 这次真实运行的阶段序列
2. `SpecBundle.plan_doc`
   - 面向人和 agent 的计划锚点

### 2.3 这不是上传附件
1. 上传附件只是背景材料
2. `SpecBundle` 是 Loom 持有版本、可修订、可引用的正式对象

---

## 3. v0 固定规则
1. 每个 active managed task 最少可绑定一个 `SpecBundle`
2. `SpecBundle` owner 固定为 `managedTaskRef`
3. 至少包含：
   - `scope_doc`
   - `plan_doc`
   - `verification_doc`
4. 文档可以修订，但修订必须留版本
5. `SpecBundle` 是活文档，不是一次性冻结文件

---

## 4. 正式对象定义
```rust
pub struct SpecBundle {
    pub spec_bundle_id: SpecBundleId,
    pub managed_task_ref: ManagedTaskRef,
    pub spec_policy_ref: SpecPolicyRef,
    pub current_revision: SpecBundleRevision,
    pub revision_history: Vec<SpecBundleRevisionRef>,
    pub status: SpecBundleStatus,
}

pub struct SpecBundleRevision {
    pub revision_id: SpecBundleRevisionId,
    pub scope_doc: SpecDocumentRef,
    pub plan_doc: SpecDocumentRef,
    pub verification_doc: SpecDocumentRef,
    pub derived_from_scope_version: TaskScopeVersion,
    pub revised_at: Timestamp,
    pub revised_reason: String,
}

pub struct SpecDocumentRef {
    pub document_id: String,
    pub kind: SpecDocumentKind,
    pub title: String,
    pub summary: String,
    pub storage_ref: ArtifactRef,
}

pub enum SpecDocumentKind {
    Scope,
    Plan,
    Verification,
}

pub enum SpecBundleStatus {
    Draft,
    Active,
    Superseded,
    Closed,
}
```

v0 固定取舍：
1. `SpecPolicyRef`
   - 不是外部 registry lookup
   - 而是 Loom 固定命名空间下的稳定模板 id
2. `spec_policy_ref`
   - 只记录这次 `SpecBundle` 采用了哪条模板策略
   - 不要求实现独立 policy store

---

## 5. 字段解释
1. `spec_bundle_id`
   - 这组文档锚点的稳定 id
2. `managed_task_ref`
   - 这组文档归属哪个任务
3. `spec_policy_ref`
   - 由哪个稳定模板 id 生成这组 spec
4. `current_revision`
   - 当前生效的 spec 文档版本
5. `revision_history`
   - 所有旧版本引用
6. `derived_from_scope_version`
   - 这版 spec 对应哪一版 `TaskScopeSnapshot`

---

## 6. 三类文档的正式语义
### 6.1 `scope_doc`
回答：
1. 当前任务范围是什么
2. 为什么是这个范围
3. 关键边界与决策是什么

### 6.2 `plan_doc`
回答：
1. 当前任务准备怎么推进
2. 阶段和关键任务如何拆解
3. 哪些是默认阶段，哪些是系统自动插入阶段

### 6.3 `verification_doc`
回答：
1. 如何验证这次任务完成
2. 哪些检查项必须过
3. 哪些证据会进入结果包

---

## 7. 和 pack 的关系
1. `PackContract` 不直接等于 `SpecBundle`
2. pack 只提供：
   - `default_spec_policy_ref`
3. 这里的 `default_spec_policy_ref`
   - 在 v0 只表示稳定模板 id
   - 不要求外部 policy registry
4. 具体任务启动后，Loom 依据 pack 和 task scope 生成该任务的 `SpecBundle`

---

## 8. 和变更的关系
1. `request_task_change` 被接受后
2. Loom 先更新 `TaskScopeSnapshot`
3. 再决定是否修订 `SpecBundle`
4. 旧 spec 版本进入 `revision_history`

---

## 9. 我的建议
1. `SpecBundle` 应成为 Loom 核心对象，而不是 coding 专属技巧
2. `verification_doc` 要和 `AcceptancePolicy`、`ReviewResult`、`ProofOfWorkBundle` 对齐
3. v0 不要求 spec 文档格式标准化到模板引擎级别，但字段语义要固定
