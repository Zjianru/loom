# 智能体绑定合同

状态：contract draft  
定位：`AgentBinding` 正式规格  
更新时间：2026-03-11

---

## 1. 目的
`AgentBinding` 定义某次任务/阶段/运行里，Loom 最终实际启用的 agent roster。

它回答：
1. 当前 run 具体让谁上场
2. 哪些角色由哪些画像和宿主映射来承接
3. `Pack`、`AgentPool`、`HostCapabilitySnapshot`、`ExecutionAuthorization` 怎样一起收敛成一次真实绑定
4. 能力漂移、暂停、重授权后，当前 roster 怎样 supersede

更深层上，它解决的是：

**把“默认角色骨架”和“这次真的派谁上场”彻底拆开，避免实现层把 roster 临时写死在 `Harness` 里。**

---

## 2. 设计边界
### 2.1 这不是 `Pack`
边界：
1. `Pack`
   - 给默认角色骨架和选人策略
2. `AgentBinding`
   - 给这次 run 当前真正生效的 roster

### 2.2 这不是 `AgentCapabilityProfile`
边界：
1. `AgentCapabilityProfile`
   - 描述某个 agent 理论上擅长什么
2. `AgentBinding`
   - 描述这次任务实际挑中了谁

### 2.3 这不是 `ExecutionAuthorization`
边界：
1. `AgentBinding`
   - 回答“谁来做”
2. `ExecutionAuthorization`
   - 回答“这次被允许做什么”

固定关系：
1. 绑定谁和授权到什么程度必须分开
2. 不允许把 agent roster 和授权边界混成一个对象

### 2.4 这不是 `IsolatedTaskRun`
边界：
1. `IsolatedTaskRun`
   - 是 run 本体
2. `AgentBinding`
   - 是该 run 当前生效的实际 roster

---

## 3. v0 固定规则
1. 每个 active `IsolatedTaskRun` 至少绑定一个当前生效的 `AgentBinding`
2. `Pack` 只给默认角色骨架，不直接等于实际 binding
3. `Harness` 必须基于：
   - `Pack`
   - `AgentPool`
   - `HostCapabilitySnapshot`
   - `ExecutionAuthorization`
   收敛出实际 `AgentBinding`
4. capability drift、pause、resume、review 升级等事件
   - 都可以导致当前 binding 被 supersede
5. `COMPLEX` v0 最小骨架中
   - 至少能表达：
   - `net`
   - `recorder`
   - `worker`
   的实际上场关系
6. 第一版不引入独立 `ResponsibilityGraph`
7. `HUGE / MAX` 如发生正式责任转移，只补最小 `HandoffContract`
8. 第二层当前只做最小冻结，不做全面多人治理
9. `coding_pack / COMPLEX` 默认路径不得创建 `HandoffContract`
10. 只有 `HUGE / MAX` 或显式责任转移时，才允许进入最小 handoff 路径

## 4. 正式对象定义
```rust
pub struct AgentBinding {
    pub binding_id: AgentBindingId,
    pub managed_task_ref: ManagedTaskRef,
    pub run_ref: IsolatedTaskRunRef,
    pub stage_run_ref: Option<StageRunRef>,
    pub pack_ref: PackRef,
    pub capability_snapshot_ref: HostCapabilitySnapshotRef,
    pub members: Vec<AgentBindingMember>,
    pub status: AgentBindingStatus,
    pub issued_reason: String,
    pub issued_at: Timestamp,
    pub supersedes: Option<AgentBindingRef>,
}

pub struct AgentBindingMember {
    pub role_kind: AgentRoleKind,
    pub profile_ref: AgentProfileId,
    pub host_mapping_ref: Option<HostMappingRef>,
    pub responsibilities: Vec<StageKind>,
    pub execution_mode: AgentExecutionMode,
    pub required: bool,
}

pub enum AgentExecutionMode {
    Inline,
    BackgroundWorker,
    ReviewOnly,
    RecorderOnly,
}

pub enum AgentBindingStatus {
    Draft,
    Active,
    Suspended,
    Superseded,
    Closed,
}
```

## 5. 关键字段分别代表什么
1. `binding_id`
   - 当前 roster 的稳定 id
2. `managed_task_ref`
   - 这份绑定归属哪个任务
3. `run_ref`
   - 这份绑定归属哪个 `IsolatedTaskRun`
4. `stage_run_ref`
   - 如果绑定是阶段级收敛结果，这里指向具体阶段运行
5. `capability_snapshot_ref`
   - 这份绑定建立时所依据的宿主能力快照
6. `members`
   - 当前 roster 中每个上场角色的正式成员
7. `supersedes`
   - 当前 binding 是否替换了旧 binding

## 6. 和其它对象的关系
### 6.1 和 `AgentPool`
固定关系：
1. `AgentPool`
   - 提供候选画像
2. `AgentBinding`
   - 从这些候选里选出当前真正上场者

### 6.2 和 `Pack`
固定关系：
1. `Pack`
   - 给默认角色骨架
2. `AgentBinding`
   - 把骨架落成实际 roster

### 6.3 和 `ExecutionAuthorization`
固定关系：
1. `AgentBinding`
   - 决定由谁执行
2. `ExecutionAuthorization`
   - 决定这些成员在当前 run 里被允许做到哪一步

### 6.4 和 capability drift
固定关系：
1. 宿主能力变化后
   - 当前 binding 可以保留、收紧、暂停或 supersede
2. 不能继续静默沿用已经依赖旧能力快照的 binding

### 6.5 和 `HandoffContract`
固定关系：
1. `AgentBinding`
   - 回答当前到底谁上场
2. `HandoffContract`
   - 回答责任如何从当前 binding 或其成员转移到新的 binding
3. 第一版不引入更重的责任拓扑对象

---

## 7. 我的建议
1. `AgentBinding` 应升为正式运行时对象，而不是散在 `Pack`、`AgentPool` 和实现代码之间的临时拼装结果
2. `Harness` 只负责生成和更新 binding，不应偷偷把 roster 硬编码成实现默认值
3. 后续实现和测试应把 `AgentBinding` 当成可追溯对象，而不是把“谁在干活”藏在日志里
4. `HUGE / MAX` 的正式交接先通过最小 `HandoffContract` 承接，不要急着把责任协作直接做成图结构
