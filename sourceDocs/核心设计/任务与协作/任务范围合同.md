# 任务范围合同

状态：contract draft  
定位：`TaskScopeSnapshot` 正式规格  
更新时间：2026-03-11

---

## 1. 目的
`TaskScopeSnapshot` 定义某个 `managedTaskRef` 当前到底被要求做什么。

它回答：
1. 这项任务当前的正式范围是什么
2. 任务要求被拆成哪些 requirement item
3. 这些要求当前处于什么状态
4. 每次接受变更后，范围版本如何递增
5. 最终交付时，结果包依据的是哪一版范围

更深层上，它解决的是：

**让任务可以持续演化，但仍然有一个正式、可追溯、可版本化的范围本体。**

---

## 2. 设计边界
### 2.1 `TaskScopeSnapshot` 不是自然语言摘要
它不是：
1. 一句模糊的“这次做什么”

它应该同时包含：
1. `scope_summary`
2. `requirement_items[]`

这是这次已经对齐的点：
1. `v0` 采用混合型
2. 不只写总述
3. 也不只写碎片 item

### 2.2 `TaskScopeSnapshot` 不是 `ResultContract`
边界：
1. `TaskScopeSnapshot`
   - 当前被要求做什么
2. `ResultContract`
   - 最后到底交付了什么

### 2.3 `TaskScopeSnapshot` 不自己推断用户语义
这次已锁定：
1. kernel 不自行理解自然语言
2. `TaskScopeSnapshot` 的修订依据来自 `SemanticDecisionEnvelope`

### 2.4 第一版工程上下文先吸收进 `TaskScopeSnapshot`
固定边界：
1. 第一版不单独新建额外工程上下文聚合
2. 这些工程上下文先作为 `TaskScopeSnapshot` 的正式字段族进入任务范围
3. 如果后续其中某一块长出独立生命周期，再考虑拆出独立对象

### 2.5 `TaskScopeSnapshot` 不是 `ExecutionAuthorization`
边界：
1. `TaskScopeSnapshot`
   - 定义任务级原则边界
2. `ExecutionAuthorization`
   - 定义当前 run 的可执法租约

固定规则：
1. run 授权只能是 task scope 的子集
2. `TaskScopeSnapshot` 不直接替代授权 lease

---

## 3. v0 固定规则
### 3.1 每次接受变更都提升 `scope_version`
这次已锁定：
1. 每次真正接受并吸收 task change
2. 都必须提升 `scope_version`

原因：
1. 不然最终结果包无法说清“最后按哪一版范围交付”

### 3.2 requirement item 必须有正式状态
这次已锁定。

我的建议最小状态集：
1. `accepted`
2. `in_progress`
3. `completed`
4. `superseded`
5. `deferred`
6. `rejected`

### 3.3 历史不能被抹掉
这条规则很重要。

一旦 scope 变更：
1. 旧要求可以被 supersede
2. 可以被 defer
3. 可以被 reject

但不能：
1. 像没发生过一样消失

### 3.4 `managedTaskRef` 稳定，scope 可以演化
这次已锁定：
1. 同一个任务可以持续变更 scope
2. 不必因为每次补充要求就新建 task

### 3.5 工程上下文是范围的一部分
这次固定：
1. `workspace_ref`
2. `repo_ref`
3. `allowed_roots`
4. `secret_classes`
都先进入 `TaskScopeSnapshot`

原因：
1. coding / operations 任务的正式边界不能只靠 prompt 文本
2. `TaskScopeSnapshot` 才是任务“原则上被允许做什么”的 owner

## 4. 正式对象定义
### 4.1 Rust 形态建议
```rust
pub struct TaskScopeSnapshot {
    pub scope_id: TaskScopeId,
    pub managed_task_ref: ManagedTaskRef,
    pub scope_version: u32,
    pub scope_summary: String,
    pub requirement_items: Vec<RequirementItem>,
    pub workspace_ref: Option<String>,
    pub repo_ref: Option<String>,
    pub allowed_roots: Vec<String>,
    pub secret_classes: Vec<String>,
    pub constraints: Vec<ScopeConstraint>,
    pub assumptions: Vec<ScopeAssumption>,
    pub source_decision_ref: SemanticDecisionEnvelopeId,
    pub created_at: Timestamp,
}
```

### 4.2 `RequirementItem`
```rust
pub struct RequirementItem {
    pub requirement_id: RequirementId,
    pub text: String,
    pub status: RequirementStatus,
    pub origin: RequirementOrigin,
    pub superseded_by: Option<RequirementId>,
    pub deferred_reason: Option<String>,
}
```

### 4.3 `RequirementStatus`
建议值：
1. `accepted`
2. `in_progress`
3. `completed`
4. `superseded`
5. `deferred`
6. `rejected`

## 5. 字段 contract
### 5.1 `scope_id`
它是什么：
1. 这份 scope snapshot 的稳定 id

### 5.2 `managed_task_ref`
它是什么：
1. 这份范围快照属于哪个任务

### 5.3 `scope_version`
它是什么：
1. 当前是任务范围的第几版

建议规则：
1. 任务启动时从 `1` 开始
2. 每次接受并吸收 change，+1

### 5.4 `scope_summary`
它是什么：
1. 当前范围的人类可读摘要

它的价值：
1. 供用户快速理解当前任务在做什么
2. 供结果包引用最终范围说明

### 5.5 `requirement_items`
它是什么：
1. 当前范围下的正式要求清单

它解决：
1. 哪些要求已经纳入
2. 哪些要求已完成
3. 哪些要求被新要求覆盖
4. 哪些要求被延期

### 5.6 `workspace_ref / repo_ref`
它们是什么：
1. 当前任务绑定的 workspace 和 repo 语义锚点

固定边界：
1. 它们是 task scope 的字段
2. 不是第一版独立 aggregate

### 5.7 `allowed_roots`
它是什么：
1. 当前任务在任务级范围内允许触达的路径根

意义：
1. 给后续 `ExecutionAuthorization.readable_roots / writable_roots` 提供上限边界

### 5.8 `secret_classes`
它是什么：
1. 当前任务被允许接触的 secret 分类

固定边界：
1. 第一版先保留分类级别，不直接展开成 secret 明细对象
2. 具体 run 能否真的读取，仍需结合宿主能力与授权 lease
3. 后续 `ExecutionAuthorization.allowed_secret_classes` 只能在它的上限内再做子集租约

### 5.9 `constraints`
它是什么：
1. 当前任务必须遵守的约束

示例：
1. 不允许写某些路径
2. 必须保留某些技术栈
3. 必须满足特定 review / compliance 要求

### 5.10 `assumptions`
它是什么：
1. 当前 scope 成立所依赖的前提假设

原因：
1. 很多 change 的影响，来自假设被改变

### 5.11 `source_decision_ref`
它是什么：
1. 这份范围快照由哪次结构化语义判断驱动生成

意义：
1. 让 `TaskScopeSnapshot` 不脱离 `SemanticDecisionEnvelope`

## 6. requirement item 的动态语义
### 6.1 `accepted`
表示：
1. 已进入当前任务正式范围

### 6.2 `in_progress`
表示：
1. 已进入执行面，正在被处理

### 6.3 `completed`
表示：
1. 已完成并仍属于当前有效范围

### 6.4 `superseded`
表示：
1. 曾经被接受
2. 后来被新要求覆盖或替代

### 6.5 `deferred`
表示：
1. 这次不再在当前交付里完成
2. 但并不等于被否决

### 6.6 `rejected`
表示：
1. 没有被纳入当前任务范围

---

## 7. 与其它对象的边界
### 7.1 和 `SemanticDecisionEnvelope`
`TaskScopeSnapshot` 不做语义理解。  
它只根据已经结构化好的 envelope 生成或修订 snapshot。

### 7.2 和 `ChangeImpactAssessment`
`ChangeImpactAssessment` 负责：
1. 一次 change 会影响哪些阶段和治理对象

`TaskScopeSnapshot` 负责：
1. 当前正式范围本体长什么样

### 7.3 和 `ResultContract`
`ResultContract.final_scope_snapshot`
1. 应引用最后一版 `TaskScopeSnapshot`

### 7.4 和 `ExecutionAuthorization`
`TaskScopeSnapshot` 回答：
1. 任务原则上允许碰什么

`ExecutionAuthorization` 回答：
1. 当前这次 run 实际被放行到什么程度

---

## 8. 我的建议
### 8.1 `TaskScopeSnapshot` 是动态任务治理的基础对象
如果没有它：
1. 任务可以被改
2. 但系统永远说不清最后到底按什么完成

### 8.2 我的建议
1. `v0` 就采用混合型范围表示
2. requirement item 一定要状态化
3. 每次接受变更都递增版本
4. 历史要求只允许 supersede / defer / reject，不允许静默消失
