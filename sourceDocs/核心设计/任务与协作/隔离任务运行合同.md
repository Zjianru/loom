# 隔离任务运行合同

状态：contract draft  
定位：Loom 执行期运行边界规格  
更新时间：2026-03-10

---

## 1. 目的
`IsolatedTaskRun` 定义单个 `managedTaskRef` 在执行期绑定的一次正式运行单元。

它回答：
1. run 是什么，和 task/session/authorization 的边界是什么
2. 一个 run 有哪些正式状态
3. run 产物、证据、失败怎样被隔离

---

## 2. 设计边界
### 2.1 run 不是 task
1. `managedTaskRef`
   - 任务 owner
2. `IsolatedTaskRun`
   - 任务执行期的一次运行单元

### 2.2 run 不是宿主 session
1. `host_session_id`
   - 宿主聊天容器
2. `IsolatedTaskRun`
   - Loom 内核里的运行边界

### 2.3 run 不是 `ExecutionAuthorization`
1. `ExecutionAuthorization`
   - 这次 run 拿到的能力租约
2. `IsolatedTaskRun`
   - 运行本体

### 2.4 run 不是 outbox
1. `OutboundDelivery`
   - 出站交互单元
2. `IsolatedTaskRun`
   - 执行期运行状态

---

## 3. v0 固定规则
1. 一个 active `managedTaskRef` 至少绑定一个当前 run
2. run 失败不能污染别的 `managedTaskRef`
3. v0 不承诺完整 OS/container 沙箱
4. 但必须承认：
   - 独立 run id
   - 独立 run artifacts
   - 独立 run event chain
   - 独立 run completion / failure

---

## 4. 正式对象定义
```rust
pub struct IsolatedTaskRun {
    pub run_id: IsolatedTaskRunId,
    pub managed_task_ref: ManagedTaskRef,
    pub status: IsolatedTaskRunStatus,
    pub authorization_ref: ExecutionAuthorizationRef,
    pub artifacts: Vec<ArtifactRef>,
    pub evidence_refs: Vec<EvidenceRef>,
    pub started_at: Timestamp,
    pub finished_at: Option<Timestamp>,
}

pub enum IsolatedTaskRunStatus {
    Pending,
    Active,
    Paused,
    Blocked,
    Cancelled,
    Failed,
    Completed,
}

pub enum IsolatedTaskRunOutcome {
    Completed,
    Blocked,
    Cancelled,
    Failed,
    Partial,
}
```

---

## 5. 字段解释
1. `run_id`
   - 一次 run 的稳定 id
2. `managed_task_ref`
   - 这次 run 归属哪个任务
3. `status`
   - 这次 run 当前处于什么状态
4. `authorization_ref`
   - 当前生效的能力租约引用
5. `artifacts`
   - run 直接产出的工件
6. `evidence_refs`
   - run 过程中产生的关键证据

---

## 6. 和其它对象的关系
1. `SpecBundle`
   - 提供 run 的文档锚点
2. `PhasePlan`
   - 提供 run 的阶段推进方案
3. `ExecutionAuthorization`
   - 提供 run 的能力边界
4. `ProofOfWorkBundle`
   - 汇总 run 最终证据

---

## 7. 我的建议
1. 第一条代码 spike 就应该显式创建 `IsolatedTaskRun`
2. v0 先把 run 作为逻辑隔离单元写清，不夸大成完整沙箱
3. 后续无论是真 workspace、真沙箱、真容器，都应挂在这个对象下扩展
