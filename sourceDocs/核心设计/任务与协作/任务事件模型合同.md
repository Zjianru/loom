# 任务事件模型合同

状态：contract draft  
定位：`TaskEventModel` 正式规格  
更新时间：2026-03-11

---

## 1. 目的
`TaskEventModel` 是 Loom 的统一事件链。

它回答：
1. 哪些事情应该被记成正式事件
2. recorder、watchdog、Harness、approval-gate 如何共享同一条真相源
3. 哪些状态变化必须留痕

更深层上，它解决的是：

**让系统不是各写各的日志，而是围绕同一条任务事件流协作。**

---

## 2. 为什么必须先补
如果没有统一事件模型，会出现 3 个问题：
1. recorder 写一套记录
2. watchdog 写一套通知
3. approval-gate 写一套审批痕迹

最后：
1. 无法统一回放
2. 无法统一审计
3. 无法稳定驱动后续 UI 和 adapter

---

## 3. 正式对象定义
```rust
pub struct TaskEvent {
    pub event_id: TaskEventId,
    pub managed_task_ref: ManagedTaskRef,
    pub event_type: TaskEventType,
    pub actor: EventActorRef,
    pub ingress_ref: Option<IngressId>,
    pub causation_id: Option<CausationId>,
    pub correlation_id: CorrelationId,
    pub payload: TaskEventPayload,
    pub occurred_at: Timestamp,
}
```

这里新增的 3 个字段分别代表：
1. `ingress_ref`
   - 这条事件最初来自哪一次宿主 ingress
   - 它让 replay、dedupe、adapter 对账有稳定锚点
2. `causation_id`
   - 这条事件由哪条上游事件或宿主请求直接触发
   - 它解决“为什么这条事件会发生”
3. `correlation_id`
   - 这条事件属于哪条更大的请求链或任务交互链
   - 它解决“哪些事件属于同一条端到端链路”

### 3.2 `HostExecutionEventPayload`
`TaskEventPayload` 应按来源域拆分。

host execution 这条线至少需要一个专门 payload：
```rust
pub struct HostExecutionEventPayload {
    pub command_id: HostExecutionCommandId,
    pub run_ref: IsolatedTaskRunRef,
    pub binding_id: AgentBindingId,
    pub role_kind: AgentRoleKind,
    pub host_session_id: HostSessionId,
    pub host_agent_id: String,
    pub host_child_execution_ref: Option<String>,
    pub host_child_run_ref: Option<String>,
    pub terminal_status: Option<HostSubagentStatus>,
    pub artifact_refs: Vec<String>,
}
```

这里几个字段分别代表：
1. `command_id`
   - 这条事件对应哪一条正式宿主派发命令
2. `run_ref`
   - 这条事件属于哪次隔离 run
3. `binding_id`
   - 这次派发引用的是哪版 `AgentBinding`
4. `role_kind`
   - 这次派发的是 `worker / recorder` 中的哪一个角色
5. `host_session_id`
   - 这条派发面向哪个宿主主会话
6. `host_agent_id`
   - 这次宿主实际使用哪个 agent 映射
7. `host_child_execution_ref`
   - 宿主真实 child execution handle 的中性正式引用
8. `host_child_run_ref`
   - 宿主真实 child run / attempt handle 的中性正式引用
9. `terminal_status`
   - 只有终态事件才填写
10. `artifact_refs`
   - 只有终态事件需要携带证据引用

固定边界：
1. `HostExecutionEventPayload`
   - 记录 lifecycle 事实和定位锚点
2. `output_summary`
   - 不复制进事件 payload
3. 终态摘要仍由 `HostExecutionCommand / ReviewResult / ResultContract`
   - 持有 authoritative narrative truth

## 4. actor 模型
### 4.1 `EventActorRef`
建议值：
1. `user`
2. `net`
3. `worker`
4. `recorder`
5. `reviewer`
6. `watchdog`
7. `approve_gate`
8. `adapter`
9. `host_runtime`
10. `system`

原因：
1. 事件必须说清是谁触发的
2. 否则回放时看不懂责任归属

### 4.2 host execution 线的 actor 口径
固定建议：
1. `host_execution_command_queued`
   - actor = `system`
2. `host_execution_command_dispatched`
   - actor = `adapter`
3. `host_execution_spawned`
   - actor = `host_runtime`
4. `host_execution_completed`
   - actor = `host_runtime`
5. `host_execution_failed`
   - actor = `host_runtime`

这里的取舍是：
1. `worker / recorder`
   - 表示角色身份
2. `host_runtime`
   - 表示“这条事实来自宿主真实执行面”
3. 两者不能混成一个 actor 口径

## 5. 与其它对象的关系
### 5.1 和 `WorkHorizon`
`WorkHorizon` 修订必须产生：
1. `work_horizon_revised`

### 5.2 和 `ControlAction`
所有用户控制动作都必须产生：
1. `control_action_received`

其中：
1. `request_task_change`
   - 应进一步驱动 `task_change_requested`
2. 如果系统接受并修订范围
   - 应写入 `task_scope_revised`
3. 如果因此重写阶段方案
   - 应写入 `phase_plan_revised`

再补一条链路边界：
1. adapter ingress 带来的 `IngressMeta`
2. 必须继续传进 `TaskEvent`
3. 否则重放、恢复、重复投递时看不清哪条事件来自哪次宿主输入

### 5.3 和 recorder
recorder 不应该单独拥有另一套事实源。  
它应该消费 `TaskEventModel`，再产出更高级的总结。

### 5.4 和 watchdog
watchdog 不应该靠猜测状态。  
它应该消费事件流，再决定通知。

### 5.5 和 `ResultContract`
`ResultContract` 不应凭空总结。  
它应该消费事件链中的：
1. `task_change_requested`
2. `task_scope_revised`
3. `phase_plan_revised`
4. `review_completed`
5. `review_result_recorded`
6. `result_compiled`

### 5.6 和 outbound delivery
delivery 相关 contract 至少要落这些事件：
1. `outbound_delivery_created`
2. `outbound_delivery_attempted`
3. `outbound_delivery_delivered_unacked`
4. `outbound_delivery_retry_scheduled`
5. `outbound_delivery_acked`
6. `outbound_delivery_terminal_failed`
7. `outbound_delivery_expired`

### 5.7 和 capability drift
宿主能力变化后的治理后果，至少要落这些事件：
1. `capability_snapshot_synced`
2. `capability_drift_assessed`
3. `execution_authorization_reissued`
4. `execution_authorization_revoked_due_to_drift`

### 5.8 和 `HostExecutionCommand`
固定关系：
1. `HostExecutionCommand`
   - 持有 authoritative aggregate 状态
2. `TaskEventModel`
   - 持有 host execution 这条线的时间序事件链
3. `HostExecutionEventPayload`
   - 是这条事件链的正式值对象

host execution 线至少要落这些事件：
1. `host_execution_command_queued`
   - 当 authoritative store 创建并排队 `HostExecutionCommand`
2. `host_execution_command_dispatched`
   - 当 `ack_host_execution` 被正式接受，且命令第一次进入 `dispatched`
3. `host_execution_spawned`
   - 当收到 `HostSubagentLifecycleEvent::Spawned`
4. `host_execution_completed`
   - 当收到 `HostSubagentLifecycleEvent::Ended(status=completed)`
5. `host_execution_failed`
   - 当收到 `HostSubagentLifecycleEvent::Ended(status=failed|timed_out|cancelled)`

补一条迟到 ack 规则：
1. 如果 `Spawned / Ended` 先到
2. 事件链优先记录更强的 lifecycle 事实
3. 之后迟到的 `ack_host_execution`
   - 只允许修正 aggregate
   - 不得再补发语义已倒序的 `host_execution_command_dispatched`

---

## 6. 我的建议
`TaskEventModel` 是所有运行线共享的基础设施 contract。  
它应该在 crate 分层之前固定。

补充一条存储边界：
1. authoritative event store 由 Loom store 持有
2. `runtime/loom/events/` 只做导出/调试 projection
3. 不允许数据库事件流和文件事件流同时充当双真相源

再补一条因果边界：
1. `TaskEvent` 必须保留 `ingress_ref / causation_id / correlation_id`
2. 这样 Loom 才能在 outbox、watchdog、recorder、adapter 重放时维持同一条因果链
