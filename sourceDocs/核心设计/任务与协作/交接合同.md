# 交接合同

状态：contract draft  
定位：`HandoffContract` 最小正式规格  
更新时间：2026-03-11

---

## 1. 目的
`HandoffContract` 定义多 agent 协作里一次正式责任交接记录。

它回答：
1. 责任从哪个当前 binding 或成员交给哪个新的 binding
2. 这次交接覆盖了哪些职责范围
3. 为什么要交接
4. 交接是否已经被接受

---

## 2. 设计边界
### 2.1 这不是 `AgentBinding`
1. `AgentBinding`
   - 回答当前实际上场 roster 是谁
2. `HandoffContract`
   - 回答责任怎样从当前 roster 转移出去

### 2.2 这不是 `ResponsibilityGraph`
1. 第一版不引入更重的责任拓扑对象
2. `HandoffContract` 只承接单次正式交接，不承接整张长期责任图

### 2.3 这不是普通事件备注
1. recorder 或事件链可以记录交接经过
2. 但正式责任转移必须有独立 `HandoffContract`

---

## 3. v0 固定规则
1. 第一版只在 `HUGE / MAX` 或显式多 worker 协作下使用。
2. 只有当责任真正从当前 binding 或其成员转移出去时，才创建 `HandoffContract`。
3. 被接受的交接，必须能追溯到：
   - `from_binding_ref`
   - `to_binding_ref`
   - `handoff_reason`
   - `handoff_scope`
4. `HandoffContract` 不替代 `AgentBinding`；它只解释为何会 supersede 到新的 binding。
5. `coding_pack / COMPLEX` 默认路径不得创建 `HandoffContract`。
6. 当前只把它作为 `HUGE / MAX` 或显式责任转移下的最小正式交接对象。

---

## 4. 正式对象定义
```rust
pub struct HandoffContract {
    pub handoff_id: HandoffId,
    pub managed_task_ref: ManagedTaskRef,
    pub run_ref: IsolatedTaskRunRef,
    pub from_binding_ref: AgentBindingRef,
    pub to_binding_ref: AgentBindingRef,
    pub handoff_scope: Vec<StageKind>,
    pub handoff_reason: String,
    pub requested_at: Timestamp,
    pub accepted_at: Option<Timestamp>,
    pub status: HandoffStatus,
}

pub enum HandoffStatus {
    Proposed,
    Accepted,
    Cancelled,
    Superseded,
}
```

---

## 5. 字段解释
1. `from_binding_ref`
   - 当前交出责任的一方
2. `to_binding_ref`
   - 当前接过责任的一方
3. `handoff_scope`
   - 这次交接覆盖哪些阶段或职责范围
4. `handoff_reason`
   - 为什么需要交接
5. `accepted_at`
   - 这次交接何时被正式接受
6. `status`
   - 交接当前是否已生效

---

## 6. 和其它对象的关系
1. `AgentBinding`
   - 持有当前实际上场 roster
2. `HandoffContract`
   - 持有单次正式责任转移事实
3. 被接受的 `HandoffContract`
   - 通常会伴随新的 `AgentBinding.supersedes`

---

## 7. 我的建议
1. 第一版先把正式交接压到最小，只回答“谁把什么交给了谁、为什么交、是否被接受”。
2. 不要把 `HandoffContract` 直接长成 `ResponsibilityGraph`。
3. `COMPLEX` 最小闭环不强制要求它，但 `HUGE / MAX` 一旦出现正式交接，就必须可追溯。
