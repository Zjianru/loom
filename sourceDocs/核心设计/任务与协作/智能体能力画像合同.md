# 智能体能力画像合同

状态：contract draft  
定位：`AgentCapabilityProfile` 正式规格  
更新时间：2026-03-10

---

## 1. 目的
`AgentCapabilityProfile` 是 `AgentPool` 中单个 agent 的正式能力画像。

它回答：
1. 这个 agent 擅长什么
2. 这个 agent 属于什么角色类型
3. 这个 agent 适合哪些阶段
4. 这个 agent 的成本、风险、延迟等级如何
5. 这个 agent 在宿主里如何被映射和绑定

更深层上，它解决的是：

**把“我们有很多 agent”收成一个可被编排、可被比较、可被静态注册的能力清单。**

---

## 2. 设计边界
### 2.1 `AgentCapabilityProfile` 不是运行时授权
它是什么：
1. 能力画像

它不是什么：
1. 这次运行时实际被允许做什么

边界：
1. `AgentCapabilityProfile`
   - 描述“理论上擅长什么、可能具备什么”
2. `ExecutionAuthorization`
   - 描述“这次运行时真的允许做什么”

### 2.2 `AgentCapabilityProfile` 不是宿主配置文件本身
这次已经锁定：
1. 具体模型配置和工具映射，和 adapter 相关
2. OpenClaw 这类宿主的真实配置可能在 `openclaw.json`

所以边界必须明确：
1. `AgentCapabilityProfile`
   - 内核里的抽象画像
2. `HostCapabilityMapping`
   - adapter 负责把画像映射成宿主真实能力

### 2.3 v0 先静态注册
这次已锁定：
1. `v0` 先静态注册
2. 不做动态发现和 marketplace

原因：
1. 这能保证编排稳定
2. 这也符合你“先做治理，不做大平台”的方向

---

## 3. 角色模型
### 3.1 系统保留角色
这次已经对齐的系统保留角色至少有：
1. `worker`
2. `recorder`
3. `reviewer`
4. `platform`
5. `facilitator`

这几个角色的意义：
1. `worker`
   - 真正做执行
2. `recorder`
   - 做过程沉淀和结构化记录
3. `reviewer`
   - 做验收、打回、质量审查
4. `platform`
   - 提供通用服务能力
5. `facilitator`
   - 提供方法、流程、协调支持

### 3.2 在保留角色上允许更细粒度专业 worker
这次也已经锁定：
1. 在这组角色基础上，要允许更细粒度专业 agent

我的建议：
1. 保留角色是顶层 taxonomy
2. 专业能力通过 `specialization_tags` 展开

例如：
1. `worker + coding.backend`
2. `worker + research.market`
3. `reviewer + compliance.security`

## 4. 正式对象定义
### 4.1 Rust 形态建议
```rust
pub struct AgentCapabilityProfile {
    pub profile_id: AgentProfileId,
    pub display_name: String,
    pub role_kind: AgentRoleKind,
    pub specialization_tags: Vec<String>,
    pub supported_task_classes: Vec<ManagedTaskClass>,
    pub supported_stage_kinds: Vec<StageKind>,
    pub desired_model_policy: DesiredModelPolicy,
    pub host_mapping_ref: Option<HostMappingRef>,
    pub tool_capability_intent: Vec<ToolCapabilityIntent>,
    pub memory_mode: MemoryMode,
    pub cost_tier: CostTier,
    pub risk_tier: RiskTier,
    pub latency_tier: LatencyTier,
    pub review_capability: ReviewCapability,
    pub metadata: AgentMetadata,
}
```

取舍：
1. 画像本身必须稳定
2. 宿主映射必须可替换
3. 不把宿主实现细节硬编码进画像

## 5. 字段 contract
### 5.1 `profile_id`
它是什么：
1. 稳定画像 ID

它表示：
1. 内核视角里的 agent 身份

建议：
1. 和宿主运行实例 ID 分离
2. 允许以后多个宿主共享同一画像

### 5.2 `display_name`
它是什么：
1. 用户或系统可读名称

### 5.3 `role_kind`
它是什么：
1. 顶层角色类型

建议值：
1. `worker`
2. `recorder`
3. `reviewer`
4. `platform`
5. `facilitator`

这是最重要的上层分类，原因是：
1. `StagePackage` 绑定的是角色，不是具体实例
2. `AgentBinding` 再从 `AgentPool` 里挑出具体上场者

### 5.4 `specialization_tags`
它是什么：
1. 角色下的细粒度专业标签

它表示：
1. 一个 `worker` 具体擅长什么
2. 一个 `reviewer` 具体审什么

建议：
1. 标签可以多值
2. 标签是分类，不是执行授权

### 5.5 `supported_task_classes`
它是什么：
1. 这个画像适用于哪些 `managedTaskClass`

它代表什么：
1. 不是每个 agent 都适合 `COMPLEX / HUGE / MAX` 全部档位

### 5.6 `supported_stage_kinds`
它是什么：
1. 这个 agent 适合哪些阶段类型

它的作用：
1. 帮助 `Harness` 从 `AgentPool` 中选人

### 5.7 `desired_model_policy`
它是什么：
1. 画像期望的模型策略

这次已经锁定的取舍：
1. 模型最终如何选，让给用户和 adapter
2. 不在内核中写死某一个模型

所以它应该表达的是：
1. 偏好大模型还是轻模型
2. 偏好高质量还是低成本
3. 是否要求长上下文
4. 是否要求更强推理或更强代码能力

而不是：
1. 只允许某个固定模型名

### 5.8 `host_mapping_ref`
它是什么：
1. 宿主映射引用

它表示：
1. 这个画像在当前宿主中映射到哪个具体 agent 配置

这条线必须由 adapter 负责，原因是：
1. OpenClaw 的映射可能落在 `openclaw.json`
2. 别的宿主可能是数据库、UI 配置或 API registry

### 5.9 `tool_capability_intent`
它是什么：
1. 这个画像希望具备哪些工具能力

注意边界：
1. 它表达的是“希望/适配需求”
2. 不是“最终已授权”

典型值：
1. `read_files`
2. `search_web`
3. `write_code`
4. `run_tests`
5. `review_artifacts`

### 5.10 `memory_mode`
它是什么：
1. 当前画像对记忆能力的预期模式

这次也已经对齐：
1. 具体是否支持 agent 自己记忆，要看宿主能力

所以建议值：
1. `none`
2. `ephemeral`
3. `host_managed`
4. `external_managed`

取舍：
1. 不在 v0 假设所有 agent 都有独立长期记忆
2. 长期沉淀仍应优先走 recorder

### 5.11 `cost_tier`
它是什么：
1. 成本等级

### 5.12 `risk_tier`
它是什么：
1. 风险等级

### 5.13 `latency_tier`
它是什么：
1. 延迟等级

这三个字段这次已锁定为必须项。  
原因不是好看，而是：
1. 后面做自动选人时，如果没有这三个字段，内核会失明

### 5.14 `review_capability`
它是什么：
1. 当前画像是否具备 review / validate 能力

原因：
1. 你已经明确要做 `review group / validate group`
2. 所以 reviewer 不能只是名字不同，必须有正式能力标记

## 6. v0 固定规则
### 6.1 静态注册
`v0` 中：
1. 所有 `AgentCapabilityProfile` 先静态注册
2. 不做运行时动态发现

### 6.2 宿主映射由 adapter 负责
固定规则：
1. 内核不直接读取 OpenClaw 的宿主配置格式
2. adapter 负责把画像映射到宿主的真实 agent 配置

这避免更深层的冲突：
1. 如果内核直接理解宿主配置，内核就会被宿主绑死
2. 你要的是 `kernel + adapter`，不是 “OpenClaw 内置插件集合”

### 6.3 记忆能力不作统一假设
固定规则：
1. 不假设所有 worker 都有长期记忆
2. 长期沉淀优先交给 recorder

---

## 7. 和其它对象的边界
### 7.1 和 `AgentBinding`
`AgentCapabilityProfile` 解决：
1. 这个 agent 是谁、能做什么、适合哪里

`AgentBinding` 解决：
1. 这次任务/阶段实际启用了谁

### 7.2 和 `ExecutionAuthorization`
`AgentCapabilityProfile` 解决：
1. 理论能力画像

`ExecutionAuthorization` 解决：
1. 本次运行时真实授权

### 7.3 和 `AdapterContract`
`AgentCapabilityProfile` 解决：
1. 抽象能力

`AdapterContract` 解决：
1. 如何把抽象能力映射到宿主实际 agent 配置

---

## 8. 我的建议
`AgentCapabilityProfile` 最容易走偏的地方是：
1. 被写成一份宿主配置镜像
2. 被写成“最终授权对象”
3. 被做成无限开放的 marketplace schema

我的建议是：
1. 先把它写成稳定的内核画像对象
2. 先静态注册
3. 先把 role、specialization、cost/risk/latency、host mapping 这几条线收稳
