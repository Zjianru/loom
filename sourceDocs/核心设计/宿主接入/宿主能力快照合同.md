# 宿主能力快照合同

状态：contract draft  
定位：`HostCapabilitySnapshot` 正式规格  
更新时间：2026-03-12

---

## 1. 目的
`HostCapabilitySnapshot` 定义宿主在某个时刻、某个宿主会话作用域下，向 Loom 同步的正式能力快照。

它回答：
1. 当前这个 `host_session_id` 真实拥有哪些 agent、model、tool、spawn、render、worker control 能力
2. 哪些能力直接影响 `AgentBinding / ExecutionAuthorization / CapabilityDriftAssessment`
3. 宿主会话当前到底处在什么 authority scope
4. 工程上下文边界怎样进入正式能力输入，而不是散在 adapter 私有逻辑里

更深层上，它解决的是：

**把“宿主当前现实能力”从松散接口事实，升级成 Loom 可复用、可漂移评估、可授权收敛的正式输入合同。**

---

## 2. 设计边界
### 2.1 这不是 `AgentCapabilityProfile`
边界：
1. `AgentCapabilityProfile`
   - 是 Loom 内核里的抽象画像
2. `HostCapabilitySnapshot`
   - 是当前宿主会话下现实可提供的能力事实

### 2.2 这不是 `AgentBinding`
边界：
1. `HostCapabilitySnapshot`
   - 回答宿主当前能给什么
2. `AgentBinding`
   - 回答这次 run 最终选了谁上场

### 2.3 这不是 `ExecutionAuthorization`
边界：
1. `HostCapabilitySnapshot`
   - 是现实能力输入
2. `ExecutionAuthorization`
   - 是在这些现实能力上收敛出的当前执行租约

### 2.4 这不是 `HostExecutionCommand`
边界：
1. `HostCapabilitySnapshot`
   - 回答“能不能派、能不能控、能在哪些根路径里做”
2. `HostExecutionCommand`
   - 回答“这次实际要把哪条执行命令派出去”

### 2.5 这不是 `CapabilityDriftAssessment`
边界：
1. `HostCapabilitySnapshot`
   - 是新旧对比前的原始事实
2. `CapabilityDriftAssessment`
   - 是 Loom 对这些事实差异的治理结论

---

## 3. v0 固定规则
1. adapter 启动时至少同步一次 `HostCapabilitySnapshot`
2. agent/model/tool/spawn/runtime authority/workspace roots/worker control 发生实质变化时，必须重新同步
3. `HostCapabilitySnapshot` 必须带 `host_session_id`
   - 因为这些能力往往是当前宿主会话、当前 agent、当前 subagent role 作用域下的事实
4. `readable_roots / writable_roots / secret_classes / max_budget_band`
   - 必须属于 `HostCapabilitySnapshot`
   - 不能只散在 adapter 实现或 `ExecutionAuthorization` 生成逻辑里
5. spawn 能力不得再只用 `supports_spawn_agents` 这种粗粒度布尔值表达
   - 必须显式建模 runtime 级能力
6. authority scope 不得再隐含在 helper session、slash command 或宿主私有状态里
   - 必须进入正式快照
7. `HostWorkerControlCapabilities` 必须是快照的一部分
   - pause/resume/cancel/interruption 文档只能引用它
8. 对未证实的 worker control 能力必须 fail closed
   - 不得乐观写成 `true`
9. 能力增强不自动扩大当前 `ExecutionAuthorization`
10. 能力缩减必须进入 `CapabilityDriftAssessment`
11. `HostSpawnCapability` 必须显式表达当前 runtime 下的 host agent scope
    - 至少区分 `All / ExplicitList / None / Unknown`
12. 不得再让 `allowed_host_agent_refs=[]` 同时承担：
    - “默认允许全部 agent”
    - “一个 agent 都不允许”
13. `session_scope` 必须显式带事实来源
    - 至少区分 `Authoritative / Derived / Unknown`
14. `session_scope.source != Authoritative`
    - 不得单独用来扩大 `ExecutionAuthorization`
    - 也不得单独用来扩大 control surface
15. `helperSessionKey`、session depth、slash command 上下文
    - 只能作为 fallback derivation 线索
    - 不得伪装成 authoritative authority state

补充说明：
1. v0 兼容 wire 里可能暂时还保留：
   - `available_agent_ids`
   - `supports_spawn_agents`
   - `supports_pause`
   - `supports_resume`
   - `supports_interrupt`
2. 这些只能作为 compat projection
3. 不得继续占据 formal owner 位置

## 4. 正式对象定义
```rust
pub struct HostCapabilitySnapshot {
    pub capability_snapshot_ref: HostCapabilitySnapshotRef,
    pub host_kind: HostKind,
    pub host_session_id: HostSessionId,
    pub available_agents: Vec<HostAgentCapability>,
    pub available_models: Vec<HostModelCapability>,
    pub available_tools: Vec<HostToolCapability>,
    pub spawn_capabilities: Vec<HostSpawnCapability>,
    pub session_scope: HostSessionCapabilityScope,
    pub readable_roots: Vec<String>,
    pub writable_roots: Vec<String>,
    pub secret_classes: Vec<String>,
    pub max_budget_band: AuthorizationBudgetBand,
    pub render_capabilities: HostRenderCapabilities,
    pub background_task_support: bool,
    pub async_notice_support: bool,
    pub worker_control_capabilities: HostWorkerControlCapabilities,
    pub recorded_at: Timestamp,
}

pub struct HostAgentCapability {
    pub host_agent_ref: HostAgentRef,
    pub display_name: String,
    pub available: bool,
}

pub struct HostModelCapability {
    pub host_model_ref: HostModelRef,
    pub provider: String,
    pub available: bool,
}

pub struct HostToolCapability {
    pub tool_name: String,
    pub available: bool,
}

pub struct HostSpawnCapability {
    pub runtime_kind: HostSpawnRuntimeKind,
    pub available: bool,
    pub host_agent_scope: HostSpawnAgentScope,
    pub supports_resume_session: bool,
    pub supports_thread_spawn: bool,
    pub supports_parent_progress_stream: bool,
}

pub enum HostSpawnRuntimeKind {
    Subagent,
    Acp,
}

pub struct HostSpawnAgentScope {
    pub mode: HostSpawnAgentScopeMode,
    pub allowed_host_agent_refs: Vec<HostAgentRef>,
}

pub enum HostSpawnAgentScopeMode {
    All,
    ExplicitList,
    None,
    Unknown,
}

pub struct HostSessionCapabilityScope {
    pub session_role: HostSessionRole,
    pub control_scope: HostSessionControlScope,
    pub source: HostCapabilityFactSource,
}

pub enum HostSessionRole {
    Main,
    Orchestrator,
    Leaf,
    Unknown,
}

pub enum HostSessionControlScope {
    Children,
    None,
    Unknown,
}

pub enum HostCapabilityFactSource {
    Authoritative,
    Derived,
    Unknown,
}

pub struct HostRenderCapabilities {
    pub supports_text_render: bool,
    pub supports_inline_actions: bool,
    pub supports_message_suppression: bool,
}

pub struct HostWorkerControlCapabilities {
    pub supports_pause: bool,
    pub supports_resume: bool,
    pub supports_cancel: bool,
    pub supports_soft_interrupt: bool,
    pub supports_hard_interrupt: bool,
}
```

## 5. 关键字段分别代表什么
1. `capability_snapshot_ref`
   - 当前能力快照的稳定引用
2. `host_kind`
   - 这份快照来自哪个宿主类型
3. `host_session_id`
   - 这份快照作用于哪个宿主会话
   - 它解决“这是谁的能力事实”
4. `available_agents / available_models / available_tools`
   - 当前宿主会话现实可见且可映射的 agent、model、tool 清单
5. `spawn_capabilities`
   - 当前宿主会话在不同 runtime 下能否 spawn
   - 也回答是否支持 `resumeSessionId`、thread spawn、parent progress stream
   - `host_agent_scope.mode`
     解决当前 runtime 下是“默认允许全部 agent”“只允许显式列表”“全部禁止”还是“宿主未知”
6. `session_scope`
   - 当前宿主会话的 authority 作用域
   - 它解决当前会话是 `main / orchestrator / leaf`，以及能否控制 children
   - `session_scope.source`
     解决这份 authority 是来自宿主 authoritative state，还是 adapter fallback derivation
7. `readable_roots / writable_roots`
   - 当前宿主现实允许 Loom worker 读取和写入的绝对根路径边界
8. `secret_classes`
   - 当前宿主现实允许 worker 接触的 secret class 上限
9. `max_budget_band`
   - 当前宿主现实允许的最高预算带
10. `render_capabilities`
    - 当前宿主文本渲染与动作呈现现实
11. `background_task_support / async_notice_support`
    - 当前宿主是否支持后台任务与异步通知
12. `worker_control_capabilities`
    - 正在运行的 worker 到底能否被 pause、resume、cancel 或 interrupt
13. `recorded_at`
    - 这份快照被 adapter 记录并同步的时间点

## 6. 和其它对象的关系
### 6.1 和 `AgentBinding`
固定关系：
1. `Harness` 先读取最新 `HostCapabilitySnapshot`
2. 再结合 `Pack / AgentPool / ExecutionAuthorization`
   收敛出 `AgentBinding`
3. `spawn_capabilities`
   - 直接影响 `BackgroundWorker` 是否能落到 `runtime=subagent` 或 `runtime=acp`
   - `host_agent_scope.mode=Unknown`
     不得被当成“默认允许全部 agent”

### 6.2 和 `ExecutionAuthorization`
固定关系：
1. `HostCapabilitySnapshot`
   - 是 lease 收敛的现实约束
2. `readable_roots / writable_roots / secret_classes / max_budget_band`
   - 直接构成授权上限输入
3. 当前宿主真实有能力
   - 不等于当前 run 自动放行
4. `session_scope.source=Derived / Unknown`
   - 只能用来收紧授权、要求重判或 fail closed
   - 不能单独用来放大 worker roundtrip 或 child control 权限

### 6.3 和 `CapabilityDriftAssessment`
固定关系：
1. 每次新快照进入后
2. Loom 都可以形成正式 drift assessment
3. assessment 再决定 keep、narrow、reissue、pause 或 require user decision
4. 以下变化都必须被视为 drift：
   - `spawn_capabilities`
   - `session_scope`
   - `worker_control_capabilities`
   - `readable_roots / writable_roots / secret_classes / max_budget_band`
   - 其中至少要显式比较：
     - `spawn_capabilities[].host_agent_scope`
     - `session_scope.source`

### 6.4 和 `HostWorkerControlCapabilities`
固定关系：
1. 它是 `HostCapabilitySnapshot` 的子对象
2. pause/resume/cancel 文档只能引用它
3. 不再单独持有字段 owner

### 6.5 和 `HostExecutionCommand`
固定关系：
1. `HostExecutionCommand`
   - 决定这次实际派给哪个宿主角色和哪个 runtime
2. `HostCapabilitySnapshot`
   - 决定这个派发在现实里是否成立
3. `helperSessionKey`
   - 只属于 adapter-local dispatch 句柄
   - 不得替代 `host_session_id` 或 `session_scope`
4. 如果 adapter 只能从 session depth 或本地 dispatch 拿到 role 线索
   - 写入正式快照时必须标成 `session_scope.source=Derived`

## 7. OpenClaw 3.11 对这份合同的修正意义
OpenClaw `v2026.3.11` 把几件以前容易被实现层“乐观处理”的语义做实了：
1. `sessions_spawn(runtime=\"acp\")`
   - 新增 `resumeSessionId`
2. subagent authority
   - 明确持久化 `leaf / orchestrator`
   - 并把 control scope 作为恢复后的共享 ownership 检查依据
3. copied-workspace sandbox
   - child session 继承真实 workspace root
4. plugin runtime security
   - 未鉴权插件路由不再自动继承 admin 级 subagent scope
5. ACP policy
   - `allowedAgents` 为空代表默认允许全部 agent
   - 不再允许用空数组偷代这一语义

因此：
1. `HostCapabilitySnapshot` 不能再只表达“宿主总体有什么”
2. 它必须表达“当前宿主会话、当前 runtime、当前 authority scope 下到底有什么”
3. `spawn_capabilities` 和 `session_scope`
   - 不只要有值
   - 还要能表达这些值来自什么业务语义

## 8. 我的建议
1. `HostCapabilitySnapshot` 继续作为宿主无关正式输入合同存在
2. 但它必须是 session-scoped snapshot，而不是宿主全局常量表
3. `HostWorkerControlCapabilities` 继续由这份合同持有 owner
4. `spawn_capabilities + session_scope`
   - 现在应升为正式字段
5. adapter 真实实现时，任何 capability sync 都应优先写成这份对象
   - 不要先写宿主私有 projection 再事后倒推 formal contract
6. `host_agent_scope.mode=All`
   - 才表示“默认允许全部 host agent”
   - 空列表本身不再拥有这个语义
7. `session_scope.source`
   - 必须跟 role/control scope 一起进入正式快照
   - 因为 3.11 之后“值是什么”和“值从哪来”都会影响主链治理
