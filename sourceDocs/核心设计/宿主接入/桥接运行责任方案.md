# 桥接运行责任方案

更新时间：2026-03-11

---

## 1. 文档定位
这份文档定义 `LocalHttpBridge` 在 Loom 体系中的运行责任、接入责任与部署责任。

它回答：
1. `LocalHttpBridge` 归谁拥有运行时生命周期
2. `loom-openclaw` plugin 到底负责什么
3. 外部 launcher / supervisor 能做什么，不能做什么
4. 第一条 `coding_pack / COMPLEX / LocalHttpBridge` spike 该怎样切实现边界

固定边界：
1. 这份文档只定义 bridge runtime owner 与责任切分。
2. bootstrap credential、session secret、轮换与失效的正式协议，继续由 [桥接启动与鉴权合同.md](桥接启动与鉴权合同.md) 持有 owner。
3. OpenClaw 首条宿主闭环验证范围，继续由 [OpenClaw适配验证方案.md](OpenClaw适配验证方案.md) 持有 owner。

---

## 2. 为什么这件事必须写死
如果只写“bridge-first”而不写“谁负责把 bridge 跑起来”，会出现 4 个更深层冲突：
1. adapter 薄层边界会被打穿
   - plugin 从接入层变成 runtime host
2. bootstrap / auth 语义会变虚
   - adapter 既是接入方，又变成 listener owner
3. 多宿主 / 多 adapter 扩展会提前打架
   - 谁占端口、谁重启、谁是 bridge owner 会失去统一答案
4. `daemon-ready` 会被工程便利性反向绑死
   - v0 为了快把 bridge 塞回 plugin，后面再抽成 Loom runtime 会发生 owner 反转

所以这件事不是“工程上谁顺手”，而是：

**`LocalHttpBridge` 到底属于哪个 bounded context。**

---

## 3. 正式结论
### 3.1 owner 固定结论
`LocalHttpBridge` 是 **Loom runtime 的本地 transport boundary**。

固定 owner：
1. 运行时 owner
   - Loom
2. 接入 owner
   - `loom-openclaw`
3. 部署 owner
   - Loom launcher / supervisor / dev tooling

固定不采用：
1. plugin 持有 bridge 进程生命周期
2. 纯外部预启动作为默认产品责任

### 3.2 责任拆分结论
#### Loom 负责
1. 启动 / 停止 `LocalHttpBridge`
2. 绑定 loopback listener
3. 生成 `bridge_instance_id`
4. 生成、轮换和失效 bootstrap / session credential
5. 持有 bridge auth authoritative state
6. 暴露 ingress / outbound HTTP surface

#### `loom-openclaw` 负责
1. 发现 bridge target
2. 使用 bootstrap material 建立 peer session
3. 同步 `HostCapabilitySnapshot`
4. 做 peer 侧 health check
5. rotation mismatch 时重新 bootstrap
6. bridge 不可达或 peer 失效时 fail-closed

#### Loom launcher / supervisor / dev tooling 负责
1. 自动拉起 Loom bridge 进程
2. 守护 bridge 进程
3. 在本地开发时提供一键启动体验

固定说明：
1. 这层“自动拉起”属于 Loom 的部署责任。
2. 它不改变 bridge runtime owner 仍然属于 Loom。

---

## 4. 关键对象分别代表什么
1. `LocalHttpBridge`
   - Loom 对宿主暴露的 loopback transport surface
   - 它不是 plugin 内部 helper，也不是宿主真相源
2. `bridge_instance_id`
   - 当前这代 bridge listener 的实例 id
   - 它解决“adapter 现在连的是不是这代 Loom bridge”
3. `BridgeBootstrapTicket`
   - 一次性引导凭据
   - 它解决“plugin 第一次凭什么合法接进来”
4. `BridgeSessionCredential`
   - bootstrap 后持续使用的 bridge peer 凭据
   - 它解决“后续请求是不是来自同一个合法 peer”
5. `HostCapabilitySnapshot`
   - 宿主现实能力快照
   - 它是 plugin 同步给 Loom 的事实，不是 bridge 自己的运行配置
6. `bridge.baseUrl`
   - plugin 连接 bridge 的目标地址
   - 它是 discovery pointer，不代表 bridge 生命周期 owner
7. `bridge.runtimeRoot`
   - plugin 读取 Loom runtime 文件树的绝对根路径
   - bootstrap ticket、probe projection 等本地 bridge 文件都必须从它派生
   - 它不得依赖 `cwd` 推导
8. `host workspace root`
   - 宿主当前 agent / runtime context 提供的工作区根
   - `workspace_ref / readable_roots / writable_roots / gateway call cwd` 都必须从它派生
   - 如果宿主给不出 repo identity，`repo_ref` 可以为空，但不得硬编码或从 `cwd` 猜

---

## 5. 为什么不把运行责任给 plugin
### 5.1 会破坏 adapter 薄层边界
`loom-openclaw` 的正式定位是 adapter 薄层。

它只应该负责：
1. ingress / egress
2. 结构化映射
3. bootstrap
4. capability sync
5. peer 连接恢复

如果 plugin 自己拉起 bridge 进程，它就从 adapter 变成了 runtime host。

### 5.2 会让 bootstrap / auth 语义变形
bootstrap / session secret 设计的前提是：
1. Loom 拥有 listener
2. plugin 作为合法 peer 接入

如果 plugin 自己拉 listener，再去 bootstrap 自己，就会让“接入鉴权”退化成“自己给自己开门”。

### 5.3 会提前制造多宿主冲突
未来 Loom 不会只有 OpenClaw 一个宿主。

如果 bridge 进程 owner 在 plugin：
1. 多 adapter 会争抢端口
2. plugin reload 会绑架 bridge 生命周期
3. transport boundary 会跟某个宿主实现耦死

---

## 6. 为什么也不把默认责任留给纯外部预启动
纯外部预启动的优点是边界干净，但它有两个问题：
1. 它不能构成默认产品闭环
   - 实现完成后仍然依赖人工先准备 bridge
2. 它会把“谁负责把系统带到可运行状态”悬空

所以固定取舍是：
1. 外部预启动可以作为部署方式之一
2. 但它不是 v0 默认产品责任
3. 默认责任仍然属于 Loom runtime / Loom launcher

---

## 7. v0 spike 的正式责任切分
### 7.1 当前默认实现切法
第一条 `coding_pack / COMPLEX / LocalHttpBridge` spike，责任固定拆成三段：

#### Loom 侧
1. 新增可运行的 bridge launcher / serve 入口
2. 持有 `LoomHarness`
3. 运行 `LocalHttpBridge`
4. 持有 bootstrap / session credential 状态

#### plugin 侧
1. 读取 `bridge.baseUrl`
2. 读取绝对路径 `bridge.runtimeRoot`
3. 从 `bridge.runtimeRoot` 派生 bootstrap ticket 与其他 Loom runtime 文件路径
4. 发现 bridge 是否可达
5. 发起 bootstrap
6. 同步 `HostCapabilitySnapshot`
7. 从宿主 runtime context / agent config 解析 `host workspace root`
8. 建立 ingress / outbound polling / ack
9. peer 失效时显式报错并停止 managed lane

#### 部署 / 开发工具侧
1. 本地开发时可以一键拉起 Loom bridge
2. 正式环境可以通过 supervisor 托管 Loom bridge

### 7.2 当前明确不做
1. 不让 plugin 自己 spawn bridge 进程后再接回自己
2. 不把“请先手工启动 bridge”当成默认产品闭环
3. 不把 bridge 进程生命周期写进 `openclaw.json`

## 8. 失败语义
### 8.1 bridge 不可达
如果 plugin 无法连接 `bridge.baseUrl`：
1. 不得静默退回匿名 bridge 模式
2. 不得假设可以跳过 Loom 继续 managed lane
3. 必须显式报 bridge 前置失败

### 8.2 bootstrap 失败
如果 bootstrap ticket 缺失、过期、被消费或校验失败：
1. 不得继续发 ingress
2. 必须请求重新 bootstrap
3. 必须保留 fail-closed
4. 如果 `bridge.runtimeRoot` 不是绝对路径，也按 bootstrap 前置失败处理

### 8.3 rotation mismatch
如果 `rotation_epoch` 落后：
1. plugin 不得继续用旧 credential 发请求
2. 必须重新 bootstrap

### 8.4 Loom bridge 重启
如果 bridge 重启导致 `bridge_instance_id` 变化：
1. 旧 peer credential 默认失效
2. plugin 必须重新建立 bridge session

## 9. 观测与验收义务
### 9.1 Loom 侧至少要有的结构化日志
1. `bridge.runtime.started`
2. `bridge.runtime.stopped`
3. `bridge.bootstrap.ticket_issued`
4. `bridge.bootstrap.accepted`
5. `bridge.credential.rotated`
6. `bridge.credential.revoked`

### 9.2 plugin 侧至少要有的结构化日志
1. `bridge.peer.connecting`
2. `bridge.peer.bootstrap_started`
3. `bridge.peer.bootstrap_succeeded`
4. `bridge.peer.capability_sync_succeeded`
5. `bridge.peer.reconnect_requested`
6. `bridge.peer.fail_closed`

### 9.3 WebUI 单入口验收最少要证明
1. 宿主闭环在 bridge 存在时能真实接通
2. bridge peer 不是匿名接入
3. bridge 失效时 plugin 不会静默伪装成成功

---

## 10. 当前冻结结论
1. `LocalHttpBridge` 的运行责任归 Loom。
2. `loom-openclaw` 只负责 bridge peer 的 bootstrap / health / capability sync / reconnect。
3. bridge 可以由 Loom launcher、supervisor 或开发工具自动拉起，但这不改变 Loom 是 runtime owner。
4. 后续若实现要修改这条边界，必须先修改这份文档和相关 ADR，再改代码。
