# 出站投递生命周期合同

状态：contract draft  
定位：Loom durable outbox 与宿主投递生命周期正式规格  
更新时间：2026-03-10

---

## 1. 目的
这份 contract 定义 Loom 出站项从创建到宿主确认的完整流转。

它回答：
1. durable outbox 里的正式对象是什么
2. `delivery_id`、`delivery_status`、`ack_outbound` 各自代表什么
3. retry / expiry / terminal failure 应该怎样表达
4. 为什么 v0 采用 `at-least-once` 而不是 `exactly-once`

更深层上，它解决的是：

**让 start card、result summary、status notice 这些治理消息在 bridge 重放和宿主重启下仍然有一致的投递真相。**

---

## 2. 设计边界
### 2.1 这不是 `KernelOutboundPayload`
边界：
1. payload contract
   - 回答“发什么”
2. delivery lifecycle
   - 回答“怎么发、发到哪一步、失败怎么办”

### 2.2 这不是 `TaskEventModel`
边界：
1. `TaskEventModel`
   - 记录领域事实
2. `OutboundDelivery`
   - 记录宿主投递单元

### 2.3 v0 不承诺 `exactly-once`
固定边界：
1. v0 采用 `at-least-once`
2. adapter 负责使用 `delivery_id` 去重
3. Loom 负责 durable outbox 真相和 ack 状态

### 2.4 `DeliveryAttempt` 是 `OutboundDelivery` 的子实体
固定边界：
1. `OutboundDelivery`
   - 是 durable outbox 的 aggregate root
2. `DeliveryAttempt`
   - 只记录该 aggregate 内的一次尝试历史
3. `DeliveryAttempt` 不应被理解成独立 delivery aggregate

---

## 3. v0 固定规则
### 3.1 authoritative store 必须先提交 state + event + outbox
固定顺序：
1. task/state 变更
2. 对应 `TaskEvent`
3. 对应 `OutboundDelivery`
4. 然后 adapter 才能读取未 ack delivery

### 3.2 `ack_outbound` 只在宿主真正完成投递后触发
固定规则：
1. 不是 adapter 读到了就 ack
2. 也不是 adapter 开始发就 ack
3. 必须是宿主真正完成目标投递后才 ack

### 3.3 retry 必须 bounded
固定规则：
1. v0 允许 retry
2. 但 retry 必须有上限和 backoff
3. 不允许无限自转

### 3.4 终态失败不能静默丢失
固定规则：
1. delivery 最终失败时，必须进入：
   - `terminal_failed`
   - 或 `expired`
2. 不能只在日志里写一句就丢掉

## 4. 正式对象定义
### 4.1 Rust 形态建议
```rust
pub enum DeliveryStatus {
    Pending,
    Delivering,
    DeliveredUnacked,
    Acked,
    RetryScheduled,
    TerminalFailed,
    Expired,
}

pub struct OutboundDelivery {
    pub delivery_id: OutboundDeliveryId,
    pub managed_task_ref: ManagedTaskRef,
    pub correlation_id: CorrelationId,
    pub causation_id: Option<CausationId>,
    pub target_host_session_id: HostSessionId,
    pub payload: KernelOutboundPayload,
    pub delivery_status: DeliveryStatus,
    pub attempt_count: u32,
    pub max_attempts: u32,
    pub next_attempt_at: Option<Timestamp>,
    pub created_at: Timestamp,
    pub expires_at: Option<Timestamp>,
    pub last_error: Option<String>,
}

pub struct DeliveryAttempt {
    pub attempt_id: DeliveryAttemptId,
    pub delivery_id: OutboundDeliveryId,
    pub started_at: Timestamp,
    pub finished_at: Option<Timestamp>,
    pub outcome: DeliveryAttemptOutcome,
    pub error_detail: Option<String>,
}

pub enum DeliveryAttemptOutcome {
    Sent,
    HostRejected,
    BridgeUnavailable,
    Timeout,
    AdapterFailure,
}
```

### 4.2 关键变量分别代表什么
1. `delivery_id`
   - 这次投递的稳定 id
   - adapter 用它做去重和 ack
2. `delivery_status`
   - 当前 delivery 走到哪一步
3. `attempt_count`
   - 已尝试投递多少次
4. `correlation_id`
   - 这条 delivery 属于哪条更大的交互链
5. `expires_at`
   - 超过这个时间点，delivery 不能再继续重试

## 5. 状态机建议
### 5.1 主状态流转
建议状态机：
1. `pending`
2. `delivering`
3. `delivered_unacked`
4. `acked`

失败分支：
1. `retry_scheduled`
2. `terminal_failed`
3. `expired`

### 5.2 典型路径
#### 成功路径
1. `pending`
2. `delivering`
3. `delivered_unacked`
4. `acked`

#### 短暂失败路径
1. `pending`
2. `delivering`
3. `retry_scheduled`
4. `delivering`
5. `delivered_unacked`
6. `acked`

#### 最终失败路径
1. `pending`
2. `delivering`
3. `retry_scheduled`
4. `delivering`
5. `terminal_failed`

### 5.3 `expired`
`expired`
1. 表示 delivery 在 bounded retry 窗口内始终无法成功
2. 或该 delivery 的业务时效已失效

---

## 6. retry 与 backoff
### 6.1 v0 建议
建议最小字段：
1. `max_attempts`
2. `next_attempt_at`
3. `last_error`

### 6.2 取舍
1. v0 不需要复杂优先级队列
2. 但必须有：
   - bounded retry
   - 明确 backoff
   - 明确终态失败

---

## 7. `ack_outbound`
### 7.1 作用
`ack_outbound`
1. 表示宿主已经完成这条 delivery 的最终投递

### 7.2 固定规则
1. 必须带 `delivery_id`
2. 应由 adapter 在宿主成功后回写
3. 如果重复 ack 已 `acked` 的 delivery，应幂等成功
4. 对 `terminal_failed / expired` 的 delivery，不得再接受正常 ack

---

## 8. 和其它对象的边界
### 8.1 和 `TaskEventModel`
建议最少写这些事件：
1. `outbound_delivery_created`
2. `outbound_delivery_attempted`
3. `outbound_delivery_delivered_unacked`
4. `outbound_delivery_acked`
5. `outbound_delivery_retry_scheduled`
6. `outbound_delivery_terminal_failed`
7. `outbound_delivery_expired`

### 8.2 和 `loom-openclaw`
adapter 负责：
1. 拉取未 ack delivery
2. 把 payload 渲染并投递给宿主
3. 以 `delivery_id` 做宿主侧去重
4. 成功后回 `ack_outbound`

adapter 不负责：
1. 自己维护 authoritative delivery truth

---

## 9. 我的建议
1. SQLite 继续作为唯一 authoritative outbox store
2. `runtime/loom/events/` 和 `runtime/loom/host-bridges/openclaw/` 只做 projection
3. v0 保持 `at-least-once + dedupe`
4. retry、expired、terminal_failed 三条失败路径必须写成正式状态，不要停留在口头约定
