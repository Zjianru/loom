# 桥接启动与鉴权合同

状态：contract draft  
定位：`LocalHttpBridge` 的启动、鉴权、轮换与失效正式规格  
更新时间：2026-03-11

---

## 1. 目的
这份 contract 定义 Loom 与 `loom-openclaw` 之间的 bridge bootstrap 和 auth 边界。

它回答：
1. `LocalHttpBridge` 第一次怎么安全接起来
2. adapter 凭什么证明“我是合法的 `loom-openclaw`”
3. bridge secret 怎么生成、轮换、失效
4. 哪些东西允许落盘，哪些绝不能写进 projection

更深层上，它解决的是：

**让“loopback-only + shared bridge secret”从一句原则变成正式运行时协议。**

补充说明：
1. bridge runtime owner、plugin peer owner 与部署责任切分，由 [桥接运行责任方案.md](桥接运行责任方案.md) 持有 owner。
2. 本文只继续持有 bootstrap credential、session credential、rotation 与 fail-closed 协议。

---

## 2. 为什么必须单独成文
如果这层不单独定义，会出现 3 个更深层风险：
1. `LocalHttpBridge` 虽然只绑 loopback，但任意本机进程都能伪造 ingress
2. bridge secret 变成长生命周期明文配置，最终重新变成第二个隐式真相源
3. adapter 重启、Loom 重启、secret 轮换时，没有正式恢复与失效路径

这会直接污染：
1. `HostSemanticBundle`
   - 表示宿主主模型给 Loom 的结构化语义判断包
2. `ControlAction`
   - 表示宿主已经结构化好的正式用户治理动作
3. `TaskEventModel`
   - 表示 Loom authoritative 事件链

## 3. 设计边界
### 3.1 这不是用户身份认证
这层只管：
1. Loom runtime
2. `loom-openclaw` adapter

之间的 bridge 互认。

它不管：
1. 终端用户是谁
2. OpenClaw 自己的账号体系
3. agent 对用户的业务权限判定

### 3.2 这不是宿主消息语义协议
`HostSemanticBundle`
1. 回答“宿主主模型把这条输入理解成什么”

Bridge bootstrap / auth
1. 回答“这条结构化请求有没有资格进 Loom”

### 3.3 这不是 `approval-gate`
`approval-gate`
1. 回答“当前这轮执行是否有写权限”

Bridge auth
1. 回答“这次 adapter 请求是否来自合法 bridge peer”

---

## 4. v0 固定规则
### 4.1 `LocalHttpBridge` 必须只绑 loopback
固定规则：
1. bridge listener 只允许 loopback
2. 不接受外部网卡绑定
3. 不暴露公网、局域网或任意网段入口

### 4.2 bootstrap credential 和 session secret 必须分离
固定规则：
1. bootstrap credential
   - 只用于第一次建立 bridge session
2. session secret
   - 用于后续正常请求鉴权

原因：
1. 否则第一次发给 adapter 的引导材料会永久变成主密钥
2. secret 轮换也会变得非常脆弱

### 4.3 raw secret 不得写入 projection
固定规则：
1. raw bootstrap secret
2. raw shared bridge secret

都不得写进：
1. `runtime/loom/events/`
2. `runtime/loom/projections/`
3. `runtime/loom/host-bridges/openclaw/` 的兼容 projection

authoritative store 里也只应保留：
1. `secret_ref`
2. `secret_hash`
3. `rotation_epoch`
4. 状态和时间戳

### 4.4 轮换必须是一等能力
固定规则：
1. bridge secret 必须可以轮换
2. 轮换后旧 secret 必须失效
3. adapter 必须能检测 rotation mismatch 并重新 bootstrap

### 4.5 失败必须 fail closed
以下情况必须 fail closed：
1. bootstrap ticket 过期
2. bridge secret 缺失
3. `rotation_epoch` 不匹配
4. secret 校验失败
5. `bridge_instance_id` 不匹配

这类失败只能导致：
1. 拒绝 ingress
2. 请求重新 bootstrap

不能导致：
1. 退回匿名模式
2. 静默接收未鉴权请求

## 5. 正式对象定义
### 5.1 Rust 形态建议
```rust
pub struct BridgeInstanceId(pub String);
pub struct BridgeBootstrapTicketId(pub String);
pub struct BridgeSecretRef(pub String);

pub enum BridgePeerRole {
    Loom,
    LoomOpenClaw,
}

pub enum BridgeCredentialStatus {
    PendingBootstrap,
    Active,
    Rotating,
    Revoked,
    Expired,
}

pub struct BridgeBootstrapTicket {
    pub ticket_id: BridgeBootstrapTicketId,
    pub bridge_instance_id: BridgeInstanceId,
    pub adapter_id: AdapterId,
    pub issued_at: Timestamp,
    pub expires_at: Timestamp,
    pub ticket_secret_hash: SecretHash,
    pub status: BridgeCredentialStatus,
}

pub struct BridgeSessionCredential {
    pub credential_id: BridgeCredentialId,
    pub bridge_instance_id: BridgeInstanceId,
    pub adapter_id: AdapterId,
    pub secret_ref: BridgeSecretRef,
    pub secret_hash: SecretHash,
    pub rotation_epoch: u32,
    pub issued_at: Timestamp,
    pub expires_at: Option<Timestamp>,
    pub status: BridgeCredentialStatus,
}

pub struct BridgeAuthEnvelope {
    pub bridge_instance_id: BridgeInstanceId,
    pub adapter_id: AdapterId,
    pub secret_ref: BridgeSecretRef,
    pub rotation_epoch: u32,
    pub signed_at: Timestamp,
    pub nonce: BridgeNonce,
    pub signature: BridgeSignature,
}
```

再补两类 bridge payload：
```rust
pub struct BridgeBootstrapRequest {
    pub bridge_instance_id: BridgeInstanceId,
    pub adapter_id: AdapterId,
    pub ticket_id: BridgeBootstrapTicketId,
    pub ticket_secret: BridgeBootstrapSecret,
    pub requested_at: Timestamp,
}

pub struct BridgeBootstrapAck {
    pub bridge_instance_id: BridgeInstanceId,
    pub credential_id: BridgeCredentialId,
    pub secret_ref: BridgeSecretRef,
    pub rotation_epoch: u32,
    pub session_secret: BridgeSessionSecret,
    pub issued_at: Timestamp,
    pub expires_at: Option<Timestamp>,
}
```

补充冻结：
1. `session_secret`
   - 只在 bootstrap 响应里明文返回一次
2. adapter 只能把它保存在进程内存
3. 不得写入 projection
4. authoritative store 也只保留：
   - `secret_ref`
   - `secret_hash`
   - `rotation_epoch`

### 5.2 关键变量分别代表什么
1. `bridge_instance_id`
   - 这一次 Loom bridge listener 的运行实例 id
   - 它解决“adapter 正在连的是哪一代 Loom bridge”
2. `ticket_id`
   - 一次 bootstrap ticket 的稳定 id
   - 它解决“一次性引导材料是否已被消费”
3. `secret_ref`
   - 当前 active bridge secret 的引用 id
   - 它解决“这次签名用的是哪把 secret”
4. `rotation_epoch`
   - 这把 secret 属于第几轮轮换
   - 它解决“secret 是否已过期或落后”
5. `nonce`
   - 单次鉴权请求的随机值
   - 它解决“同一签名被重放”

## 6. v0 bootstrap 流程建议
### 6.1 启动
1. Loom 启动 `LocalHttpBridge`
2. 生成新的 `bridge_instance_id`
3. 生成一次性 `BridgeBootstrapTicket`
4. authoritative store 写入 ticket hash 和状态

固定说明：
1. 这里的“Loom 启动”
   - 指 Loom runtime 或 Loom launcher 负责拉起 bridge
2. 不由 `loom-openclaw` plugin 持有 bridge 进程生命周期

### 6.2 本地引导材料交付
v0 建议：
1. bootstrap ticket 只通过本地受限 bootstrap 渠道交给 `loom-openclaw`
2. 例如：
   - 当前用户私有权限的临时 bootstrap 文件
   - 或启动时显式注入的进程环境

取舍：
1. 这层材料可以是临时引导物
2. 但它不是 projection truth，也不是长期配置

当前默认路径冻结为：
1. `runtime/loom/bootstrap/openclaw/bootstrap-ticket.json`
2. 文件权限固定 `0600`

本地路径 owner 补充冻结为：
1. `bridge.runtimeRoot`
   - 是 Loom runtime 文件树的绝对根路径
   - 推荐入口是 `plugins.entries.loom-openclaw.config.bridge.runtimeRoot`
2. bootstrap ticket 路径不是独立配置 owner
   - 必须由 `join(bridge.runtimeRoot, "loom/bootstrap/openclaw/bootstrap-ticket.json")` 派生
3. `loom-openclaw` 不得依赖进程 `cwd` 去解析 bootstrap 路径
4. 相对路径只允许作为本地开发回退，不作为正式验收成功路径
5. 以后其他 Loom runtime 本地文件也必须从 `bridge.runtimeRoot` 派生，不允许再长出第二套单文件路径配置

### 6.3 建立 session secret
1. `loom-openclaw` 使用 bootstrap ticket 调用 bridge bootstrap endpoint
2. Loom 校验：
   - loopback 来源
   - ticket 是否存在
   - ticket 是否未过期
   - ticket 是否未被消费
3. Loom 激活 `BridgeSessionCredential`
4. adapter 改用 session secret 做后续请求签名
5. bootstrap ticket 立即失效

固定说明：
1. adapter 负责的是 peer 接入建立，不是 bridge runtime owner
2. 如果 bridge 不可达，adapter 必须请求重新 bootstrap 或 fail-closed，不得尝试自建 bridge listener

### 6.4 后续请求
后续每次 ingress/outbound ack 都必须带：
1. `BridgeAuthEnvelope`
2. 有效 `rotation_epoch`
3. 未复用的 `nonce`

当前 header contract 冻结为：
1. `x-loom-bridge-instance-id`
2. `x-loom-adapter-id`
3. `x-loom-secret-ref`
4. `x-loom-rotation-epoch`
5. `x-loom-signed-at`
6. `x-loom-nonce`
7. `x-loom-signature`

签名字符串固定为：
1. `method`
2. `path`
3. `body_sha256`
4. `signed_at`
5. `nonce`

按换行拼接后做 `HMAC-SHA256`。

## 7. 轮换与失效
### 7.1 何时轮换
建议触发：
1. Loom 重启
2. adapter 重连
3. secret 疑似泄露
4. 达到最大 session 生命周期

### 7.2 轮换后的处理
1. Loom 生成新的 `secret_ref + rotation_epoch`
2. 旧 secret 状态改成 `Rotating` 或 `Revoked`
3. adapter 下一次请求收到 epoch mismatch 后必须重新 bootstrap

### 7.3 失效后的处理
如果 bridge auth 失败：
1. 当前请求拒绝
2. 记录 bridge auth failure
3. 不推进任何 Loom 治理分支
4. 不创建新的 `managedTaskRef`
5. 不消费任何 `ControlAction`

## 8. 与其它对象的边界
### 8.1 和 `IngressMeta`
`IngressMeta`
1. 回答“这次 ingress 的幂等和因果链是什么”

Bridge auth
1. 回答“这次 ingress 是否来自合法 adapter peer”

两者必须同时存在，不能互相替代。

### 8.2 和 `TaskEventModel`
桥接认证本身不应该污染普通任务事件。  
但 bridge 层仍应产出自己的审计记录，至少包括：
1. bootstrap accepted/rejected
2. rotation started/completed
3. auth failed

### 8.3 和 `runtime/loom/host-bridges/openclaw/`
这个目录可以存在：
1. bridge mapping projection
2. delivery ack projection
3. bootstrap/reconnect 状态 projection

但不能存：
1. raw bootstrap secret
2. raw session secret

---

## 9. 我的建议
### 9.1 v0 最稳的取舍
我建议 v0 固定成：
1. `LocalHttpBridge`
2. loopback-only
3. bootstrap ticket + session secret 两阶段
4. authoritative store 只存 hash/ref/epoch
5. projection 只存状态，不存 raw secret

### 9.2 为什么不直接“一个本地 secret 用到底”
因为那会把：
1. bootstrap
2. 正常请求鉴权
3. 轮换
4. 重启恢复

全压到一把长期 secret 上。  
短期看简单，长期一定会变成隐式配置地雷。

### 9.3 我的建议
1. 这份 contract 要在真实代码 spike 前补上
2. 否则 `LocalHttpBridge` 会先跑起来，再补安全
3. 到那时 adapter 和 Loom 之间的 owner 边界就又会被“临时偷跑”污染
