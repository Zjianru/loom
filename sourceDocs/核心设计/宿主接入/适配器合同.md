# 适配器合同

状态：contract draft  
定位：`AdapterContract` 正式规格  
更新时间：2026-03-10

---

## 1. 目的
`AdapterContract` 解决一个最关键的问题：

**Loom 如何接入不同 personal AI terminal，而不被任何一个宿主绑死。**

它回答：
1. 宿主最少必须提供什么能力
2. 宿主负责什么
3. 内核负责什么
4. `session` 和 `managedTaskRef` 如何分离
5. 结构化协议如何在宿主里落成用户可见文本

---

## 2. 设计边界
### 2.1 Adapter 负责宿主映射，不负责治理判断
这次已经锁定：
1. adapter 负责和宿主 AI assistant 的一切映射
2. 语义判断由宿主大模型负责，adapter 负责把语义判断结果结构化转交给 kernel

但边界要明确：
1. 宿主语义层负责：
   - 从自然语言与上下文中得出 `interactionLane / taskActivationReason / managedTaskClass / WorkHorizon / task_change_classification / boundary_recommendation`
2. adapter 负责：
   - 消息输入输出
   - 宿主身份映射
   - 宿主语义判断结果映射
   - agent / model / tool / memory 能力映射
   - 结构化卡片落成宿主可见文本
3. kernel 负责：
   - 消费 `SemanticDecisionEnvelope`
   - 基于结构化判断做治理分发
   - task owner
   - stage/runtime/acceptance/governance

再补一条值域映射边界：
1. 宿主 schema 可使用小写 wire 值：
   - `complex / huge / max`
2. adapter 负责把它们归一化到 Loom 内部的正式档位：
   - `COMPLEX / HUGE / MAX`
3. adapter 不得把值域映射留给 kernel 临时猜

如果不拆开，会出现更深层冲突：
1. 宿主插件逻辑会反向污染内核真相源
2. 内核会被 OpenClaw 之类单一宿主绑死

### 2.2 协议层结构化，展示层文本化
这次已明确：
1. 协议层必须结构化
2. 展示给用户的必须是文本

这意味着：
1. 内核和 adapter 之间交流的不是自然语言拼接
2. 宿主最终给用户看的，不一定是结构化卡 UI，也可以是文本渲染

### 2.3 v0 只支持宿主消息流通知
这次已锁定：
1. `v0` 通知只走宿主消息流

不做：
1. 系统通知
2. 邮件
3. 独立 IM webhook

---

## 3. 最小适配能力
这次已经基本锁定，最小 adapter 能力应包括：

1. 消息输入
2. 任务卡展示
3. 用户动作回传
4. 通知通道
5. 结果回传
6. session / identity 映射
7. 后台任务支持
8. 异步通知支持
9. 宿主语义判断结果回传

这里最关键的两个不是 UI，而是：
1. 后台任务支持
2. 异步通知支持

如果宿主不支持这两点，你要的“任务扔出去后继续做别的事”就不成立。

## 4. 正式接口边界
### 4.1 Inbound
adapter 至少要能把这些东西送给 kernel：
```rust
pub struct CurrentTurnEnvelope {
    pub turn_id: CurrentTurnId,
    pub host_session_id: HostSessionId,
    pub host_message_id: HostMessageId,
    pub host_message_ref: Option<HostMessageRef>,
    pub actor_ref: HostActorRef,
    pub content: String,
    pub message_kind: HostMessageKind,
    pub host_context: HostContextSnapshot,
    pub received_at: Timestamp,
}
```

它表示：
1. 这是宿主里的哪条消息
2. 由谁发出
3. 在哪个宿主会话里
4. 当前宿主上下文是什么

固定分层：
1. `HostInboundTurn`
   - 是宿主 hook/bridge 侧的 transport 载体
2. `CurrentTurnEnvelope`
   - 是 adapter 归一化后真正送进 Loom 的正式入站对象
3. 不允许这两层各自长成一套并行 formal owner

详细字段以：
1. [当前轮输入合同.md](../语义与入口/当前轮输入合同.md)
   为准。

此外 adapter 还至少要能提交：
```rust
pub struct SemanticDecisionIngress {
    pub host_session_id: HostSessionId,
    pub host_message_ref: Option<HostMessageRef>,
    pub managed_task_ref: Option<ManagedTaskRef>,
    pub decision_envelope: SemanticDecisionEnvelope,
}
```

它表示：
1. adapter 已经把宿主的 `HostSemanticBundle` 归一化成了 kernel 可消费的 decision 单元
2. kernel 不再自己猜自然语言

adapter 还必须支持正式控制动作回传：
```rust
pub struct ControlActionIngress {
    pub host_session_id: HostSessionId,
    pub host_message_ref: Option<HostMessageRef>,
    pub managed_task_ref: Option<ManagedTaskRef>,
    pub action: ControlAction,
}
```

这里的边界是：
1. `ControlAction`
   - 仍然来自宿主语义层的结构化 judgment
2. adapter
   - 只转运、映射、补宿主上下文
3. kernel
   - 只消费正式动作，不解释用户原始文本

再补一条 ingress 层级边界：
1. `SemanticDecisionIngress / ControlActionIngress`
   - 只是 adapter transport carrier
2. 正式领域 owner
   - 仍是 `SemanticDecisionEnvelope / ControlAction`
3. `HostCapabilitySnapshot`
   - 作为正式宿主能力输入对象
   - 字段 contract 以 [宿主能力快照合同.md](宿主能力快照合同.md)
     为准

### 4.2 Outbound
kernel 至少要能把这些东西交给 adapter：
```rust
pub enum KernelOutboundPayload {
    StartCard(StartCardPayload),
    BoundaryCard(BoundaryCardPayload),
    ResultSummary(ResultSummaryPayload),
    ApprovalRequest(ApprovalRequestPayload),
    SuppressHostMessage(SuppressHostMessagePayload),
    ToolDecision(ToolDecisionPayload),
}
```

这些 payload 的正式字段定义，统一以：
1. [内核出站载荷合同.md](内核出站载荷合同.md)

为准。

取舍：
1. kernel 输出结构化出站 payload
2. adapter 负责把它变成宿主可展示的文本
3. `StatusNotice`
   - 仍只作为可选 watchdog 扩展存在
   - 当前不属于本轮最小正式 outbound payload 集
4. `SuppressHostMessagePayload / ToolDecisionPayload`
   - 已属于当前最小正式 outbound payload 集
   - 不能再只停留在 landing/spike 草稿里

再补一条 token 边界：
1. `StartCardPayload`
   - 必须带 `decision_token`
2. `BoundaryCardPayload`
   - 必须带 `decision_token`
3. 后续 `approve_start / keep_current_task / replace_active` 等控制动作
   - 必须显式回带该 token

再补一条 delivery 边界：
1. `KernelOutboundPayload`
   - 只表示出站内容
2. durable outbox 的创建、retry、ack、expired、terminal failure
   - 统一由 [出站投递生命周期合同.md](出站投递生命周期合同.md)
     定义

### 4.3 User Action Callback
adapter 必须支持把用户对卡片或文本动作的回复回传给 kernel。

例如：
1. 批准启动
2. 修改任务候选
3. 取消任务候选
4. 保留当前 active task
5. 替换当前 active task

这很重要，因为：
1. 你后面所有治理点都依赖结构化回传
2. 否则系统就会退回自然语言猜测

## 5. `session` 与 `managedTaskRef` 必须分离
这是整个 `AdapterContract` 里最关键的架构决定之一。

### 5.1 `session`
它是什么：
1. 宿主聊天容器

它表示：
1. 用户当前在哪个会话里聊天

### 5.2 `managedTaskRef`
它是什么：
1. 内核里的任务容器

它表示：
1. 单个受管任务的 owner

### 5.3 为什么必须分离
原因：
1. 用户可以在一个 session 里继续聊天
2. active task 只是这个 session 下挂着的一个受管任务
3. 如果把 session 和 task 混成同一个 owner，系统就会重新掉回“聊天被冻结、任务状态串线”的旧坑

我的建议：
1. adapter 只提供宿主 session identity
2. kernel 独立生成 `managedTaskRef`
3. adapter 再维护 host session 与 task 的映射视图

---

## 6. Agent / Model / Tool / Memory 映射职责
这次已锁定：
1. 这些映射都交给 adapter

### 6.1 Agent 映射
adapter 负责：
1. 把 `AgentCapabilityProfile` 映射到宿主真实 agent 配置

OpenClaw 示例：
1. 宿主真实 agent 配置可能来自 `openclaw.json`

### 6.2 Model 映射
adapter 负责：
1. 根据宿主提供的方式，把 `desired_model_policy` 映射成真实模型选择

### 6.3 Tool 映射
adapter 负责：
1. 把工具意图映射成宿主实际工具能力

### 6.4 Memory 映射
adapter 负责：
1. 把 `memory_mode` 映射成宿主真实可用记忆能力

更深层的意义：
1. kernel 不假设宿主一定支持 agent memory
2. kernel 只消费“已映射后的事实能力”

## 7. v0 OpenClaw 假设
### 7.1 当前唯一宿主是 OpenClaw
这次已明确：
1. `v0` 现在只有 OpenClaw 一个宿主

### 7.2 v0 不做宿主自定义卡片样式
这次已锁定：
1. `v0` 不做宿主自定义卡样式

这意味着：
1. 先固定语义
2. 不在 UI 皮肤层扩张

### 7.3 v0 通知只走宿主消息流
固定规则：
1. `watchdog` 的通知先进入宿主消息流
2. 不做独立通知通道

## 8. 建议的数据对象
### 8.1 `HostContextSnapshot`
它是什么：
1. 宿主当前能力和上下文快照

建议字段：
1. `available_agents`
2. `available_models`
3. `available_tools`
4. `memory_capabilities`
5. `render_capabilities`

### 8.2 `HostMappingRegistry`
它是什么：
1. adapter 维护的宿主映射表

它表示：
1. 哪个内核画像映射到哪个宿主 agent
2. 哪类模型偏好映射到哪个宿主模型
3. 哪类工具意图映射到哪个宿主工具

### 8.3 `RenderedTextPayload`
它是什么：
1. adapter 最终返回给用户的文本化内容

这很重要，因为：
1. 协议层是结构化
2. 用户层必须文本化
3. 两者中间需要一个稳定渲染层

---

## 9. 和其它对象的边界
### 9.1 和 `StagePackage`
`AdapterContract` 不定义阶段语义。  
它只负责：
1. 宿主如何展示阶段状态
2. 宿主如何把用户动作回传给 kernel

### 9.2 和 `AgentCapabilityProfile`
`AdapterContract` 不定义 agent 画像。  
它只负责：
1. 把画像映射到宿主真实 agent 配置

### 9.3 和 `AcceptancePolicy`
`AdapterContract` 不定义什么叫完成。  
它只负责：
1. 把结果渲染成宿主可见文本

### 9.4 和 `SemanticDecisionEnvelope`
`AdapterContract` 不定义语义 schema 本身。  
它只负责：
1. 把宿主大模型给出的结构化语义判断送到 kernel

kernel 负责：
1. 校验
2. 留痕
3. 治理执行
1. 把验收相关结构化事件展示给用户

---

## 10. 我的建议
`AdapterContract` 最容易被低估，但其实它决定你到底是在做：
1. 一个通用内核
2. 还是一套 OpenClaw 专用插件

我的建议是：
1. 从第一天起就把 adapter 写成正式 contract
2. 让内核只拥有治理真相
3. 让宿主只拥有映射和展示真相

这样以后你要支持别的 personal AI terminal 时，才不会把现有内核全部重写
