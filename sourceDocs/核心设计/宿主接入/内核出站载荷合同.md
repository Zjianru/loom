# 内核出站载荷合同

状态：contract draft  
定位：Loom 到宿主 adapter 的正式出站载荷规格  
更新时间：2026-03-12

---

## 1. 目的
这份 contract 定义 Loom 向宿主 adapter 发出的结构化 payload 到底长什么样。

它回答：
1. `StartCard / BoundaryCard / ResultSummary / ApprovalRequest / SuppressHostMessage / ToolDecision / StatusNotice` 这 7 类出站对象的最小字段是什么
2. 哪些字段是协议层强约束
3. 哪些字段只服务 adapter 文本渲染，而不是 UI 样式系统
4. 如何让 `decision_token`、`managed_task_ref`、`acceptance_verdict` 这些关键治理对象在出站时不丢失

更深层上，它解决的是：

**让 adapter 不再自己发明 payload 结构，而是消费 Loom 正式定义的出站协议。**

---

## 2. 设计边界
### 2.1 这不是 UI contract
这份 contract 只定义：
1. 协议层结构
2. 必填字段
3. 治理语义边界

它不定义：
1. 卡片组件长什么样
2. OpenClaw 最终用按钮、纯文本还是别的渲染形式
3. 宿主的样式和交互布局

### 2.2 这不是 `ResultContract`
边界：
1. `ResultContract`
   - 回答“任务最终交付什么结果包”
2. `ResultSummaryPayload`
   - 回答“这一轮 outbound 向宿主投递什么结果摘要”

### 2.3 这不是 `ControlAction`
边界：
1. 这份 contract 定义的是 Loom 发出去的 payload
2. `ControlAction`
   - 定义的是宿主回传给 Loom 的结构化治理动作

两者通过 `decision_token` 闭环，但不应混成同一对象。

再补一条窗口 owner 边界：
1. `StartCardPayload / BoundaryCardPayload / ApprovalRequestPayload`
   - 只表示用户可见 projection
2. authoritative pending window state
   - 仍由 Loom runtime 持有
   - 正式 owner 以 [决策窗口合同.md](../治理策略/决策窗口合同.md)
     为准
3. control action
   - 负责消费该窗口

---

## 3. v0 固定规则
### 3.1 payload 必须是协议层结构，不是自由文本
固定规则：
1. Loom 先产出结构化 payload
2. adapter 再把结构化 payload 渲染成宿主文本
3. 不允许 kernel 直接把“已拼好的用户文案”当成唯一 truth

### 3.2 所有消费 pending decision window 的 payload 都必须带 `decision_token`
这条规则至少覆盖：
1. `StartCardPayload`
2. `BoundaryCardPayload`
3. `ApprovalRequestPayload`

原因：
1. `decision_token`
   - 表示当前 pending decision window 的绑定令牌
2. 没有它，迟到批准或迟到替换就会串线

### 3.3 payload 只承载治理所需最小用户信息
固定规则：
1. payload 必须足够让 adapter 文本化
2. 但不应把全量内部调试信息直接塞进 payload
3. 详细内部证据继续留给：
   - `TaskEventModel`
   - `ResultContract.internal_appendix`
   - projections

### 3.4 `StatusNotice` 收口为最小正式 async notice payload
固定规则：
1. `watchdog` notice 的概念、投递语义和 runtime projection 已经存在
2. 当前分支先把 `StatusNoticePayload` 的字段合同冻结成最小正式协议
3. 这次只允许两类 notice：
   - `StageEntered`
   - `Blocked`
4. 它属于正式 `KernelOutboundPayload`
5. 但不等于“完整 watchdog cadence 已落稳”
   - aging / waiting-user / approval pending 等 notice 仍不在本轮进入
6. `StatusNotice`
   - 必须复用现有 durable outbox 主链
   - 在 adapter 侧归类为 `async_notice`
   - 不得抬成新的 control surface 或第二套 notice owner

## 4. 正式对象定义
### 4.1 Rust 形态建议
```rust
pub enum KernelOutboundPayload {
    StartCard(StartCardPayload),
    BoundaryCard(BoundaryCardPayload),
    ResultSummary(ResultSummaryPayload),
    ApprovalRequest(ApprovalRequestPayload),
    SuppressHostMessage(SuppressHostMessagePayload),
    ToolDecision(ToolDecisionPayload),
    StatusNotice(StatusNoticePayload),
}

pub struct RenderHint {
    pub tone: RenderTone,
    pub emphasis: RenderEmphasis,
    pub host_text_mode: HostTextMode,
}

pub enum ApprovalScope {
    ToolExecution,
    TaskChangeConfirmation,
    CapabilityDriftResolution,
    ReviewResolution,
}

pub enum ApprovalRequestAction {
    ApproveRequest,
    RejectRequest,
}

pub enum ToolDecisionValue {
    Allow,
    Deny,
    RequiresUserApproval,
}

pub enum SuppressionReason {
    InternalPrelude,
    InternalGovernanceTrace,
    ToolNoise,
    DuplicateHostMessage,
    ReplacedByStructuredPayload,
}
```

## 5. `StartCardPayload`
### 5.1 目的
`StartCardPayload`
1. 表示 candidate managed task 的正式用户入口卡

### 5.2 Rust 形态建议
```rust
pub struct StartCardPayload {
    pub managed_task_ref: ManagedTaskRef,
    pub decision_token: DecisionToken,
    pub managed_task_class: ManagedTaskClass,
    pub work_horizon: WorkHorizonKind,
    pub task_activation_reason: TaskActivationReason,
    pub title: String,
    pub summary: String,
    pub expected_outcome: String,
    pub recommended_pack_ref: Option<PackRef>,
    pub allowed_actions: Vec<StartCardAction>,
    pub render_hint: RenderHint,
}

pub enum StartCardAction {
    ApproveStart,
    ModifyCandidate,
    CancelCandidate,
}
```

### 5.3 最小字段含义
1. `managed_task_ref`
   - 这张 start card 绑定的是哪个 candidate task
2. `decision_token`
   - 这张卡所属的 pending decision window 令牌
3. `managed_task_class`
   - 当前建议档位，值为 `COMPLEX / HUGE / MAX`
4. `work_horizon`
   - 当前工作意图，如 `maintenance / improvement / extension / disruption`
5. `summary`
   - 给用户看的候选任务摘要
6. `allowed_actions`
   - 告诉 adapter 这张卡允许哪些结构化控制回复

### 5.4 固定规则
1. `decision_token` 必填
2. `summary` 必须用户可读
3. `allowed_actions` 至少要包含：
   - `ApproveStart`
   - `ModifyCandidate`
   - `CancelCandidate`

## 6. `BoundaryCardPayload`
### 6.1 目的
`BoundaryCardPayload`
1. 表示 active task 和第二项重任务之间的边界确认卡

### 6.2 Rust 形态建议
```rust
pub struct BoundaryCardPayload {
    pub managed_task_ref: ManagedTaskRef,
    pub candidate_managed_task_ref: ManagedTaskRef,
    pub decision_token: DecisionToken,
    pub active_task_summary: String,
    pub candidate_task_summary: String,
    pub boundary_reason: BoundaryReason,
    pub allowed_actions: Vec<BoundaryCardAction>,
    pub render_hint: RenderHint,
}

pub enum BoundaryCardAction {
    KeepCurrentTask,
    ReplaceActive,
}
```

### 6.3 最小字段含义
1. `managed_task_ref`
   - 当前 active managed task owner
2. `candidate_managed_task_ref`
   - 当前边界窗口里等待用户拍板的 candidate task owner
3. `decision_token`
   - 当前边界确认窗口的绑定令牌
4. `active_task_summary`
   - 当前 active task 的用户可见摘要
5. `candidate_task_summary`
   - 新进候选任务的用户可见摘要
6. `boundary_reason`
   - 为什么这次必须走边界确认

### 6.4 固定规则
1. `decision_token` 必填
2. 必须同时展示 active 和 candidate 两侧摘要
3. `allowed_actions` 只允许：
   - `KeepCurrentTask`
   - `ReplaceActive`
4. 它必须是当前 `pendingBoundaryConfirmation` 的用户可见 projection，而不是第二套边界真相源

## 7. `ResultSummaryPayload`
### 7.1 目的
`ResultSummaryPayload`
1. 表示最终回宿主聊天区的用户可见结果摘要

### 7.2 Rust 形态建议
```rust
pub struct ResultSummaryPayload {
    pub managed_task_ref: ManagedTaskRef,
    pub outcome: ResultOutcome,
    pub acceptance_verdict: AcceptanceResult,
    pub summary: String,
    pub final_scope_version: TaskScopeVersion,
    pub scope_revision_headline: Option<String>,
    pub proof_of_work_excerpt: ProofOfWorkExcerpt,
    pub next_actions_excerpt: Vec<NextActionItem>,
    pub render_hint: RenderHint,
}

pub struct ProofOfWorkExcerpt {
    pub run_summary: String,
    pub evidence_refs: Vec<EvidenceRef>,
    pub review_summary: Option<ReviewSummary>,
    pub artifact_manifest_excerpt: Vec<ArtifactManifestItem>,
    pub acceptance_basis_excerpt: Vec<String>,
}
```

### 7.3 最小字段含义
1. `outcome`
   - 这次任务最终终态，如 `completed / completed_with_notes / partial / blocked / cancelled / rework_exhausted`
2. `acceptance_verdict`
   - 正式验收结论
3. `summary`
   - 用户默认先看到的短总结
4. `final_scope_version`
   - 最终按哪一版 `TaskScopeSnapshot` 交付
5. `scope_revision_headline`
   - 动态变更后给用户看的最短范围修订摘要
6. `proof_of_work_excerpt`
   - 从完整 `ProofOfWorkBundle` 提炼出的宿主摘要 section
7. `next_actions_excerpt`
   - 后续建议动作的简版

### 7.4 固定规则
1. `acceptance_verdict` 必填
2. `summary` 必填
3. 它不能代替完整 `ResultContract`
4. `proof_of_work_excerpt` 只能摘录正式 `ProofOfWorkBundle`
5. adapter 文本渲染时不得把 internal appendix 直接暴露给用户

## 8. `ApprovalRequestPayload`
### 8.1 目的
`ApprovalRequestPayload`
1. 表示 Loom 对高风险动作的显式审批请求

### 8.2 Rust 形态建议
```rust
pub struct ApprovalRequestPayload {
    pub managed_task_ref: ManagedTaskRef,
    pub decision_token: DecisionToken,
    pub approval_scope: ApprovalScope,
    pub requested_action: String,
    pub why_now: String,
    pub risk_summary: String,
    pub allowed_actions: Vec<ApprovalRequestAction>,
    pub render_hint: RenderHint,
}
```

### 8.3 固定规则
1. `decision_token` 必填
2. `approval_scope`
   - 表示当前请求覆盖哪个边界范围
   - v0 最小值固定为：
     - `ToolExecution`
     - `TaskChangeConfirmation`
     - `CapabilityDriftResolution`
     - `ReviewResolution`
3. `why_now`
   - 必须解释为什么现在触发审批，而不是之前
4. `allowed_actions`
   - 只允许：
     - `ApproveRequest`
     - `RejectRequest`

## 9. `SuppressHostMessagePayload`
### 9.1 目的
`SuppressHostMessagePayload`
1. 告诉 adapter 哪些宿主原始消息不应该继续透给用户

### 9.2 Rust 形态建议
```rust
pub struct SuppressHostMessagePayload {
    pub host_message_ref: HostMessageRef,
    pub reason: SuppressionReason,
    pub replacement_outbound_ref: Option<OutboundDeliveryId>,
}
```

### 9.3 固定规则
1. 必须显式指出 `host_message_ref`
2. suppression 只能由 Loom 正式发出
3. adapter 不得因为“看起来像内部消息”就自己静默吞消息

## 10. `ToolDecisionPayload`
### 10.1 目的
`ToolDecisionPayload`
1. 表示 Loom 对某次真实工具调用的治理结果

### 10.2 Rust 形态建议
```rust
pub struct ToolDecisionPayload {
    pub tool_intent_ref: HostToolIntentRef,
    pub decision: ToolDecisionValue,
    pub rationale_summary: String,
    pub approval_request_ref: Option<DecisionToken>,
}
```

### 10.3 固定规则
1. `decision`
   - 只能是：
     - `Allow`
     - `Deny`
     - `RequiresUserApproval`
2. `RequiresUserApproval` 时，必须能关联到一个正式审批窗口

## 11. `StatusNoticePayload`
### 11.1 目的
`StatusNoticePayload`
1. 表示任务推进过程中的最小正式异步 notice
2. 当前只承接：
   - 阶段进入
   - 阶段阻塞

### 11.2 Rust 形态建议
```rust
pub struct StatusNoticePayload {
    pub managed_task_ref: ManagedTaskRef,
    pub notice_kind: StatusNoticeKind,
    pub stage_ref: PhasePlanEntryId,
    pub headline: String,
    pub summary: String,
    pub detail: Option<String>,
    pub render_hint: RenderHint,
}

pub enum StatusNoticeKind {
    StageEntered,
    Blocked,
}
```

### 11.3 最小字段含义
1. `managed_task_ref`
   - 这条 notice 归属于哪个 `managedTaskRef`
2. `notice_kind`
   - 这条 notice 的正式语义类型
   - 当前只允许：
     - `StageEntered`
     - `Blocked`
3. `stage_ref`
   - 指向本次任务当前阶段条目的稳定标识
   - 它必须绑定到 `PhasePlanEntryId`
   - 不能退回成自由文本阶段名，也不能混成顶层 `workflowStage`
4. `headline`
   - 紧凑显示时的短标题
   - 给 adapter / WebUI 的 notice 收纳视图使用
5. `summary`
   - 用户默认先看到的一句话说明
6. `detail`
   - 可选补充解释
   - 只承载最小用户可见说明，不承载内部调试证据

### 11.4 固定规则
1. `stage_ref` 必填
2. `headline` 必填
3. `summary` 必填
4. `StageEntered`
   - 只应针对 `PhasePlanEntry.visibility=user_visible` 的阶段发出
5. `Blocked`
   - 允许引用 user-visible 或 internal 阶段
   - 但文案必须翻译成用户可理解的阻塞说明
6. `StatusNoticePayload`
   - 必须走现有 durable outbox + `ack_outbound` 主链
   - adapter 侧固定归类为 `async_notice`
   - 不参与 `current control surface` 投影
7. 它不得额外引入：
   - `notice_id`
   - `occurred_at`
   - `priority`
   - `cooldown_until`
   - `source_event_ref`
8. 上述 notice 去重、冷却和打断预算
   - 继续由 `watchdogNoticeState` 持有
   - 不得反向泄漏到用户可见 payload
9. `StatusNoticePayload`
   - 只表示“过程中的升级或进展”
   - 不得代替最终 `ResultSummaryPayload`
   - 尤其 `Blocked` notice 不等于最终 `outcome=blocked`

## 12. 和其它对象的边界
### 12.1 和 `OutboundDelivery`
这些 payload 只是：
1. 出站内容本体

真正的 delivery 生命周期由：
1. `OutboundDelivery`
2. `DeliveryAttempt`
3. `delivery_status`

负责。

### 12.2 和 `ResultContract`
`ResultSummaryPayload`
1. 是面向宿主文本层的摘要出站

完整结果由：
1. `ResultContract`
   - 持有
2. `ProofOfWorkBundle`
   - 作为正式证据包被 `ResultContract` 持有

### 12.3 和 `ReviewResult`
`ResultSummaryPayload.proof_of_work_excerpt.review_summary`
1. 应引用最近一次有效 `ReviewSummary`
2. 不应自己再发明一套 review 文案

### 12.4 和 `ProofOfWorkBundle`
`ResultSummaryPayload.proof_of_work_excerpt`
1. 只是 `ProofOfWorkBundle` 的宿主摘要切片
2. 可以摘录：
   - `run_summary`
   - `evidence_refs`
   - `review_summary`
   - `artifact_manifest`
   - `acceptance_basis`
3. 不能替代完整 `ProofOfWorkBundle`

---

## 13. 我的建议
1. 先把这 7 类 payload 写死成正式协议对象
2. adapter 只做文本渲染，不再自己拼 payload 结构
3. 所有 pending decision window 都必须通过 `decision_token` 闭环
4. `SuppressHostMessagePayload` 必须继续保留
   - 否则宿主内部治理噪音还会重新漏进主聊天区
5. `StatusNoticePayload`
   - 先按最小正式 async notice 收口
   - 不要一上来把完整 watchdog cadence 一次塞进 payload
