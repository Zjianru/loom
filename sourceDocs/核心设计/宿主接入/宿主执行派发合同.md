# 宿主执行派发合同

状态：contract draft  
定位：宿主无关的执行派发与子 agent 生命周期正式规格  
更新时间：2026-03-12

---

## 1. 目的
这份 contract 定义 Loom 把执行工作派发给宿主，再由宿主回传子 agent 生命周期事实的正式流转。

它回答：
1. `HostExecutionCommand` 是什么、不是什么
2. `ack_host_execution` 到底确认哪一步
3. `pending / dispatched / running / completed / failed` 各自代表什么
4. dispatch 失败、ack 失败、生命周期迟到或重复上报时应该怎样处理

更深层上，它解决的是：

**让“执行命令排队”“宿主真正接受派发”“子 agent 真实开始/结束”这三层事实各自有稳定 owner，而不是继续混在 adapter 临时状态里。**

---

## 2. 设计边界
### 2.1 这不是 `AgentBinding`
边界：
1. `AgentBinding`
   - 回答“这次 run 绑定了哪些角色和画像”
2. `HostExecutionCommand`
   - 回答“其中某个角色现在要怎样被派发给宿主执行”

### 2.2 这不是 `IsolatedTaskRun`
边界：
1. `IsolatedTaskRun`
   - 是 Loom 内核的执行期 run owner
2. `HostExecutionCommand`
   - 是该 run 之下的一条宿主派发单元
3. `HostSubagentLifecycleEnvelope`
   - 是宿主回传的执行观察事实

### 2.3 这不是 `HostCapabilitySnapshot`
边界：
1. `HostCapabilitySnapshot`
   - 回答宿主“现在能给什么能力”
2. `HostExecutionCommand`
   - 回答 Loom “这次实际要把哪项执行派出去”

### 2.4 v0 不承诺 `exactly-once` dispatch
固定边界：
1. v0 采用 `at-least-once` 派发
2. `command_id` 是这条派发命令的稳定幂等锚点
3. adapter 必须把 `command_id` 映射到宿主原生去重手段
4. Loom 继续持有 authoritative command truth 与终态

### 2.5 adapter-local dispatch handle 不是 formal owner
固定边界：
1. 不同宿主都可能需要一个 adapter-local 的派发句柄
2. OpenClaw 当前实现把它落成 `helperSessionKey`
3. 这个句柄只解决 bridge transport 与宿主调度，不是 Loom 正式对象
4. 只有宿主真实观测到的 child handle，才允许进入正式 lifecycle 对象
5. `helperSessionKey`
   - 也不得被拿来倒推 `session_scope` 或 authority source

### 2.6 `host_child_execution_ref / host_child_run_ref` 才是宿主无关正式字段
固定边界：
1. `host_child_execution_ref`
   - 语义上表示“宿主真实给出的 child execution handle”
2. `host_child_run_ref`
   - 语义上表示“宿主真实给出的 child run / attempt handle”
3. `child_session_key / child_run_id`
   - 只允许作为 session 型宿主的 compat alias 或 transport 字段
4. OpenClaw 当前 `childSessionKey`
   - 统一映射到 `host_child_execution_ref`
5. 兼容字段不得继续占据 formal owner 位置

## 3. v0 固定规则
### 3.1 authoritative store 必须先提交 command
固定顺序：
1. task / run / authorization 等状态变更
2. 对应 `TaskEvent`
3. 对应 `HostExecutionCommand`
4. 然后 adapter 才能读取待派发命令

### 3.2 `ack_host_execution` 只在宿主真正接受 dispatch 后触发
固定规则：
1. 不是 adapter 读到了命令就 ack
2. 也不是 adapter 刚准备发就 ack
3. 必须是宿主已经成功接受该次 dispatch 后，adapter 才能 ack

### 3.3 dispatch 失败前不得脱离 `pending`
固定规则：
1. 如果宿主还没真正接受 dispatch
2. 命令状态必须继续保持 `pending`
3. 不得因为一次 adapter 内部失败直接进入 `failed`

### 3.4 `Spawned / Ended` 是比 `ack_host_execution` 更强的宿主事实
固定规则：
1. `ack_host_execution`
   - 只证明“宿主已经接受派发”
2. `HostSubagentLifecycleEvent::Spawned`
   - 证明 child execution 已真实存在
3. `HostSubagentLifecycleEvent::Ended`
   - 证明该 child execution 已真实结束
4. 如果 lifecycle 先于 ack 到达，Loom 必须优先接受 lifecycle 事实，不得把它当无效噪音丢掉

### 3.5 一条命令在 v0 只对应一条逻辑 child execution 线
固定规则：
1. 一条 `HostExecutionCommand`
   - 代表一次逻辑派发
2. 它可以经历不确定重放
3. 但不得因为重放产生两条都被视为成功的逻辑 child execution 线

### 3.6 `ack_host_execution` 与终态 lifecycle 都必须幂等
固定规则：
1. 重复 ack 已进入 `dispatched / running / completed / failed` 的命令，应幂等成功
2. 重复 `Spawned` 或重复 `Ended`，不得把同一命令推进出两条平行终态
3. 不得因为重复事件再次创建新的 review / result 链

## 4. 正式对象定义
### 4.1 Rust 形态建议
```rust
pub enum HostExecutionCommandStatus {
    Pending,
    Dispatched,
    Running,
    Completed,
    Failed,
}

pub struct HostExecutionCommand {
    pub command_id: HostExecutionCommandId,
    pub managed_task_ref: ManagedTaskRef,
    pub run_ref: IsolatedTaskRunRef,
    pub binding_id: AgentBindingId,
    pub role_kind: AgentRoleKind,
    pub host_session_id: HostSessionId,
    pub host_agent_id: String,
    pub prompt: String,
    pub label: String,
    pub status: HostExecutionCommandStatus,
    pub host_child_execution_ref: Option<String>,
    pub host_child_run_ref: Option<String>,
    pub output_summary: Option<String>,
    pub artifact_refs: Vec<String>,
    pub issued_at: Timestamp,
    pub acked_at: Option<Timestamp>,
    pub completed_at: Option<Timestamp>,
}

pub struct HostSubagentLifecycleEnvelope {
    pub meta: IngressMeta,
    pub command_id: HostExecutionCommandId,
    pub managed_task_ref: ManagedTaskRef,
    pub run_ref: IsolatedTaskRunRef,
    pub role_kind: AgentRoleKind,
    pub event: HostSubagentLifecycleEvent,
}

pub enum HostSubagentLifecycleEvent {
    Spawned(SubagentSpawnedPayload),
    Ended(SubagentEndedPayload),
}

pub struct SubagentSpawnedPayload {
    pub host_child_execution_ref: String,
    pub host_child_run_ref: Option<String>,
    pub host_agent_id: String,
    pub observed_at: Timestamp,
}

pub struct SubagentEndedPayload {
    pub host_child_execution_ref: String,
    pub host_child_run_ref: Option<String>,
    pub host_agent_id: String,
    pub status: HostSubagentStatus,
    pub output_summary: String,
    pub artifact_refs: Vec<String>,
    pub observed_at: Timestamp,
}

pub fn next_host_execution(
    host_session_id: HostSessionId,
) -> LoomResult<Option<HostExecutionCommand>>;

pub fn ack_host_execution(
    command_id: HostExecutionCommandId,
) -> LoomResult<()>;
```

### 4.2 关键变量分别代表什么
1. `command_id`
   - 这条派发命令的稳定 id
   - 它是 ack、重放去重和 lifecycle 关联的主锚点
2. `managed_task_ref`
   - 这条命令属于哪个受管任务
3. `run_ref`
   - 这条命令属于哪次隔离 run
4. `binding_id`
   - 这条命令引用的是哪版 `AgentBinding`
   - 它不是 child execution 自己的 id
5. `role_kind`
   - 这条命令正在派发哪个角色
   - v0 典型值是 `worker / recorder`
6. `host_session_id`
   - 这条命令面向哪个宿主主会话或主执行容器
7. `host_agent_id`
   - 在当前宿主里实际要使用哪个 agent 映射
8. `status`
   - 这条命令当前走到哪一步
9. `acked_at`
   - 宿主成功接受 dispatch 的时间点
   - 它不是 child 真正启动时间
10. `host_child_execution_ref`
    - 宿主真实返回的 child execution handle
    - 它是宿主无关的中性正式引用
11. `host_child_run_ref`
    - 宿主真实返回的 child run handle
12. `output_summary`
    - 这次 child execution 结束后回传的摘要
13. `artifact_refs`
    - 这次 child execution 结束后回传的证据引用

### 4.3 adapter-local 变量的边界
1. `helperSessionKey`
   - OpenClaw 当前实现里，用于把内部 dispatch prompt 送给宿主的 helper 会话键
   - 它只属于 adapter runtime，不得进 authoritative store 当正式字段
   - 也不得被当成 `session_scope.source=Authoritative` 的依据
2. 宿主原生去重 token
   - adapter 依据 `command_id` 派生出的宿主原生幂等键
   - 它的宿主实现方式可以不同，但语义必须稳定

### 4.4 compat alias 的边界
固定规则：
1. authoritative aggregate 与 lifecycle contract
   - 一律使用 `host_child_execution_ref / host_child_run_ref`
2. `child_session_key / child_run_id`
   - 只能出现在宿主兼容 transport、旧 HTTP 字段或 adapter 内部映射层
3. compat alias 的值必须与正式字段一一对应
4. 不允许新文档、新领域模型继续把 compat alias 写成正式字段

## 5. 状态机建议
### 5.1 主状态流转
建议主链：
1. `pending`
2. `dispatched`
3. `running`
4. `completed / failed`

### 5.2 每个状态分别代表什么
1. `pending`
   - Loom 已排队
   - 但宿主还没有被正式确认已接受 dispatch
2. `dispatched`
   - adapter 已确认宿主接受了 dispatch
   - 但还没观测到 child execution 真正开始
3. `running`
   - 宿主已回传 `Spawned`
   - child execution 已真实存在
4. `completed`
   - 宿主已回传 `Ended`
   - 且 child execution 终态成功
5. `failed`
   - 宿主已回传 `Ended`
   - 且 child execution 终态失败、超时或取消

### 5.3 成功路径
1. `pending`
2. adapter 成功 dispatch
3. `ack_host_execution`
4. `dispatched`
5. `Spawned`
6. `running`
7. `Ended(status=completed)`
8. `completed`

### 5.4 lifecycle 先到的路径
允许路径：
1. `pending`
2. dispatch 已在宿主成功，但 ack 回写暂时失败
3. `Spawned`
4. `running`
5. 之后迟到的 `ack_host_execution`
   - 幂等成功
   - 不得把状态回退成 `dispatched`

### 5.5 终态直达路径
允许路径：
1. `pending` 或 `dispatched`
2. `Ended`
3. 直接进入 `completed` 或 `failed`

原因：
1. v0 不能要求宿主一定先稳定送到 `Spawned`
2. 如果 `Ended` 是目前能拿到的最强事实，就必须接受

### 5.6 明确禁止的状态误用
1. dispatch 失败
   - 不得把命令记成 `failed`
2. `ack_host_execution`
   - 不得代表“任务执行成功”
3. `completed / failed`
   - 只能由终态 lifecycle 推进
   - 不得只靠 adapter 自己猜测

## 6. 失败矩阵与恢复建议
### 6.1 adapter 读到命令，但 dispatch 失败
处理建议：
1. 命令继续保持 `pending`
2. `acked_at` 不写
3. adapter 可重试读取与 dispatch

### 6.2 dispatch 成功，但 `ack_host_execution` 回写失败
处理建议：
1. authoritative command 仍可能保持 `pending`
2. adapter 后续重试 dispatch 时，必须复用同一个 `command_id` 派生出的宿主幂等键
3. 宿主不得把这类重放视为新的逻辑 child execution
4. 如果 `Spawned / Ended` 已先到达，Loom 应直接以前者推进状态

### 6.3 bridge 或 adapter 重启
处理建议：
1. 恢复锚点优先级：
   - `command_id`
   - 宿主幂等键
   - 宿主回传的 child handle
2. 不得因为重启再生成新 `command_id`
3. 不得把旧 helper handle 当 authoritative owner

### 6.4 `Spawned` 缺失，但 `Ended` 已到达
处理建议：
1. 接受 `Ended`
2. 从 `Ended` 填回 `host_child_execution_ref / host_child_run_ref`
3. 直接进入终态

### 6.5 重复 `Ended`
处理建议：
1. 第一条终态为准
2. 后续重复终态只做幂等吸收
3. 不得重复创建 review/result 链

## 7. 和其它对象的边界
### 7.1 和 `TaskEventModel`
固定关系：
1. `HostExecutionCommand`
   - 持有 authoritative aggregate 状态
2. `TaskEventModel`
   - 持有这条链的时间序事件事实
3. 不允许 aggregate 状态和事件链分别再长出两套平行 owner

`TaskEventModel` 必须至少记录这些事件：
1. `host_execution_command_queued`
2. `host_execution_command_dispatched`
3. `host_execution_spawned`
4. `host_execution_completed`
5. `host_execution_failed`

补一条迟到事件规则：
1. 如果 `Spawned / Ended` 先于 `ack_host_execution` 到达
2. 事件链优先记录更强的 lifecycle 事实
3. 之后迟到的 `ack_host_execution`
   - 只更新 aggregate
   - 不得再补发语义上过时的 `host_execution_command_dispatched`

### 7.2 和 `ExecutionAuthorization`
固定关系：
1. authorization 决定这次 run 是否允许 spawn agent
2. `HostExecutionCommand`
   - 只承接一条已经被允许的派发
3. dispatch lifecycle 不得反向篡改 authorization owner

### 7.3 和 `ResultSummaryPayload`
固定关系：
1. worker / recorder 的终态输出
   - 可以成为后续 `ReviewResult / ProofOfWorkBundle / ResultSummaryPayload` 的输入证据
2. 但 `HostExecutionCommand` 本身不是最终结果包

### 7.4 和具体宿主 adapter
adapter 负责：
1. 把 `command_id` 映射成宿主原生幂等手段
2. 把宿主 child handle 映射回 lifecycle payload
3. 维护 adapter-local dispatch handle

adapter 不负责：
1. 自己维护 authoritative command truth
2. 自己猜测 `running / completed / failed`

---

## 8. 我的建议
1. 先把这条链收成宿主无关 contract，再让 OpenClaw 只负责映射
2. v0 先坚持 `at-least-once + command_id` 去重，不先引入 claim/release
3. `attempts / backoff / terminal dispatch failure` 可以在多消费者或高失败率出现后再补
4. 当前最重要的不是做复杂，而是先保证：
   - dispatch 失败不丢单
   - lifecycle 事实不被 adapter 临时状态覆盖
