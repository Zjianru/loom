# 敏捷模式吸收说明

状态：design draft  
定位：方法论吸收与映射文档  
更新时间：2026-03-11

---

## 1. 文档目的
这份文档回答 4 个问题：
1. 我们应该从哪些敏捷/组织方法里吸收概念。
2. 我们具体吸收什么，不吸收什么。
3. 这些概念应该映射到 Loom 的哪些正式对象。
4. 哪些属于内核主语义，哪些只适合进入特定 `StagePackage` 或 `Pack`。

这份文档不回答：
1. 当前代码已经实现了哪些方法论
2. 哪个框架更“正统”
3. 是否把外部框架原样搬进产品

我们的原则不是“复刻敏捷框架”，而是：

**把适合个人 AI 终端协作治理的部分吸收进来，把不适合的部分留在外面。**

---

## 2. 总体结论
Loom 最适合吸收的，不是某一个完整框架，而是一组分层零件：

1. `unFIX`
   - 用来定义 agent 编排、crew、forum、依赖治理、投资意图
2. `Scrum`
   - 用来定义阶段目标、阶段收口、完成标准、inspect/adapt
3. `Kanban`
   - 用来定义任务流、限流、显式策略、阻塞升级、主动提醒
4. `Management 3.0`
   - 用来定义授权等级和边界透明度
5. `Team Topologies`
   - 用来定义 agent 类型和 agent 间交互模式
6. `Shape Up`
   - 用来定义预算、下注、止损、冷却
7. `TDD / XP / ATDD`
   - 用来定义 coding package 的验证优先级和返工闭环

更深层的取舍是：
1. `unFIX` 和 `Team Topologies` 解决“谁和谁怎么合作”
2. `Scrum` 解决“阶段怎么设目标和收口”
3. `Kanban` 解决“工作怎么流动和怎么升级”
4. `Management 3.0` 解决“授权边界怎么透明”
5. `Shape Up` 解决“预算与止损”
6. `TDD / XP / ATDD` 解决“编码类任务怎么保证验证先行”

---

## 3. 设计原则
### 3.1 吸收语义，不吸收仪式负担
我们要吸收的是：
1. 目标
2. 边界
3. 反馈
4. 授权
5. 限流
6. 验收

不是：
1. 一整套人类团队仪式
2. 一堆要用户亲自参加的会议
3. 企业组织层的复杂治理

### 3.2 用户不该被 PM 术语淹没
普通用户未来看到的应该是：
1. 工作模式
2. 任务流程
3. 阶段目标
4. 进展提醒
5. 结果与验收

不应该被迫理解：
1. Scrum ceremony 全家桶
2. SAFe/LeSS 级组织术语
3. 人类团队管理游戏

### 3.3 方法论要服务内核对象
每次吸收一个概念，都要回答：
1. 映射到哪个正式对象
2. 由哪个运行线持有
3. 是不是进入 `kernel`，还是只进某个 `StagePackage`

---

## 4. 我们已经有的内核对象
为了避免方法论漂浮，先固定当前已经讨论出的对象：

1. `interactionLane`
   - 表示当前输入留在 `chat`，还是进入 managed task 路径。
2. `managedTaskClass`
   - 表示单个 managed task 属于 `COMPLEX / HUGE / MAX` 哪种协作框架。
3. `managedTaskRef`
   - 表示单个受管任务的一等对象。
4. `activeManagedTaskRef`
   - 表示当前唯一活跃任务指针。
5. `taskConversationPolicy`
   - 表示 active task 存在时主 assistant 是否继续开放，当前建议值为 `chat_open`。
6. `StagePool`
   - 表示可复用的阶段模板池。
7. `StagePackage`
   - 表示单个阶段模板。
8. `PhasePlan`
   - 表示任务本次选择了哪些阶段、顺序和依赖是什么。
9. `AgentPool`
   - 表示 agent 能力池。
10. `AgentCapabilityProfile`
    - 表示单个 agent 的能力画像。
11. `AgentBinding`
    - 表示某次任务/阶段真正启用的 agent roster。
12. `RecorderAgentRef`
    - 表示当前任务的记录者角色。
13. `watchdogNoticeState`
    - 表示通知到哪一步、哪些事件已发、哪些需要升级。
14. `ExecutionAuthorization`
    - 表示当前任务/阶段允许的工具、写路径、风险边界。
15. `AcceptancePolicy`
    - 表示完成标准、验收者、返工上限和升级条件。

## 5. unFIX：吸收 agent 组织，而不是整套组织设计
### 5.1 为什么适合我们
unFIX 官方把 `Crew` 定义成“带明确目标的小团队”，同时强调：
1. 任务有目标
2. 团队规模较小
3. 团队在较大范围内自治
4. 没有线经理在队内盯梢

这个定义和我们的 `AgentBinding` 很接近。  
来源：[unFIX Crew Types](https://unfix.com/crew-types)

### 5.2 我们吸收什么
#### `Crew`
我们吸收它的“带使命的小队”语义。

映射建议：
1. `AgentPool`
   - 是可选人手池。
2. `AgentBinding`
   - 是当前阶段真正上场的 crew。

取舍：
1. 我们不把 crew 理解成长期组织单元。
2. 我们把它理解成“阶段或任务内的短期执行小队”。

#### `Forum`
unFIX 把 forum 看成“讨论和做重要决策的地方”。  
这个概念非常适合我们。

映射建议：
1. `BoundaryForum`
   - 对应边界确认、替换 active task 的决策点。
2. `ReviewForum`
   - 对应验收与打回。
3. `EscalationForum`
   - 对应任务冲突、预算超限、风险升级。

取舍：
1. forum 不需要先变成 UI 大模块。
2. 但它很适合成为内核里的“结构化决策点”对象。

#### `Investment Horizons`
unFIX 的 `Maintenance / Improvement / Extension / Disruption` 值得吸收。  
来源：[unFIX Investment Horizons](https://unfix.com/investment-horizons)

映射建议：
1. `WorkHorizon`
   - 表示当前任务或阶段是在做维护、改进、扩展还是探索/颠覆。

它的价值：
1. 让系统知道这个阶段到底在保守推进，还是在探索性试错。
2. 影响 `AcceptancePolicy`、`BudgetPolicy`、`ReviewPolicy`。

#### `Dependency Breakers`
unFIX 的 `Contain / Decouple / Self-service / Swarm / Throttle / Visualize` 这类模式非常适合作为阶段设计检查表。  
来源：[unFIX Dependency Breakers](https://unfix.com/dependency-breakers)

映射建议：
1. `DependencyPolicy`
   - 表示当前阶段主要的依赖治理策略。
2. `StageDesignHints`
   - 作为 `StagePackage` 的设计元数据。

它的价值：
1. 当一个阶段依赖过多时，系统知道该拆阶段、换 agent、还是限流。

### 5.3 我们暂时不吸收什么
1. `Base`
2. `League`
3. 复杂的组织归属结构
4. 人类组织层面的线经理和 HR 结构

原因：
1. 这些更适合公司组织设计，不适合个人 AI 终端 v0。

### 5.4 我的建议
`unFIX` 最值得先进入 v0 的有 4 个：
1. `Crew`
2. `Forum`
3. `Investment Horizons`
4. `Dependency Breakers`

## 6. Scrum：吸收阶段目标和完成标准，不吸收会议负担
### 6.1 为什么适合我们
Scrum 官方定义了：
1. `Product Goal`
2. `Sprint Goal`
3. `Definition of Done`
4. artifacts 与 commitments

这些都很适合变成我们的阶段治理对象。  
来源：[Scrum Guide](https://scrumguides.org/), [2020 Scrum Guide PDF](https://scrumguides.org/docs/scrumguide/v2020/2020-Scrum-Guide-US.pdf)

### 6.2 我们吸收什么
#### `Product Goal`
映射建议：
1. `ManagedTaskGoal`

它表示：
1. 整个受管任务最终想达成的结果。

#### `Sprint Goal`
映射建议：
1. `StageGoal`

它表示：
1. 当前阶段的唯一核心目标。

取舍：
1. 每个 `StagePackage` 都应该要求一个明确的 `StageGoal`。
2. 否则阶段会变成杂乱的工作列表。

#### `Definition of Done`
映射建议：
1. `AcceptancePolicy.doneDefinition`

它表示：
1. 什么条件满足后，阶段或任务才算完成。

#### `Sprint Backlog is a forecast`
这是 Scrum 里非常值得吸收的一点。  
Sprint Backlog 是朝目标推进的计划，不是不可变的合同。

映射建议：
1. `PhasePlan`
   - 允许阶段内调整范围和工作项，只要不破坏 `StageGoal`。

#### `Review` 与 `Retrospective`
映射建议：
1. `ReviewStagePackage`
2. `RetroArtifact`

它们的价值：
1. 让结果被 review，而不是直接宣告完成。
2. 让 recorder 有机会沉淀阶段经验。

### 6.3 关于“3355”的建议
中文语境里经常用“3355”概括 Scrum。  
这个记忆术有帮助，但我不建议直接把它变成产品 schema。

原因：
1. 它更像教学记忆法
2. 我们真正需要的是：
   - 目标
   - 工件
   - 完成标准
   - inspect/adapt
3. 如果直接把“3355”原样做成 schema，产品会过于教条

我的建议：
1. `3355` 可以作为内部设计记忆术
2. 不要直接暴露给普通用户

### 6.4 我们暂时不吸收什么
1. 让用户显式参加所有 ceremony
2. Daily Scrum 式高频显式会议
3. 把术语直接塞进产品 UI

### 6.5 我的建议
`Scrum` 应该成为 `StagePool` 的主要语义来源。  
最优先吸收的对象是：
1. `ManagedTaskGoal`
2. `StageGoal`
3. `doneDefinition`
4. `Review`
5. `Retrospective` 产物

## 7. Kanban：吸收流动治理，这会直接增强 watchdog
### 7.1 为什么适合我们
Kanban University 的官方指南强调：
1. 流动要被真实展示
2. WIP limit 是显式策略的一种
3. pull 比 push 更重要
4. 要定期回顾服务与策略

来源：[The Official Guide to The Kanban Method](https://kanban.university/kanban-guide/)

### 7.2 我们吸收什么
#### `WIP limits`
映射建议：
1. `WipPolicy`

它表示：
1. 同时允许多少 active managed task
2. 某个阶段里同时允许多少活跃 worker
3. 某类任务是否需要更保守的并发上限

更深层的作用：
1. 把“只能一个 `activeManagedTaskRef`”上升成正式流动治理原则
2. 防止系统任务暴走

#### `Pull`
映射建议：
1. `ReplenishmentPolicy`

它表示：
1. 什么时候允许新的工作进入系统
2. 当前没有容量时，系统应该阻塞新 managed task 还是排队

#### `Make Policies Explicit`
Kanban 明确强调显式策略。  
来源：[Kanban Guide - Make Policies Explicit](https://kanban.university/kanban-guide/)

映射建议：
1. `FlowPolicy`
2. 各治理策略里的 `...escalation_policy`
3. `ReviewPolicy`
4. `ApprovalPolicy`

#### `Blocked / Aging / Escalation`
虽然官方指南不把 “aging” 单独当核心术语讲得很重，但 Kanban 实践里对阻塞和超时工作非常敏感。

映射建议：
1. `WorkItemState`
   - `queued / in_progress / blocked / waiting_review / done`
2. `AgingPolicy`
   - 某项工作超过多久没推进，就视为 aging。
3. 各治理策略里的 `...escalation_policy`
   - blocked 或 aging 到什么程度，`watchdog` 必须升级给用户。

### 7.3 为什么它和 `watchdog` 特别适配
`watchdog` 不该只是“定时发消息”。  
它应该是：

**一个流动治理观察者。**

它要观察的是：
1. 工作是否卡住
2. 是否超时未推进
3. 是否超出并发限制
4. 是否长期在 review 中反复打回
5. 是否到了必须通知用户的风险级别

映射结果：
1. `watchdogNoticeState`
   - 记住已经通知过什么。
2. `WorkItemState`
   - 描述任务/阶段当前流转状态。
3. `AgingPolicy`
   - 描述多久没推进就算异常。
4. 各治理策略里的 `...escalation_policy`
   - 描述异常到什么程度必须升级。

### 7.4 我们暂时不吸收什么
1. 把看板 UI 当成产品核心
2. 让用户必须自己维护看板

### 7.5 我的建议
`Kanban` 应该成为 `watchdog` 的主要方法论底盘。  
v0 最值得直接落成正式对象的是：
1. `WipPolicy`
2. `WorkItemState`
3. `AgingPolicy`
4. `...escalation_policy`
5. `ReplenishmentPolicy`

## 8. Management 3.0：吸收授权等级，不吸收团队游戏
### 8.1 为什么适合我们
Management 3.0 的 `Delegation Poker` 和 `Delegation Board` 非常适合翻译成执行边界。  
官方材料强调：
1. 七级授权
2. 关键决策区域
3. 通过 board 可视化授权边界

来源：[Delegation Poker & Delegation Board](https://management30.com/practice/delegation-poker/)

### 8.2 我们吸收什么
#### `Seven Levels of Delegation`
官方七级从 `Tell` 到 `Delegate`。  
来源：[Management 3.0 seven levels](https://management30.com/practice/delegation-poker/)

映射建议：
1. `DelegationLevel`

它表示：
1. 某类动作在当前阶段到底授权到什么级别。

典型应用：
1. 读文件
2. 写文件
3. 调用外部 API
4. 派生 worker
5. 修改任务计划

#### `Delegation Board`
映射建议：
1. `DecisionArea`
2. `AuthorizationMatrix`

它表示：
1. 不同决策区当前授权到什么程度。

这和 `approval-gate` 的关系非常紧密：
1. `approval-gate` 不是只回答“允许/拒绝”
2. 更成熟的做法是回答“在这类决策区，此时授权到哪一级”

### 8.3 为什么它很适合 `approval-gate`
`approval-gate` 现在最核心的原则是：
1. 未获同意，永不写入

但后续它一定会变得更细：
1. 哪些写可以自动做
2. 哪些写必须二次确认
3. 哪些写永远禁止
4. 哪些工具允许建议、但不允许直接执行

`DelegationLevel` 能给这条线一个稳定的语义。

### 8.4 我们暂时不吸收什么
1. Kudo Cards
2. 团队文化类小游戏
3. 泛化的激励框架

原因：
1. 这些更偏人类团队文化，不是内核主路径。

### 8.5 我的建议
`Management 3.0` 在 v0 最值得吸收的是：
1. `DelegationLevel`
2. `DecisionArea`
3. `AuthorizationMatrix`

## 9. Team Topologies：吸收 agent taxonomy 和交互模式
### 9.1 为什么适合我们
Team Topologies 官方给出了：
1. 四种 team topology
2. 三种 interaction mode

来源：[Team Topologies key concepts](https://teamtopologies.com/key-concepts)

### 9.2 我们吸收什么
#### 四种 team topology
1. `Stream-aligned`
2. `Enabling`
3. `Complicated Subsystem`
4. `Platform`

映射建议：
1. `AgentRoleTopology`

它的价值：
1. 让 `AgentPool` 的 agent 不只是“名字不同”，而是职责类型不同。

举例：
1. `Stream-aligned agent`
   - 某个工作流的主 worker
2. `Enabling agent`
   - 教练、reviewer、方法顾问
3. `Complicated subsystem agent`
   - 高专精 worker
4. `Platform agent`
   - 通用服务型 agent，例如连接器、知识检索、文档处理

#### 三种 interaction modes
1. `Collaboration`
2. `X-as-a-Service`
3. `Facilitation`

映射建议：
1. `AgentInteractionMode`

它表示：
1. 两个 agent 是短期共创
2. 还是服务调用
3. 还是辅导/帮助关系

### 9.3 对我们的价值
这套模型能避免 `AgentPool` 变成一堆无差别 agent。  
它还能帮助我们明确：
1. 哪些 agent 应该被长期当服务使用
2. 哪些 agent 只应短期加入某个阶段
3. 哪些 agent 只是 reviewer 或 facilitator

### 9.4 我们暂时不吸收什么
1. 全套组织设计语言
2. 大规模 team-of-teams 图谱

### 9.5 我的建议
`Team Topologies` 应该主要进入：
1. `AgentCapabilityProfile`
2. `AgentBinding`
3. `AgentInteractionMode`

## 10. Shape Up：吸收下注、预算和止损
### 10.1 为什么适合我们
Shape Up 强调：
1. `Bets, not backlogs`
2. 固定周期
3. cool-down
4. circuit breaker

来源：[Shape Up - The Betting Table](https://basecamp.com/shapeup/2.2-chapter-08)

### 10.2 我们吸收什么
#### `Bet`
映射建议：
1. `TaskBet`

它表示：
1. 这次不是无限堆需求，而是明确下注某个结果。

#### `Appetite`
映射建议：
1. `BudgetPolicy`

它表示：
1. 用户愿意投入多少时间、模型成本、子 agent 预算、返工轮次。

#### `Circuit Breaker`
映射建议：
1. `KillSwitchPolicy`

它表示：
1. 如果超预算、长期 blocked、review 多次失败，就停止推进并升级给用户。

#### `Cool-down`
映射建议：
1. `CooldownPolicy`

它表示：
1. 阶段结束后是否需要整理、复盘、归档，而不是立刻连续压榨系统。

### 10.3 为什么适合我们的 managed task
这和你的产品体验很吻合：
1. 用户把任务扔出去，不应该意味着系统无限自转
2. 任务必须有预算
3. 任务必须有止损
4. 任务不该靠无尽 backlog 驱动

### 10.4 我的建议
`Shape Up` 很适合进入：
1. `AcceptancePolicy`
2. `BudgetPolicy`
3. `KillSwitchPolicy`
4. `Pack` 的默认执行预算

## 11. TDD / XP / ATDD：进入 coding package 的验证治理
### 11.1 为什么适合我们
你刚才补的判断非常对：
1. coding task 不能等写完再想验证
2. 测试必须先行
3. reviewer 不能只看代码，还要看验证是否覆盖真实需求

Martin Fowler 对 TDD 的定义强调：
1. 先写一个会失败的测试
2. 写最少量代码让它通过
3. 再重构

来源：[Martin Fowler on TDD](https://martinfowler.com/bliki/TestDrivenDevelopment.html)

### 11.2 我们吸收什么
#### `TestFirstPolicy`
映射建议：
1. `TestPolicy.testFirst = true`

它表示：
1. 当前 coding package 的实现阶段必须先提出测试或可执行验证。

#### `Definition of Done` 与测试结合
映射建议：
1. `AcceptancePolicy.doneDefinition`
2. `VerificationStagePackage`

它表示：
1. 通过测试只是 done 的组成部分，但它必须先被形式化。

#### `Reviewer + Rework`
映射建议：
1. `ReviewAgentRefs`
2. `ReworkPolicy`

它表示：
1. reviewer 发现不满足测试或验收标准时，任务必须返工。

#### `ATDD / Specification by Example`
映射建议：
1. `AcceptanceExamples`

它表示：
1. 在写实现之前，先把需求转成可验证案例。

### 11.3 为什么它应该进 package，而不是进 kernel 全局
取舍很重要：
1. `TDD` 非常适合 coding task
2. 但不适合研究、内容、运营等所有任务

所以我的建议是：
1. `TDD / XP / ATDD` 主要进入 `coding package`
2. kernel 只提供：
   - `TestPolicy`
   - `VerificationStagePackage`
   - `ReviewAgentRefs`
   - `ReworkPolicy`
3. 具体执行规则由 package 决定

### 11.4 我的建议
coding package 最值得吸收的对象是：
1. `TestPolicy`
2. `VerificationStagePackage`
3. `ReviewAgentRefs`
4. `ReworkPolicy`
5. `AcceptanceExamples`

---

## 12. 我们暂时不建议吸收的框架
### 12.1 SAFe / LeSS / Disciplined Agile
原因：
1. 太偏大型组织扩展框架
2. 会把个人产品做重
3. 会把用户拉进企业流程术语

### 12.2 过多的人类团队仪式
原因：
1. 用户不是来学 PM 方法论
2. 我们的目标是自动化治理，不是让用户自己主持 ceremony

### 12.3 把“3355”硬编码成 schema
原因：
1. 更像记忆术，不像稳定 runtime contract
2. 容易让产品教条化

## 13. 推荐的吸收优先级
### 13.1 v0 必须吸收
1. `unFIX`
   - `Crew`
   - `Forum`
2. `Scrum`
   - `StageGoal`
   - `doneDefinition`
3. `Kanban`
   - `WipPolicy`
   - `WorkItemState`
   - `...escalation_policy`
4. `Management 3.0`
   - `DelegationLevel`
5. `TDD / XP`
   - `TestPolicy`
   - `VerificationStagePackage`

### 13.2 v0.5 很值得补
1. `unFIX`
   - `Investment Horizons`
   - `Dependency Breakers`
2. `Team Topologies`
   - `AgentRoleTopology`
   - `AgentInteractionMode`
3. `Shape Up`
   - `BudgetPolicy`
   - `KillSwitchPolicy`

### 13.3 后续再补
1. 更丰富的 retrospective 资产化
2. 更复杂的 forum 类型
3. 更细粒度的 review / rework matrix

## 14. 最终建议
如果我要把这份 intake 转成一句设计建议，我会这样说：

**用 unFIX 定义谁协作，用 Scrum 定义阶段，用 Kanban 定义流动，用 Management 3.0 定义授权，用 Team Topologies 定义 agent 类型，用 Shape Up 定义预算和止损，用 TDD/XP 定义 coding package 的验证优先级。**

这里最关键的两个新增结论是：
1. `watchdog` 最应该吸收 `Kanban` 的流动治理，而不是只做消息推送
2. `coding package` 最应该吸收 `TDD / review / rework`，把“能写代码”升级成“能稳定交付代码”
