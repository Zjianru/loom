## 1. 研究边界
这份文档回答 4 个问题：
1. 截至 2026-03-10，市面上主流的开源、开源+商业、闭源商业 multi-agent / agent workflow 方案有哪些。
2. 它们在多 agent 治理、阶段任务编排、可观测性、权限边界、记忆沉淀、用户入口这些维度上分别强调什么。
3. 和我们当前正在讨论的 `net + recorder + worker + watchdog + Harness + approval-gate` 方案相比，差异点在哪里。
4. 这个方向有没有商业价值，价值更可能来自哪里。

### 1.1 方法
本轮优先使用官方文档、官方产品页、官方 GitHub、官方新闻稿。

这意味着：
1. 信息更可信。
2. 但不保证覆盖所有边缘项目。
3. 某些产品公开披露较少，只能按公开材料判断。

### 1.2 一个必要前提
“所有方案”在绝对意义上不可穷尽。  
这里的“尽量全”按下列范围执行：
1. 主流开源框架
2. 主流开源+商业平台
3. 主流企业商业平台
4. 中文互联网中能公开核实、且与多 agent / agent workflow 明显相关的代表性产品

## 2. 先给结论
### 2.1 市场已经很拥挤
主流市场里，已经不缺：
1. supervisor + subagent
2. graph / workflow 编排
3. observability / tracing
4. low-code agent builder
5. agent + knowledge base + tools + integrations

也就是说，如果我们只是再做一个：
1. “多 agent 工作流平台”
2. “可视化编排器”
3. “带监控的 agent builder”

商业差异很弱。

### 2.2 我们真正不一样的，不是“多 agent”，而是“任务治理”
和外部方案相比，我们真正有机会做出差异的，不是：
1. 多几个 agent
2. 再加一个 workflow 画布

而是这几个更深层的点：
1. **`chat lane` 与 `managed task lane` 明确分离**
2. **`net` 在任务委派后保持自由，不被执行态锁死**
3. **`recorder` 成为 first-class role，而不是只靠 tracing/logging**
4. **`watchdog` 负责主动通知，但不抢主聊天**
5. **`Harness`、`approval-gate`、`watchdog`、`recorder` 职责分离**
6. **任务的暂停、替换、补充、边界确认是显式治理对象**

这不是“agent builder 差异”，而是“agentic work governance” 差异。

### 2.3 商业价值是存在的，但不是通用型大平台逻辑
如果我们想做“人人都能搭 agent”的大而全平台，会直接撞上：
1. LangChain / LangGraph
2. CrewAI
3. Google ADK
4. Dify
5. n8n
6. 腾讯 ADP
7. 百度千帆 AppBuilder
8. 阿里百炼
9. Agentforce
10. Copilot Studio
11. Bedrock Agents
12. Writer AI HQ
13. ServiceNow / SAP / IBM

但如果我们把自己定义成：
1. **任务治理内核**
2. **高价值复杂工作的协作操作系统**
3. **用户入口、委派执行、过程沉淀、异步通知、边界控制的统一治理层**

就有机会避开“通用搭建器”红海。

## 4. 主流方案到底怎么分
如果只看“是否有多个 agent”，会看不出真正差异。  
更重要的是它们的**控制结构**。

### 4.1 Supervisor / router 型
代表：
1. LangChain subagents
2. Bedrock supervisor/collaborators
3. Copilot Studio connected agents
4. Zapier agent-to-agent
5. 腾讯 ADP 自由转交

特点：
1. 有一个主 agent 决定把问题交给谁。
2. 子 agent 负责专业执行。
3. 大部分系统默认主 agent 一直处于 orchestration hot path。

优点：
1. 易理解。
2. 易扩展 agent catalog。
3. 易做企业权限与连接管理。

缺点：
1. 主 agent 很容易被持续占用。
2. 用户聊天与任务执行经常混在一个主流程里。
3. 过程沉淀通常退化成 logs / traces。

### 4.2 Graph / workflow 型
代表：
1. LangGraph
2. Google ADK
3. LlamaIndex custom planner
4. Spring AI Alibaba Graph
5. Dify Workflow
6. n8n
7. 腾讯 ADP 工作流编排
8. 阿里百炼 Workflow

特点：
1. 强调节点、边、状态、分支、并行。
2. 更像 orchestration engine，而不是会话系统。

优点：
1. 可控、可回放、可观测。
2. 适合阶段任务。
3. 企业交付感强。

缺点：
1. 很容易把“任务流程”做强，但把“用户入口体验”做弱。
2. 很多系统没有很好解决“任务执行时主聊天还要继续”的问题。

### 4.3 Role / SOP / digital workforce 型
代表：
1. MetaGPT
2. CAMEL
3. AgentScope
4. Manus
5. Agentforce
6. SAP Joule
7. IBM watsonx Orchestrate
8. Writer AI HQ

特点：
1. 强调角色、部门、数字劳动力、业务流程专家。
2. 比 graph 更接近“组织化协作”。

优点：
1. 更贴近业务方理解。
2. 更容易做“数字员工 / 数字劳动力”叙事。

缺点：
1. 很多系统把“组织结构”讲清楚了，但没有把“任务治理边界”讲清楚。
2. recorder / watchdog / approval 这些角色往往没有被真正拆成第一类对象。

## 5. 和我们的核心差异
这里的“我们”，指当前讨论中的 to-be：
1. `chat lane`
2. `managed task lane`
3. `COMPLEX / HUGE / MAX`
4. `net`
5. `recorder`
6. `worker`
7. `watchdog`
8. `Harness`
9. `approval-gate`

### 5.1 `interactionLane`
它是什么：
1. 顶层仲裁变量。

它表示：
1. 这条输入留在普通聊天。
2. 还是进入任务治理。

市场主流现状：
1. 大多数产品默认“既然装了 agent，就尽量让 agent 接所有输入”。
2. 少有系统把“chat lane”当成正式产品对象。

我们的差异：
1. 默认静默。
2. 默认先是聊天。
3. 只有需要治理的任务才激活插件。

这是一个非常重要的差异，因为它决定了产品是不是“默认侵入”。

### 5.2 `managedTaskClass`
它是什么：
1. 单个 managed task 的协作拓扑定义。

我们现在的方向：
1. `COMPLEX` = `1 + 1`
2. `HUGE` = `1 + n`
3. `MAX` = 分层树状协作

市场主流现状：
1. 大多数系统不是按协作拓扑分型，而是按能力包、业务域、或固定模板分工。
2. 很少有产品把“任务类型”直接定义为协作结构。

这会带来一个潜在优势：
1. 你的产品语言不是“任务大小”，而是“组织形态”。
2. 这更接近真实企业协作。

### 5.3 `net`
它是什么：
1. 主用户入口。

我们的设想：
1. `net` 负责用户对接、分派、变更入口、最终汇总。
2. 任务一旦委派出去，`net` 仍保持自由。

市场主流现状：
1. 很多 supervisor 系统让主 agent 一直挂在执行链上。
2. 这会导致：
   - 主聊天被占用
   - 用户补充需求容易被误吞
   - 长任务体验变差

这是我们一个很重要的产品差异点。

### 5.4 `recorderAgentRef`
它是什么：
1. managed task 的记录/沉淀角色。

市场主流现状：
1. 大多数系统有 tracing、logs、session history、knowledge graph。
2. 但很少把“记录者”当成显式角色。

这正是我们可能做出花样的地方：
1. 记录者不是被动日志。
2. 记录者是主动沉淀经验、阶段摘要、交付记忆的角色。

### 5.5 `watchdogNoticeState`
它是什么：
1. 异步通知去重与通知状态对象。

市场主流现状：
1. 有监控，有 tracing，有 runtime dashboards。
2. 但很少有产品把“对用户的主动通知”做成清晰的一等产品能力。

我们如果做对了，会和“企业监控台”不一样，而更像：
1. 任务活跃，但不打断主聊天。
2. 需要时主动汇报。

### 5.6 `approval-gate`
它是什么：
1. 写操作 / 执行边界的专门治理层。

市场主流现状：
1. Guardrails 很多。
2. 但很多 guardrails 是内容安全、模型输出规范、平台权限。
3. 真正把“写操作边界”单独做成运行时 follower 的并不多。

这是我们技术设计上相对硬核的地方，但商业包装上不一定显眼。

## 6. 我们能从他们身上学什么
### 6.1 学 LangGraph / ADK / Spring AI Alibaba
应该学：
1. graph/state runtime 的稳定性
2. long-running stateful workflows
3. nested / parallel / routing / supervisor 这些可复用编排原语

不要学：
1. 让 graph 成为整段聊天的默认宿主

### 6.2 学 AgentScope / AutoGen / OpenAI Agents SDK
应该学：
1. handoff 的简洁抽象
2. agent-to-agent conversation 的可组合性
3. traces / observability 的最小侵入设计

不要学：
1. 把所有复杂对话都自动扔进 multi-agent

### 6.3 学 CrewAI / Dify / n8n / 腾讯 ADP / 百炼
应该学：
1. builder 体验
2. workflow 与 agent 的桥接方式
3. 企业用户能理解的“模式切换”

尤其腾讯 ADP 的一点很值得看：
1. 它把 Multi-Agent 模式下的协同方式拆成：
   - 自由转交
   - 工作流编排
2. 这说明市场已经认识到：
   - “Agent 自由协作”
   - “稳定流程编排”
   其实不是同一个东西

这和我们的思路很接近，但我们要走得更远：
1. 再往上拆出 `chat lane`
2. 再把 `recorder` 和 `watchdog` 正式化

### 6.4 学 Agentforce / Writer / IBM / ServiceNow / SAP
应该学：
1. enterprise governance 的叙事方式
2. agent catalog / monitoring / ROI dashboard 的交付方式
3. “digital labor” 这个商业包装

不要学：
1. 仅仅把平台包装成“万能企业 agent 平台”

因为那会把我们卷入最贵、最泛的竞争层。

### 6.5 学 Manus
应该学：
1. 并行专用 agent
2. fresh context window
3. 主 agent 负责合成

这和你定义 `COMPLEX / HUGE / MAX` 时的分工特别接近。

不要学：
1. 只强调任务完成，不强调治理边界

### 6.6 世界上最像我们的，不是框架，而是几类“复杂工作代理”
如果只看“最像我们正在讨论的方向”，我认为最接近的不是通用 multi-agent framework，而是下面 5 类：

1. **Devin**
   - 像我们的地方：
     - 任务被明确委派到后台持续执行。
     - 用户入口和执行过程在体验上是分离的。
     - 更像“任务承包者”而不是一次性 chat tool call。
   - 不像我们的地方：
     - 公开材料里，它更像单强 agent + tooling + async session，不是我们这种显式 `net + recorder + watchdog + governed worker topology`。
     - 它强调 coding outcome、session continuity、knowledge，而不是任务边界治理。
   - 它的进展：
     - 已经从“能做 demo”走向商业交付与团队协作叙事，说明“后台持续执行型 agent”确实有付费市场。
   - 我们能学到：
     - 用户愿意为“我把事交出去，系统自己跑并回来交付”买单。
     - 但如果没有清晰治理边界，就更像强执行 agent，而不是复杂协作操作系统。

2. **Manus / Genspark 这类 general-purpose autonomous agent**
   - 像我们的地方：
     - 强调把任务正式交给系统，在后台持续推进。
     - 往往会并行调用不同能力模块，再由主代理整合。
   - 不像我们的地方：
     - 公共信息里更强调“自动完成任务”，而不是“受管协作中的边界确认、替换 active task、聊天不断线”。
     - recorder 通常不是 first-class role。
   - 它们的进展：
     - 市场热度很高，说明“复杂任务不是一次问答”这个方向被验证了。
     - 但公开产品叙事仍更偏 magic automation，而不是 explicit governance。
   - 我们能学到：
     - “把任务委派出去”本身有很强吸引力。
     - 但如果要做企业级和可持续产品，不能只卖 magic，要卖 controllability。

3. **Relevance AI / Writer AI HQ / Agentforce 这类 digital workforce 平台**
   - 像我们的地方：
     - 都在卖“数字劳动力”“agent team”“多角色协作”。
     - 都在强调监控、治理、可观测性、组织级交付。
   - 不像我们的地方：
     - 它们更偏平台、catalog、builder、ops。
     - 还没有把“用户单入口 + chat lane 持续开放 + active task 显式治理 + recorder 沉淀”做成强主叙事。
   - 它们的进展：
     - 商业化路径最清楚，说明企业愿意为 agent governance 付费。
   - 我们能学到：
     - 商业包装应该讲 ROI、治理、审计、数字劳动力，而不只是 agent 技术。
     - 后期一定要补监控、权限、报表、组织级控制。

4. **腾讯 ADP / 百炼 / 千帆 这类“workflow + multi-agent”平台**
   - 像我们的地方：
     - 已经把“自由转交”和“流程编排”分开。
     - 说明市场也意识到自由协作和稳定流程不是一回事。
   - 不像我们的地方：
     - 它们的主轴仍是 builder/platform。
     - 还没有把 `chat lane` 与 `managed task lane` 的分离做成产品第一原则。
   - 它们的进展：
     - 在中文互联网和企业交付侧推进很快，尤其是和知识库、模型、连接器、低代码一起卖。
   - 我们能学到：
     - 如果后续商业化，要非常清楚“自由协作”和“阶段编排”的模式切换。
     - 但不能把产品重心滑向通用 builder。

5. **Magentic-One / MetaGPT / AgentScope 这类“多角色复杂任务框架”**
   - 像我们的地方：
     - 都承认复杂任务需要角色分工、协调者、阶段或 ledger。
     - 比普通 supervisor-subagent 更接近“团队化任务执行”。
   - 不像我们的地方：
     - 它们大多还是 developer-facing framework 或 research artifact。
     - 很少把“主入口持续自由”“异步汇报不打断聊天”“recording 作为正式角色”做成产品行为。
   - 它们的进展：
     - 对研究和原型价值很高，但产品化和大规模商业落地相对弱于企业平台。
   - 我们能学到：
     - role/ledger/stage 这些思想是有用的。
     - 但必须把研究框架语言翻译成用户能买单的产品语言。

## 7. 商业价值判断
### 7.1 为什么我认为有价值
外部信号已经足够说明“agent”不是伪需求：
1. [MIT AI Agent Index](https://aiagentindex.mit.edu/) 已经把 30 个 prominent agents 做成持续索引。
2. [Gartner 2025 新闻稿](https://www.gartner.com/en/newsroom/press-releases/2025-06-11-gartner-predicts-that-guardian-agents-will-capture-10-15-percent-of-the-agentic-ai-market-by-2030) 直接预测到 2028 年 70% 的 AI app 会用 multi-agent。
3. [Zapier 2026 survey](https://zapier.com/blog/ai-agents-survey/) 显示 72% 企业已在使用或测试 AI agents，84% 准备继续加大投入。
4. [OpenAI enterprise report](https://openai.com/business/guides-and-resources/the-state-of-enterprise-ai-2025-report/) 也显示企业正把 AI 放进 repeatable, multi-step workflows。

所以价值不是“agent 有没有市场”，而是：
1. **哪一层还有空位**

### 7.2 哪些层已经太拥挤
已经很拥挤的层：
1. 通用 agent builder
2. workflow canvas
3. RAG + tools + actions
4. no-code AI automation
5. 企业 agent catalog

如果我们卖的是这些，差异不够。

### 7.3 哪些层仍有空位
我认为最有价值的空位有 3 个：

#### A. 用户入口与任务执行解耦
市场大量产品擅长“把任务跑起来”，但不擅长：
1. 任务跑着时，主对话还自然继续
2. 用户继续闲聊、补充、改任务
3. 旧任务又不丢失治理边界

这正是你的 `net + chat_open + watchdog` 思路。

#### B. recorder 驱动的任务记忆沉淀
市场大多有：
1. logs
2. traces
3. dashboards

但少有：
1. 主动形成阶段经验
2. 主动生成交付记忆
3. 把“过程沉淀”从“日志”升级成“系统能力”

这是我们很有希望形成产品差异的地方。

#### C. 复杂协作任务的边界治理
很多系统能并行 agent，但不擅长：
1. single active managed task
2. boundary confirmation
3. task mutation during execution
4. pause / replace / continue 的显式治理

这块更接近真正复杂工作的现实。

### 7.4 真正的商业前提
这个方向有价值，但前提是：
1. 不做“万金油 agent builder”
2. 要锚定高价值复杂工作

更适合的落地方向：
1. 编码与工程治理
2. 深度研究与报告生产
3. 复杂知识工作流
4. 有审计、权限、沉淀需求的企业内流程

如果只是做：
1. 聊天助手
2. 轻量自动化
3. FAQ / 客服小助手

你的设计会显得太重。

## 8. 我对我们方向的判断
### 8.1 值得继续，但要更聚焦
我建议继续，因为你现在的方向已经不是“再造一个 LangGraph”。

更准确地说，你在定义的是：
1. **complex work operating model**
2. **task governance kernel**

### 8.2 最可能成为卖点的，不是 `MAX`
最可能最先有商业价值的，反而是：
1. `COMPLEX`
2. `HUGE`

原因：
1. `COMPLEX` 对应“把一件专业任务正式委派出去，但我还要继续工作”的真实高频场景。
2. `HUGE` 对应“多角色协作，但仍需要单入口管控”的真实企业场景。
3. `MAX` 太强，容易先把系统做复杂。

### 8.3 你最有可能打穿的差异点
如果只选一个最有潜力的差异点，我会选：
1. **`net` 继续自由 + `recorder` 正式沉淀 + `watchdog` 异步通知**

因为这是一套完整的体验闭环：
1. 用户不被长任务绑死
2. 长任务不丢状态
3. 过程不只是 logs，而是可交付记忆

这比单纯“并行多 agent 更强”更像商业卖点。

## 9. 我现在最想问你的问题
### 9.1 你想卖的是“平台”，还是“复杂工作操作系统”
这是最重要的问题。

两种路线完全不同：
1. 平台路线
   - 会直接撞上 LangGraph、Dify、n8n、腾讯、百炼、Agentforce
2. 复杂工作操作系统路线
   - 更强调：
     - net 入口
     - recorder
     - watchdog
     - task governance
     - user continuity

### 9.2 recorder 到底是不是正式产品角色
如果是，就要正式定义：
1. 它的输入
2. 它的输出
3. 它的触发时机
4. 它如何进入记忆系统

如果不是，我们现在很多差异化判断会回退。

### 9.3 `COMPLEX` 默认要不要总是带 recorder
我倾向于：
1. `COMPLEX` 默认就带 recorder
2. 但 recorder 不必一直显性存在

### 9.4 `HUGE` 的阶段是固定模板，还是动态生成
这直接决定：
1. `Harness` 以后是 workflow runtime，还是 governance runtime

### 9.5 `MAX` 第一阶段要不要先降级
我建议：
1. 第一个可商用版本，不要把 `MAX` 做成 fully recursive managed-task tree
2. 先把它做成“更深的 worker tree，但仍属于同一个 `managedTaskRef`”
