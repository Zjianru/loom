# 分层治理装配模型

状态：background note  
定位：Loom 治理深化设计吸收后的装配视角背景说明  
更新时间：2026-03-11

---

## 1. 这份文档要回答什么
这份文档回答三件事：
1. 为什么 Loom 需要比当前更深的治理层
2. 为什么这些治理层不能对所有任务一刀切地全部启用
3. 已经确认的深化方向，应该优先复用哪些正式对象族，而不是继续长新 noun

这份文档不是正式 contract。

固定边界：
1. Loom 当前正式主链仍以 `managedTaskRef / PendingDecisionWindow / TaskScopeSnapshot / PhasePlan / SpecBundle / IsolatedTaskRun / ProofOfWorkBundle / OutboundDelivery` 为主
2. 这里讨论的六组深化能力，暂时都是**可装配治理深度**
3. 它们未来可以进入正式合同，但现在不应该被误读成“所有 managed task 的默认必选件”
4. 这轮已经冻结的收敛原则是：
   - 先复用 `Assessment / Policy / Scope / Snapshot / Binding / Ref`
   - 没有独立生命周期和独立真相源，就不要升成独立 noun
5. 当前第一层/第二层/第三层的推进状态，统一以实施与验收主链为准，本文件不再记录现行推进 gate

---

## 2. 为什么要把“更深治理”做成装配，而不是全局负担
如果把所有治理能力都做成全局默认，系统会立刻变重：
1. 轻量 `COMPLEX` 任务会背上不必要的状态和证明负担
2. 普通用户会被过多治理概念淹没
3. 实现层会被迫在所有路径都持久化大量不总是需要的状态
4. 很多 pack 的默认体验会变得僵硬

但如果完全不做这些更深治理，又会丢掉 Loom 最重要的产品差异点：
1. 决策为什么成立
2. 授权为什么给出
3. 多 agent 协作的责任如何界定
4. 结果为什么可信
5. 用户什么时候应该被打断
6. 工程上下文到底绑定了什么

所以当前冻结的总原则是：

**一套治理内核，按任务类型、风险等级、任务域，启用不同治理层。**

这里三个维度分别代表什么：
1. `PackContract`
   - 回答这类任务默认应该怎么治理
2. `managedTaskClass`
   - 回答这次需要多深、多严的协作结构
3. `RiskAssessment`
   - 回答这次具体任务是否需要再抬治理强度

重要边界：
1. `PackContract` 不是运行时状态
2. `managedTaskClass` 不是 pack 名，也不是任务大小
3. `risk_profile` 当前只作为 `RiskAssessment` 的结果视角存在，不再作为平行对象族

---

## 3. 哪些治理层属于所有 managed task 的基础层
当前所有 managed task 都应该默认具备的基础治理层是：
1. `managedTaskRef`
2. `PendingDecisionWindow`
3. `TaskScopeSnapshot`
4. `PhasePlan`
5. `SpecBundle`
6. `IsolatedTaskRun`
7. `ProofOfWorkBundle`
8. `OutboundDelivery`

这些对象代表的是 Loom 的最低治理骨架。  
后面的六组深化能力，都是建立在这层基础之上的可选深化层，而不是替代这条主链。

---

## 4. 当前冻结的装配优先级
模块挂载优先级先固定为：
1. `HostCapabilitySnapshot`
   - 先给出现实能力上限
2. 内核强制安全规则
   - 给出不可绕过的底线
3. `PackContract`
   - 给出场景默认挂载
4. `managedTaskClass`
   - 给出协作深度叠加
5. `RiskAssessment`
   - 给出本次任务或本次动作的风险升级
6. 用户显式覆盖
   - 但只能在 `override_policy` 允许范围内生效

固定规则：
1. 被能力约束挡住的深化层不能被强行启用
2. 被安全规则强制拉起的深化层不能被用户关闭
3. `managedTaskClass` 不直接替代 pack，而是在 pack 内抬高治理强度

## 5. 六组治理深化方向
### 5.1 决策依据层
这一层回答：
1. 为什么打开这个窗口
2. 为什么要求审批
3. 为什么判定需要返工或升级

当前冻结的收敛方式：
1. 复用 `Assessment` 家族
2. 以 `RiskAssessment` 作为风险判断 owner
3. 不新增平行记录对象族，具体禁用清单见第 8 节

`RiskAssessment` 当前代表什么：
1. 任务基线风险
   - 影响治理层默认挂载
2. 动作覆盖风险
   - 影响 `approval-gate` 与 `ExecutionAuthorization` 是否要继续收紧

当前已吸收进主线的固定取舍：
1. 风险固定分两层：`TaskBaseline` 与 `ActionOverride`
2. `overall_risk_band` 第一版固定为 `Low / Medium / High / Critical`
3. 用户可以显式上调风险，但不能把风险压到系统底线以下

第一版最小形态：
1. `overall_risk_band`
2. `risk_dimensions`
3. `trigger_reason`
4. `evidence_refs`
5. `derived_consequences`
6. `supersedes`

局限和代价：
1. 会引入额外留痕成本
2. 轻任务如果也全量启用，会显得过度官僚
3. 如果把它做成第二套大状态机，会和现有治理主链抢 owner

### 5.2 授权租约层
这一层回答：
1. 这次执行到底被允许做什么
2. 哪些行为只是读，哪些行为是写
3. 哪些动作必须二次确认
4. 授权会在什么条件下过期或被收回

当前冻结的收敛方式：
1. 继续深化 `ExecutionAuthorization`
2. 不新增平行授权范围对象，具体禁用清单见第 8 节
3. 把工具、写入、时间、预算四维继续压进 `ExecutionAuthorization.granted_areas`

固定关系：
1. `HostCapabilitySnapshot`
   - 现实能力上限
2. `TaskScopeSnapshot`
   - 任务级边界
3. `ExecutionAuthorization`
   - 当前 `IsolatedTaskRun` 的租约子集

统一公式：

`实际可执行范围 = HostCapabilitySnapshot ∩ TaskScopeSnapshot ∩ ExecutionAuthorization`

局限和代价：
1. 这层会显著增加授权状态复杂度
2. 如果 scope 设计得过细，用户会很难理解
3. 如果和任务范围边界混层，会长成两套授权系统

### 5.3 责任协作层
这一层回答：
1. 谁对这项任务最终负责
2. 谁只是执行者
3. 谁能 review 谁
4. handoff 后责任如何转移

当前冻结的收敛方式：
1. 第一版先复用 `AgentBinding`
2. 先把“谁实际上场”说清楚
3. 如确有 handoff 需求，再补最小 `HandoffContract`
4. 当前不引入更重的责任拓扑对象，具体禁用清单见第 8 节

如果未来真的需要更重责任对象，固定边界：
1. owner 必须在任务级
2. 阶段里只能保留 overlay 或 assignment
3. 不允许每个阶段各自持有一套责任真相

适合什么任务：
1. `HUGE`
2. `MAX`
3. 多 worker 协作任务
4. 对 review 公正性要求高的任务

局限和代价：
1. 太早引入会让 `COMPLEX` 的单 worker 体验过重
2. review 独立性和 quorum 现在还没有冻结，不宜提前做成重 contract

### 5.4 证明增强层
这一层回答：
1. 结果为什么可以被接受
2. 哪些证据支撑了当前交付
3. 哪些验证约束已经满足

当前冻结的收敛方式：
1. 第一版继续以 `AcceptancePolicy + ReviewResult + ProofOfWorkBundle + verification_doc` 形成闭环
2. 不引入独立接受账本对象，具体禁用清单见第 8 节
3. 如果后续确需正式接受记录，只能作为 `ProofOfWorkBundle` 的引用对象或快照来源

适合什么任务：
1. coding
2. research
3. 需要正式交付和复核的任务

局限和代价：
1. 如果 claim 粒度过细，会让结果链变重
2. 如果接受记录抢走 `ProofOfWorkBundle` 的 owner 地位，会形成双真相源

### 5.5 注意力治理层
这一层回答：
1. 什么时候应该打断用户
2. 什么时候应该合并提醒
3. 什么时候应该沉默
4. 用户的打断预算是否已经被耗尽

当前冻结的收敛方式：
1. 第一版先扩 `watchdogNoticeState`
2. 同时扩现有 `WipPolicy / ReworkPolicy / AcceptancePolicy` 里的 `...escalation_policy`
3. 不引入总注意力对象，具体禁用清单见第 8 节

如果未来引入用户级注意力对象，优先级固定为：
1. 安全强打断
2. 用户硬偏好
3. 任务级收紧
4. pack 默认

第一版最小能力：
1. `notice_priority`
2. `cooldown_until`
3. `interruption_budget_counter`
4. `interruption_budget_window_started_at`

适合什么任务：
1. 个人生活任务
2. 长期跟踪任务
3. 需要 `watchdog` 主动通知，但不能扰民的任务

局限和代价：
1. 这层对普通个人用户重要，但对纯工程任务并非第一优先级
2. 它天然需要更强的用户偏好数据，隐私和默认值都需要谨慎

### 5.6 工程上下文绑定层
这一层回答：
1. 这项任务到底绑定了哪个工作区、仓库和资源边界
2. 哪些路径和密钥类别属于这次任务的正式作用域

当前冻结的收敛方式：
1. 第一版先并入 `TaskScopeSnapshot`
2. 不先拆独立工程上下文对象，具体禁用清单见第 8 节

`TaskScopeSnapshot` 第一版当前承接：
1. `workspace_ref`
2. `repo_ref`
3. `allowed_roots`
4. `secret_classes`

适合什么任务：
1. coding
2. operations
3. 需要正式工程上下文的 research 任务

局限和代价：
1. 如果边界设计不稳，会和 `ExecutionAuthorization` 形成双重授权系统
2. 如果完全退回 prompt 文本，就会失去 Loom 的治理优势

## 6. 推荐装配矩阵
### 6.1 `coding_pack / COMPLEX / medium-risk`
建议默认：
1. 基础层
2. `RiskAssessment` 基础版
3. `TaskScopeSnapshot` 工程上下文基础版
4. 扩展版 `ExecutionAuthorization`

可选：
1. `ProofOfWorkBundle` 证据增强

暂不默认：
1. 重责任协作层
2. 注意力治理层

### 6.2 `coding_pack / HUGE / high-risk`
建议默认：
1. 基础层
2. `RiskAssessment`
3. 扩展版 `ExecutionAuthorization`
4. `AgentBinding` 强化协作约束
5. `TaskScopeSnapshot` 工程上下文强化
6. `ProofOfWorkBundle` 证据增强

如确有 handoff 需求，再补：
1. `HandoffContract`

### 6.3 `research_pack / COMPLEX / medium-risk`
建议默认：
1. 基础层
2. `RiskAssessment` 轻量版
3. `ProofOfWorkBundle` 证据增强

通常不优先：
1. 工程上下文完整版
2. 重责任协作层

### 6.4 `operations_pack / HUGE / high-risk`
建议默认：
1. 基础层
2. `RiskAssessment`
3. 扩展版 `ExecutionAuthorization`
4. `AgentBinding` 强化协作约束
5. `ProofOfWorkBundle` 证据增强

如果有明确工程或外部系统边界，再补：
1. `TaskScopeSnapshot` 工程上下文字段强化

### 6.5 未来个人任务 pack / `COMPLEX / low-risk`
建议默认：
1. 基础层
2. `watchdogNoticeState` 的注意力增强字段

可选：
1. `RiskAssessment` 轻量版

不建议默认：
1. 重证明增强层
2. 重责任协作层

## 7. 它和 `PackContract` 的关系
`PackContract` 当前已经非常适合承接这套装配思想，因为它本来就在回答：
1. 默认阶段骨架是什么
2. 默认角色骨架是什么
3. 默认验收、预算、WIP、delegation 策略是什么

往下自然延伸，就是：
1. 默认启用哪些治理深化层
2. 哪些深化层是可选件
3. 哪些深化层在高风险时必须被强制拉起

当前取舍：
1. 先承认 pack 是装配第一入口
2. 暂不把治理层挂载规则冻结成 `PackContract` 新字段
3. 等第一轮边界、owner、持久化策略稳定后，再决定是否增加正式字段

---

## 8. 当前明确不做的事
1. 不把六组深化能力全部升级成 v0 首条 spike 的默认正式对象
2. 不建立外部脚本式治理模块系统
3. 不让 adapter 或宿主直接决定这些深化层的开启与关闭
4. 不在 owner 未统一前继续发明平行 noun
5. 不让深化层长出第二条 result 链或第二条 notice 链

当前明确不新增的名字包括：
下面这组清单只作为“不要新增”的禁用 noun 清单，不代表未来待选主对象：
1. `PolicyEvaluationRecord`
2. `DecisionBasis`
3. `AuthorizationLeaseScope`
4. `ResponsibilityGraph`
5. `AcceptanceLedger`
6. `AttentionPolicy`
7. `WorkspaceBinding / RepoBinding / ResourceScope / SecretScope` 第一版独立对象

---

## 9. 当前最值得优先推进的顺序
1. 决策依据层
   - 直接解释“为什么这么治理”
2. 工程上下文绑定层
   - 最贴近 coding pack 的现实边界
3. 授权租约层
   - 最贴近安全和长期驻留内核
4. 证明增强层
   - 最贴近 Loom 的结果差异化
5. 责任协作层
   - 主要服务 `HUGE / MAX`
6. 注意力治理层
   - 更适合在个人任务 pack 成型后发力

这个顺序不是“价值排序”，而是“当前最适合进入设计收敛”的顺序。

## 11. 仍开放的背景主题
1. 模块 profile 最终是否需要进入 `PackContract` 正式字段，还是先保持实现外 profile id
2. `ProofOfWorkBundle` 若未来需要接受快照，应该保留到 claim 级还是 result 级
3. `watchdog` 的用户可见注意力配置语言后续是否要暴露更细颗粒度
4. `TaskScopeSnapshot` 后续是否值得把文件、仓库、外部服务边界再拆细
5. 何时值得把 `HandoffContract` 从讨论推进到正式合同

## 12. 当前判断
如果 Loom 未来要从任务治理内核往上长控制面，这六组深化能力不是可有可无的锦上添花，而是决定 Loom 是否真的有“深根”的关键能力。

但它们不应该被理解成：
1. 六个必须立刻落地的新 contract
2. 六套所有任务都要背上的固定负担

更合理的理解是：

**它们是 Loom 内核未来要支持的六组可装配治理深度。**
