# 建模收敛与边界治理讨论

状态：background note  
定位：以 OOP / DDD 视角解释 Loom 建模收敛原则的背景说明  
更新时间：2026-03-11

---

## 1. 这份文档要回答什么
这份文档回答四件事：
1. Loom 现在为什么容易继续长出新名词
2. 哪些情况下应该复用已有对象族，而不是继续新造 noun
3. 现有文档里哪些对象可以压缩成值对象、子实体、字段族或上下文内扩展
4. 哪些边界即使看起来相近，也不能压，否则会破坏 bounded context

这份文档不是正式 contract。  
当前生效的对象 owner、层级和字段边界，统一以：
1. [架构决策索引.md](../运行与治理/架构决策索引.md)
2. [正式对象总表.md](../运行与治理/正式对象总表.md)
3. [核心设计目录说明.md](../核心设计/目录说明.md)
为准。

本文件只保留“为什么要这样收敛”的背景说明与排除项，不再承接现行设计决定。
当前第一层/第二层/第三层的推进状态，统一以实施与验收主链为准。

## 2. 当前根问题
Loom 现在的文档并不是“概念不够”，而是已经进入了第二阶段风险：

1. 领域对象开始增多
2. 每个对象都看起来有道理
3. 但如果不收敛，就会出现名词森林
4. 名词一多，排查问题时就很难判断：
   - 这个对象是不是 owner
   - 它是不是只是别人的字段、子结构或 projection
   - 它和相邻对象到底差在哪个 context

这就是典型的 DDD 建模风险：
1. 不是“对象太少”
2. 而是“边界还没彻底理顺前，对象已经开始膨胀”

## 3. 我的建模收敛规则
### 3.1 只有满足三件事，才配升格成独立 noun
一个概念只有同时满足下面三件事，才值得在 Loom 里升级成独立正式对象：
1. 它有独立生命周期
2. 它有独立真相源
3. 它有独立状态转移或独立消费关系

只要缺一项，就优先降级为：
1. 值对象
2. 子实体
3. 嵌入字段族
4. 某个正式对象里的枚举或摘要

这条规则的意义是：
1. 避免“因为名字重要就单独立对象”
2. 避免“为了说明清楚，再造一个解释性 noun”

### 3.2 先复用对象族，再谈新对象
如果当前设计已经存在稳定对象族，优先复用它，而不是另开新族。

当前 Loom 已经存在的稳定对象族包括：
1. `...Assessment`
   - 评估结果族
2. `...Policy`
   - 规则族
3. `...Scope`
   - 作用域与边界族
4. `...Snapshot`
   - 某时刻事实快照族
5. `...Binding`
   - 实际绑定结果族
6. `...Ref`
   - 引用族

固定理解：
1. `Assessment`
   - 回答“评估完的治理结论是什么”
2. `Policy`
   - 回答“规则是什么”
3. `Scope`
   - 回答“原则边界是什么”
4. `Snapshot`
   - 回答“当前事实是什么”
5. `Binding`
   - 回答“这次真实上场了什么”

如果新概念其实只是这几类的局部展开，就不应该再发明新的顶层 noun。

### 3.3 先看 bounded context，再看字段长相
在 Loom 里，看起来字段相近，不代表应该合并。

先看它们处在哪个 context：
1. 宿主输入 context
2. 任务治理 context
3. 执行与授权 context
4. review / result context
5. notice / outbox context

只要处在不同 context，即使字段相似，也不能轻易压成一个对象。

## 4. Loom 当前可以稳定识别的 bounded contexts
### 4.1 宿主输入 context
owner 文档集中在：
1. `CurrentTurnEnvelope`
2. `HostSemanticBundle`
3. `SemanticDecisionEnvelope`
4. `HostCapabilitySnapshot`

它回答：
1. 当前输入事实是什么
2. 宿主结构化判断是什么
3. 宿主现实能力是什么

这里不能和任务治理 context 混。

### 4.2 任务治理 context
owner 文档集中在：
1. `managedTaskRef`
2. `TaskScopeSnapshot`
3. `PhasePlan`
4. `PendingDecisionWindow`
5. `ChangeImpactAssessment`

它回答：
1. 当前任务是什么
2. 当前范围是什么
3. 当前阶段结构是什么
4. 当前等什么决策
5. 当前 change 会影响什么

### 4.3 执行与授权 context
owner 文档集中在：
1. `AgentBinding`
2. `IsolatedTaskRun`
3. `ExecutionAuthorization`
4. `CapabilityDriftAssessment`

它回答：
1. 这次到底谁上场
2. 当前 run 是什么
3. 当前 run 被放行到什么程度
4. 宿主能力变化后会怎样影响执行

### 4.4 review / result context
owner 文档集中在：
1. `ReviewResult`
2. `ProofOfWorkBundle`
3. `ResultContract`

它回答：
1. review 怎么判
2. 证据包长什么样
3. 最终交付包长什么样

### 4.5 notice / outbox context
owner 文档集中在：
1. `watchdogNoticeState`
2. `KernelOutboundPayload`
3. `OutboundDelivery`

它回答：
1. 通知状态如何去重与升级
2. 内核想发什么
3. 这些东西投递到了哪一步

## 5. 哪些现有对象我认为可以压
### 5.1 `ReviewSummary`
当前状态：
1. 它已经作为 `ReviewResult.summary` 嵌在 `ReviewResult` 里
2. 但在部分文档阅读时，容易被误读成平行 formal object

我的判断：
1. `ReviewSummary` 更适合被视作 `ReviewResult` 的值对象
2. 它有结构，但不应该被当成和 `ReviewResult` 平级的 aggregate

原因：
1. 它没有独立生命周期
2. 它没有独立真相源
3. 它只是 `ReviewResult` 中供结果链消费的结构化摘要

固定建议：
1. 保留这个名字
2. 但把它理解成 `ReviewResult` 的成员值对象，而不是平行顶层对象

### 5.2 `DeliveryAttempt`
当前状态：
1. 它在 [出站投递生命周期合同.md](../核心设计/宿主接入/出站投递生命周期合同.md) 中有独立 struct
2. 但它始终围绕某个 `OutboundDelivery`

我的判断：
1. `DeliveryAttempt` 更像 `OutboundDelivery` aggregate 里的子实体
2. 它不应该被理解成独立投递 aggregate

原因：
1. 它没有离开 `OutboundDelivery` 的独立语义
2. 它只是在 delivery 生命周期里记录尝试历史

### 5.3 工程上下文绑定
我的判断：
1. 第一版不要立刻拆成一串新 noun
2. 更合理的是先并入 `TaskScopeSnapshot`

原因：
1. 工程上下文本质上就是任务范围的一部分
2. 先并进 `TaskScopeSnapshot`，更符合“任务当前到底被允许做什么”的 bounded context
3. 后面如果某一块真的长出独立生命周期，再拆不迟

第一版建议直接长字段：
1. `workspace_ref`
2. `repo_ref`
3. `allowed_roots`
4. `secret_classes`
5. 已排除的旧命名见第 7 节

### 5.4 决策依据层
我的判断：
1. 第一版不应该另开一条“记录对象族”
2. 更合理的是复用现有 `Assessment` 家族

原因：
1. 现在已经有：
   - `ChangeImpactAssessment`
   - `CapabilityDriftAssessment`
2. 未来的风险判断，也应该优先进入 `RiskAssessment`
3. 说明、依据、证据先作为 assessment 的字段族落地
4. 已排除的旧命名见第 7 节

### 5.5 注意力治理
我的判断：
1. 第一版不应该急着造一个大注意力聚合
2. 更合理的是先扩现有：
   - `watchdogNoticeState`
   - `WipPolicy.escalation_policy`
   - `ReworkPolicy.escalation_policy`
   - `AcceptancePolicy.escalation_policy`

原因：
1. Loom 现在已经有 notice state 和 escalation policy 链
2. 如果再单独造一条 attention 主链，很容易重复建模
3. 已排除的旧命名见第 7 节

### 5.6 `PendingBoundaryConfirmation`
当前状态：
1. 它在 [决策窗口合同.md](../核心设计/治理策略/决策窗口合同.md) 中是独立 struct
2. 但它始终绑定某个 `PendingDecisionWindow(kind=BoundaryConfirmation)`

我的判断：
1. 它可以继续保留名字
2. 但概念上更适合理解成 `PendingDecisionWindow` 的专门扩展子实体
3. 不应被理解成与 `PendingDecisionWindow` 平级的第二个窗口 aggregate

这里的取舍是：
1. 先不改 contract 名字
2. 先把它的层级理解收紧

## 6. 哪些现有对象不能压
### 6.1 `HostCapabilitySnapshot` 和 `ExecutionAuthorization`
这两个看起来都在讲“能力”，但不能压。

原因：
1. `HostCapabilitySnapshot`
   - 回答宿主现实能给什么
2. `ExecutionAuthorization`
   - 回答当前 run 真正拿到了什么

这是两个不同 bounded context：
1. 一个是现实事实输入
2. 一个是执行租约结果

### 6.2 `TaskScopeSnapshot` 和 `ChangeImpactAssessment`
这两个看起来都在讲“任务变化”，但不能压。

原因：
1. `TaskScopeSnapshot`
   - 回答当前正式范围本体
2. `ChangeImpactAssessment`
   - 回答这次 change 会影响哪些治理对象

一个是本体，一个是评估结论。

### 6.3 `AgentBinding` 和 `ExecutionAuthorization`
这两个看起来都在讲“当前执行面”，但不能压。

原因：
1. `AgentBinding`
   - 回答谁上场
2. `ExecutionAuthorization`
   - 回答上场后被放行到什么程度

一个是 roster，一个是 lease。

### 6.4 `PendingDecisionWindow` 和 `ControlAction`
这两个看起来都围绕用户决策，但不能压。

原因：
1. `PendingDecisionWindow`
   - 回答系统当前在等什么
2. `ControlAction`
   - 回答用户怎样消费这个等待态

这是典型的 state 和 command 分离。

### 6.5 `ReviewResult` 和 `ResultContract`
这两个看起来都在讲结果，但不能压。

原因：
1. `ReviewResult`
   - 回答 review 怎么判
2. `ResultContract`
   - 回答整次任务最终交付了什么

review 是结果链输入，不是最终交付包本体。

## 7. 排除项/禁用项
这些名字当前只保留为“不要新增”的排除项，不再作为未来候选主对象在正文里反复讨论：
1. `WorkspaceBinding / RepoBinding / ResourceScope / SecretScope`
   - 第一版工程上下文已经并入 `TaskScopeSnapshot`
2. `PolicyEvaluationRecord / DecisionBasis`
   - 第一版决策依据直接复用 `Assessment` 家族与 `RiskAssessment` 字段族
3. `AuthorizationLeaseScope`
   - 第一版授权细化继续收敛在 `ExecutionAuthorization.granted_areas`
4. `ResponsibilityGraph`
   - 第一版责任协作先由 `AgentBinding` 承接，必要时再补最小 `HandoffContract`
5. `AcceptanceLedger`
   - 第一版接受链继续留在 `AcceptancePolicy + ReviewResult + ProofOfWorkBundle`
6. `AttentionPolicy`
   - 第一版注意力治理先并入 `watchdogNoticeState + ...escalation_policy`

## 8. 我建议的第一轮压缩动作
### 7.1 只压“层级理解”，先不急着全量改名
更稳的动作顺序应该是：
1. 先重新标注哪些是 aggregate、哪些是值对象、哪些是子实体
2. 再决定是否真的要改 contract 名称

原因：
1. 很多时候问题不是名字本身
2. 而是大家把一个成员对象当成顶层 aggregate 在用

### 7.2 第一轮最值得收口的 5 个点
1. `ReviewSummary`
   - 明确成 `ReviewResult` 的值对象
2. `DeliveryAttempt`
   - 明确成 `OutboundDelivery` 的子实体
3. `PendingBoundaryConfirmation`
   - 明确成 `PendingDecisionWindow` 的边界确认扩展子实体
4. 工程上下文
   - 第一版并进 `TaskScopeSnapshot`
5. 注意力治理
   - 第一版并入 `watchdogNoticeState + escalation_policy` 现有族

## 9. 对代码实现的直接影响
如果后面按这个规则编码，代码层会更接近 DDD：

1. aggregate root 数量更少
2. 运行时真相源更集中
3. 状态排查时更容易知道该读哪个 owner
4. 不容易再长出多个近义对象抢语义

对实现者的直接指导是：
1. 先找 aggregate root
2. 再找子实体和值对象
3. 最后才决定要不要新建 noun

不要反过来：
1. 先觉得“这个概念挺重要”
2. 再补一个 struct 和文档

---

## 10. 我的当前判断
Loom 现在最需要的，不是继续扩展新对象，而是：

**在不破坏 bounded context 的前提下，把已有对象压到正确层级。**

这才是真正符合 DDD 的治理方式。
