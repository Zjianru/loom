# 概念模型

更新时间：2026-03-11

---

## 1. 文档作用
这份文档只做一件事：

**把 Loom 顶层对象之间的关系说清楚。**

它不负责字段细节。  
字段、协议、接口和示例，统一下沉到 [目录说明.md](../核心设计/目录说明.md)。

---

## 2. 基础概念
### 2.1 Chat
Chat 是用户和 assistant 的默认交互形态。

特点：
1. 默认存在
2. 不要求任务治理
3. 即使有 active managed task，也继续可用

### 2.2 `chat lane`
`chat lane` 是顶层产品边界，不是底层实现优化。

它回答：
1. 这条输入要不要激活 Loom

### 2.3 `managed task lane`
`managed task lane` 是 Loom 真正介入的地方。

它负责：
1. start card
2. task scope
3. phase plan
4. agent binding
5. review
6. result
7. boundary

---

## 3. 五个 bounded contexts
Loom 现在固定按 5 个 bounded contexts 建模：
1. 宿主输入 context
   - `CurrentTurnEnvelope / HostSemanticBundle / SemanticDecisionEnvelope / HostCapabilitySnapshot`
2. 任务治理 context
   - `managedTaskRef / TaskScopeSnapshot / PhasePlan / PendingDecisionWindow / ChangeImpactAssessment`
3. 执行与授权 context
   - `AgentBinding / IsolatedTaskRun / ExecutionAuthorization / CapabilityDriftAssessment / RiskAssessment`
4. review / result context
   - `ReviewResult / ProofOfWorkBundle / ResultContract`
5. notice / outbox context
   - `watchdogNoticeState / KernelOutboundPayload / OutboundDelivery`

固定规则：
1. 字段相似不等于应该跨 context 合并
2. 没有独立生命周期和独立真相源的概念，不应继续升格成平行 noun

## 4. 宿主输入、宿主语义与任务三者分离
### 4.1 `CurrentTurnEnvelope`
它是什么：
1. 当前这条宿主输入的结构化入站事实

它回答：
1. 谁说了什么
2. 来自哪个宿主会话

### 4.2 `HostSemanticBundle`
它是什么：
1. 宿主语义层对当前输入给出的综合结构化判断包

它回答：
1. `interactionLane`
2. `taskActivationReason`
3. `managedTaskClass`
4. `WorkHorizon`
5. 必要时的 change / boundary / control judgment

### 4.3 `SemanticDecisionEnvelope`
它是什么：
1. `loom-openclaw` 归一化后交给 Loom 的 bounded judgment

它回答：
1. Loom 这一步到底消费哪条结构化判断
2. 当前 judgment 的 schema、source、confidence 和 payload 是什么

固定边界：
1. kernel 不直接消费自然语言
2. kernel 也不直接消费原始 `HostSemanticBundle`
3. kernel 正式消费的是 `SemanticDecisionEnvelope`

### 4.4 `host_session_id`
它是什么：
1. 宿主聊天容器

它不代表：
1. 任务 owner

### 4.5 `managedTaskRef`
它是什么：
1. 单个受管任务 owner

它回答：
1. 这项任务当前的范围、阶段、授权、结果属于谁

### 4.6 `activeManagedTaskRef`
它是什么：
1. 当前唯一 active task 的指针

取舍：
1. 单活跃任务仍保留
2. 但不再锁死主聊天

---

## 5. 顶层判断对象
### 5.1 `interactionLane`
它是什么：
1. 这条输入进聊天，还是进受管任务治理

### 5.2 `taskActivationReason`
它是什么：
1. 为什么从 `chat lane` 升级成 managed lane

### 5.3 `managedTaskClass`
它是什么：
1. 一旦进入 managed lane，这项任务采用哪种协作拓扑

固定值：
1. `COMPLEX`
2. `HUGE`
3. `MAX`

### 5.4 `WorkHorizon`
它是什么：
1. 这次工作的投资意图

取舍：
1. 它不是任务规模
2. 它不是 pack 名
3. 它会影响 review、budget、delegation 等治理策略

---

## 6. 任务生命周期对象
### 6.1 `workflowStage`
它是什么：
1. 单个 managed task 当前处在哪个粗粒度治理生命周期

固定边界：
1. `workflowStage` 只使用：
   - `candidate`
   - `execute`
   - `review`
   - `result`
   - `closed`
2. `clarify / source_collection / synthesis / deliver`
   - 这些是 `StagePackageId` 或 `PhasePlan` 层对象
   - 不是 `workflowStage` 扩展枚举
3. `execute / review`
   - 在某些 pack 的 `default_stage_sequence` 里也可能作为具体 `StagePackageId` 出现
   - 但 `workflowStage=execute/review` 仍然只表示粗粒度治理生命周期
   - 不能把二者混成同一个 owner 或同一个枚举层

顶层只关心：
1. `candidate`
2. `execute`
3. `review`
4. `result`
5. `closed`

### 6.2 `pendingUserDecision`
它是什么：
1. 当前任务 projection 上指向 open `PendingDecisionWindow` 的指针

### 6.3 `pendingBoundaryConfirmation`
它是什么：
1. `BoundaryConfirmation` 窗口的专门扩展子实体

它至少绑定：
1. 当前 active task owner
2. 当前 boundary candidate owner
3. 当前窗口 `decision_token`
4. 触发本次确认的 `boundary_reason`

固定边界：
1. `BoundaryCardPayload`
   - 只是这个窗口的用户可见 projection
2. `replace_active` 消费的是当前 open `PendingDecisionWindow` 及其边界扩展状态

### 6.4 `decision_token`
它是什么：
1. 绑定当前 pending decision window 的令牌

它解决：
1. 迟到回复串线
2. 重复消费
3. 生命周期始终由 `PendingDecisionWindow` 持有

### 6.5 `TaskScopeSnapshot`
它是什么：
1. 当前任务范围与工程上下文快照

它回答：
1. 当前 requirement 是什么
2. 当前任务级路径和 secret 边界是什么

第一版固定吸收：
1. `workspace_ref`
2. `repo_ref`
3. `allowed_roots`
4. `secret_classes`
5. 它们继续作为 run 级 `readable_roots / writable_roots / allowed_secret_classes` 的上限

### 6.6 `SpecBundle`
它是什么：
1. 围绕当前 `TaskScopeSnapshot` 的正式执行锚点文档组

它回答：
1. 怎样把范围、计划、验证锚在活文档里

### 6.7 `ProofOfWorkBundle`
它是什么：
1. 任务交付时的正式证据包

它回答：
1. 最终交付到底基于哪些 review、artifact、evidence 和 acceptance basis
2. 第一版整体结果由谁在什么时候接受

固定边界：
1. 第一版整体结果接受快照直接进入 `ProofOfWorkBundle`
2. 不引入独立 `AcceptanceLedger`

### 6.8 `IsolatedTaskRun`
它是什么：
1. 一个 `managedTaskRef` 在执行期绑定的一次隔离受管 run

它回答：
1. 这轮执行的 run boundary、run artifacts、run evidence 和 run outcome

---

### 6.9 `RiskAssessment`
它是什么：
1. 当前任务基线风险或动作覆盖风险的正式评估结果

它回答：
1. 风险总体处在哪个档位
2. 为什么需要升级治理、收紧授权或要求审批

固定边界：
1. 它优先复用 `Assessment` 对象族
2. 第一版不再另开平行记录对象
3. 第一版固定分成 `TaskBaseline / ActionOverride`
4. 总体风险档位固定为 `Low / Medium / High / Critical`

---

## 7. 资源池与角色
### 7.1 `Stage Pool`
它回答：
1. 这个系统有哪些可复用阶段模板

### 7.2 `Agent Pool`
它回答：
1. 系统有哪些可实例化 agent 能力单元

### 7.3 `Pack`
它回答：
1. 某个场景默认如何组合阶段、角色和治理策略

当前写透的 pack：
1. `coding_pack`
2. `research_pack`

### 7.4 关键角色
当前固定角色：
1. `net`
2. `recorder`
3. `worker`
4. `reviewer`
5. `platform`
6. `facilitator`

---

## 8. 运行线
### 8.1 `Harness`
负责：
1. pack 装配
2. phase plan 推进
3. agent 角色绑定
4. review 与结果主链
5. `SpecBundle / IsolatedTaskRun / ProofOfWorkBundle` 主链

### 8.2 `approval-gate`
负责：
1. 写边界
2. 高风险动作审批
3. `ExecutionAuthorization` 约束

### 8.3 `watchdog`
负责：
1. 进展通知
2. 卡点升级
3. notice 去重

### 8.4 `HandoffContract`
只在 `HUGE / MAX` 或显式多 worker 协作下启用：
1. 它回答单次责任如何从当前 `AgentBinding` 转移到新的 binding
2. 它不是 `ResponsibilityGraph`
3. 第一版只承接最小正式交接

---

## 9. 当前结论
Loom 的概念边界现在已经固定成三层：
1. 宿主输入与语义判断
2. 任务对象与治理状态
3. 资源池、角色和运行线

未来实现只允许在这三层内细化。
