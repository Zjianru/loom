# 端到端示例

更新时间：2026-03-10

---

## 1. 示例 A：普通聊天不触发 Loom
### 用户输入
```text
解释一下 术语表 里的 interactionLane。
```

### 关键判断
1. 宿主产出 `HostSemanticBundle`
2. `interactionLane=chat`
3. 不创建 `managedTaskRef`

### 用户可见结果
1. 主 agent 直接解释术语
2. 不出现 start card
3. 不出现 watchdog notice

## 2. 示例 B：`coding_pack / COMPLEX` 启动并完成
### 用户输入
```text
开始一个任务：重构 sourceDocs 的索引和导航，交给子 agent 持续推进。
```

### 宿主结构化判断
1. `interactionLane=managed_task_candidate`
2. `managedTaskClass=COMPLEX`
3. `taskActivationReason=explicit_start_task`
4. `pack=coding_pack`
5. adapter 把 `HostSemanticBundle` 归一化成 `SemanticDecisionEnvelope`

### 主链
1. Loom 消费 `SemanticDecisionEnvelope`
2. Loom 创建 `managedTaskRef`
3. Loom 打开 `PendingDecisionWindow(kind=StartCandidate)`
4. Loom 写入 `pendingUserDecision` projection
5. Loom 发 `StartCardPayload`
6. 用户输入 `/loom approve`
7. `activeManagedTaskRef` 生效
8. `Harness` 依据 `coding_pack` 生成 `SpecBundle`
9. Loom 创建 `IsolatedTaskRun`
10. `Harness` 生成默认 `PhasePlan`
   - `clarify -> execute -> review -> deliver`
11. `net + recorder + 1 worker` 上场
12. `review` 产出 `ReviewResult`
13. `ProofOfWorkBundle` 绑定 run、review、artifact、acceptance 证据
14. `ResultContract` 生成结果包
15. adapter 渲染 `ResultSummaryPayload`

### 用户可见结果
1. 一张 start card
2. 任务执行期间主聊天仍可继续
3. 最终收到摘要、proof-of-work 摘要、交付物清单、review 摘要

## 3. 示例 C：运行中变更范围
### 用户输入
```text
把导航重构范围补充到 核心设计 目录，也把 目录说明 一起改掉。
```

### 主链
1. 宿主把这条输入结构化成 `request_task_change`
2. Loom 更新 `TaskScopeSnapshot`
3. Loom 修订 `SpecBundle`
4. Loom 生成 `ChangeImpactAssessment`
4. `scope_revision_summary` 记录：
   - 新增 `核心设计/目录说明.md`
5. 如有必要，重排 `PhasePlan`
6. 如有必要，reopen 当前 `IsolatedTaskRun` 的 review/acceptance
7. review 时检查新增范围是否也被覆盖

### 用户可见结果
1. 主聊天里收到简短确认
2. 最终结果包明确说明最终按哪版 scope 交付

## 4. 示例 D：第二项 managed task 触发边界确认
### 前置条件
1. 当前已有一个 active task 正在执行

### 用户输入
```text
再开始一个任务：把 OpenClaw adapter 的 bridge 协议重新设计。
```

### 主链
1. 宿主判断这是第二项 managed task 候选
2. Loom 打开 `PendingDecisionWindow(kind=BoundaryConfirmation)`
3. Loom 创建 `pendingBoundaryConfirmation`
4. 发 `BoundaryCardPayload`
5. 用户二选一：
   - `/loom keep`
   - `/loom replace`

### 用户可见结果
1. 先看到边界确认卡
2. 不会静默覆盖当前任务

## 5. 示例 E：`research_pack / COMPLEX` 完成
### 用户输入
```text
开始一个任务：研究个人 AI 终端市场里的多 agent 治理产品，做成简报。
```

### 主链
1. 宿主给出：
   - `interactionLane=managed_task_candidate`
   - `managedTaskClass=COMPLEX`
   - `pack=research_pack`
2. Loom 创建 candidate
3. 用户输入 `/loom approve`
4. 默认阶段：
   - `clarify -> source_collection -> synthesis -> review -> deliver`
5. recorder 沉淀来源与阶段总结
6. Loom 生成 research 场景的 `SpecBundle`
7. Loom 为任务创建 `IsolatedTaskRun`
8. review 检查来源质量、论据一致性、范围收口
9. 最终输出带 `ProofOfWorkBundle` 的结果包

### 用户可见结果
1. 收到研究结论摘要
2. 收到主要来源和 artifact 清单
3. 收到 review 摘要与 acceptance verdict

---

## 6. 结论
这几条示例共同证明：
1. 普通聊天默认留在 `chat lane`
2. managed task 先有 start card，再进入 execute
3. `COMPLEX` 已经能表达最小正式委派协作
4. review 和 result 是正式主链，不是附属物
5. 变更和边界都是正式治理路径，不再是实现细节
