# 目标规格

更新时间：2026-03-11

---

## 1. 总体原则
Loom 的目标不是替代 assistant，而是给个人 AI 终端补上一层正式的任务治理内核。

固定原则：
1. 默认走 `chat lane`，插件默认隐身。
2. 只有明确的受管任务才进入 `managed task lane`。
3. kernel 不做自然语言语义推断，只消费宿主结构化判断。
4. `Loom` 是治理真相源，`loom-openclaw` 是首个薄 adapter。
5. `runtime/loom/` 是 authoritative truth root。

---

## 2. 语义入口
### 2.1 `CurrentTurnEnvelope`
它是什么：
1. 宿主原始入站事实

它回答：
1. 这条输入是谁发的
2. 来自哪个宿主会话
3. 对应哪个宿主消息引用

### 2.2 `HostSemanticBundle`
它是什么：
1. 宿主语义层交给 adapter 的综合结构化判断包

它至少回答：
1. `interaction_lane`
2. `task_activation_reason`
3. `managed_task_class`
4. `work_horizon`
5. 需要时的 change / boundary / control judgment

### 2.3 `SemanticDecisionEnvelope`
它是什么：
1. adapter 归一化后交给 Loom 的 bounded judgment

它代表：
1. Loom 已经可以直接消费的治理输入

取舍：
1. 宿主负责理解原始自然语言。
2. `HostSemanticBundle` 负责把多条宿主判断绑成一个语义包。
3. Loom 只做 schema 校验、状态推进、治理执行。

---

## 3. 双车道模型
### 3.1 `chat lane`
`chat lane` 负责：
1. 普通闲聊
2. 解释、追问、轻量处理
3. 有 active managed task 时的并行聊天

它不负责：
1. 创建 `managedTaskRef`
2. 启动 `Harness`

### 3.2 `managed task lane`
`managed task lane` 负责：
1. 正式受管任务
2. 阶段推进
3. agent 委派
4. review、结果包、通知与边界治理

---

## 4. 核心对象
### 4.1 `interaction_lane`
它是什么：
1. 当前输入到底留在聊天，还是进入 managed task 治理

### 4.2 `managedTaskRef`
它是什么：
1. 单个受管任务的正式 owner

它持有：
1. start card
2. `PhasePlan`
3. `ExecutionAuthorization`
4. recorder / review 产物
5. 最终结果包

### 4.3 `activeManagedTaskRef`
它是什么：
1. 当前唯一 active managed task 的指针

取舍：
1. 只约束单活跃任务
2. 不冻结主聊天

### 4.4 `managed_task_class`
它是什么：
1. managed lane 内采用哪种协作拓扑和治理强度

正式值固定为：
1. `COMPLEX`
2. `HUGE`
3. `MAX`

### 4.5 `TaskScopeSnapshot`
它是什么：
1. 当前任务范围与工程上下文的正式快照

它至少回答：
1. 当前任务被允许触达什么范围
2. 当前任务绑定了哪个 `workspace_ref`
3. 当前任务绑定了哪个 `repo_ref`
4. 当前任务允许读写哪些 `allowed_roots`
5. 当前任务允许接触哪些 `secret_classes`

固定边界：
1. 它是任务级作用域 owner
2. 第一版工程上下文先并入它，不另开独立工程上下文对象
3. run 级 `ExecutionAuthorization` 只能在它的边界内再做子集租约
4. `allowed_roots / secret_classes` 继续作为 run 级 `readable_roots / writable_roots / allowed_secret_classes` 的上限

### 4.6 `RiskAssessment`
它是什么：
1. 任务基线风险或动作覆盖风险的正式评估结果

它回答：
1. 当前总体风险档位是什么
2. 风险主要来自哪些维度
3. 这次风险结论会触发哪些治理后果

固定边界：
1. `risk_profile` 当前只作为 `RiskAssessment` 的结果视角存在
2. 它不单独升格成任务主 aggregate
3. pack、`managed_task_class` 和运行时升级都只能消费 `RiskAssessment` 的正式结论
4. 第一版固定分成：
   - `TaskBaseline`
   - `ActionOverride`
5. 总体风险档位固定为：
   - `Low`
   - `Medium`
   - `High`
   - `Critical`
6. `critical` 风险不得继续 silent execution

### 4.7 `SpecBundle`
它是什么：
1. 围绕单个 `managedTaskRef` 的正式执行锚点文档组

它至少包含：
1. `scope_doc`
2. `plan_doc`
3. `verification_doc`

固定边界：
1. owner 固定为 `managedTaskRef`
2. 它是活文档锚点，不是上传附件
3. 它由 Loom 持有版本和修订历史

### 4.8 `ProofOfWorkBundle`
它是什么：
1. 任务交付时的正式证据包

它至少包含：
1. `run_summary`
2. `evidence_refs`
3. `review_summary`
4. `artifact_manifest`
5. `acceptance_basis`
6. 第一版整体结果接受快照：
   - `accepted_by_ref`
   - `accepted_at`
   - `accepted_scope_version`

固定边界：
1. 它是 `ResultContract` 的正式 section
2. `ResultSummaryPayload` 只能摘录它，不能替代它
3. 第一版不引入独立 `AcceptanceLedger`

### 4.9 `IsolatedTaskRun`
它是什么：
1. 单个 `managedTaskRef` 在执行期对应的一次隔离受管 run

固定边界：
1. task 不是 run
2. run 不是宿主 session
3. `ExecutionAuthorization` 只是 run 的能力租约，不是 run 本体

### 4.10 治理层装配
它是什么：
1. Loom 在当前任务上最终启用了哪些治理深度

固定规则：
1. 挂载优先级固定为：
   - `HostCapabilitySnapshot` 约束
   - 内核强制安全规则
   - `PackContract` 默认
   - `managed_task_class`
   - `RiskAssessment`
   - 用户在 `override_policy` 允许范围内的显式覆盖
2. 被能力约束挡住的层不能被强行启用
3. 被安全规则或风险升级强制拉起的层不能被用户关闭
4. 第一版不引入独立 `ResponsibilityGraph / AttentionPolicy / AcceptanceLedger`

## 5. 三档正式定义
### 5.1 `COMPLEX`
表示：
1. `1 + 1` 型受管协作

默认形状：
1. `net + recorder + 1 worker`
2. 最小阶段骨架：
   - `clarify`
   - `execute`
   - `review`
   - `deliver`

### 5.2 `HUGE`
表示：
1. `1 + n` 型受管协作

特点：
1. 多 worker
2. 更强阶段治理
3. 更严格 review、budget、delegation
4. 如发生正式责任转移，先由最小 `HandoffContract` 承接

### 5.3 `MAX`
表示：
1. 项目级、工程级分层协作

特点：
1. 更深协作层级
2. 更强的治理、预算和边界控制
3. 正式交接仍先以任务级 `HandoffContract` 承接，而不是直接引入图结构

## 6. 任务开始与决策
### 6.1 `managed task start card`
当系统进入 `managed_task_candidate`，第一条用户可见输出必须是：
1. `managed task start card`

这张卡至少说明：
1. 当前建议是哪个 `managed_task_class`
2. 为什么会激活 managed lane
3. 这条任务大概会怎样被治理

### 6.2 start card 决策集合
固定为：
1. `approve_start`
2. `modify_candidate`
3. `cancel_candidate`

### 6.3 `pendingUserDecision`
它是什么：
1. 当前任务 projection 上指向 open `PendingDecisionWindow` 的指针

### 6.4 `decision_token`
它是什么：
1. 当前决策窗口的绑定令牌

它代表：
1. 迟到或重复回复不能串到新的窗口上
2. 它的生命周期由 `PendingDecisionWindow` 持有，而不是独立漂浮

---

## 7. 运行中任务
### 7.1 `workflowStage`
它是什么：
1. 单个 managed task 的粗粒度治理生命周期

顶层目标只固定这组：
1. `candidate`
2. `execute`
3. `review`
4. `result`
5. `closed`

硬约束：
1. `clarify / source_collection / synthesis / deliver`
   - 这些是 `StagePackageId` 或 `PhasePlan` 层对象
   - 不是 `workflowStage` 扩展枚举
2. `execute / review`
   - 在某些 pack 的细粒度阶段骨架里也可能作为 `StagePackageId` 出现
   - 但 `workflowStage=execute/review` 仍然只表示粗粒度治理生命周期

### 7.2 execute 期的正式对象
一旦 candidate 被 `approve_start` 消费，Loom 必须：
1. 依据 pack preset 生成 `SpecBundle`
2. 建立当前 `TaskScopeSnapshot`
3. 为这个 `managedTaskRef` 创建 `IsolatedTaskRun`
4. 再把 `PhasePlan`、`AgentBinding`、`ExecutionAuthorization` 绑到当前 run 上
5. 同时建立首个任务基线 `RiskAssessment`

### 7.3 `taskConversationPolicy`
默认固定为：
1. `chat_open`

它代表：
1. 任务执行中，主 assistant 仍然继续和用户聊天

固定边界：
1. 这是一条顶层产品默认值
2. 当前不把它建成 formal runtime object
3. v0 不要求持久化、单独 API 字段或独立 contract

### 7.4 `request_task_change`
运行中的 active task 必须允许：
1. 用户补充要求
2. 用户修改任务细节
3. 用户调整尚未执行和已执行部分的目标

但取舍固定为：
1. Loom 不自己判断自然语言变更语义
2. 宿主先在同 bundle 中给出：
   - `task_change`
   - `control_action(request_task_change)`
3. `task_change` 只负责治理判断
4. patch 内容留在 `control_action.payload`
5. Loom 再做 `ChangeImpactAssessment` 与 `TaskScopeSnapshot` 修订

## 8. 边界确认
### 8.1 `pendingBoundaryConfirmation`
它是什么：
1. `PendingDecisionWindow(kind=BoundaryConfirmation)` 的专门边界确认扩展子实体

它至少绑定：
1. 当前 active task owner
2. 当前 boundary candidate owner
3. 当前窗口 `decision_token`
4. 本次确认的 `boundary_reason`

### 8.2 边界决策集合
固定为：
1. `keep_current_task`
2. `replace_active`

取舍：
1. 不允许第二项重任务被 silent 吞进当前任务
2. 边界决策必须先显式消费，再允许替换 active task

---

## 9. Loom 套件职责边界
### 9.1 `Harness`
负责：
1. 阶段流转
2. pack 默认装配
3. agent 角色绑定
4. review / result / boundary 主治理链

### 9.2 `approval-gate`
负责：
1. 写操作和高风险动作边界
2. `ExecutionAuthorization` 的执行约束

### 9.3 `watchdog`
负责：
1. 进展通知
2. 卡点升级
3. notice 去重与节流

### 9.4 `recorder`
负责：
1. 阶段纪要
2. 结果摘要素材
3. 结构化沉淀

---

## 10. Pack 与结果
### 10.1 pack
当前正式写透的 pack 只有：
1. `coding_pack`
2. `research_pack`

### 10.2 结果包
最终结果统一以 `ResultContract` 交付，并至少包含：
1. `acceptance_verdict`
2. `final_scope_snapshot`
3. `scope_revision_summary`
4. `proof_of_work`

其中：
1. `proof_of_work` 必须是正式 `ProofOfWorkBundle`
2. 它至少包含：
   - `run_summary`
   - `evidence_refs`
   - `review_summary`
   - `artifact_manifest`
   - `acceptance_basis`
