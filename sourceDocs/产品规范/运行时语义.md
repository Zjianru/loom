# 运行时语义

更新时间：2026-03-11

---

## 1. 文档定位
这份文档解释 Loom 运行时的正式时序和状态推进规则。

---

## 2. 总体时序
Loom 运行时固定分成 8 段：
1. ingress
2. host semantic
3. lane arbitration
4. candidate/start
5. execute
6. review
7. result
8. delivery/outbound

这 8 段里，最重要的 3 个边界是：
1. 自然语言理解在宿主，不在 Loom
2. start/review/result 是正式阶段，不是实现细节
3. authoritative truth 先提交，再允许 adapter 发消息

---

## 3. ingress 与 host semantic
### 3.1 ingress
每条宿主输入先被写成：
1. `CurrentTurnEnvelope`

它固定包含：
1. `host_session_id`
2. `host_message_ref`
3. 当前输入原文
4. `IngressMeta`

`IngressMeta` 至少包含：
1. `ingress_id`
2. `causation_id`
3. `correlation_id`
4. `dedupe_window`

它的作用：
1. 防止 hook 重放
2. 防止 control action 重复消费

### 3.2 host semantic
宿主语义层必须把这条输入先收敛成：
1. `HostSemanticBundle`

固定规则：
1. Loom 不直接消费自然语言
2. adapter 不直接猜自由文本
3. 缺少结构化判断时，默认保守退化

---

## 4. lane 仲裁
### 4.1 `interaction_lane`
adapter 把 `HostSemanticBundle` 归一化成 `SemanticDecisionEnvelope[]` 后，Loom 做 lane 仲裁。

固定值：
1. `chat`
2. `managed_task_candidate`
3. `managed_task_active`

### 4.2 lane 仲裁规则
1. 若 bundle 明确给出 `chat`
   - 直接留在聊天主线
2. 若 bundle 给出 candidate 激活
   - 创建 `managedTaskRef`
3. 若当前已有 `activeManagedTaskRef`
   - 优先判断这是：
     - active task follow-up
     - second managed task candidate
     - 普通聊天

### 4.3 fail-closed / fail-open
取舍固定为：
1. 对“是否启 Loom”
   - 默认 `fail-open to chat`
2. 对“是否用第二项重任务覆盖当前 active task”
   - 默认 `fail-closed to boundary confirmation`

### 4.4 治理层挂载优先级
固定顺序：
1. `HostCapabilitySnapshot` 约束
2. 内核强制安全规则
3. `PackContract` 默认
4. `managed_task_class`
5. 当前生效 `RiskAssessment`
6. 用户在 `override_policy` 允许范围内的显式覆盖

---

## 5. candidate 与 start card
### 5.1 candidate 创建
进入 `managed_task_candidate` 后：
1. 先创建 `managedTaskRef`
2. 写 `workflowStage = candidate`
3. 绑定 `managed_task_class`
4. 绑定 `task_activation_reason`
5. 绑定 `work_horizon`

这里的阶段层级固定为：
1. `workflowStage`
   - 只表示粗粒度治理生命周期：
     - `candidate / execute / review / result / closed`
2. `PhasePlan`
   - 才表示任务内部真正跑的细粒度阶段序列
   - 例如 `clarify / source_collection / synthesis / deliver`
3. adapter、Harness 和结果链都不得把细粒度阶段名重新抬成 `workflowStage` 枚举

### 5.2 start card
随后 Loom 必须产出：
1. `StartCardPayload`

同时固定完成：
1. 先打开 `PendingDecisionWindow(kind=StartCandidate)`
2. 再把 `pendingUserDecision` 投影到当前 open window
3. `StartCardPayload` 必须绑定该窗口的 `decision_token`
4. 写入 `confirmationPromptSentAt`

### 5.3 start card 决策消费
固定决策集合：
1. `approve_start`
2. `modify_candidate`
3. `cancel_candidate`

消费规则：
1. `approve_start`
   - 写 `activeManagedTaskRef`
   - 进入 `execute`
2. `modify_candidate`
   - 更新 candidate
   - 重新发 start card
3. `cancel_candidate`
   - 关闭 candidate
   - 返回聊天主线

---

## 6. execute 主链
### 6.1 execute 阶段
进入 `execute` 后，`Harness` 开始按 `PhasePlan` 和 `Pack` 推进。

固定语义：
1. `approve_start` 被消费后，Loom 必须先依据 pack preset 生成 `SpecBundle`
2. 同时为当前 `managedTaskRef` 创建 `IsolatedTaskRun`
3. 当前 run 再绑定：
   - `PhasePlan`
   - `AgentBinding`
   - `ExecutionAuthorization`

这里再强调一次：
1. `PhasePlan` 里的 `clarify / source_collection / synthesis / review / deliver`
   - 都是 `StagePackageId` 层对象
2. `workflowStage = execute`
   - 只表示当前任务已经进入执行主链
   - 不表示当前细粒度阶段一定叫 `execute`
3. `execute / review`
   - 在某些 pack 里也可能作为细粒度 `StagePackageId` 出现
   - 这不改变 `workflowStage` 只承接粗粒度生命周期的规则

### 6.2 `SpecBundle`
`SpecBundle`
1. 不是 pack 附件
2. 而是当前 task 的正式活文档锚点
3. 它围绕：
   - `TaskScopeSnapshot`
   - `PhasePlan`
   - `AcceptancePolicy`
   组织 `scope_doc / plan_doc / verification_doc`
4. 当前第一版工程上下文也先并入 `TaskScopeSnapshot`
   - `workspace_ref`
   - `repo_ref`
   - `allowed_roots`
   - `secret_classes`
5. run 级 `ExecutionAuthorization` 第一版至少继续表达：
   - `allowed_tools`
   - `readable_roots`
   - `writable_roots`
   - `allowed_secret_classes`
   - `budget_band`

### 6.3 `IsolatedTaskRun`
`IsolatedTaskRun`
1. 是当前执行期的独立运行单元
2. 它不等于宿主 session
3. 也不等于 `ExecutionAuthorization`

它最少承接：
1. `run_id`
2. run boundary
3. run artifacts
4. run 级事件链
5. run outcome

### 6.4 `AgentBinding`
`Harness` 依据：
1. `Pack`
2. `Agent Pool`
3. `HostCapabilitySnapshot`
4. `ExecutionAuthorization`

建立本次实际 `AgentBinding`。

当 `HUGE / MAX` 或显式多 worker 协作发生正式责任转移时：
1. Loom 继续用新的 `AgentBinding` 表达“谁实际上场”
2. 并用最小 `HandoffContract` 记录“责任怎样转移”
3. 第一版不引入独立 `ResponsibilityGraph`

### 6.5 `RiskAssessment`
execute 主链至少维护两类正式风险评估：
1. task baseline risk
   - 影响治理层装配和默认强度
2. action override risk
   - 影响当前高风险动作是否需要额外收紧

它的作用：
1. 给出当前 `overall_risk_band`
2. 说明风险主要来自哪些 `risk_dimensions`
3. 给出这次风险结论触发的 `derived_consequences`
4. 供 `approval-gate`、`ExecutionAuthorization`、review/result 链共同消费

固定边界：
1. `risk_profile` 当前只作为 `RiskAssessment` 的结果视角存在
2. 不另开平行记录对象族
3. 第一版固定分成 `TaskBaseline / ActionOverride`
4. 总体风险档位固定为 `Low / Medium / High / Critical`
5. `critical` 风险不得继续 silent execution

### 6.6 `approval-gate`
execute 期间，所有高风险动作都先经过：
1. `approval-gate`

它读取：
1. `ExecutionAuthorization`
2. `DelegationLevel`
3. `DecisionArea`
4. 当前生效 `RiskAssessment`

然后产出：
1. `ToolDecisionPayload`

### 6.7 `watchdog`
execute 期间，`watchdog` 独立监听：
1. 阶段进入
2. 阶段完成
3. `blocked`
4. aging
5. 需要升级给用户的条件

---

## 7. review 主链
### 7.1 review 是正式阶段
Loom 的最小运行时主链固定包含：
1. `review`

这不是实现优化，而是治理要求。

### 7.2 `ReviewResult`
review 阶段至少产出：
1. `ReviewResult`
2. 其中内嵌 `ReviewSummary`

它供：
1. `AcceptancePolicy`
2. `ResultContract`
3. recorder

### 7.3 review verdict
正式 verdict 固定为：
1. `approved`
2. `approved_with_notes`
3. `rework_required`
4. `blocked`
5. `escalate_to_user`

### 7.4 `escalate_to_user`
若 review 结论是 `escalate_to_user`：
1. 任务保持在 `review` 相关治理链
2. Loom 打开新的 `PendingDecisionWindow(kind=ApprovalRequest)`
3. 更新 `pendingUserDecision` 指向该 open window
4. 输出 `ApprovalRequestPayload`
5. 不直接编译 `ResultContract / ResultSummaryPayload`
6. 待用户结构化回复 `approve_request / reject_request` 后，再进入：
   - result 收口
   - 或 reopen execute / review

### 7.5 review 与返工
若 review 要求返工：
1. Loom 进入 `rework` 路径
2. 触发 `ReworkPolicy`
3. 必要时 reopen execute 或 reopen review

---

## 8. result 主链
### 8.1 `AcceptancePolicy`
任务进入 `result` 前，必须先过正式验收。

完成是“与关系”，至少联合判断：
1. `requirements_satisfied`
2. `review_approved`
3. 必要时的 verification/test 通过
4. 当前生效 `RiskAssessment` 落在阈值内

### 8.2 `ResultContract`
进入 `result` 后，Loom 生成正式结果包。

结果包至少包含：
1. `final_scope_snapshot`
2. `scope_revision_summary`
3. `proof_of_work`
4. `acceptance_verdict`

其中：
1. `proof_of_work` 必须是正式 `ProofOfWorkBundle`
2. 它至少包含：
   - `run_summary`
   - `evidence_refs`
   - `review_summary`
   - `artifact_manifest`
   - `acceptance_basis`
   - 整体结果接受快照

### 8.3 `ResultSummaryPayload`
最终回宿主聊天区的文本摘要必须来自：
1. 结构化结果包
2. `ResultSummaryPayload`

adapter 只能渲染，不能临时发明字段。

### 8.4 `ProofOfWorkBundle`
`ProofOfWorkBundle` 在结果链里的固定关系：
1. `ReviewResult`
   - 提供 review 事实
2. `AcceptancePolicy`
   - 提供完成判定规则
3. `IsolatedTaskRun`
   - 提供 run summary 与 evidence
4. `ProofOfWorkBundle`
   - 把 review、验证、artifact、证据绑定成正式交付证据包
5. 第一版整体结果接受快照直接并入 `ProofOfWorkBundle`
6. 第一版不引入独立 `AcceptanceLedger`

---

## 9. active task 期间的变更与边界
### 9.1 `request_task_change`
active task 期间，用户可以持续补充和修改要求。

运行时固定顺序：
1. 宿主语义层先产出成对的：
   - `task_change`
   - `control_action(request_task_change)`
2. Loom 先记录 `task_change_requested`
3. Loom 生成 `ChangeImpactAssessment`
4. 如变更被正式接受，再更新 `TaskScopeSnapshot`
5. 必要时：
   - 改 `PhasePlan`
   - 重做 review
   - 重算 acceptance
   - 调整 budget

### 9.2 `pendingBoundaryConfirmation`
若 active task 存在，又出现第二项重任务：
1. 先打开 `PendingDecisionWindow(kind=BoundaryConfirmation)`
2. 再创建 `pendingBoundaryConfirmation` 扩展子实体
3. 产出 `BoundaryCardPayload`
4. 等待用户结构化决策

这不是一句“待确认”的备注，而是正式边界窗口上的扩展子实体。  
它最少要绑定：
1. `active_managed_task_ref`
2. `candidate_managed_task_ref`
3. `decision_token`
4. `boundary_reason`

固定边界：
1. `BoundaryCardPayload`
   - 只是该窗口的用户可见投影
2. `replace_active`
   - 只能消费当前有效窗口里绑定的 candidate task
   - 不允许 adapter 或 kernel 在窗口外自由猜测替换目标

边界决策固定为：
1. `keep_current_task`
2. `replace_active`

---

## 10. outbound 与 delivery
### 10.1 durable outbox
Loom 出站采用 durable outbox。

authoritative store 先提交：
1. state 变更
2. `TaskEvent`
3. `OutboundDelivery`

然后 adapter 再读取 delivery。

### 10.2 `OutboundDelivery`
delivery 生命周期固定为：
1. `pending`
2. `delivering`
3. `delivered_unacked`
4. `acked`
5. `retry_scheduled`
6. `terminal_failed`
7. `expired`

### 10.3 `ack_outbound`
只有宿主真正完成投递后，才允许：
1. `ack_outbound`

### 10.4 durable 判定规则
固定规则：
1. 只要对象会影响 `ControlAction` 是否有效、`review/result` 是否成立、或 run 是否允许继续执行，它就必须 durable。
2. 解释类背景、研究类判断、用户可见 projection 不得伪装成平行 owner。
3. 第一版不引入独立 `AttentionPolicy / ResponsibilityGraph / AcceptanceLedger` 主链。

---

## 11. runtime 真相源
### 11.1 authoritative truth
Loom authoritative truth 只允许在：
1. `runtime/loom/`

### 11.2 compatibility projection
宿主兼容投影只允许作为 compatibility projection 出现。

### 11.3 kernel 边界
kernel 仍然固定：
1. 不做自然语言推断
2. 不直接渲染宿主文本
3. 不依赖 compatibility projection 当真相源

---

## 12. 当前结论
Loom 运行时语义已经固定成：
1. 宿主判断输入
2. Loom 仲裁 lane
3. candidate/start 正式建模
4. execute/review/result 正式建模
5. durable outbox + authoritative truth 收口
