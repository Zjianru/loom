## 产品简介
Loom 是一个面向个人 AI 终端的协作治理内核。

它不试图替代模型，不试图取代各类专业 agent，也不试图再造一个大而全的平台。  
它要解决的是另一类问题：

当用户已经拥有一个强大的 assistant 之后，如何让这个 assistant 不再只是“会聊天、会写一点东西、会跑一点工具”，而是能够把复杂任务正式扔出去，交给一组更专业的 agent 持续推进，并在整个过程中保持可控、可汇报、可记录、可验收。

一句话定义：

**它让个人 AI 终端第一次具备后台持续工作的治理能力。**

## 本轮吸收的固定结论
这份产品概览已经吸收本轮设计对齐纪要中的正式结论，固定为：
1. assistant 继续做唯一自然语言入口
2. Loom 不做自然语言语义推断，只消费宿主结构化判断
3. 后台持续工作的核心卖点不是“更强模型”，而是“更强治理”
4. 产品卖点固定落在：
   - 正式任务对象
   - 阶段推进
   - 多智能体协作治理
   - 主动汇报
   - 过程沉淀
   - 验收与返工
   - 高风险边界

## 我们为什么做它
过去一年里，个人 AI 终端快速爆发。  
OpenClaw、类 OpenClaw、Chat-first assistant、coding assistant、research assistant、creative assistant 不断出现，能力本身也在迅速增强。

但一个稳定的现实问题也越来越明显：

1. 用户会在开始时对 assistant 很兴奋。
2. 一旦尝试把更复杂、更长期、更贴近真实工作的事情交给它，能力曲线会迅速下滑。
3. 问题并不总出在模型不够强，而是系统缺少治理。

今天大多数 assistant 的问题，不是不会回答问题，而是不会真正“承接任务”。

它们通常缺少：
1. 正式的任务对象
2. 持续推进的机制
3. 阶段和边界
4. 多 agent 的正式协作
5. 主动汇报机制
6. 过程沉淀机制
7. 独立的验收机制
8. 写操作和高风险行为的稳定边界

结果就是：
1. 用户仍然需要自己盯着
2. 用户仍然要自己做项目经理
3. 用户仍然要自己做人肉 QA
4. 用户很难把复杂工作真正扔出去

Loom 的目标，就是补上这一层。

## 产品定位
### 我们卖什么
我们卖的不是模型，也不是 agent marketplace。  
我们卖的是：

**面向个人 AI 终端的任务编排与治理能力。**

更具体地说，我们提供：
1. 复杂任务的正式激活与接管
2. 阶段化推进与阶段组合
3. 多 agent 的协作治理
4. 主动提醒与卡点上报
5. 过程记录与知识沉淀
6. 验收与返工机制
7. 写操作与高风险行为边界

### 我们不卖什么
我们不想做：
1. 通用低代码 builder
2. 通用企业 agent 平台
3. 通用大模型能力平台
4. 各类专业 agent 的执行能力本身

那些能力应该交给更专业的模型和 agent。  
我们只负责让它们在个人 AI 终端里被可靠地组织起来。

## 目标用户
Loom 面向的是：
1. 已经在使用 AI assistant 的普通个人用户
2. 想把 assistant 从聊天工具升级成工作入口的用户
3. 想把复杂任务正式委派出去，但不想自己盯全过程的用户
4. 希望自己的 assistant 真正能成为“一人公司入口”的用户

这意味着我们的目标人群，不只是开发者，也不只是 enterprise builder 用户。  
它面向的是任何已经开始依赖个人 AI 终端的人。

## 核心产品主张
### 1. assistant 继续做入口
用户不需要离开自己熟悉的 personal AI terminal。  
assistant 仍然是唯一的自然语言入口，并负责把自然语言理解成结构化语义判断。  
Loom 不再猜用户原话，它只消费宿主已经给出的结构化判断并执行治理。

### 2. 复杂工作被正式扔出去
一旦进入受管任务模式，任务就不再是一次聊天，不再是一次 tool call，而是一个可持续推进的对象。

### 3. 后台持续工作，但前台仍然自由
任务跑着时，assistant 仍然继续可用。  
用户可以用它处理别的事情，继续聊天、补充、修改、暂停、取消。

### 4. 系统主动汇报，而不是要求用户 babysit
watchdog 主动汇报进展、卡点、阶段完成，不让用户靠刷新和追问来了解状态。

### 5. 过程会沉淀成资产
recorder 不是日志系统，而是正式的记录角色。  
它会把阶段过程、经验、结论、结构化材料沉淀出来，为后续交付和长期记忆服务。

### 6. 复杂任务不是黑箱自治，而是可治理自治
用户不必盯细节，但仍保留治理权：
1. 启动权
2. 变更权
3. 暂停权
4. 替换权
5. 取消权
6. 最终接收权

## 产品运行模型
Loom 的运行模型可以概括为：

**三条运行线，两类资源池。**

### 三条运行线
#### `Harness`
主运行时。  
它负责任务激活、阶段流转、阶段编排、agent 编排和任务推进。

#### `watchdog`
通知运行时。  
它负责主动提醒、卡点上报、阶段完成通知和需要用户拍板时的升级通知。

#### `approval-gate`
边界运行时。  
它始终守住执行边界。  
在没有得到同意的情况下，系统永远不能执行写入。

### 两类资源池
#### `StagePool`
阶段池。  
系统提供一组可复用、可组合的阶段模板。  
用户未来可以像拼积木一样拼出适合自己的任务流程。

#### `AgentPool`
agent 池。  
系统提供一组可实例化的 agent 能力单元。  
不同阶段可以绑定不同的 agent 组合，形成适合当前任务的人才池。

#### `Pack`
工作模式包。  
系统按场景对外提供可直接启动的承包故事点，例如编码、研究、运营。  
用户先选“要做什么”，再在 pack 内选择要用 `COMPLEX / HUGE / MAX` 哪种协作规模来处理这个场景。

---

## 为什么是阶段池和 agent 池
大多数系统会把流程和角色写死在一起。  
Loom 要做的，是把这两件事拆开：

1. `StagePool` 回答“这件事怎么推进”
2. `AgentPool` 回答“这件事谁来做”

这样做的好处是：
1. 用户能复用流程
2. 用户能替换人手
3. 不同 assistant 宿主都能共用同一个治理内核
4. 不同模型和不同 agent 能力可以灵活替换

Pack 在这里的作用是：
1. 给普通用户一个直接可用的默认组合
2. 不要求用户自己从 `StagePool` 和 `AgentPool` 手动组装
3. 允许有限定制，但不允许拆掉治理骨架

## 三种受管任务模式
Loom 里的 managed task 不再沿用旧的大小分级。  
新的产品语言是三种协作框架：

### `COMPLEX`
`COMPLEX` 用于 `1 + 1` 型受管协作。

它适合：
1. 明确委派给某个专业 agent 的任务
2. 不是一次轻量对话就能结束的工作
3. 需要在后台持续推进，但又不值得上升到大型项目编排的任务

典型角色组合：
1. `net`
2. 一个专业 worker
3. 一个 recorder
4. `watchdog`

它的关键体验是：
1. 任务被正式扔出去
2. assistant 继续可用
3. 用户在结束时收到一份结果，而不是一串过程日志

### `HUGE`
`HUGE` 用于 `1 + n` 型受管协作。

它适合：
1. 多阶段的任务
2. 多角色协作的任务
3. 需要阶段性推进和阶段性总结的任务
4. 需要用户在执行中途继续补充、调整、暂停、取消的任务

在 `HUGE` 中：
1. 多个 agent 被视作人才池
2. 阶段被视作一个个 sprint
3. recorder 继续负责沉淀
4. watchdog 继续负责主动通知
5. `net` 仍然是统一入口

### `MAX`
`MAX` 用于项目级、工程级的分层协作。

它适合：
1. 拆解粒度更细
2. 协作拓扑更深
3. 某些 agent 需要继续调用或派生更多 agent 的任务

`MAX` 与 `HUGE` 的区别不只是数量更多，而是治理层级更深。  
它更接近项目级编排，而不是单次任务执行。

## recorder 与 watchdog 的角色分离
这是 Loom 的一个关键设计原则。

### recorder
recorder 是正式的工作角色，而不是 tracing 插件。

它负责：
1. 过程记录
2. 阶段纪要
3. 经验沉淀
4. 面向记忆系统的结构化产物
5. 为最终汇总提供高质量材料

### watchdog
watchdog 不是记录者，也不是执行者。  
它是主动通知者。

它负责：
1. 进展提醒
2. 卡点上报
3. 阶段完成播报
4. 需要用户拍板时的升级提醒

这两个角色必须分开。  
否则系统要么通知过多，要么沉淀能力不足，要么所有责任又回压到主 assistant 身上。

---

## approval-gate 为什么必须独立存在
很多多 agent 方案的真正风险，不在生成质量，而在执行越界。

Loom 的原则非常简单：

**在没有得到同意的情况下，永远不能做写入。**

这意味着：
1. 编排能力和写入能力必须分离
2. 强执行 agent 不能自己决定高风险写操作
3. 阶段推进不等于自动获得写权限
4. 所有写入和关键高风险工具都必须经过 approval-gate

这是产品可信度的底线，也是未来开源后仍然能被放心采用的前提。

## 典型使用场景
想象这样一个场景：

用户接到一个复杂任务。  
他把任务扔给自己常用的 assistant，选择 `HUGE` 模式。  
系统从 `StagePool` 选择一组阶段，从 `AgentPool` 绑定一组专业 agent。

随后：
1. worker agents 在后台持续推进任务
2. recorder 持续生成阶段纪要和经验沉淀
3. watchdog 主动通知进展和卡点
4. reviewer agents 可以对关键产物进行严格验收和打回
5. `net` 在整个过程中继续可用，继续和用户处理别的事情

最终，用户在 assistant 上收到的是：
1. 一份清楚的结果汇总
2. 一组经过 review 的产物
3. 一条清晰的任务推进记录

而不是一段必须自己逐行复盘的 agent 过程日志。

## 为什么它适合开源
Loom 的核心价值，不在模型闭源，而在治理设计。

开源反而可能更适合它：
1. 用户可以把它接到不同 assistant 宿主上
2. 用户可以接不同模型
3. 用户可以接不同 agent 实现
4. 社区可以共建 stage package 和 agent profile
5. 开源有助于建立对治理逻辑和安全边界的信任

从这个角度说，Loom 更像基础设施能力，而不是黑箱产品。

---

## 为什么内核建议用 Rust
Loom 最终会是一个长期运行、强状态、强并发、强边界的运行时。

Rust 适合它，不是因为它更时髦，而是因为它更适合承载：
1. 长生命周期 daemon
2. 多任务状态推进
3. 并发调度
4. 事件流与通知
5. 严格的授权边界
6. 面向插件和 adapter 的稳定协议

Rust 的优势在这里主要是：
1. 内存安全
2. 并发安全
3. 强类型状态建模
4. 能把高风险路径压缩到很少的模块里

我的建议是：
1. 内核用 Rust
2. adapter 按宿主生态实现
3. 通过稳定接口连接，而不是把全部治理逻辑塞回宿主插件

---

## 与现有 assistant 的关系
Loom 不是要替代 assistant。  
它是要扩展 assistant。

它与 personal AI terminal 的关系，应该是：
1. 宿主负责用户入口、基础交互和语义理解
2. Loom 负责复杂任务治理与运行时状态
3. 更专业的 agent 负责具体执行能力

从这个角度看，OpenClaw 不是终点，而是第一个 adapter。  
未来任何类 OpenClaw 的个人 AI 终端，都可以接入这一层能力。

---

## 产品原则
1. 默认静默，不默认接管聊天
2. 复杂任务要被正式承接，而不是临时拼接
3. assistant 在后台任务运行时仍然保持可用
4. recorder 与 watchdog 必须分离
5. 编排能力与写入能力必须分离
6. 阶段池与 agent 池必须分离
7. 普通用户要先获得预置工作模式，而不是先学复杂配置
8. 用户退出微操，但保留治理权
