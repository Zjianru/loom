# 术语表

更新时间：2026-03-11

---

## 1. 顶层对象
### `Loom`
个人 AI 终端的任务治理内核。

### `Harness`
Loom 的核心治理运行时。  
负责阶段流转、agent 绑定、review、结果与边界主链。

### `loom-openclaw`
OpenClaw 的 Loom adapter。  
负责宿主 ingress/egress、结构化语义转运、文本渲染与执行桥接。

### `runtime/loom/`
Loom 的 authoritative truth root。

---

## 2. 输入与判断
### `CurrentTurnEnvelope`
宿主原始入站事实对象。

### `HostSemanticBundle`
宿主语义层给 adapter 的综合结构化判断包。

### `SemanticDecisionEnvelope`
adapter 归一化后给 Loom 的 bounded judgment。

### `HostCapabilitySnapshot`
宿主无关的正式能力输入快照。

### `interactionLane`
当前输入进 `chat lane` 还是 `managed task lane`。

### `taskActivationReason`
为什么从聊天升级为受管任务。

### `WorkHorizon`
工作的投资意图，而不是任务规模。

### `RiskAssessment`
任务基线风险或动作覆盖风险的正式评估结果。  
它持有 `overall_risk_band`、风险维度和风险后果，而不是宿主 risk hint。

---

## 3. 任务对象
### `managedTaskRef`
单个受管任务 owner。

### `activeManagedTaskRef`
当前唯一 active task 的指针。

### `workflowStage`
单个 managed task 的粗粒度治理生命周期。

固定只包含：
1. `candidate`
2. `execute`
3. `review`
4. `result`
5. `closed`

不包含：
1. `clarify`
2. `source_collection`
3. `synthesis`
4. `deliver`

这些细粒度阶段都属于 `PhasePlan / StagePackageId`。

### `PendingDecisionWindow`
待用户结构化消费的正式决策窗口族。

### `pendingUserDecision`
当前任务 projection 上挂着的 open `PendingDecisionWindow` 指针。

### `pendingBoundaryConfirmation`
`BoundaryConfirmation` 窗口的专门扩展子实体。

### `decision_token`
绑定某个 `PendingDecisionWindow` 的令牌，生命周期由窗口持有。

### `watchdogNoticeState`
`watchdog` 的正式通知状态对象。  
它记录哪些 notice 已发、哪些升级仍待发、哪些通知已被去重，以及最小优先级/冷却/打断预算状态。

### `taskConversationPolicy`
顶层产品默认值。  
v0 固定为 `chat_open`，表示 active task 执行中主 assistant 仍继续聊天。  
它当前不是 formal runtime object，不要求持久化或单独 contract。

---

## 4. 协作类型
### `COMPLEX`
`1 + 1` 型受管协作。

### `HUGE`
`1 + n` 型受管协作。

### `MAX`
项目级、工程级分层协作。

---

## 5. 资源池与角色
### `Stage Pool`
可复用阶段模板池。

### `Agent Pool`
可复用 agent 能力池。

### `Pack`
面向用户的工作模式包，定义默认阶段、角色和治理骨架。

### `AgentBinding`
某次任务/阶段实际启用的 agent roster。

### `TaskScopeSnapshot`
当前任务范围与工程上下文快照。  
第一版同时持有 `workspace_ref / repo_ref / allowed_roots / secret_classes`。

### `SpecBundle`
围绕单个 `managedTaskRef` 的正式执行锚点文档组，持有 `scope_doc / plan_doc / verification_doc`。

### `IsolatedTaskRun`
单个 `managedTaskRef` 在执行期对应的一次隔离受管 run。

### `ExecutionAuthorization`
当前 `IsolatedTaskRun` 真正拿到的执行租约。

### `recorder`
记录与沉淀正式角色。

### `review_group`
review 规则或 reviewer 角色组。

---

## 6. 结果与控制
### `managed task start card`
进入 candidate 后第一条用户可见消息。

### `approve_start / modify_candidate / cancel_candidate`
start card 的正式决策集合。

### `keep_current_task / replace_active`
边界确认的正式决策集合。

### `ResultContract`
最终结果包的正式结构。

### `ProofOfWorkBundle`
结果链里的正式证据包，绑定 `run_summary / evidence_refs / review_summary / artifact_manifest / acceptance_basis`。

### `ReviewResult`
review 阶段的正式产物。

### `ReviewSummary`
`ReviewResult` 内嵌的结构化摘要值对象。  
它供结果链消费，不是与 `ReviewResult` 平级的正式对象。

### `ApprovalRequestPayload`
高风险动作或 review 升级给用户时的正式审批请求。

### `HostWorkerControlCapabilities`
宿主对 pause / resume / cancel / interrupt 的现实控制能力。

---

## 7. 固定边界
1. Loom 不做自然语言语义推断。
2. adapter 不持有治理真相源。
3. 主聊天默认可与 active task 并行存在。
4. pack preset 是默认装配规则，不是最终 `PhasePlan`。
5. result examples 是正式实例，不是随手举例。
